# Doc Router

구글 문서의 혼합 콘텐츠를 KA / MP patterns / MP tips로 분류하여 분리한다. 분리 후 변환은 각 프로젝트에서 별도로 수행한다.

## 입력

사용자에게 구글 문서를 **PDF와 MD 두 형식으로 export**하도록 요청한다. 이미 제공된 경우 건너뛴다.

- PDF: 이미지, 레이아웃, 서식 확인용
- MD: 텍스트 추출, 하이퍼링크 URL 확보용

### PDF Read가 실패하는 경우

`Read` tool로 PDF를 못 여는 환경(`pdftoppm not found` 등)에서는 MD-only로 진행한다. 이미지가 본문 핵심 정보(다이어그램, 도식, 의존성 그래프 등)를 담고 있다면 사용자에게 해당 페이지의 스크린샷을 별도로 요청한다.

### MD 파일이 256KB를 초과하는 경우

base64 이미지 임베드 등으로 MD가 Read tool 한도(256KB)를 초과하면, 텍스트 라인만 추출하여 처리한다.

- `Grep`을 헤딩·텍스트 패턴(`^#{1,6}\s|^[A-Za-z가-힣]`)으로 호출하여 본문 구조 파악
- 또는 `Bash`로 `awk 'length($0) < 500'` 같은 라인 길이 필터로 텍스트 라인만 추출
- 이미지 정보가 필요하면 PDF 또는 사용자 요청 스크린샷에서 보충

## 분류 기준

KA / MP patterns / MP tips 행선지 판단 기준은 MP `docs/meta/placement.md`에 정의돼 있다. 분류를 시작하기 전에 그 파일을 Read하여 분업 표를 확인한다. doc-router에는 별도 분류 기준을 두지 않는다.

## 분류 단위

문서마다 구조가 다르다. 고정된 단위(Q&A, 섹션 등)를 가정하지 않는다. 문서의 실제 구조를 파악한 뒤 내용의 성격이 바뀌는 경계를 찾아 분리한다.

## 버리는 내용

KA / MP patterns / MP tips 어디에도 속하지 않는 내용은 산출물에서 제외한다. 제외한 내용 목록은 사용자에게 알려준다. 구체적인 버림 기준은 gotchas 참고.

## 애매한 내용

행선지 경계가 불분명한 내용은 가장 적합한 행선지를 제안하고 사용자 확인을 받는다. 사용자가 판단한 결과는 즉시 아래 gotchas 섹션에 기록한다.

## TODO 처리

문서 내 TODO 항목은 사용자에게 묻지 않고 자동 처리한다. 분류 흐름이 끊기면 안 된다.

- **명백히 무가치한 TODO** (의미 없는 낙서, 중복, 이미 해결된 잔재 등) → 산출물에서 제외하고 "버리는 내용" 목록에 포함해 사용자에게 알린다. 판단은 doc-router 자체 권한으로 내린다.
- **그 외 TODO** → KA `inbox.md`(없으면 신설)에 누적한다. 출처 문서 제목을 헤딩으로 두고 그 아래에 TODO들을 묶는다:

  ```markdown
  ## <원본 문서 제목>

  - TODO 항목 1
  - TODO 항목 2
  ```

  같은 문서가 다시 등장하면 동일 헤딩 아래에 추가한다.

**비중 가드**: 한 문서에서 TODO/개인 계획이 본문의 절반 이상을 차지하면 자동 처리하지 않는다. 비중을 보고하고 처리 방향(전체 inbox 적재 vs 일부 추출)을 사용자에게 확인받는다.

## 산출물

분류 결과를 KA / MP patterns / MP tips로 명확히 분리하여 출력한다. 사용자가 각 프로젝트에서 변환 작업을 할 때 바로 입력으로 사용할 수 있어야 한다.

## 후속 변환 수행

분류 결과를 사용자에게 확인받은 후, 서브에이전트를 사용하여 각 프로젝트에서 변환을 직접 수행한다.

- KA 내용 → 서브에이전트가 KA 프로젝트(`~/WebstormProjects/main/knowledge-archive/`)의 convert 스킬 파일을 찾아 읽고, 그 절차대로 변환 수행
- MP 내용 → 서브에이전트가 MP 프로젝트(`~/WebstormProjects/main/monorepo-playground/`)의 `docs/meta/placement.md`를 읽고, 작성 규칙에 맞게 다음 중 적합한 곳에 추가:
  - `docs/patterns/<주제>.md` — 판단 기준·구현 패턴 (예시 코드 동반)
  - `docs/tips/<주제>.md` — 한 줄 결론·함정·체크포인트 (디렉토리는 첫 콘텐츠 추가 시점에 생성)
  - `docs/best-practices-map.md` — 인덱스

스킬 간 하드코딩 의존 없이, 대상 프로젝트의 스킬·메타 파일을 런타임에 읽어서 수행한다.

### 변환을 직접 수행하는 경우 (단일 문서 등 작은 입력)

서브에이전트 없이 doc-router에서 변환까지 직접 처리하더라도, 위 두 프로젝트의 스킬·메타 파일을 동일하게 읽고 절차를 따른다. 특히 KA 변환에서는 KA convert 스킬의 **"기존 knowledge 병합" 규칙**을 반드시 적용한다.

신규 KA 파일을 만들기 전:

- 같은 개념을 다루는 기존 Q&A가 있는지 `Glob`/`Grep`으로 검색 (예: `'use client'` 관련은 `react/rendering/`, `publishing/`, `bundler/` 등)
- 기존 파일이 같은 Official Answer(영문 원문)를 이미 인용 중이면, 신규 파일에서 그 원문을 다시 인용하지 않는다 — 기존 파일을 Reference로 가리키고 User Annotation/User Answer만 추가
- 동일 개념의 Q가 양쪽에 들어가지 않게 한다

## KA Q&A 꼬리질문 참조 양식

꼬리질문이 길어져 별도 파일로 분리할 때 원본 Questions 목록의 참조 양식:

- 같은 문서 내 꼬리질문: 내부 앵커 링크 `[질문](#앵커)`
- 별도 문서로 분리된 꼬리질문: 외부 파일 링크 `[질문](./분리파일.md)`

```markdown
# Questions
- [배열이란 무엇인가?](#배열이란-무엇인가)
  - [부분 배열이란?](#부분-배열이란)
  - [캐시 친화적인 이유는?](./cache-friendliness.md)
```

## gotchas

- 문제 풀이 기록, 접근방법 로그 → 버린다
- 개인 메모 → 버린다
- 버전 추적 로그 (`(0.0.x)` 식 본인 패키지 버전별 작업 단계) → 버린다. 일반화 가능한 결론만 추출
- 디버깅 감정·소회 ("당황", "ㅋㅋ", "ㅠ", "해맸음", "구라쳐서") → 버린다
- 자기 프로젝트 고유명사 (개인 패키지명·repo명·계정명) → 일반화·익명화. 산출물에 그대로 두지 않는다
- AI 채팅 캡처는 무조건 버리지 않는다. 본문(텍스트·결론) 추출 후 다른 항목과 동일하게 재판정
