# 서브에이전트 위임 구조 Eval 가이드

skill-creator의 표준 eval(서브에이전트에서 스킬 실행)은 **스킬 내부에서 Agent tool을 사용하는 패턴**을 테스트할 수 없다. 서브에이전트는 Agent tool이 없기 때문이다.

## 해결: 메인 에이전트가 직접 오케스트레이션

최상위 세션의 메인 에이전트가 스킬의 지시를 따라 직접 실행한다.

| | skill-creator 표준 | 메인 직접 실행 |
|---|---|---|
| 실행 주체 | 서브에이전트 | 최상위 세션 |
| Agent tool | 없음 | 있음 |
| 병렬 A/B | 가능 | 불가 (순차) |
| 적합한 대상 | 단일 에이전트로 완결되는 스킬 | 내부에서 서브에이전트를 띄우는 스킬 |

## 절차

1. 가상 프로젝트 준비 (git init, 최소한의 파일, 스킬이 요구하는 산출물)
2. 스킬 파일을 읽고 지시를 그대로 따름
3. 스킬이 서브에이전트 위임을 지시하면 실제로 Agent tool로 실행
4. 각 단계에서 검증 항목 확인
5. 서브에이전트의 토큰/시간을 task notification에서 수집

## 유효 범위

**검증 가능**: 위임 트리거 인식, 오케스트레이션 사이클, 서브에이전트 간 산출물 전달, 커밋 순서

**검증 불가**: before/after 정량 비교 (병렬 A/B 불가), 처음 보는 AI의 반응 (메인이 스킬 내용을 이미 앎)

## gotchas

- **방향 확인**: "출력이 맞는가"가 아니라 "분기·순서·에이전트 구성이 맞는가"를 검증한다. 테스트 입력은 각 분기 경로를 타도록 설계한다.
- **시나리오 격리**: 독립된 테스트 시나리오는 별도 세션에서 실행한다. 한 세션에서 TeamCreate를 여러 시나리오에 걸쳐 쓰면 팀 컨텍스트가 오염된다.
- **선행 산출물 체크**: 테스트 대상이 이전 step의 산출물에 의존하면, mock을 만들어야 한다. 테스트 대상 스킬만 보면 이 의존성을 놓친다.
- **mock 데이터는 스킬의 모든 조건 분기를 트리거해야 한다.** "오케스트레이션만 보면 된다"고 클린 데이터만 넣으면 차단/에러 경로가 발동하지 않는다. 게이트, 필터링, 루프 종료 등 분기가 있으면 각 경로를 트리거하는 입력을 포함한다.
