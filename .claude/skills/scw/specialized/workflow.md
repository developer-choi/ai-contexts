# Workflow 특화 체크리스트

`skills/workflow/` 리뷰 시 추가로 점검하는 항목.

## 신설 룰의 일반화 — 세션·파일명 직접 호명 회피

### 왜

workflow의 새 절차·룰·sub-section을 박을 때 세션 이름(`PLANNING_SESSION` 등)이나 산출물 파일명(`overview`, `logic` 등)을 본문에 직접 명시하면, 그 룰이 명시한 이름에 묶인다. 세션은 학습 세션 같은 새 종류가 추가되면 룰의 적용 범위에서 누락되고, 산출물 파일명은 step 정비 과정에서 자주 바뀐다. 룰 본문이 그 이름을 가리키고 있으면 변경 때마다 다시 손봐야 하고, 손보지 않으면 룰이 깨진 채 남는다.

SKILL.md 「자가 검토 필수」가 첫 줄에 "세션 이름·개수에 의존하지 않는다"를 박은 것도 같은 이유다. 이미 선언된 원칙이므로 새로 박는 룰도 같은 형태를 따라야 SKILL.md 안에서 일관성이 깨지지 않는다.

### 어떻게

리뷰 시 본문에 다음이 등장하면 일반화 후보다:

- **세션 이름 직접 호명** — `BACKGROUND_SESSION`, `PLANNING_SESSION`, `IMPLEMENTATION_SESSION`, `WRITING_SESSION` 등
- **산출물 파일명 열거** — `overview`, `logic`, `interface`, `implementation`, `test-cases`, `markup` 등 그 시점의 step 산출물명
- **step 번호 직접 호명** — 절차가 step 번호 변경에 묶이는 경우 (`step-3에서`, `step-5의 마무리에서` 등)

일반화 표현 매핑:

| 직접 명시 (지양) | 일반화 표현 (권장) |
|---|---|
| `PLANNING_SESSION의 모든 산출물에 적용` | `산출물 파일을 만드는 세션의 모든 산출물에 적용` |
| `overview·logic·interface·implementation·test-cases·markup` | `그 세션에서 작성하는 모든 산출물 파일` |
| `BACKGROUND_SESSION이 끝나면` | `해당 세션의 마지막 step이 끝나면` |
| `step-3에서` | `과제 정의 단계에서` 또는 `해당 step에서` |

### 예외

세션·파일명·step을 직접 가리켜야 자연스러운 위치는 SKILL.md 상단 `## 세션` 표·`## 작업 진행 순서` 섹션처럼 **그 자체가 인덱스·참조표**일 때다. 인덱스는 이름을 직접 가리키는 게 본질이므로 일반화 대상 아님. 절차·룰의 본문에서만 직접 호명을 피한다.

## 산출물 정의의 옵션 인식 유발 문구

산출물 정의(step-4 표 등)에서 "여부와 무관하게", "필요시", "옵션", "선택적으로" 같은 약한 어휘를 쓰면, 그 산출물을 소비하는 다음 단계가 "쓸 수도 안 쓸 수도"로 인식한다. 시간 압박이나 자체 검열이 들어왔을 때 자연스럽게 줄이는 흐름이 나온다.

이전 사고: `test-cases.md` 정의에 "테스트코드 작성 여부와 무관하게"가 있어 PLAN이 implementation.md의 단위테스트를 "옵션 3건"으로 줄였고 IMPL이 0건 채택했다.

리뷰 시 산출물 정의 본문에 위 표현이 보이면 디폴트(작성 필수/금지)를 명시하도록 보강 제안한다.

## 세션 경계 정합성

세션 분리와 관련된 구조가 일관되는지.

### 세션 회고 시점

- 세션 회고가 있는 step의 회고 내용이 해당 세션 성격에 맞는지 (구현 세션 → 코드 패턴·패키지 변동 등)

### 세션 전환 안내

- SKILL.md 세션 테이블의 세션 종류와 전환 안내 문구가 **1:1 대응**하는지 (세션 타입이 N개면 안내 문구도 N-1개)
- 안내 문구의 step 번호·세션명이 세션 테이블과 일치하는지

### 세션 간 산출물 생존

- 세션 경계를 넘어서 참조되는 산출물이 **이전 세션에서 삭제되고 있지 않은지**
- 각 step의 "산출물 정리/소비" 시점이 해당 산출물의 마지막 소비자 이후인지

## 산출물 라이프사이클

산출물이 살아있는 동안 결정 진화에 따라 동기화 부담이 누적되는 패턴 방지. 「세션 간 산출물 생존」이 "마지막 소비자 이후 정리"의 하한선을 본다면, 본 섹션은 살아있는 동안의 부담과 상한선을 본다.

라이프사이클 폴더(`persistent/`·`retained/`·`consumable/`)는 그 자체가 정리 시점을 표현하는 메커니즘이다. 본문 룰은 그 메커니즘을 보조하는 형태로, 폴더 분류가 일관·정합·최소 수명으로 유지되는지를 본다. 폴더 구조 정의는 workflow SKILL.md 「/plan/ 폴더 구조」 참조.

### 폴더 분류 부재 검토

새로 도입되거나 기존에 존재하는 산출물이 라이프사이클 폴더 중 하나에 분류되어 있는지 본다.

- `pr{N}/` 또는 `background/` **직속에 박혀 있는 산출물**(폴더 분류 없음)이 있는지 — 미분류 항목은 정리 시점이 메커니즘으로 표현되지 못하므로 별도 식별해 분류 결정을 유도한다.
- 분류 보류 사유가 산출물 정의(또는 폴더 구조 다이어그램 주석)에 적혀 있는지 — 없으면 "왜 미분류인지" 사유를 함께 박도록 보강 제안.

### 폴더 디폴트 매핑 가이드

새 산출물을 도입하거나 기존 산출물의 폴더 분류를 검토할 때 적용하는 디폴트:

| 산출물 특성 | 디폴트 폴더 |
|---|---|
| 한 step만 소비 / 큐 모델 적용 가능 (절 단위 소비) | `consumable/` |
| 본 PR 여러 step이 소비, PR 종료 후 폐기 OK | `retained/` |
| PR·프로젝트 종료 후에도 가치 (회사 컨텍스트 의존 결정, 컨벤션 인덱스 등) | `persistent/` |

`consumable/`이 기본값이다. `retained/`·`persistent/`로 올리는 산출물은 그 사유가 산출물 정의에 명시되어 있는지 본다 — 사유 없이 올라가 있으면 수명을 부풀리는 방향으로 흘러간 흔적이다.

### 폴더 분류와 사용처의 정합

산출물의 폴더 분류와 사용처 step 목록이 일관되는지 본다.

- **`consumable/`인데 복수 step이 소비**하는 경우 — 큐 모델(절 단위 소비)이 실제로 적용 가능한지 확인. 절 단위로 깔끔히 갈라지지 않으면 `retained/`로 올리는 게 맞다.
- **`retained/`인데 한 step만 소비**하는 경우 — 수명을 늘려 잡은 흔적. `consumable/`로 내릴 수 있는지 본다.
- **`persistent/`인데 PR 라이프타임 안에서만 소비**되고 외부 가치가 없는 경우 — `retained/`로 내리는 게 맞다.

폴더가 정리 시점을 표현하므로 본문 규칙으로 "어느 step 직후 정리" 같은 정리 시점을 반복 명시하지 않는다. 그 대신 폴더 분류 자체의 적정성에 무게를 둔다.

### 큐 모델 적용 검토

`retained/`에 묶여 일괄 폐기되는 산출물 중, 절 단위로 소비 가능해서 `consumable/` 큐 모델로 내릴 수 있는 후보가 있는지 본다.

- 절 단위로 소비되고 절 사이 의존이 없다면 `consumable/` 큐 모델이 동기화 부담을 더 잘 줄인다 (소비 즉시 절 삭제 → 살아있는 동안 부담 감소).
- 절 사이 의존이 있어 절 단위 분리가 부자연스러우면 `retained/`가 맞다.

### 산출물에 코드와 1:1 중복 작성 금지

폴더 라이프사이클과 무관하게, 산출물 본문이 코드 진실 원천과 중복되면 살아있는 동안 양쪽 동기화가 강제된다. 다음이 포함되면 위반:

- **폴더 트리 박스** (예: `_fsd/entities/call/ ├─ api/...`) — `ls` 또는 stub 파일 자체가 진실 원천
- **코드 박스** (예: `export const callQueries = queryOptions({...})`) — `.ts` 파일이 진실
- **`index.ts` 외부 공개 범위 박스** — 실제 `index.ts` 파일이 진실
- 그 외 코드와 1:1 매핑되는 모든 인용

산출물에는 narrative(거부된 대안·근거, 컨벤션 인용 + 라인 번호, TODO, 회귀 체크리스트, 비포함 scope)만 유지. 코드 진실 원천과 중복되는 부분은 처음부터 작성하지 않거나 즉시 제거한다.

이유: 코드 결정이 진화하면 코드만 진화하고 산출물 narrative는 작성 시점 snapshot으로 둘 수 있다. 중복 인용을 산출물에 넣으면 결정 진화마다 양쪽 동기화가 강제된다.

### before/after 사례

**before** — overview.md 결정 1을 적을 때 폴더 트리 박스 + index.ts 외부 공개 코드 박스를 함께 박았고, stub 작성 중 결정 진화(예: useSuspenseQuery → server fetchQuery, namespace → class, lib/mapper.ts 미생성 등) 시 매번 박스들을 사후 동기화. 코드 1:1 중복 + 수명 길게가 합쳐져 부담 누적.

**after** — overview.md 결정 1에는 거부된 대안·근거·컨벤션 인용(라인 번호)만 적고, 실제 폴더 구조·코드는 "`_fsd/entities/call/` 디렉토리 + 각 stub 파일 참조"로 안내. 결정 진화는 stub 코드에만 반영. overview는 `consumable/` 폴더의 큐 모델로 소비 즉시 정리되고, PR 본문 작성 단계에서 "PLAN 대비 진화" 회고로 녹인다.

## step-4 vs step-5 코드 작성 경계

step-4(stub)와 step-5(IMPL) 모두 코드를 작성하므로 경계가 흐려진다. 항목별 경계를 직접 명시.

### 모든 파일·함수·컴포넌트 stub 필수

PR에서 만들 **모든 파일·함수·컴포넌트는 step-4에서 stub으로 생성**한다. 외부 공개 여부 무관 — 내부 헬퍼·내부 컴포넌트도 시그니처 + `throw new Error('not implemented')`로 stub.

두 가치:
- **구조 가시화**: 페이지 컴포넌트가 `<Header />`, `<Body />`, `<Footer />`를 렌더링하면 H/B/F도 stub으로 만들어 한눈에 구조 파악. IMPL은 본문만 채우면 됨
- **PR 병렬 기반**: 외부 공개 인터페이스가 stub에 드러나야 후속 PR이 의존 가능

### 항목별 경계 매핑

| 항목 | step-4 | step-5 |
|---|---|---|
| 외부 공개 모듈 시그니처 (export 함수·컴포넌트·hook) | ○ 필수 | — |
| 내부 함수·컴포넌트 시그니처 | ○ 필수 (시그니처 + throw) | — |
| 함수 본문 (짧고 결정됨 — 3~5줄) | ○ 직접 적어도 OK | — |
| 함수 본문 (길거나 분기 많음) | `// TODO [AI_IMPL]:` 한글 명세 | ○ 본문 작성 |
| 결정된 분기 (짧은 가드) | ○ 직접 | — |
| 복잡 분기 (가드 + switch + 매핑) | 한글 명세 | ○ 본문 |
| 타입·상수 정의 | ○ | — |
| 테스트 케이스 시그니처 | ○ `it.todo('자연어')` | — |
| 테스트 본문 | — | ○ `it(...)` |
| scss layout (`display: flex`, `grid-template-columns`) | ○ | — |
| scss 디자인 값 (gap·color·padding 토큰) | — | ○ (markup.md 「figma 링크·체크리스트」 보고 채움) |

### 점검 항목 (scw 리뷰)

- step-4 stub에 함수 본문이 길게 박혀 있는지 — 한글 명세로 바꿔야 하는 후보
- step-4에서 **내부 함수·컴포넌트 stub이 누락**됐는지 — 구조 가시화 깨짐 신호 (PR 안 호출 그래프·컴포넌트 트리가 stub에 드러나지 않음)
- step-5에서 **외부 공개 시그니처가 변경**됐는지 — step-4 누락 신호 (후속 PR 의존 깨짐)
- step-5에서 결정된 분기·매핑·상수가 새로 추가됐는지 — step-4 누락 또는 step-5에서 새 결정 발생
- step-4에서 scss 디자인 값(토큰)이 미리 박혔는지 — step-5 시점 결정인데 추측한 흔적 (figma 검수 후 교체 부담)

### 사례

- ✓ Good: 페이지 컴포넌트 + 내부 H/B/F 컴포넌트 모두 stub (시그니처 + throw) — 구조 한눈
- ✓ Good: 짧은 가드(`if (!user) return null;`)는 step-4 stub에 직접
- ✓ Good: 복잡 매핑(10줄 switch + guard)은 `// TODO [AI_IMPL]: 한글 명세` + step-5 본문 작성
- ❌ Bad: step-4에 외부 공개만 stub하고 내부 H/B/F 누락 — 구조 파악 불가
- ❌ Bad: step-5에서 `add(a, b)` → `add(a, b, options)` 시그니처 변경 — step-4 누락. 후속 PR 의존 깨짐
- ❌ Bad: step-4에서 `gap: var(--semantic-gap-md)` 추측해 박음 — figma 검수 전 추측이라 IMPL 시점 교체 부담

### 엄격도

- **필수 (위반 카운트)**: 모든 파일·함수·컴포넌트 stub 생성, 외부 공개 시그니처 안정성, scss 디자인 값 step-4 추측 금지
- **방향 가이드 (후보 식별 → 사용자 보고)**: 함수 본문이 짧으면 직접 / 길면 한글 명세 — 분량 경계는 case by case

## 절차 의존성 분석 — 병렬 가능 절차 명시

### 왜

새 step·sub-step을 박을 때 의존성을 점검하지 않고 순차로 적으면 LLM 실행이 강제로 직렬화된다. 의존 없는 두 단계는 "X 다음에 Y" 대신 "X·Y 병렬 진행 가능"으로 명시한다.

### 어떻게

새 sub-step을 추가하거나 기존 순서를 손볼 때:

- **의존성 점검**: 이 단계의 산출물·상태 변화가 다른 단계의 입력이 되는가? history rewrite가 working tree에 영향을 주는가?
- **의존 없으면 병렬 명시**: 본문에 "X와 의존 없음 — 병렬 진행 가능"
- **부분 의존이면 충돌 조건만 명시**: 완전 병렬은 아니지만 특정 조건만 피하면 되는 경우, 그 조건만 짚는다 (전체 순서 강제 X)

### 사례

커밋 정리·재정렬 단계(stub 커밋 정리 + rebase + force-push)와 decisions.md 최신화 단계: `/plan/`은 대부분 `.gitignore`라 후자의 working tree 변경이 전자의 rebase 대상이 아니다. 의존 없음 → "병렬 진행 가능"으로 명시한다.

## 인덱스 동기화

step 파일의 추가/삭제/제목·역할 변경이 `SKILL.md`의 작업 진행 순서 테이블에 반영되어 있는지.

- step 파일이 추가/삭제되었는데 테이블에 반영되지 않은 경우
- step 파일의 제목/역할이 변경되었는데 테이블의 설명이 이전 버전인 경우
- 테이블의 step 순서와 실제 파일의 논리적 순서가 불일치하는 경우

## Step-Skill 오케스트레이션

step이 스킬을 오케스트레이션하는 구조를 따르고 있는지.

- 오케스트레이터 step에 **스킬 선택 테이블**이 있는지 (작업 종류 → 스킬 매핑)
- 새로 추가된 스킬이 해당 step의 스킬 선택 테이블에 등록되어 있는지
- 스킬이 워크플로우 없이 **독립 호출 가능**한 구조인지 (스킬 내부에 step 컨텍스트 의존이 없는지)
- step이 스킬의 **내부 관심사를 대신 분기**하고 있지 않은지 — step이 입력 유형·프로젝트 유형에 따라 절차가 동일한 스킬을 별도 호출하고 있으면, 스킬 하나로 통합하고 분기를 스킬 내부로 옮겨야 한다

### 산출물 체이닝

각 step의 산출물이 다음 step의 입력으로 연결되는지.

- 같은 step에서 호출되는 스킬들이 **동일한 산출물 디렉토리**를 사용하는지 (예: step-1 스킬 → `/plan/background/`)
- 산출물 경로가 스킬의 SKILL.md와 step 파일 **양쪽에** 명시되어 있는지
- 다음 step의 **참고 자료**가 이전 step의 산출물 경로와 일치하는지
- Lead가 `/plan/` 하위를 탐색하여 산출물을 파악하고 팀에게 분배하는지
- 파생 산출물이 소비자 테이블(step-4)대로 올바른 에이전트에게 전달되는지
- 구현 완료 후 소비된 산출물이 삭제되는지 (overview.md 제외)

### 스킬 로드 테이블 정합성

- 오케스트레이션 step에 스킬 로드 테이블이 있는가
- `실행` 컬럼이 판별 기준(상호작용 필요 → 메인, 원본 맥락 필요 → 메인, 둘 다 아니면 → 서브에이전트)과 일치하는가
- 서브에이전트 스킬의 입력/산출물이 파일로 정의되어 있는가

### 컨텍스트 비대화 검토

- 하나의 세션/에이전트가 이후 step에서 불필요한 컨텍스트를 누적하고 있지 않은가
- 분리가 필요하다면 서브에이전트(일회성 분석)로 충분한가, 별도 세션(지속적 작업)이 필요한가
- 에이전트가 스스로 폴더를 탐색(glob/grep)하지 않고, Lead가 주입한 경로의 파일만 읽는지
- Markup/Feature Implementer의 컨텍스트가 격리되는지 (CSS 컨텍스트가 Feature에 오염되지 않는지)

### 팀 에이전트 소통 구조

- Figma Reviewer ↔ Implementer, Advanced Reviewer ↔ Implementer가 Lead 개입 없이 직접 루프하는지
- Coding-Standards Reviewer ×N 결과만 Lead가 종합하는지 (sonnet 오탐 필터링)
- Markup 파이프라인이 3단계(Figma → Coding-Standards → Advanced)인지
- Feature 파이프라인이 2단계(Coding-Standards → Advanced)인지
- 리뷰어는 code-review SKILL.md의 절차를 따르는지
- 계획과 달라지는 경우 사용자에게 보고하는지

### 컨벤션 커버리지

`contexts/coding-standards/map.md`를 읽고, 각 규칙이 workflow step에서 적절한 시점에 참조되고 있는지 점검한다.

- 어떤 step에서도 참조되지 않는 코딩 규칙이 있는지
- 참조 시점이 부적절한 규칙이 있는지 (예: 코딩 컨벤션이 구현 step이 아닌 곳에만 있는 경우)
- 새로 추가된 규칙이 적절한 step에 참조되지 않은 경우

## 세션 상호 의심

각 세션이 이전 세션·이전 PR 산출물을 무비판적으로 따르지 않고 자기 영역에서 비판적 검토하는 절차가 있는지.

- IMPL 세션이 PLAN 산출물을 비판적으로 검토하는 룰이 있는가 (현 step-5 본문)
- PLAN 세션이 이전 PR 산출물(코드·결정)을 비판적으로 검토하는 룰이 있는가 (현 step-4 본문)
- 검토 범위가 자기가 참조하는 부분에 **한정**되는가 — 전부 다 의심하면 비대화
- 검토 근거가 자기가 이미 로드한 컨벤션 문서로 **제한**되는가 — 추가 컨벤션 로드 시 영역 확장으로 간주
- "다른 부분 검증을 위해 다른 컨벤션을 추가 로드하지 않는다" 명시가 룰에 들어가 있는가
