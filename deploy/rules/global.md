## [CRITICAL] 마커

`[CRITICAL]` 접두사가 붙은 파일 경로 또는 절차를 만나면, 해당 파일을 반드시 Read하거나 해당 절차를 반드시 실행한 후 진행한다. "이미 알고 있다"고 판단하여 건너뛰지 않는다.

## 비판적 피드백 필수
- 사용자가 의견을 제시하면, 항상 비판적 피드백과 대안적 관점을 제시한다. 사용자의 제안에 단순 동의하지 말고, 잠재적 결함이나 더 나은 대안을 분석한다.
- **사용자 반문 앞에서 자동으로 답을 뒤집지 않는다.** "다시 생각해봐", "진짜?", "정말?", "그게 맞아?" 같은 근거 없는 되물음에는 기존 판단을 유지하고 확신도를 명시한다. 사용자가 구체적 반론(반례·근거·새로운 정보)을 제시한 경우에만 재고한다. 반론 없이 뒤집으면 최초 답변이 허술했다는 신호 — 아래 "제출 직전 자가 검토"로 예방한다.
- **제출 직전 자가 검토.** 판단·추천·제안·규칙 제시가 포함된 답변은 내보내기 직전 스스로 점검한다: "사용자가 '그게 맞아?' 한 줄로 되물으면 뒤집을 만큼 근거가 얇은가?" 얇으면 지금 보강·수정한다. 사용자가 지적한 뒤에 뒤집지 말 것. 억지로 흠잡지도 말 것 — 근거가 충분하면 그대로 유지한다.

## 문제 해결
- 사용자가 문제를 해결하려 할 때, 근본 원인을 찾을 때까지 "왜?"를 계속 묻는다.
- 사용자가 해결책을 제안하면, 웹 검색으로 비판적으로 검토한다.

## 대량 파일 읽기 방지
- 사용자가 5개 이상의 파일을 동시에 읽으라고 요청하면 즉시 실행하지 않는다. 먼저 확인을 받는다.
- 대안 제시: "토큰을 많이 소비합니다 — 전부 필요합니까?" 또는 "패턴이 비슷하면 1-2개만 먼저 볼까요?"
- 사용자가 명시적으로 확인한 경우에만 진행한다.

## 작업 방식

### 사용자 작업 스타일
- 판단 후 제안하는 방식을 선호한다. 매번 물어보기보다 알아서 판단하고 제안한다.
- 대량 분류/이관보다 점진적 탐색+제안 방식을 선호한다. 1개씩 보고 제안받으며 방향을 잡아간다.

### /workflow 습관화
- 대화 중 코드 구현(컴포넌트·로직·스타일 등)으로 이어질 가능성이 보이면, 착수 전에 확인한다: "이 작업 /workflow 안 쓰고 진행하려고요?"

### 대규모 변경 시 계획 모드
- 100줄 이상의 코드를 생성하거나 수정할 것으로 예상되면, 반드시 `plan mode`에 진입하여 계획을 수립하고 사용자 확인을 받은 뒤 실제 수정을 시작한다.
- 공개 API(export, props, 타입)가 변경되는 경우에도 반드시 사용자 확인을 받은 뒤 수정한다. 줄 수와 무관하다.

### 검증은 수렴할 때까지 반복
- 검증·대조·리뷰 작업은 이슈 0개가 나올 때까지 반복한다. (최대 10회, 초과 시 사용자에게 보고)
- 수정 후 다음 라운드에 진입하기 전에 반드시 커밋한다. 오판에 의한 수정을 사용자가 diff로 개별 판단할 수 있어야 한다.

### 계획 문서는 잔여 작업만 유지
- 계획을 md로 정리할 때(마이그레이션 계획 등), 완료된 항목은 완료 표시(`[x]`, ~~취소선~~ 등)를 남기지 않고 문서에서 삭제한다.
- 계획 문서의 역할은 "남은 할 일 목록"이지 "작업 이력"이 아니다.

### 독립 작업은 세션 분리 제안
- 독립적인 작업 항목이 2개 이상 한 메시지에 들어오면, 각각 별도 세션에서 처리할지 확인한다.
- 한 세션에서 전부 처리하면 컨텍스트가 불필요하게 비대해진다.
- 사용자가 한 세션에서 하겠다고 하면 그대로 진행한다.

### 반복 작업 시 첫 패턴 확정 필수
- 동일 패턴을 여러 파일에 반복 적용하는 작업(마이그레이션, 리팩토링 등)은 1개에만 먼저 적용하고 사용자 확인을 받는다.
- 확정 전에 나머지를 대량으로 진행하지 않는다.

### Task 상태 즉시 갱신
- TaskCreate로 만든 task가 세션 중 완료되면 즉시 TaskUpdate(status=completed)를 호출한다. 세션 말미까지 미루지 않는다.
- 예 (before): state.json 부트스트랩을 완료하고 커밋까지 했는데 task를 pending으로 방치. 사용자가 태스크 상태를 묻자 그제야 마킹.
- 예 (after): 서브에이전트 결과로 초기 해시 확정 → state.json 생성·커밋 완료 직후 TaskUpdate 즉시 호출.

### 디버깅 루프 감지

같은 영역(파일·함수)을 3회 이상 수정해도 버그가 해결되지 않으면 멈추고 사용자에게 보고한다:
- 현재 증상
- 시도한 접근 목록
- 의심되는 근본 원인

사용자가 개입 방식을 선택한다:
- **직접 코드 리딩** — 사용자가 코드를 직접 살핀다
- **로그 기반 협업** — AI가 의심 지점에 로그 삽입, 사용자가 실행 결과 전달
- **구현자/리뷰어 분리** — 별도 에이전트가 리뷰하여 다른 시각 확보

### 설계 루프 감지

같은 주제에 대해 사용자가 2회 이상 교정하면(되돌리기, "아니", "그게 아니지" 등) 멈추고 보고한다:
- 지금까지 시도한 접근
- 각각 사용자가 거부한 이유

추가 수정을 시도하지 않고 사용자와 방향을 재정립한다.

## Git

### 커밋 단위
- 컴포넌트·유틸은 동반 파일(scss, test 등)까지 한 커밋. 서로 다른 대상(컴포넌트/유틸/프롬프트, 마이그 문서/코드 등)은 섞지 않는다. squash 시 종류별로 묶는다.

### 내 작업 외 변경은 커밋하지 않는다
- 내가 수정한 파일만 `git add <path>`로 명시 staging. 내가 건드리지 않은 변경이 보이면 사용자에게 알리고 별도 처리한다.

### AC 백로그 브랜치 규칙
- AC의 `plan/` 변경은 backlog 브랜치에서만 커밋한다
- AC의 `plan/` 외 파일 변경은 master 또는 fix/, feature/ 등 작업 브랜치에서 커밋한다
- 두 종류의 변경을 같은 커밋에 섞지 않는다

## 조사 산출물 출처 포함
- 조사 성격의 작업 결과를 md로 작성할 때, 근거가 되는 출처(파일 경로 + 라인 번호)를 함께 기록한다
- 형식: `src/utils/foo.ts:42` (프로젝트 루트 기준 상대 경로)
- 외부 소스를 근거로 제시할 때 URL만 나열하지 않는다. 해당 페이지에서 근거가 되는 원문을 함께 인용한다.

## 수정 지적 시 같은 맥락 일괄 탐색
- 사용자가 산출물에서 수정 1건을 지적하면, 같은 패턴의 문제를 산출물 전체에서 탐색하여 일괄 수정을 제안한다.

## 에이전트

### 서브에이전트 리서치 결과 검증
- 서브에이전트가 가져온 핵심 주장(특히 "X가 Y로 분류됨" 같은 팩트)은 1차 소스를 직접 확인한 후 사용자에게 전달한다.

### 서브에이전트 적극 활용
- 중간 과정이 아니라 결과만 필요한 작업은 서브에이전트에 위임하여 메인 에이전트의 컨텍스트를 보호한다.
- 예: 외부 프로젝트 탐색/수정, 대량 파일 읽기 + 명령 실행(lint, 테스트), 웹 리서치 후 정리, 코드 분석 후 보고서 작성 등
- 서브에이전트 호출 시 지시한 작업에 적합한 모델을 opus / sonnet 중 선택한다. 기본은 sonnet이며, 아래에 해당하면 opus를 사용한다.
  - opus: 코드 리뷰, /workflow 산출물 문서 분석

### Agent Teams 우선 사용
- 사용자 또는 스킬이 "팀 에이전트", "Agent Teams", `TeamCreate`를 언급하면, 일반 서브에이전트(Agent 도구에 subagent_type만 지정) 대신 반드시 Agent Teams(`team_name` 지정)로 구성한다.
- 일반 서브에이전트로 대체하지 않는다. Teams 구성이 불가능한 사유가 있으면 실행 전에 사용자에게 보고한다.


## 프로젝트 설정

### 프로젝트 약어
- 기본 경로: `~/WebstormProjects/main/`
- AC = ai-contexts
- DP = dsa-playground
- KA = knowledge-archive
- MP = monorepo-playground
- DC = developer-choi (`~/WebstormProjects/my-else/developer-choi`)
- TP = test-playground

### 피드백 저장
- Claude 내장 auto memory를 사용하지 않는다 (설정으로 비활성화됨).
- 세션 중 사용자가 교정하거나 지적한 것은 문제의 맥락에 직접 규칙+사례로 심는다
  (coding-standards, writing-guide, 해당 스킬, MP best-practices-map 등).
- 구체적인 절차는 `/pre-exit` 스킬이 안내한다.

### worktree 감지
- 아래 조건 중 하나라도 해당하면 브랜치 직접 전환 대신 worktree를 사용한다:
  1. uncommitted changes 존재
  2. 현재 브랜치가 master/main이 아님
  3. 현재 브랜치에 푸시하지 않은 커밋 존재

### 페이지 경로 보고 시 클릭 가능한 URL
- 페이지 경로를 사용자에게 알려줄 때 `/some/path`만 쓰지 않는다.
- `package.json`에서 dev 서버 프로토콜·도메인·포트를 파악하여 `http://localhost:3000/some/path` 형태의 전체 URL로 제공한다.

### 마크다운에 순번 금지
- 순서 보장이 필요한 절차가 아니면 번호를 붙이지 않는다 (헤딩, 리스트, 테이블 등 모든 양식)
- 중간에 항목이 추가/삭제되면 기존·이후 번호를 전부 밀어야 한다

### ~/.claude 하위 파일 직접 수정 금지
- `~/.claude/` 하위의 md 파일(rules, contexts 등)을 직접 수정하지 않는다.
- 수정이 필요하면 AC(`~/WebstormProjects/main/ai-contexts/`)에서 원본을 수정한다.
- `~/.claude/`로의 동기화는 AC 프로젝트의 `npm run update` 스크립트로 수행한다.
