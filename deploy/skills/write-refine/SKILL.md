# write-refine — 다듬기 세션용

write-init이 만든 패키지를 받아 톤·구조·분량을 다듬는다. 작업 히스토리는 없고, 패키지의 frontmatter가 유일한 컨텍스트 소스다.

## 입력

`/write-refine <파일 경로>` 형태로 호출한다. 파일은 write-init이 만든 패키지(frontmatter + 본문 단일 .md).

frontmatter 형식·필드 정의는 [write-init SKILL.md "패키지 형식"](../write-init/SKILL.md#패키지-형식)을 따른다 (필수·선택 필드, `rendering_env`·`placeholder_policy` 의미 등).

## writing-guide 로드

[CRITICAL] `../../contexts/writing-guide/map.md`를 읽고 탐색 절차를 따른다.

## 에이전트 구조

[CRITICAL] [team-agent](../../contexts/team-agent.md) 규칙을 따른다.

- 메인 = 리뷰어 (검토)
- refine-writer = 수정자 (파일 직접 다듬기)
- refine-writer는 전역 agent 파일에 의존하지 않는다. 아래 "내장 refine-writer 지시문"을 위임 지시로 사용한다.
- refine-writer 위임은 team-agent 규칙의 현재 런타임 경로를 따른다. Claude Code에서는 Agent Teams, Codex에서는 허용된 subagent 도구를 사용하고, 위임 경로가 없으면 메인이 같은 규칙을 직접 수행한다.

## 내장 refine-writer 지시문

refine-writer는 텍스트 다듬기 작업자다. 메인(리뷰어)의 위임을 받아 입력 파일의 본문을 다듬는다.

### 권한과 범위

- writing-guide 규칙과 사례를 적용해 본문의 말투·구조·분량을 다듬는다.
- 사실 검증·사실 수정은 하지 않는다. 표현·구조·분량만 다룬다.
- frontmatter(`---`부터 다음 `---`까지)는 절대 수정하지 않는다. 이 영역을 건드리면 작업 실패로 본다.
- 별도 파일을 만들지 않는다. 결과는 입력 파일에 직접 반영한다.

### 작업 순서

1. 메인이 전달한 파일 경로와 위임 지시를 확인한다.
2. 파일을 Read해서 frontmatter와 본문을 파악한다.
3. writing-guide 규칙·사례와 위임 지시를 본문에 적용한다.
4. 본문만 Edit으로 수정한다.
5. 완료 후 메인에게 적용한 규칙·사례 요약과 사실 오류 발견 사항을 보고한다.

### 사실 오류 발견 시

본문에 명백한 사실 오류(예: 함수명 오타, 날짜·숫자 오류, 누락된 주요 정보)를 발견하면 수정하지 말고 메인에게 보고한다. write-refine은 톤·구조·분량만 다루며, 사실은 사용자/write-init 책임이다.

## 작업 흐름

### 0. 입력 검증

파일 존재 확인. frontmatter 파싱. 필드 정의는 write-init SKILL.md "패키지 형식" 참조.

- frontmatter 자체가 없거나 형식이 깨져 파싱 실패 → "frontmatter가 없거나 형식이 깨졌습니다. write-init SKILL.md '패키지 형식'을 참고해 추가한 뒤 재호출하거나, write-init을 먼저 호출하세요"로 종료
- 필수 필드(write-init "패키지 형식" 정의) 누락 → "이 파일은 write-init 패키지 형식이 아닙니다. write-init부터 다시 호출하세요"로 종료
- 선택 필드 누락은 통과

### 0.5. 초기 맥락 확정

작업을 시작하기 전에 아래 항목을 사용자에게 확인하거나 명시적으로 기록한다. 이 단계를 건너뛰면 세션 중반에 방향 전환이 일어나 앞선 작업이 낭비된다.

- **최종 산출물**: 결과물이 무엇이고 어디에 쓰이는가 (기존 파일 덮어쓰기 / 별도 파일 / 외부 시스템 입력 등)
- **저장 위치**: 확정본이 어디에 반영되는지 미리 합의. 확정분을 대화에만 쌓지 않는다.
- **최신본 존재**: 사용자가 외부 도구(이력서 플랫폼·문서 서식 등)에 이미 편집한 최신본이 있는지 물어본다. 있으면 그 버전을 기준으로 삼는다.
- **원본 직접 확인**: 원본·레퍼런스 문서가 있으면 메인이 반드시 직접 읽고 톤을 대조한다. 다른 에이전트가 제공한 맥락 요약("내부 문서이므로 한다체 허용" 등)만으로 톤을 판단하지 않는다.
- **refiner 위임 기본값**: 톤·구조 다듬기는 refiner에 위임한다. 메인이 직접 시안을 쓰지 않는다. 사용자가 명시적으로 "메인이 직접 해"라고 지시하거나 현재 런타임에서 위임 경로가 없는 경우에만 예외.

### 1. 패키지 파악

frontmatter에서 컨텍스트(독자·목적·분량 목표·참조 컨텍스트 등) 추출. 본문 분량과 구조 파악.

**type 분기 — resume-item**: frontmatter `type`이 `resume-item`이면 `../../contexts/writing-guide/resume-guide.md`와 `../../contexts/writing-guide/examples/resume.md`를 추가로 Read하고 "시안 다양화 모드"에 진입한다 (시안 다양화 워크플로우 정의는 `resume-guide.md`, 모드 진입·운영은 "모드별 확장 > 시안 다양화 모드" 참조).

**type 분기 — decision**: frontmatter `type`이 `decision`이면 `../../contexts/writing-guide/decision-guide.md`를 추가로 Read하고, "비교 구조 강제 규칙"(방법 2개 이상 / 장점·단점 / 결론에 비선택 이유)을 메인 검토 체크리스트에 추가한다. 위반 시 writer에게 보강 위임.

### 2. refine-writer 위임

team-agent 규칙에 맞는 방식으로 refine-writer에게 위임:
- 파일 경로 (writer가 직접 Read·Edit)
- 다듬기 지시: "writing-guide 규칙·사례 적용해서 본문 다듬기. **frontmatter는 절대 건드리지 마.** 사실 검증·수정은 안 함. 이모지를 추가하지 않는다. 원본에 이모지가 있어도 제거하지 않는다. frontmatter `rendering_env`·`placeholder_policy`는 write-init SKILL.md '패키지 형식' 정의대로 처리한다 (`rendering_env: plain-text`면 마크다운 문법 제거, `placeholder_policy` 명시 없으면 `keep`). 사용자 원본의 bullet/섹션 개수는 기본 보존 — 통합·삭제·분할이 필요하면 메인에게 보고. frontmatter `tone_hint`·`audience`를 기준으로 원본을 선제 검토한다 — tone_hint에 \"담담\", \"자기 자랑 X\" 등이 있으면 인트로·어필 단락부터 확인하고, audience가 \"비개발자\"면 코드 레벨 용어·기술 링크·기술 괄호 부연을 우선 탐색한다. Edit 완료 후 파일 끝에 `==>`, `<!--`, `TODO:` 등 편집자 코멘트가 남아 있지 않은지 확인한다."
- **type 분기**: frontmatter `type`이 `resume-item`이면 위 기본 다듬기 지시 대신, `resume-guide.md`의 "시안 다양화 워크플로우"를 따르는 **시안 3개 생성 지시**로 교체한다. 세부 포맷은 "모드별 확장 > 시안 다양화 모드" 참조.
- **누적 블록 첨부**: 세션 중 쌓인 "사용자 톤 선호 누적 블록"(5단계 참조)이 있으면 위임 메시지에 포함한다.

**핑퐁 카운터**: 매 위임 시 메인이 `+1` 카운트. 카운트는 메인이 보유.

### 3. 메인이 검토

writer 작업 완료 통보 받으면 파일 다시 Read해서 검토:
- writing-guide 규칙·사례와 대조
- **frontmatter 변경 여부 확인** — 변경됐으면 즉시 writer에게 복원 지시 (메인 카운트 소비 X)
- **복원 시도 캡: 2회**. 2회 후에도 frontmatter 변경되면 작업 중단하고 사용자에게 "writer가 frontmatter를 반복 변경하여 작업 중단됨"으로 보고

#### 메인 검토 체크리스트

**등재 기준**: frontmatter 의존 OR 다중 룰을 가로지르는 절차만 이 체크리스트에 둔다.

**examples 단일 룰**(예: 비교 서술 차별점·코드 블록 표기 일관성·문단 길이 상한선 등)은 등재하지 않는다 — `writing-guide/map.md` 탐색 + writing-guide 규칙·사례 대조 단계(`### 3. 메인이 검토` 첫 항목)에서 자연 적용된다.

위 대조와 별개로 아래 4가지를 반드시 점검한다.

- **고유명사 정의 누락**: 본문에 등장하는 컴포넌트명·내부 용어가 정의 없이 서술되는지 확인. frontmatter `audience_knowledge`에 포함된 용어가 아니면 정의가 선행되어야 한다. (examples/tone.md "고유명사·내부 용어를 정의 없이 사용" 참조)
- **결론 섹션 중복 검토**: 결론/마무리 섹션은 앞 섹션(요지·본문)과 논점·표현이 겹치는지 대조한다. 결론 + 앞 섹션 두 위치를 가로지르는 절차라 등재. 리드 문장과 핵심 판정 외 문장은 "앞에서 이미 다뤘는가?"를 기준으로 삭제 후보로 본다. (examples/structure.md "결론 섹션이 앞 섹션의 논점을 반복" 참조)
- **문서 전체 일괄 점검**: 한 지점의 이슈를 발견하면 같은 패턴이 문서 전체에 있는지 grep·Read로 일괄 확인한다 (용어 정의 유무, 비교 차별점, 섹션 간 중복 등). 다중 룰을 가로지르는 절차.
- **audience_knowledge ↔ 본문 용어 1:1 대조**: frontmatter `audience_knowledge`에 "X 모름"으로 명시된 용어가 본문에 정의·풀이 없이 첫 등장하는 곳을 grep으로 식별한다. 누락 시 첫 등장 위치에 한 줄 풀이를 보강 위임. **위임 1라운드 단계에서도 사전 점검**하여 위임 메시지에 "이 용어들의 첫 등장 풀이 보강"을 별도 항목으로 명시하면 검토 라운드 한 번을 줄일 수 있다.

### 4. 응답 처리

- **이슈 없음** → 결과 파일 경로를 사용자에게 보고하고 종료
- **이슈 있음** → 가공된 피드백을 같은 writer에게 후속 지시로 전달 → 2번부터 반복 (카운트 +1)
- **카운트 5 도달 → 6번째 위임 시도 시 강제 종료**, 현재 결과 + 미해결 이슈 + 카운트 함께 보고

### 5. 사용자 피드백 처리

사용자가 결과 보고 자연어 피드백 줄 때, 메인이 **1차 가공**한다. 매번 writing-guide map.md를 다시 탐색해서 가장 가까운 규칙·사례에 매핑.

| 사용자 자연어 (예시) | 가공된 지시 |
|--------------|-----------|
| "단정 표현 줄여" | "tone.md '단정 표현 회피' 규칙 적용 — `~이다`를 `~로 보인다` 등으로" |
| "이 부분 더 짧게" | "X 섹션을 N문단으로 축소. 핵심 메시지 보존" |
| "구조 어색해" | "structure.md 사례 대조 — 전제→결론 순서 점검" |

위 표는 **예시**이며 폐쇄 목록 아니다. 표에 없는 자연어가 들어오면 메인이 writing-guide 탐색 후 매핑하고, 매핑 결과를 사용자에게 한 번 확인받고 위임한다.

**문장 반려 시 여러 시안 제시.** 사용자가 특정 문장을 반려하면, 단일 수정안이 아니라 여러 시안을 가져와서 선택하게 위임 지시에 명시한다.

가공된 지시를 같은 writer에게 후속 지시로 전달한다. **메인은 직접 파일 수정하지 않는다.** 단, team-agent 규칙상 위임 경로가 없으면 메인이 수행한다.

**5단계 진입 시 핑퐁 카운터 리셋.** 사용자 자연어 피드백 1건당 새 핑퐁 라운드(최대 5회) 시작.

### 사용자 톤 선호 누적 블록

세션 중 사용자가 자연어 피드백으로 교정한 톤 선호(예: "한자어 빼자", "명사형 종결", "이 단어 말고 저 단어로")는 **누적 블록**으로 보관하고, 이후 모든 위임 지시에 함께 첨부한다.

누적 블록 예시:

```
## 사용자 톤 선호 (세션 누적)
- 딱딱한 한자어 회피 — 유사 → 비슷한, 파편화 → 흩어져
- AI 맥락에서 "학습" 대신 "분석"
- 문제·해결은 한 줄 압축, 디테일은 성과에 모음
```

- writing-guide 규칙을 매번 다시 매핑하는 것과 별개로, 이 블록은 세션 맥락으로 계속 전달한다.
- 시리즈 작업 모드에서 여러 항목을 순차 처리할 때도 누적만 하고 재삭제하지 않는다.
- **거부한 표현 재투입 금지.** 사용자가 한 번 거부한 결의 표현(예: 추상 압축어)은 다음 시안에 또 넣지 않는다.

### 사실 오류 발견 시

writer가 명백한 사실 오류(예: 함수명 오타, 날짜·숫자 오류, 누락된 주요 정보) 발견 시 **수정하지 말고 메인에게 보고** → 메인이 사용자에게 알림. write-refine은 톤·구조·분량만 다룸. 사실은 사용자/write-init 책임.

### 사용자 원본 구조 존중

다듬기는 톤·문장 단위 정리가 기본 범위다. 사용자 원본의 다음 구조 요소는 기본값으로 보존한다:

- bullet 개수
- 섹션·헤딩 개수
- 항목 분리/통합

위 요소를 변경해야 한다고 판단되면 writer가 직접 수정하지 말고 메인에게 보고한다. 메인이 사용자에게 한 번 확인받고 위임 지시에 명시한다.

## 출력

writer가 Edit으로 입력 파일을 **직접 다듬는다**. 별도 파일 생성 X. 사용자에게 같은 파일 경로 안내.

(시안 다양화 모드에서는 예외 — 모드별 확장 섹션 참조.)

## 모드별 확장

### 시리즈 작업 모드

여러 개의 유사한 항목을 순차로 처리할 때 진입한다 (예: 이력서 여러 주제 다듬기, PR 본문 시리즈 작성).

- **이전 확정본 첨부**: 각 항목 위임 시 이전 항목의 확정본(전체 텍스트)을 컨텍스트로 함께 전달한다.
- **재사용 금지 재료 목록**: 이전 항목에서 쓴 핵심 키워드·수치를 "재사용 금지" 목록으로 writer에게 강제 입력한다. 같은 수치·표현이 여러 항목에 반복되면 어필이 희석된다.
- **주제별 관점 분리**: 여러 항목이 같은 도메인 재료를 공유할 때는 관점을 먼저 나눈다. (예: 한 주제 = 개발팀 관점 / 다른 주제 = 사용자·운영팀 관점) 관점이 나뉘면 재료도 자연스럽게 나뉜다.

### 시안 다양화 모드

한 주제에 대해 서로 대비되는 **3축 시안**을 생성하는 모드. `type: resume-item` 패키지에서 기본 진입한다.

- 각 시안 앞에 `### 시안 X — 관점 라벨` 헤더
- 시안 하단에 두 블록:
  - `공통 준수 사항`: 모든 시안이 지킨 톤·포맷 규칙 요약
  - `판단 필요 사항`: 사실관계·수치·범위·톤 선택에서 사용자 확인이 필요한 지점
- 시안 간 대비가 뚜렷하게 (예: 설계 관점 / 확장성 관점 / 운영 관점)
- writer는 파일을 수정하지 않고 **시안 3개를 텍스트로 메인에 회신**한다.
- 메인이 사용자에게 전달 → 사용자 선택·조합 → 확정본이 실제 저장 위치(0.5 단계에서 합의한 곳)에 반영.

## 교정 사례 축적

다듬기 중 메인은 사용자의 교정·지적을 현 세션 대화 컨텍스트에 남긴다. 같은 규칙·사례에 해당하는 건은 대표 예 1개 + 나머지 횟수로 묶는다. 규칙화·writing-guide 반영은 `/pre-exit`가 담당한다.

## write-init과의 관계

write-init이 내용 채움, write-refine이 톤·구조·분량 다듬기. 두 스킬을 한 작업의 두 단계로 사용한다. write-refine은 작업 히스토리 없이 frontmatter만으로 출발.
