# 점검 항목

## 1. 목적/배경 파악

기획서에서 이 기능의 목적과 배경을 추출한다.

- 왜 이 기능이 필요한지 명시되어 있는가?
- 이 기능이 해결하려는 사용자 문제가 무엇인가?
- 성공 기준(KPI, 목표 지표)이 정의되어 있는가?

> 목적이 불명확하면 구현 방향 자체가 흔들린다. 가장 먼저 확인한다.

## 2. 사용자 플로우 검증

상세 점검은 [flow.md](./flow.md) 참조.

- 진입점(entry point)이 명확한가?
- 시작부터 완료까지의 전체 경로가 그려져 있는가?
- 분기 조건(if/else)이 모두 명시되어 있는가?
- "뒤로가기"와 "취소" 시 동작이 정의되어 있는가?

## 3. 상태 정의 확인

상세 점검은 [flow.md](./flow.md) 참조.

- 화면/데이터가 가질 수 있는 상태가 모두 나열되어 있는가?
- 상태 간 전이 조건이 명확한가?
- 각 상태에서 사용자가 할 수 있는 액션이 정의되어 있는가?

## 4. 엣지케이스 확인

상세 점검은 [edge-cases.md](./edge-cases.md) 참조.

- 빈 상태(empty state) UI가 정의되어 있는가?
- 에러 상태별 UI와 복구 방법이 정의되어 있는가?
- 권한별 화면 차이가 정의되어 있는가?
- 데이터가 극단적일 때(0건, 1000건, 매우 긴 텍스트) 처리가 정의되어 있는가?

## 5. 데이터/API 확인

상세 점검은 [data.md](./data.md) 참조.

- 화면에 필요한 데이터가 무엇인지 파악 가능한가?
- 해당 데이터를 제공하는 API가 존재하는가, 새로 만들어야 하는가?
- 사용자 입력값의 유효성 검증 규칙이 정의되어 있는가?

## 6. 비기능 요구사항

- 성능 요구사항이 있는가? (로딩 시간, 대량 데이터 처리 등)
- 보안 관련 고려사항이 있는가? (인증, 인가, 민감 데이터 등)
- 접근성 요구사항이 있는가?
- 다국어/로컬라이제이션 대응이 필요한가?

## 8. URL / 라우팅 / 렌더링 확인

상세 점검은 [routing.md](./routing.md) 참조.

- 각 페이지의 URL 경로가 정의되어 있는가?
- 서버 캐싱 전략과 컴포넌트 유형(Server/Client) 선택 근거가 타당한가?
- 여러 페이지가 공유하는 레이아웃이 식별되어 있는가?

## 9. 일정 영향도 판단

위 1~8의 결과를 종합하여 분류한다:

- 명확하여 바로 구현 가능한 부분
- 기획자 확인이 필요한 부분 (질문 목록)
- 불명확하여 일정 리스크가 되는 부분
