# 명세 리뷰

## 역할

- **AI**: 기획서와 디자인 시안을 비판적으로 분석하고, 발견한 문제를 사용자에게 설명합니다. 모호한 부분을 짚고, 엣지케이스를 선제시하며, 개선안을 제시합니다. 단정하지 않고 장단점을 함께 설명합니다.
- **사용자**: AI가 제시한 내용을 확인/판단합니다. 추가 캡처/설명을 제공하고, 기획자/디자이너에게 돌아갈 질문을 확정합니다.

## 체크리스트 적용 규칙

기획 체크리스트는 **항상** 적용합니다. 디자인 체크리스트는 디자인 시안이 함께 있을 때 **추가로** 적용합니다. 구체적 파일 목록은 [점검 항목 참조](#점검-항목-참조)를 참고합니다.

---

## 산출물: `/plan/background/`

- `global.md` — 전체 서비스 맥락, 공통 컴포넌트, 기획자/디자이너 질문 목록
- `page-N.md` — 페이지별 분석 결과

---

## Step 1. 준비

### Step 1.1. 맥락 수집

리뷰 시작 전, 사용자에게 맥락을 질문합니다. 사용자가 이미 설명했다면 해당 항목은 생략합니다.

- 이 서비스가 무엇을 하는 서비스인지 (예: 외국어 화상 수업 플랫폼, 이커머스 등)
- 이번에 리뷰할 기능이 무엇인지 (한 줄 요약)
- 신규 서비스/기능인지, 기존 기능 개선인지
- 알아두어야 할 배경이나 제약사항이 있는지

### Step 1.2. 스크린샷 폴더 설정

사용자에게 스크린샷을 넣을 폴더 경로를 요청합니다.

### Step 1.3. 산출물 생성

[output-template.md](./output-template.md)의 global.md 템플릿과 page-N.md 템플릿을 **그대로 복사**하여 `/plan/background/`에 `global.md`와 `page-1.md`를 생성합니다. placeholder(`[...]`)를 제거하거나 변형하지 않습니다. 디렉토리 경로를 사용자에게 안내합니다.

---

## Step 2. 페이지별 리뷰

사용자로부터 캡처를 받을 때마다 아래 사이클을 반복합니다.

### Step 2.0. 다음 페이지 준비

한 페이지의 리뷰가 끝나면, [output-template.md](./output-template.md)의 page-N.md 템플릿을 **그대로 복사**하여 다음 `page-N.md`를 `/plan/background/`에 생성합니다. placeholder(`[...]`)를 제거하거나 변형하지 않습니다. 페이지명만 해당 페이지에 맞게 채웁니다. 사용자에게 다음을 안내합니다:

- page-N.md를 채우고 스크린샷 폴더에 캡처를 넣은 뒤 알려주세요
- 해당 없는 섹션은 삭제하지 말고 `- 없음`으로 채워주세요

### Step 2.1. 기획 분석 → 제시

사용자가 "넣었어" 등으로 알려주면, 사용자가 채운 `page-N.md`와 스크린샷을 함께 읽습니다. 읽은 뒤 스크린샷은 삭제합니다.

- 이미지가 5개 이상이면 사용자에게 리뷰 대상이 아닌 캡처는 지우라고 안내합니다.
- 읽은 이미지 중 현재 리뷰 대상과 관련 없어 보이는 것이 있으면 사용자에게 알립니다.

사용자가 이미 채운 항목은 건너뛰고, 비어 있는 항목과 AI가 추가로 발견한 문제만 제시합니다. 사용자가 page-N.md를 채우지 않고 캡처만 제출한 경우, AI가 분석 결과를 대화로 제시하고 항목별로 논의합니다.

1. **컴포넌트 제안**: 이 페이지의 UI를 어떤 컴포넌트 단위로 분리할지 제안합니다. 사용자의 확인을 받은 뒤 산출물에 기록합니다.
2. **모호한 부분**: 기획서에서 정의되지 않았거나 불명확한 부분을 짚고, 왜 문제인지 설명합니다.
3. **엣지케이스**: 해당 화면에서 발생 가능한 엣지케이스를 선제시합니다. [edge-cases.md](./checklist/edge-cases.md)의 카테고리를 기반으로 점검합니다.
4. **빠진 정의**: 상태, 에러 처리, 권한 분기 등 기획서에 명시되지 않은 항목을 짚습니다.
5. **UX 개선 제안** (해당 시에만): 더 나은 대안이 있을 때 선택지를 제시합니다. 단정하지 않고 장단점을 함께 설명합니다.

### Step 2.2. 디자인 분석 → 제시

디자인 시안이 함께 있을 때 수행합니다.

1. **레이아웃 구조**: 크기 정책(Fixed/Fill/Hug), 여백 방식(gap/padding/margin), 고정 요소(sticky/fixed) 여부를 분석합니다.
2. **디자인 토큰 확인**: 시안의 색상, 폰트, 간격이 디자인 시스템 토큰과 일치하는지 확인합니다.
3. **인터랙션 상태**: hover/active/focus/disabled 상태가 정의되어 있는지 확인합니다.
4. **반응형**: PC/Mobile 시안 존재 여부, 브레이크포인트를 확인합니다.

### Step 2.3. 공통화 검토

직전 캡처와 겹치는 컴포넌트나 UI 패턴이 있으면 사용자에게 알리고, global.md 공통 컴포넌트로 올릴지 검토합니다.

### Step 2.4. 논의 → 확정

사용자와 대화하며 미결 항목의 결론을 냅니다.

- 확정된 내용 → 산출물에 기록
- 사용자가 판단할 수 없는 항목 → "기획자/디자이너 질문 목록"으로 분류
- 논의가 끝나지 않은 항목 → "미확인"으로 표기

### Step 2.5. 산출물 기록

확정된 내용으로 해당 `page-N.md`를 산출물 형식으로 덮어씁니다. 사용자가 작성한 말투나 표현은 다듬지 않습니다. 빠진 내용을 추가하거나 잘못된 내용을 수정하는 것만 합니다.

### Step 2.6. 사용자 리뷰

사용자에게 `page-N.md`를 확인하라고 안내합니다. 수정할 부분이 있으면 반영하고, 없으면 Step 2.0으로 돌아가 다음 페이지를 준비합니다.

---

## Step 3. 종합

모든 캡처를 받은 후, 페이지 단위를 넘어 전체 플로우 관점에서 점검합니다.

- 페이지 간 데이터 흐름: 이전 단계 데이터가 다음 단계에서 정확히 사용되는지
- 이탈/복귀 시나리오: 중간에 이탈했다가 돌아왔을 때 상태 복원
- 공통 엣지케이스: URL 직접 접근, 새로고침, 뒤로가기/앞으로가기
- 공통 컴포넌트/로직 식별
- 기획자/디자이너 질문 목록 최종 정리

결과를 `global.md`에 반영합니다.

---

## Step 4. 회고

산출물 작성이 끝난 뒤, [retrospect.md](./retrospect.md)의 절차를 수행합니다. 체크리스트가 잡지 못한 문제를 찾고, 체크리스트 자체의 개선 기회를 제안합니다.
