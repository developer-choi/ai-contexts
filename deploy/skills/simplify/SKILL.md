# AI 행동 가이드

## 목적

라이브러리를 학습에 필요한 구현만 남겨 단순화한다.

> **용어 정의**
> - **SIMPLIFY_SOURCE**: 이 스킬 폴더 (`deploy/skills/simplify/`). 워크플로우, 패턴 참조, 문서 양식 등 규칙 파일이 있는 곳.
> - **SIMPLIFY_TARGET**: 간소화 대상 프로젝트 (`simplified-[라이브러리명]`). 실제 코드를 읽고 단순화하는 작업 프로젝트.

## [CRITICAL] 지킬 원칙

이 스킬의 목적은 **학습에 필요한 구현만 남기는 것**이다. 아래 원칙이 다른 모든 지침(형제 컴포넌트 컨벤션, 패턴 카탈로그 포함)에 우선한다.

### 간소화는 기능 단위까지 간다

Scope 제거·forwardRef 제거 같은 구조적/기계적 변환은 **최소선**이지 목표가 아니다. 코드가 짧거나 동작이 유지된다고 해서 간소화할 게 없는 것이 아니다. 기준은 "코드가 얼마나 긴가"가 아니라 **"학습에 필요한 구현만 남아있는가"**이다.

### 제거/유지 판정 — 학습 범위 테스트

각 기능·서브시스템마다 묻는다: **"이 대상의 핵심 개념을 학습하는 데 이 구현이 필요한가?"**

- 핵심 흐름 이해에 없어도 되는 것(장식성 기능, 드문 고급 케이스, UX 폴리시성 타이밍, 중첩·멀티인스턴스 변형 등) = **제거 대상**
- 핵심 동작·접근성·합성 구조에 필수인 것 = **유지**
- **delta-over-sibling**: 이미 간소화된 더 단순한 형제(예: dropdown)와 비교한다.
    - 빼면 그 형제와 **구별되지 않게 되는** 기능 = 이 대상이 형제보다 더 가르치는 **delta** = 학습 포인트 = **유지**. 더 단순한 형제에 **아예 없는** 고유 기능일수록 delta일 가능성이 높다(없다고 제거 후보가 아니다).
    - 더 단순한 형제도 **이미 제거했거나 기본값으로 고정한** 기능 = 제거 선례 있음 = **제거 후보**.
    - 애매하면 **제거 쪽으로 추천 판정**하고 근거를 `plan.md`/커밋 계획에 남긴다(사용자에게 묻지 않는다). 학습에 불필요하면 제거가 기본값이다.
- "동작이 바뀐다"는 사실 자체는 유지 사유가 아니다. 단, 기능을 제거할 때 **유지하기로 한 다른 기능을 떠받치는 필수 코드**까지 함께 지우지 않는다(부분 해부 — `common-keep-patterns.md`).

### 형제는 컨벤션 참고일 뿐, 제거 범위의 기준이 아니다

이미 간소화된 형제는 **코드 스타일·구조적 변환 패턴을 미러링**할 때만 참고한다(throw 패턴, ref-as-prop 형태 등). **형제가 기능을 안 지웠다는 사실은 이 대상에서 기능을 유지할 근거가 못 된다** — 형제 작업이 보수적이었을 수 있고, 형제는 기계적 패스만 거쳤을 수 있다. 형제의 최종 상태에 맞추는 것(sibling-mimicry)은 금지한다.

### 제거 범위는 AI가 판단해 추천대로 진행한다 (init 제외)

간소화 단계에서는 어떤 기능을 제거/유지할지 **AI가 학습 범위 테스트로 판단해 추천대로 곧장 진행**한다. **도중에 사용자에게 묻거나 승인을 기다리지 않는다** — 판정 근거는 계획서(`plan.md`/커밋 계획)와 `*-simplified.md`에 명시한다. 공개 API가 바뀌어도 근거를 적고 진행한다. (이 항목은 "여러 항목 하나씩 제시"·"대규모 변경 시 승인" 류의 일반 선호를 간소화 단계에 한해 덮어쓴다.)

- **예외**: 새 프로젝트 **초기 세팅(`initialize.md`)** 단계에서는 기존대로 사용자 확인을 받는다.
- AI가 단독으로 "기능은 안 건드린다"고 **축소** 결정하지도 않는다 — 학습에 불필요하면 제거가 기본값이다(위 학습 범위 테스트).

### 큐는 배치로 처리하고 정지 조건에서 멈춘다

`analysis-queue.md`의 "다음 분석 목록"을 처리할 때:

- **배치 단위**: 한 번에 **최대 10개 대상**을 연속 처리한다(1개씩 멈추지 않는다). 대상마다 `main.md`의 단계(계획→original→단순화 커밋→simplified→큐 갱신)를 적용하되 사용자 확인 없이 이어간다.
- **정지 조건** — 아래 중 하나면 멈추고 사용자에게 **종합 보고**한다:
    - 큐의 "다음 분석 목록"이 비었다.
    - 남은 대상이 **전부** "여기서 더 간소화하면 학습 목적에 부합하지 않는" 것들뿐이다(예: 순수 재export 진입점, 타입 정의만 있는 파일 중 제거할 기능 참조가 없는 경우 등). 이 판정도 AI가 내리고 근거를 보고에 적는다.

## 작업 공통 규칙 (모든 단계 적용)

### 코드 읽기 전용

이 프로젝트의 목적은 소스 코드를 읽고 이해하는 것이다. 의존성 설치·실행·빌드를 하지 않는다 — `npm/yarn install`, `yarn test`, `tsc`, `npm run build` 등을 실행하지 않으며, package.json의 실행 관련 설정도 초기 세팅에서 제거한다(`initialize.md`).

### 산출물 문서는 별도 커밋

분석 문서(`*-original.md`, `*-simplified.md`, `analysis-queue.md` 등)는 코드 변경과 같은 커밋에 섞지 않고 별도 커밋한다. 단순화 커밋만 따로 추적·리뷰할 수 있어야 하기 때문이다.

## 단순화 작업 진행

1. **작업 프로세스**: `main.md` 읽기
2. **유지/삭제 패턴 참고**:
   - SIMPLIFY_SOURCE `common-keep-patterns.md` — 공통 유지 패턴
   - SIMPLIFY_SOURCE `common-simplification-patterns.md` — 공통 삭제 패턴
   - SIMPLIFY_TARGET `instructions/keep-patterns.md` — 라이브러리 특화 유지 패턴
   - SIMPLIFY_TARGET `instructions/simplification-patterns.md` — 라이브러리 특화 삭제 패턴
3. **문서 양식 참고** (필요 시): `[타입]/format/` 하위 파일 읽기
   - `[타입]`: 컴포넌트 라이브러리 → `component`
   - 비컴포넌트(상태관리·hooks·유틸 등)도 `component` 양식을 쓰되, 컴포넌트 전용 필드(렌더링 구조·Props)는 대상에 맞게 적응한다(모듈/클래스 공개 API·호출 구조 등). 별도 타입 폴더는 만들지 않는다.
4. **코드 변환**: `convert.md` 읽기
