# 계획 단계 — 심층 분석 (복잡한 대상)

`main.md` 계획 작성 절차의 ②(제거/유지 판정)를 **복잡한 대상**에서 서브에이전트로 심층 수행하는 절차다. 기능·서브시스템이 여럿이고 학습가치 판단이 모호할 때만 쓴다. 단순한 대상(독립 제거 후보가 거의 없는 소형 컴포넌트)은 메인이 직접 판정하고 이 절차를 생략한다.

## 1. 범위 사전 질문 (fan-out 전)

계획 작성 전 사용자에게 간소화 강도를 묻는다:

- 기계적만 / 공격적(기능 제거 포함) / 항목별로 추천받고 결정
- "기계적만" 선택 시 fan-out 생략, 구조적 패턴만으로 계획한다.

## 2. 서브에이전트 fan-out (공격적·추천 선택 시)

메인이 소스를 읽어 제거 후보 서브시스템을 식별한 뒤 다음을 **병렬**로 띄운다. 메인이 이미 읽은 소스는 슬라이스(서브시스템명 + 파일 경로 + 질문)만 전달하고 전체 재로딩을 시키지 않는다.

- **제거 분석가** (서브시스템당 1) — 필수 출력:
  - 이 서브시스템이 하는 일(요약)
  - 전용 코드/상태/props/effect와 **coupling map**(제거 시 깨지는 것·같이 고쳐야 할 것)
  - 제거 vs 유지 **양면 논증** 후 verdict + confidence
- **학습핵심 수호자** (1) — `analysis-queue.md`와 더 단순한 형제 `*-simplified.md`를 읽고, 이 대상이 더 단순한 형제 대비 갖는 **delta(=학습 포인트)**를 지목한다. 이 목록은 **거부권**이다 — 자동 제거 불가, CONTESTED로만 넘긴다.

> 분석가는 고립되면 모든 걸 "제거 가능"으로 기우는 경향이 있다(서브시스템은 따로 보면 다 선택적으로 보인다). 수호자의 delta 거부권이 이 편향을 잡는다.

## 3. 종합 (메인)

분석가 verdict와 수호자 delta 목록을 대조해 분류한다:

- **KEEP** — 수호자 delta이거나 분석가가 유지로 판정
- **REMOVE** — 분석가·수호자 모두 제거 가능
- **CONTESTED** — 분석가 "제거" vs 수호자 "학습 핵심" 충돌

## 4. 범위 확정 질문 (fan-out 후)

**CONTESTED 항목만** 사용자에게 판정받는다(기본값 = KEEP). 확정된 범위로 `main.md` 계획 작성 절차 ④(커밋 계획)로 진행한다.
