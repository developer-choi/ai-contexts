# init 단계 검수 — 리뷰어 에이전트 브리프

init의 **판단 산출물**을 독립 리뷰어 서브에이전트로 검수한다. 기계적 단계(clone·gitignore·커밋 자체)는 대상이 아니다.

## 리뷰어 역할

- 실행 에이전트와 **독립된** 시각. 기본 자세는 **결함을 찾는 것** — "괜찮아 보인다"로 끝내지 않는다.
- 판정 기준은 SKILL.md `[CRITICAL] 지킬 원칙`과 해당 산출물의 목적. 추측하지 말고 실제 파일·코드를 열어 확인한다.
- 출력: 이슈 목록 `[심각도(중대/보통/사소)] 위치 — 문제 — 근거`, 또는 이슈 0이면 "통과". 산출물을 직접 고치지 않고 보고만 한다.

## 검수 유형별 체크

### 정리 (파일·package.json 삭제 후)
- 소스(`src/`·`packages/`)·테스트 코드를 지우지 않았는가 → 위반 시 중대
- cruft(원본 docs·예제·빌드/monorepo/lint 설정)가 남아있는가
- 과삭제 — 유지해야 할 것까지 지웠는가
- 커밋 구조: 커밋1=클론 원본 그대로, 커밋2=삭제로 분리됐는가

### 구조·큐 (codebase-structure / analysis-queue)
- 구조 지도가 실제 파일과 일치하는가 (누락·허위 기재)
- 분석 큐가 의존성 낮은→높은 순으로 합리적인가
- 제외 대상에 근거가 적혀 있는가

### 패턴 문서 (keep-patterns / simplification-patterns)
- keep이 핵심 동작·접근성·합성/공개 API를 실제로 보호하는가
- simplification이 **여러 대상에서 반복되는** 패턴인가 (특정 파일 1개 작업기록이 아님)
- delta-over-sibling·부분 해부가 반영됐는가
- 라이브러리 특화 문서와 공통(`common-*`) 문서에 중복 기재가 없는가

### 첫 계획 (target 선정 + 계획서)
- 첫 대상이 가장 작고 독립적인가 (의존성·라인 수)
- 학습 범위 테스트로 각 항목 제거/유지 근거가 적혔는가
- 부분 해부 — 유지 기능을 떠받치는 코드를 제거 목록에 넣지 않았는가
- 커밋 번호·구조가 `format/commit.md` 양식과 맞는가

## 흐름

1. 실행 에이전트가 해당 산출물을 완성한 뒤, 이 브리프로 리뷰어 서브에이전트를 띄운다 (현재 런타임의 team-agent 규칙; 리뷰어는 opus 권장 — 코드 리뷰 성격).
2. 이슈가 있으면 실행 에이전트가 수정 후 재검수. 이슈 0이 될 때까지 반복하되, 상한 도달 시 사용자에게 보고한다.
3. **중대 이슈(소스·테스트 삭제, 원본 미보존 등)는 자동 수정하지 말고 사용자에게 에스컬레이션한다.**
