# 기존 프로젝트 현행화 (renew)

이미 만들어진 `simplified-<라이브러리명>` 프로젝트의 **인프라**(정리 상태·폴더 구조·분석 문서 양식·패턴 분류)를 현행 표준으로 끌어올리는 모드다. 소스는 재간소화하지 않는다.

모델과 이 스킬은 계속 개선되므로, 옛 스킬로 init된 프로젝트는 시간이 지나면 인프라가 낡은 표준이 된다. renew는 **소스·완료기록·패턴 내용을 보존한 채** 재생성 가능한 인프라만 현행 표준으로 다시 적용한다.

## 절차 — `initialize.md` Step 3~8 재실행

renew는 새 절차를 정의하지 않는다. `initialize.md`의 **Step 3~8을 기존 프로젝트에 재실행**하되, 아래 "스킵"과 "단계별 차이"를 적용한다. 각 단계의 본문·검수 체크포인트(`init-review.md`)는 `initialize.md`를 단일 출처로 따른다.

### 스킵하는 단계 (재실행하지 않음)

- **Step 1 (shallow clone)** — 이미 클론돼 있다.
- **Step 2 (git 재초기화 + 원본 커밋)** — 히스토리를 다시 만들지 않는다. (아래 「한계」 참조)
- **Step 9 (첫 간소화)** — 소스를 재간소화하지 않는다.

### 단계별 차이 (기존 프로젝트 재적용)

| Step | renew에서의 의미 |
|---|---|
| 3~5 정리 (README·package.json·파일·.gitignore) | 옛 스킬이 안 지운 cruft만 추가로 삭제·교정한다. 변경이 있으면 별도 커밋. 이미 정리된 항목은 건드리지 않는다. |
| 6 구조 | 폴더 구조를 현행 표준에 맞춘다. 기존 분석 문서·소스 위치는 보존하고 누락된 구조만 보강. |
| 7 구조·큐 문서 | `docs/codebase-structure.md`·`docs/analysis-queue.md`를 현행 양식으로 **재포맷**한다. 양식·형식만 현행화하고 내용은 보존한다. |
| 8 패턴 문서 | `instructions/keep-patterns.md`·`instructions/simplification-patterns.md`를 현행 규칙으로 **재분류·재검증**한다. 예: `[반복]`(여러 파일) vs `[단일]`(한 파일 서브시스템) 분류, lanes류 delta-over-sibling `CONTESTED` 표기. |

## [CRITICAL] 보존 — 절대 건드리지 않는 것

- **간소화된 소스 파일 전체** — 재간소화하지 않는다.
- **`analysis-queue.md`의 `완료` 기록** — 어느 대상이 끝났는지.
- **패턴 문서의 축적된 라이브러리 고유 내용** — 형식·분류만 현행화하고 내용은 보존한다. 패턴은 실제 간소화 작업에서 학습된 것이라, 작업을 다시 하지 않으면 재발굴이 불가능하다. renew는 "재포맷·재검증"이지 "재발굴"이 아니다.

## 한계 — renew로 못 고치는 것

git 히스토리는 renew로 고칠 수 없다. 옛 프로젝트의 가장 큰 인프라 결함은 "원본 미보존 커밋 구조"(첫 커밋이 이미 정리된 상태라 원본을 history에서 diff할 수 없음)인데, 이는 히스토리 재작성 없이는 불가능하다 → **re-init(전체 재초기화)만 해결한다.** renew는 재생성 가능한 인프라에 한정한다.

"renew하면 커밋 구조도 고쳐지겠지"라고 기대하지 말 것. 커밋 구조 결함을 고치려면 re-init이 필요하고, 그 경우 소스 재간소화 비용을 감수해야 한다.
