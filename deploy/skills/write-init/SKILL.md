# write-init — 내용 작성

## 목적

외부 공개용 글(PR 본문·README·PR 코멘트·이력서·결정문)의 **내용**을 채운다. 골격(헤딩)을 세우고, 작업 맥락(git log·코드·대화)에서 끌어올 수 있는 건 직접 채우고, 사용자 의도가 필요한 곳만 받는다. 톤·구조·분량 다듬기는 다음 단계 `write-refine`. 이 스킬은 작업 컨텍스트를 가진 작업 세션에서만 호출한다.

## 입력

`/write-init <type> [<subtype>]` 형태로 호출한다.

| type | subtype | 필수/선택 | 설명 |
|------|---------|----------|-----|
| `pr-body` | (없음) | 선택 | 일반 PR 본문 |
| `pr-body` | `recruitment` | 선택 | 채용용 PR 본문 — 어필 포인트 강조 |
| `readme` | — | — | README (subtype 미사용) |
| `pr-comment` | `reviewer` | **필수** | 검토자가 남의 PR에 코멘트 |
| `pr-comment` | `author` | **필수** | 작성자가 자기 PR에 답글 |
| `resume-intro` | — | — | 이력서 한줄소개 — 수치 중심 bullet 다단 |
| `resume-item` | — | — | 이력서 경력기술서 한 항목 — 제목/문제/해결/성과 4줄 블록. write-refine에서 시안 다양화 모드로 진입 |
| `decision` | — | — | 기술적 의사결정 문서 — 선택지 2개 이상 비교 + 장단점 + 선택 근거 |

subtype 누락·오타 시 한 번 묻고 진행한다.

## 경로 판정 — 먼저, 사실 질문으로 결정론적으로

모델 추론으로 넘겨짚지 않는다. 아래를 **사실 질문**으로 물어 경로를 고정한다:

> "쓰실 글의 **골격(섹션 구조)이 이미 있으세요**, 아니면 **재료만** 있으세요? 혹은 **기존에 쓴 글을 재활용**하시나요?"

- **막연** (재료만, 뭘 쓸지 추상적) → 골격 게이트 **ON**
- **확실** (내용·구조 다 앎, 사용자가 초안 보유) → 골격 게이트 **OFF**
- **재활용** (기존 확정본 + 새 요구) → **델타 게이트** (추가·삭제되는 헤딩만 미니 검수)

## 작업 흐름

### 1. 템플릿·가이드 로드 (on-demand)

- `templates/<type>.md`가 있으면 Read (pr-body·decision·resume-intro·resume-item).
- type별 가이드가 있으면 Read: `resume-item`·`resume-intro` → `../../contexts/writing-guide/resume-guide.md`, `decision` → `../../contexts/writing-guide/decision-guide.md`, `readme` → `../../contexts/writing-guide/readme-guide.md`.
- 해법·선택을 담는 글(`decision`·`resume-*`)이면 `../../contexts/writing-guide/tradeoff-guide.md`도 Read한다 (트레이드오프 점검 — 아래 [가드4]).
- **무조건 전부 로드하지 않는다.** 해당 type에 필요한 것만.
- 예외 — `pr-comment`(템플릿 없음, 자유 형식: subtype에 따라 질문/제안/해명/감사 구조를 동적 결정), `readme`(템플릿 없음, `readme-guide.md` 구조 따름. 1차 소스는 작업 컨텍스트가 아니라 대상의 SKILL.md/핵심 문서. SKILL.md에 있는 내용은 되묻지 말고 번역하고 빈칸을 최소화).
- **대상 레포 규약 따르기**: 글이 다른 레포(MP·AC·DC 등)에 들어가면 그 레포의 CLAUDE.md 운영방침·frontmatter 스키마를 따른다. 출처 문서가 있으면 그 구조를 골격으로 삼고, 시리즈면 cross-link 구조를 보존하며 N개 skeleton을 동시에 만든다.
- HTML 코멘트(`<!-- -->`) 영역은 메타 가이드 — 참조만, 본문 출력 제외. `## 본문 시작 ↓` 아래만 본문.

### 2. 골격 + 뼈대 초안

경로별로:

- **막연 (게이트 ON)**: 헤딩 계층(골격)을 먼저 세워 사용자에게 보인다. **헤딩 구조만** 검수받는다(내용 아님). OK 후 내용 채움.
- **확실 (게이트 OFF)**: 사용자 초안을 골격으로 받아 바로 내용 단계로.
- **재활용 (델타)**: 기존 확정본을 기준 골격으로, 추가·삭제되는 헤딩만 짚어 확인.

### 3. 내용 채움 — 두 가드 준수

작업 맥락(git log·diff·코드·대화)과 템플릿을 결합해 채운다.

- **(가) 과잉 생성 기본**: 근거 있으면 왕창 채운다. "창조는 어렵고 삭제는 쉽다" — 사용자가 덜어내면 됨.
- **(나) 빈칸 마크 예외**: AI가 추론 불가한 정보(개인사 등 원문·맥락에 근거 없는 것)는 채우지 말고 `[채울 내용]` 마크로 남겨 사용자에게 받는다.
- **(가)/(나) 경계 = 날조가드**: 원문·맥락에 근거 있으면 (가), 지어내야 하면 (나). **날조할 바엔 빈칸.**

**[가드1] 날조 방지**: 원문·컨텍스트(git diff·코드·대화) 어디에도 없는 수치·지표·고유명사·도구명·라이브러리명을 추가하지 않는다. "왜 그 선택을 했나"는 풀되 "무엇을 더 했다"는 근거 있을 때만. 보강 문단의 각 문장이 "근거가 어디 있나" 자체 점검을 통과해야 한다.

**[가드2] 사용자 목소리 보존**: 사용자가 쓴 텍스트는 그대로 두고 컨텍스트로 보강만. 의도 왜곡 금지.

**[가드3] 리서치 디테일을 저자 경험으로 차용 금지** (1인칭 글 — 이력서·블로그 등): 외부 리서치(서브에이전트·웹)로 얻은 구체 사례(고유명사·증상·수치)를 저자가 실제 겪은 일처럼 본문에 1인칭으로 넣지 않는다. 날조가드의 1인칭 확장이다. 저자 확인 전이면 일반화된 표현으로만 쓰거나 `[채울 내용]`으로 묻는다 — 생생함이 그럴듯해도 출처가 "저자 경험"이 아니면 (나)다.

- **before**: 서브에이전트가 찾은 "스크린리더(VoiceOver/NVDA)에서 접근성 실패" 사례를 저자가 겪은 직접 개발 단점으로 서술.
- **after**: 저자에게 확인 → 실제 경험은 "iOS·안드로이드·크롬 등 모던 브라우저 간헐 버그·디버깅 고통"이라 그것으로 교체, 스크린리더 사례는 제거.

**[가드4] 트레이드오프 누락 점검** (해법·선택을 담는 글 — `decision`·`resume-*`·블로그 등): 해법·선택을 제시하는 절마다 그 선택에 딸려 치른 단점·비용(이득과 맞바꾼 것)이 있는지 점검한다. 빠졌으면 보강하되, 그 트레이드오프는 **저자 고유 판단**이므로 컨텍스트에 근거가 없으면 일반론 교과서 단점으로 메우지 말고 `[채울 내용]`으로 저자에게 받는다(날조가드 상속). 세부·서술 규칙은 [tradeoff-guide.md](../../contexts/writing-guide/tradeoff-guide.md). `decision`은 decision-guide의 비교 구조 강제가 우선한다. **문서 전체 type이 위 목록에 없어도(`readme`·`pr-body` 등) 본문에 "검토한 접근법"·"방법 A vs B"·"왜 X 대신 Y" 같은 비교·선택 절이 있으면 그 절에만 이 점검을 건다** — 문서 전체가 아니라 해당 절 단위로.

- **before**: 모노레포 선택의 단점을 "빌드 시간이 길어지고 빌드 경계를 나눠 관리해야 한다"는 일반론으로 채움.
- **after**: 저자의 실제 판단으로, 교환으로 교체 — "모노레포로 묶으면 코드 공유는 쉬워지는 대신, 공유 안 하는 코드까지 한 저장소에 얽혀 깃 로그가 섞이는 손해를 같이 떠안는다". 근거 없었으면 `[채울 내용]`으로 받았다.

### 4. 사용자 채움·리뷰

마크가 있으면 채워달라 안내하고 명시적 신호("OK"·"다 적었어")까지 대기. 신호 오면 사용자 텍스트를 보존한 채 컨텍스트로 펼친다. 빠진 내용/시각을 묻고 반영(캡 없음). **이 단계는 내용만 — 톤 수정 요청은 "그건 write-refine에서"로 안내.** 산출물 주석·지적이 단어 선택·표현 다듬기("이 단어 별로", "표현 어떻게 다듬지")면 init에서 리라이트 안을 만들지 말고 write-refine으로 안내한다. 한 주석에 내용(무엇을·어디에)과 표현(어떤 단어로)이 섞이면 내용만 init에서 확정하고 표현은 refine으로 넘긴다. 사용자 인라인 주석을 반영한 뒤에는 반영한 주석 마커를 그 자리에서 모두 제거한다 — 일부만 지우지 않으며, 미반영(보류) 주석만 남긴다.

분량은 frontmatter `length_target`이 있으면 그 값을, 없으면 `templates/<type>.md` default를, 그것도 없으면 사용자 텍스트의 2~3배를 상한으로 삼는다.

### 5. 패키지 저장

확정 본문을 frontmatter + 본문 단일 .md로 저장하고 경로를 안내한다. 이 파일이 write-refine 입력.

## 패키지 형식

이 섹션이 패키지(frontmatter + 본문 단일 .md) 형식의 SSOT다. write-refine 등 다른 스킬은 형식 정의를 본문으로 재인용하지 말고 이 섹션을 포인터로 가리킨다.

```markdown
---
type: pr-body
subtype: recruitment
audience: 채용담당자
audience_knowledge: 무한스크롤 도메인 모름, 코드베이스 처음
purpose: 성능 최적화 어필
key_message: 무한스크롤 도입으로 초기 렌더 시간 60% 단축
length_target: 주요 섹션 3-4개, 섹션당 3-4문단
rendering_env: markdown
placeholder_policy: keep
refs:
  git_log: a1b2c3..d4e5f6
  related_files:
    - src/components/InfiniteScroll/
    - docs/perf.md
---

# PR 본문

## 변경 사항
...
```

**필수 필드** (누락 X): `type`, `subtype`, `audience`, `purpose`, `key_message`
**선택 필드** (불확실 시 비우거나 `미상` 표기 허용): `audience_knowledge`, `length_target`, `refs`, `rendering_env`, `placeholder_policy`

- `rendering_env`: `markdown` (default, 생략 시 가정) | `plain-text` | `rich-text`. 본문이 출력될 환경. `plain-text`(이력서 입력 필드, 텍스트 폼 등)면 백틱·코드펜스·헤딩 마크 등 마크다운 문법을 본문에 사용하지 않는다.
- `placeholder_policy`: `keep` (default, 생략 시 가정) | `drop` | `mark`. 사용자가 채울 자리(`n`, `[채울 내용]` 등) 처리 방식. `keep`은 그대로 두고, `drop`은 검증 불가 수치 등을 제거하며, `mark`는 `[...]` 표기만 사용한다.

frontmatter는 새 세션의 write-refine이 작업 히스토리 없이 출발할 수 있게 해주는 핵심 인터페이스다.

## 산출 후 1회 안내

패키지 저장 후 아래를 한 번만 안내하고 끝낸다 (반복 넛지 없음):

- **다듬기는 새 세션에서**: 톤·구조·분량은 다음 단계다. **새 세션을 열어** `/write-refine <패키지 경로>`로 호출한다 (작업 히스토리 없이 frontmatter만으로 출발).
- 단어 빼기 같은 **표면 교정은 직접** 하시는 게 빠릅니다.
- 수정 지시는 채팅으로 하나씩 말고 **산출물에 주석으로 일괄** 달아주세요 — refine이 모아서 한 번에 반영합니다.
- **잘못된 내용은 refine 전에 여기(init)서 확정**하세요. refine은 표현만 다루고 내용 오류는 고치지 않습니다.

## write-refine과의 관계

write-init은 내용, write-refine은 톤·구조·분량. frontmatter 패키지로 인계한다. 두 스킬을 한 작업의 두 단계로 사용한다.
