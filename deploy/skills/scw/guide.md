# 스킬 생성/개선 시 따르는 규칙

skill-creator 플러그인을 사용하되, 이 프로젝트에서는 추가로 아래를 따른다.

## 사전 학습

스킬을 만들기 전에 아래 블로그를 읽어라. how 최소화, gotchas 축적 등 이 가이드의 원칙이 여기서 나왔다:
- https://claude.com/blog/improving-skill-creator-test-measure-and-refine-agent-skills

## skill-discovery

신규 스킬을 만들기 전에, 기존 스킬 중 유사한 것을 탐색하여 구조/패턴을 참고한다.

- `deploy/skills/` 하위 SKILL.md들의 description을 읽어 유사 스킬 탐색
- 유사 스킬이 있으면 구조를 참고하여 신규 스킬 설계

## 설계 토론 (Capture Intent 보강)

skill-creator의 Capture Intent / Interview 단계에서, 사용자에게 바로 질문하지 않는다. 에이전트팀을 만들어 설계 토론을 거친 뒤 정제된 제안을 가져온다.

### 팀 구성

| 역할 | 책임 |
|------|------|
| 설계자 | 사용자 목표(마이그레이션 문서 등)를 달성하는 설계안을 만든다. 목표 수호 책임 |
| 토론자 | 설계안의 결함을 찾고 도전한다. 구현 후에는 리뷰어 역할도 겸한다 |
| 구현자 | 합의된 설계안을 실행한다. 설계자와 세트로 움직인다 |

설계자+구현자가 한 세트, 토론자가 독립이다.

### 인프라

설계 토론 팀은 Agent tool의 서브에이전트가 아니라 **TeamCreate로 팀을 구성**한다. 서브에이전트는 1회 응답 후 종료되지만, 팀 에이전트는 계속 살아있어 다라운드 토론이 가능하다.

### 절차

1. **사용자 의도 최소 확인** — 뭘 하고 싶은지, 왜 하는지만 파악한다
2. **설계 토론** — 설계자가 기존 파일을 분석하고 설계안을 만든다. 토론자가 도전한다. 2-3라운드 토론 후 합의. 폴더/파일 구조가 관련된 작업이면, 구조의 합리성(파일 위치, 공용 vs 개별, 참조 경로)도 토론 범위에 포함한다.
3. **사용자에게 제안** — 합의된 설계안을 가져온다. 사용자는 승인/수정/반려한다
4. **구현** — 구현자가 실행한다. 토론자가 리뷰한다

### 왜 이 구조인가

- **설계자가 목표를 수호한다**: 토론자 2명이면 누구도 사용자 목표에 책임이 없다. 설계자가 사용자 목표 달성을 책임지면서 토론자의 도전을 받는다
- **설계 토론이 구현 전에 일어난다**: 구현 후 리뷰에서 설계 문제가 터지면 수정 비용이 높다. 설계안 단계에서 토론하면 비용이 낮다
- **토론자가 설계 맥락을 안다**: 설계 토론에 참여했으므로 구현 리뷰 품질이 높다

## how 최소화 원칙

- 새 스킬이든 기존 스킬 개선이든, what(무엇을 하라)만 남기고 how(어떻게 하라)는 최소화한다
- 기존 스킬 개선 시: OLD(현재) vs NEW(how 제거) 비교 테스트를 거친다
- how를 남길지 판단 기준: "이걸 빼면 AI가 못 하는가?" — 반드시 비교 테스트로 검증한다. 자체 판단만으로 넘어가지 않는다

## 비교 테스트는 AI가 수행

- 테스트 대상(엉망인 코드 등)은 AI가 만든다
- 테스트 실행 후 비교 분석도 AI가 한다
- 사용자는 AI의 분석 결과를 보고 최종 판단만 한다
- 테스트 버전은 넉넉하게 만든다. OLD/NEW 2개에 한정하지 말고, 변형을 여러 개 만들어서 동시에 비교해도 된다 (예: OLD/NEW/NEW2/NEW3). 동시에 20개 미만으로
- 테스트 케이스는 최소 3개
- 테스트 결과 분석 후, "제거한 항목이 실제로 필요한 상황을 테스트했는가?" 되돌아본다. 테스트 케이스가 제거 항목을 스트레스 테스트하지 못했다면 추가 테스트를 돌린다
- "테스트 필요 없어 보인다"는 판단을 하지 않는다. 어떤 스킬/step이든 테스트할 거리를 찾아내서 검증한다
- 리뷰 완료 후, 자신의 리뷰를 비판적으로 되돌아본다. 테스트 케이스가 충분했는지, 빠진 엣지 케이스는 없는지 스스로 따져본다. 부족한 테스트 케이스를 발견하면 사용자 승인 없이 바로 추가 벤치마크를 돌린다

## 산출물 양식 원칙

- 산출물 예시(마크다운 코드블록)로 양식을 고정하지 않는다
- 대신 **필수 필드만 명시**하고, 구체적 마크다운 형태는 AI 자유
- 이유: 예시로 고정하면 더 나은 구조를 만들 기회를 막는다. 필수 필드만 지정하면 양식 일관성과 유연성을 둘 다 잡을 수 있다
- 예시를 제거할 때, 그 예시가 유도하던 필수 필드/행동이 있는지 확인한다. 있으면 예시는 제거하되 필수 필드명은 남긴다

## 산출물 규칙

- 스킬은 `deploy/skills/{skill-name}/`에 생성
- 드래프트 상태면 `STATUS.md`를 함께 생성하여 진행 상황과 남은 작업 기록
- 실사용 후 안정되면 `STATUS.md` 삭제
- commitlint scope에 스킬명 추가
- 완료된 백로그는 `plan/backlog/`에서 삭제

## gotchas 축적

- 스킬을 처음 만들 때 gotchas는 비워둔다
- 실사용 중 AI가 반복 실패하는 패턴을 발견하면 SKILL.md에 gotchas 섹션으로 추가

## 서브에이전트 위임 구조 테스트

스킬 내부에서 Agent tool로 서브에이전트를 띄우는 패턴은 skill-creator 표준 방식(서브에이전트에서 스킬 실행)으로 테스트할 수 없다. 이 경우 메인 에이전트가 직접 오케스트레이터 역할을 수행한다. 상세는 [eval-delegation.md](eval-delegation.md) 참조.

## 정리

- **벤치마크/테스트 완료 후 workspace 폴더 삭제 필수**: 작업 중 생성한 workspace 폴더(테스트 문서, eval 결과, 임시 산출물 등)를 반드시 삭제한다. 커밋 전에 `rm -rf`로 제거하고, 삭제 여부를 확인한 뒤 다음 단계로 넘어간다. workspace 폴더가 남은 채로 작업을 종료하지 않는다.

## gotchas

- **벤치마크 인프라 실패 시 스킵 금지**: OLD 스냅샷 경로 오류 등으로 비교 대상이 무효화되면, 원인을 수정하고 해당 run만 재실행한다. "additive 변경이라 괜찮다"는 합리화로 넘어가지 않는다.
