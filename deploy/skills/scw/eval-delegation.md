# 서브에이전트 위임 구조 Eval 가이드

skill-creator의 표준 eval(서브에이전트에서 스킬 실행)은 **스킬 내부에서 Agent tool을 사용하는 패턴**을 테스트할 수 없다. 서브에이전트는 Agent tool이 없기 때문이다.

## 해결: 메인 에이전트가 직접 오케스트레이션

최상위 세션의 메인 에이전트가 스킬의 지시를 따라 직접 실행한다.

| | skill-creator 표준 | 메인 직접 실행 |
|---|---|---|
| 실행 주체 | 서브에이전트 | 최상위 세션 |
| Agent tool | 없음 | 있음 |
| 병렬 A/B | 가능 | 불가 (순차) |
| 적합한 대상 | 단일 에이전트로 완결되는 스킬 | 내부에서 서브에이전트를 띄우는 스킬 |

## 절차

1. 가상 프로젝트 준비 (git init, 최소한의 파일, 스킬이 요구하는 산출물)
2. 스킬 파일을 읽고 지시를 그대로 따름
3. 스킬이 서브에이전트 위임을 지시하면 실제로 Agent tool로 실행
4. 각 단계에서 검증 항목 확인
5. 서브에이전트의 토큰/시간을 task notification에서 수집

## 유효 범위

**검증 가능**: 위임 트리거 인식, 오케스트레이션 사이클, 서브에이전트 간 산출물 전달, 커밋 순서

**검증 불가**: before/after 정량 비교 (병렬 A/B 불가), 처음 보는 AI의 반응 (메인이 스킬 내용을 이미 앎)

## 사례: workflow 서브에이전트 위임 구조 eval

### 1단계: 위임 트리거 (skill-creator 표준 방식 병행)

서브에이전트에서 스킬을 읽고 "어떻게 하겠다"고 설명하게 하여, 위임 패턴을 인식하는지 테스트. 실제 위임은 불가하지만 인식 여부는 확인 가능.

- 6개 시나리오 × with/without skill = 12 runs
- 결과: 위임 인식 6/6 (100%)
- 발견: 스킬 문서 모호성 3건 (병렬/순차, 코드 접근 경계, assertion 설계 오류)

**주의**: without_skill baseline 에이전트가 working directory의 스킬 파일을 자체 발견하여 오염됨 (5/6). worktree 격리 필요.

### 2단계: E2E (메인 직접 실행)

> ⚠️ 아래 결과는 이전 구조(implementation 서브에이전트 사용) 기준. 현재 step-5는 메인 에이전트가 직접 구현하고, 서브에이전트는 리뷰만 담당.

메인 에이전트가 가상 프로젝트(sum/multiply)에서 step-5 오케스트레이션을 직접 수행.

- implementation 서브에이전트 → 코드 작성 (현재는 메인이 직접 수행)
- review 서브에이전트 → diff 리뷰
- 2 phase 사이클 완료, 커밋 순서 확인

**한계**: toy 프로젝트라 복잡도 부족. 컨벤션별 N개 리뷰 병렬은 미검증.

### 3단계: 세션 분리 인식

"Step 4 끝났습니다" → IMPLEMENTATION_SESSION 전환 안내 여부. skill-creator 표준으로 검증 가능. PASS.

### 성과

eval 과정에서 스킬 문서 개선 6건 반영:
1. N개 서브에이전트 인스턴스 병렬 실행 규칙
2. step-5 리뷰 서브에이전트 병렬 명시
3. step-6 병렬 + Gap Analysis 코드 접근 경계
4. 세션 전환 안내 구체적 문구
