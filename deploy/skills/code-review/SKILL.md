# Code Review

코딩 스탠다드 기반으로 코드 품질을 검증하고 리뷰 보고서를 작성합니다.

---

## 준비

아래 두 항목을 사용자에게 **반드시 확인**받은 뒤 검증을 시작합니다. 인자에서 유추 가능해도 스킵하지 않습니다.

1. **리뷰 모드** — 타인 코드 리뷰인가, 본인 코드 점검인가?
2. **리뷰 범위** — PR 전체 / 특정 커밋 / 특정 파일?

확인 후 [coding-standards/README.md](../../contexts/coding-standards/README.md)에서 리뷰 대상에 관련된 컨벤션을 찾아 로드합니다.

### 타인 코드 리뷰 시 리뷰 범위

- **리뷰 대상은 해당 작성자가 실제로 수정한 코드에 한정합니다.**
- `git diff`로 변경된 라인만 식별하고, 그 범위 내에서만 검증을 수행합니다.
- 변경 라인의 전후 맥락(주변 코드)은 이해를 위해 읽되, 기존 코드의 문제를 리뷰 항목에 포함하지 않습니다.

---

## 검증 수행 항목

1. **테스트 케이스 도출**:
   - Happy Path, Edge Case 시나리오 리스트업
   - 사용자가 놓칠 수 있는 예외 상황 포함

2. **코드 논리 점검 (Static Analysis)**:
   - 예측 가능한 에러 (Null Reference, 비동기 경쟁 상태 등)
   - 구체적인 위험 요소 지적
   - **[CRITICAL]** 별도 요청이 없어도 항상 비판적으로 검토해야 합니다.
   - **실제 문제(Critical, Minor)만 기록**합니다. 단순 정보 전달(Info)이나 "문제 없음" 항목은 적지 마세요.

---

## 산출물 예시

```markdown
## 1. 코드 논리 점검 (잠재적 위험)
- **[Critical]** `userData`가 `null`일 경우 크래시 가능성. (`UserProfile.tsx:45`)
  - 옵셔널 체이닝(`?.`) 추가 필요
- **[Minor]** API 실패 시 에러 처리 누락. 무한 로딩 가능성

## 2. 도출된 테스트 케이스

### 필수 검증
- [ ] **[결제]** 잔액 부족 시 "잔액 부족" 알림 노출
- [ ] **[결제]** 결제 완료 후 뒤로가기 시 중복 결제 방지

### 엣지 케이스
- [ ] **[입력]** 수량 음수/0 입력 시 방어 로직 작동
- [ ] **[네트워크]** 요청 중 연결 끊김 시 앱 멈추지 않음
```
