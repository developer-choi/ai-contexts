# Code Review

코드 변경사항을 검증하고 이슈 목록을 산출하는 스킬. **수정은 하지 않는다** — 이슈 목록만 반환하고, 수정 여부와 방법은 호출자가 결정한다.

## 입력

| 입력 | 설명 |
|------|------|
| **context** | 리뷰 대상 (PR diff, 커밋, 파일) + 컨벤션 |
| **mode** | 내 코드 / 남의 코드 |

- **독립 호출 시**: 사용자에게 리뷰 대상과 모드를 확인받는다
- **step-6에서 호출 시**: step-6이 context(컨벤션 포함)와 mode를 전달한다. 1~2단계(컨벤션 로드, 리뷰 대상 파악)를 건너뛴다.

## 출력

이슈 목록. severity (Critical / Minor / Suggestion) 포함.

---

## 절차

### 1. 컨벤션 로드

[coding-standards/map.md](../../contexts/coding-standards/map.md)에서 리뷰 대상과 관련된 컨벤션을 확인하고 로드한다.

### 2. 리뷰 대상 파악

- PR 본문(description)과 커밋 히스토리를 읽고 변경 의도를 파악한다
- diff를 읽는다

**남의 코드 모드 추가 절차:**

- 이미 달린 코멘트를 읽고, 중복 지적을 피한다
- 리뷰 대상은 해당 작성자가 실제로 수정한 코드에 한정한다. 기존 코드의 문제는 리뷰 항목에 포함하지 않는다.

### 3. 리뷰 수행

코딩 스탠다드 기반으로 검증하되, 스탠다드에 없는 문제도 전문가 판단으로 짚는다.

#### 내 코드 모드

에이전트를 spawn하여 병렬 리뷰:

- `convention-reviewer` ×N (sonnet) — 컨벤션 영역별 분할
- `advanced-reviewer` (opus) — coding standard 판단 + 자유 리뷰 (구현 의도·컨벤션 없이 코드만 보는 외부 리뷰어 시점)

Lead가 결과를 종합한다 (중복 제거, 이상한 지적은 사용자에게 확인).

#### 남의 코드 모드

단일 리뷰. 컨벤션 기반 검증 + 자유 리뷰를 함께 수행한다.

### 4. 이슈 목록 산출

발견한 이슈를 severity별로 정리하여 반환한다.

- **Critical**: 버그, 보안, 로직 오류
- **Minor**: 컨벤션 위반, 가독성
- **Suggestion**: 개선 제안

컨벤션 기반 지적에는 근거가 되는 컨벤션 파일 경로와 라인번호를 함께 명시한다 (예: `coding-standards/naming.md:12 — camelCase 규칙 위반`).

---

## 후속 연결 (사용자 판단)

이슈 목록 산출 후, 다음 단계는 호출 맥락에 따라 다르다:

- **step-6에서 호출**: step-6이 Implementer에게 수정 지시 → 재리뷰 루프
- **남의 PR 독립 호출**: 사용자가 코멘트할 항목을 선택하면 `/pr-comment-write`로 넘긴다
