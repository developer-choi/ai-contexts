# 채용 과제 제출 마무리

## 목적

채용 과제 제출을 마무리한다 — 커밋 정리, PR 작성, README, 배포 안내까지.

`/plan/`에서 과제 요구사항과 각 PR의 pr-body를 찾아 참조한다.

PR 분할 안내·최종 제출 QA체크리스트·어필·회고 양식은 완성본 라이브러리의 `meta-submission` 주제에 재사용 자산으로 모은다 — 있으면 [pr-body/](pr-body/)에서 글롭해 복사 기점으로 삼는다.

## 1. 커밋 정리 — 사용자에게 안내

메시지 최종화·오배치 커밋 재배치·슬라이스 재정렬은 진입 전 최종화 세션이 이미 끝냈다 — 여기서 다시 하지 않는다. 본 절은 제출 리뷰만 남긴다.

- 커밋 히스토리를 코드리뷰 관점으로 훑어 이상 유무 확인 (커밋 재배치·메시지 재작성은 하지 않는다). `/plan/`은 전역 gitignore라 애초에 커밋되지 않으므로 산출물 커밋을 삭제하는 단계는 없다.

## 2. 제출 직전 수정 점검 — AI + 사용자

커밋 정리 과정에서 마지막에 수정한 내용이 있으면, 제출 전에 반드시 점검합니다.

1. PR 순서대로 커밋 목록을 나열하고, 커밋 정리 중 추가/수정한 커밋을 표시한다
2. 사용자에게 확인한다:
   > "커밋 정리 중 코드를 수정한 부분이 있나요?"
3. 수정이 있으면 AI가 해당 변경의 영향 범위를 분석하여 보고한다:
   - 수정된 파일을 참조하는 다른 파일이 있는지
   - 기존 동작이 변경될 가능성이 있는지
4. 영향 범위가 크다고 판단되면 해당 부분을 다시 테스트하도록 안내한다

## 3. PR 작성 — 사용자에게 안내

- `/plan/`의 pr-body를 기반으로 GitHub PR 작성

## 3-1. PR 본문 검증 — AI 작업

사용자가 GitHub PR URL을 제공하면, `gh pr view`로 실제 올라간 본문을 읽고 검증한다.

- 원본 프로젝트 용어 잔재 (상품, product, 과제 등)
- 커밋 링크가 올바른 레포/커밋을 가리키는지
- 이미지가 깨지지 않는지 (URL 유효성)
- TODO, 플레이스홀더가 남아있지 않은지
- 수치 불일치 (제목의 수치와 본문의 수치가 다른 경우)
- 이미지 플레이스홀더가 남아있지 않은지 (`[좌측 이미지]`, `[스크린샷]` 등 캡처 전 임시 텍스트)

## 4. README.md / MAIL.md 작성 — AI 작업

`/plan/`에서 과제 요구사항과 각 PR의 pr-body를 모두 읽고 다음 문서를 작성하여 사용자에게 제시합니다.

1. **README.md** — 과제 요구사항에 맞는 프로젝트 소개 문서
2. **MAIL.md** — 제출 시 보낼 메일 본문 초안
   - `.env.local` 등 환경변수 파일을 메일에 첨부하도록 안내 포함

**README.md 공통 스펙:**

1. 개요
2. 배포 링크
3. 실행 방법
4. 폴더 구조
5. PR 링크별 요약
6. 과제 요구사항에 따라 추가 섹션 (기술 선택 근거, 완성/미완성 등)

## 4-1. 산출물 문서 채용담당자 시점 리뷰 — AI 작업

### 리뷰 대상

- **README.md** — 외부 채용 과제 프로젝트의 저장소 루트
- **GitHub PR 본문** — `gh pr view {PR_URL}`로 fetch한 본문 (PR 분할이면 각 PR 본문)

### 정제 절차 (리뷰 전)

- PR 본문에서 `docs/pr*.md` / `docs/*.md` 패턴 grep 후 삭제 — 채용담당자 시점에 내부 문서 링크 노출은 노이즈
- README의 이미지 참조는 유지 (위 「1. 커밋 정리」 룰과 정합 — `plan/` 내 이미지 참조는 보존)

### 별도 에이전트로 분리하는 이유

채용담당자 시점 리뷰는 **프로젝트 맥락 없이** 읽는 에이전트가 수행한다. 이 에이전트에는 과제 요구사항, 내부 계획, 구현 맥락을 일절 전달하지 않는다.

**왜**: 채용담당자는 코드를 보기 전에 문서를 먼저 본다. 문서가 한 번에 읽히지 않으면 코드까지 도달하지 않고 탈락시킬 수 있다.

### 에이전트 프롬프트 핵심

- "당신은 채용담당자입니다. 이 지원자의 README.md와 PR 본문을 처음 보는 사람의 시선으로 위에서부터 읽어주세요."
- 각 문서에 대해 다음 4가지 관점 보고:
  - **첫인상** — 문서를 처음 본 사람이 느끼는 흐름
  - **한 번에 이해되지 않는 문장** — 재독·문맥 추론이 필요한 표현
  - **빠져있는 정보** — 채용담당자가 기대하는데 빠진 항목
  - **불필요하게 긴 부분** — 압축 가능한 영역
- **수정 없이 리뷰 결과만 보고**

### 3-1 vs 4-1 책임 분리

- **3-1 (메인 에이전트, 사실 검증 관점)** — 원본 프로젝트 용어 잔재, 커밋 링크 유효성, 이미지 URL, TODO/플레이스홀더 잔존, 수치 불일치. 위 「3-1. PR 본문 검증」 참조
- **4-1 (별도 에이전트, 첫인상 관점)** — 위 4가지 관점. 프로젝트 맥락 차단

사용자가 리뷰 결과를 확인하고, 필요한 수정은 직접 판단한다.

## 5. 제출 절차 — AI + 사용자

`/plan/background/persistent/`의 과제 요구사항과 메일 원본을 읽고, 해당 과제의 제출 방식에 맞는 체크리스트를 생성하여 사용자에게 안내한다.

제출 방식은 과제마다 다르다. 아래는 자주 등장하는 사례:

| 사례 | 절차 |
|------|------|
| zip 제출 | node_modules 등 제외하고 압축, 파일명 규칙 확인, 메일 발송 |
| GitHub 제출 | Private 확인, .env.local 미포함 확인, collaborator 초대, 배포 |
| 혼합 | 위 사항 조합 |

공통 점검:
- 빌드(`npm run build` 등) 실행하여 에러 없이 완료되는지 확인
- `.env.local` 등 환경변수 파일이 제출물에 포함되지 않는지 확인
- `plan/` 등 AI 산출물이 제출물에 포함되지 않는지 확인
- README.md의 이미지·링크가 정상 동작하는지 확인
- 제출 메일 본문 초안(MAIL.md) 작성

## 6. 완성본 라이브러리 갱신 (AI + 사용자)

제출까지 끝난 뒤, 이번 과제 각 PR의 최종 본문을 주제별 완성본 라이브러리([pr-body/](pr-body/))에 반영한다. **최종 본문은 GitHub 게시본**(`gh pr view <N> --json body`)이다 — 로컬 `pr-body.md`는 게시 시점 이전 판일 수 있다.

**PR 단위가 아니라 재료(섹션) 단위로 분해해 분류한다.** 한 PR 본문은 여러 `#` 섹션으로 나뉘고 섹션마다 다른 주제(재료)를 담으므로, PR 하나가 완성본 재료 여러 개를 기여할 수 있다. 각 PR 본문을 섹션별 재료로 갈라, 재료마다:

- **주제 식별** — 그 재료(섹션)의 성격으로 슬러그를 정하고 `pr-body/`를 글롭해 해당 완성본을 찾는다.
- **없으면 신규** — 이 재료를 기점으로 그 주제 완성본을 새로 만든다.
- **있고 이번이 더 나으면 교체** — 기존 완성본보다 나은 부분(더 정확한 근거·트레이드오프, 더 나은 섹션 구조·표현)만 골라 그 부분을 갈아끼운다. 나머지는 두고, 새 벌을 더하지 않는다(주제당 한 벌).

- **자족 인라이닝** — 제출 레포는 삭제되므로 완성본에 제출 레포 PR·커밋 링크를 남기지 않고, 근거·인용의 핵심 내용을 파일 안에 직접 담는다. 레포가 사라져도 재료가 홀로 살아 있어야 한다.
- AI가 주제별 변경(신규/교체) 초안을 diff로 제시하고, 반영은 사용자가 판단한다 — 라이브러리는 사용자 자산이라 임의로 덮어쓰지 않는다.
- "더 나은가"는 억지 교체를 피한다 — 명백한 개선일 때만. 애매하면 유지.

## 7. 자산 회수 (AI + 사용자)

제출까지 끝난 뒤, 완성본 라이브러리 갱신(6번) 외의 자산 — 원본 자료, 인터뷰 준비 자료, 미병합 브랜치, 이번 과제 패턴, 세션 대화 기록 — 을 회수한다. 제출 레포는 삭제해도 무방하도록 여기서 전부 별도 보존한다.

### 7-1. 아카이브 레포 생성 + 원본 자산 이설

로컬 작업 디렉토리를 복사해 제출 레포(collaborator 초대된 실제 제출용)와 별개의 **영구 보존용 private 레포**를 만들어 push한다.

- **전제**: 이 복사는 로컬 저장소에 더 이상 커밋이 추가되지 않는 시점(3-1·4-1 수정 반영이 끝난 뒤)에 한다. 복사 이후 로컬에 추가 커밋이 생기면 아카이브가 그 시점을 놓친다 — 순서(1~6번을 먼저 다 끝낸 뒤 7번)를 지키면 자연히 충족된다.
- **이름 규칙**: `recruitment-{slug}-{실행일:YYYYMMDD}` (예: `recruitment-wisebirds-20260505`). `{slug}`는 회사명·과제명 중 아무거나.
- **복사 범위**: `.git/`을 포함해 전체 복사한다 (미병합 로컬 브랜치가 `.git/refs`에만 존재하므로 브랜치 보존을 위해 필수). `node_modules/`·`.next/`(그 외 빌드 산출물) 등 git 미추적 재생성 가능 디렉토리만 제외.
- **워크트리 메타데이터 정리**: 복사본에서 `.git/worktrees/`를 지우고 `git worktree prune`을 돌린다. `.git`을 통째로 복사하면 이 메타데이터가 원본 워크트리 경로를 계속 가리켜, 복사본이 단독 저장소로 동작하지 않는다(그 브랜치들이 다른 워크트리에 체크아웃된 것으로 취급된다).
- **remote 교체 — 반드시 이 순서로**:
  1. 복사본에서 기존 `origin`(제출 레포를 가리킴)을 **먼저 제거**한다.
  2. 새 아카이브 레포를 만들고 `origin`으로 등록한다.
  3. **여기서 멈추고 `git remote -v` 결과를 사용자에게 보여준다** — origin이 제출 레포가 아니라 새 아카이브 레포를 가리키는지 확인받은 뒤에만 다음 단계(브랜치 push)로 진행한다. 로컬 브랜치가 잘못된 원격(제출 레포)으로 나가면 되돌리기 어려우므로, 이 확인 없이 push하지 않는다.
- **레포 구성**:
  - `original/` — 과제 메일·요구사항 문서·피그마 캡처 등 원본 (`/plan/background/persistent/`에서 복사)
  - 인터뷰 준비 자료 — 아래 「인터뷰 준비」
  - 병합 안 된 로컬 브랜치 전부 push (remote 확인 완료 후)
  - 제출 레포의 PR 복원 — 아래 「PR 재생성」
  - `HANDOFF.md` — 위 갈래에 안 들어가는 자유 메모
- **`/plan/` 이설**: `/plan/`은 전역 gitignore라 그대로 두면 push되지 않는다. 아카이브에 남길 산출물은 추적되는 경로로 옮겨 담아야 실제로 보존된다.
- **이설 산출물은 PR 본문과 중복 제거(SSOT=게시본)**: 옮겨 담는 산출물이 PR 본문과 같은 주제를 다루면(특히 `decisions.md`), PR 게시본이 진실 원천이므로 **본문에 이미 있는 내용은 빼고 본문에 없는 측면만 남긴다** — 안 그러면 같은 내용을 두 벌 보게 된다. 불필요한 폴더 뎁스는 평탄화한다(예: PR별 하위에 흩어진 `decisions.md`를 `decisions/pr{N}.md`로).
- **이설 산출물은 `plan` 브랜치에 — 기본 브랜치는 제출 미러 유지**: 아카이브 기본 브랜치는 제출 레포를 그대로 미러(제출된 코드·README)해야 조회 시 제출 시점이 그대로 보인다. 제출물이 아닌 계획 산출물(`archived-plan/`)은 기본 브랜치에 얹지 않고 거기서 뻗은 `plan` 브랜치에 커밋해 보존·복구한다. 기본 브랜치엔 제출 관련 변경(README·제출 레포 링크 교체 등)만 얹는다.
- **완료 판정**: 레포가 GitHub에 존재 + 위 항목이 실제로 커밋·push됐는지로 확인.

#### PR 재생성

제출 레포를 삭제하면 **PR 본문은 함께 사라진다** — 코드는 브랜치로 보존되지만 본문은 브랜치에 없다. 아카이브 레포에 같은 PR을 다시 만들어 **닫힌 상태**로 남긴다.

- **본문의 진실 원천은 GitHub 게시본이다.** 로컬 `pr-body.md`가 아니라 `gh pr view <N> --json body`로 받은 본문을 쓴다 — 게시 후 GitHub에서 직접 손본 문장이 로컬엔 없을 수 있다(로컬은 게시 시점 이전 판일 수 있다). 옮긴 뒤 원본과 대조해 일치를 확인한다.
- **번호를 맞춘다** — 새 레포는 이슈·PR 번호가 1부터 붙으므로, 원본과 같은 순서로 만들면 번호가 그대로 재현된다. 다른 이슈·PR을 먼저 만들지 않는다.
- **스택을 복원한다** — 각 PR의 커밋 범위를 기본 브랜치 히스토리에서 찾아 head 브랜치를 만들고, base는 앞 PR의 head로 잡는다. 첫 PR의 base는 작업 시작 지점(스타터 커밋)에 만든 브랜치다. 범위 매핑은 원본 PR의 커밋 목록(`gh pr view <N> --json commits`)과 메시지로 대조해 확인한다.
- **제목은 외부 독자가 읽을 서술형으로** — 원본 제목이 브랜치명에서 자동 생성된 것(`Refactor/folder` 등)이면 그대로 옮길 가치가 없다. README의 PR 요약 제목을 쓴다.
- 만든 PR은 **닫는다**. 머지하면 아카이브 기본 브랜치가 제출 시점과 달라진다. 닫아도 본문·diff·커밋은 그대로 보인다.
- **제출 레포를 가리키는 링크 교체** — README 등 아카이브 레포 안의 문서가 제출 레포 PR·커밋을 링크하고 있으면 아카이브 레포 링크로 바꾼다. 레포가 삭제되면 전부 죽는 링크다. 교체 후 잔존 0건을 grep으로 확인한다.

#### 인터뷰 준비

면접관은 제출된 PR 본문을 읽고 "왜 그렇게 했느냐"를 묻는다. 그러므로 인터뷰 준비의 **1차 소스는 PR 본문**이다 — decisions.md가 아니다.

- **1차: PR 본문(GitHub 게시본) 정독** — 각 PR 본문이 내세운 근거·트레이드오프·거부한 대안을 예상 질문("왜 X 대신 Y?", "이 판단의 리스크는?")과 짝지어 정리한다. PR 본문은 "이 근거로 이렇게 했다"를 외부 독자에게 대놓고 밝힌 글이라, 면접 질문이 여기서 그대로 나온다. 본문의 진실 원천은 게시본이다(「PR 재생성」과 동일 — 로컬 `pr-body.md`가 아니라 `gh pr view <N> --json body`).
- **2차: `decisions.md`로 보강** — PR 본문엔 지면상 못 담은 판단·기각 사유를 각 PR `persistent/decisions.md`에서 주워 1차 정리에 덧댄다.

### 7-2. best-practice 갱신

이번 과제에서 얻은 패턴·결정을 MP(monorepo-playground) `docs/best-practices/{분야}.md`(`api-query.md`·`component-ui.md`·`data-validation.md`·`form.md`·`rendering.md`·`setup.md`·`testing.md`)에 반영한다. 6번 절과 동일한 패턴(diff 제시 → 사용자 판단, 명백한 개선일 때만 교체) 재사용.

- **완료 판정**: 각 파일에 대해 diff 제시 여부 + 사용자 반영/스킵 결정이 남았는지로 확인.

### 7-3. 세션 자기 회고 호출

과제 종료는 workflow 종료와 같은 시점이므로, workflow를 타고 진행한 각 세션에 다시 들어가 `session-retro`로 자기 회고를 돌리도록 안내한다. 절차는 [session-retro/SKILL.md](../session-retro/SKILL.md). 채용 특유는 두 가지:

- 과제가 대화내역 제출을 요구하면 뽑기(단계 1)의 「제출용 정제」 모드를 켠다.
- 회고 산출물(`plan/retrospective/*.md`)을 7-1 아카이브 레포에 함께 회수한다.

- **완료 판정**: 대상 세션마다 `plan/retrospective/*.md` 존재.

### 7-4. 제출 레포 로컬 폴더 삭제 (사용자 실행 — 마지막 절차)

7-1~7-3으로 원본·산출물·회고가 아카이브 레포에 전부 보존됐음을 확인한 뒤, 제출 레포 로컬 작업 폴더는 삭제해도 된다. 채용 마무리의 **마지막 절차**다.

- **AI는 실행하지 않는다** — 그 폴더(또는 그 폴더에서 만든 워크트리)를 cwd로 둔 세션·프로세스가 파일을 잡고 있어 삭제가 막힌다. 사용자가 **해당 세션들을 모두 닫은 뒤** 직접 지운다.
- 삭제 전 아카이브가 GitHub에 실제로 push됐는지(7-1 완료 판정) 재확인한다 — 로컬 폴더가 유일본이 아님을 확인한 뒤 지운다.
