# AI가 작성한 코드를 사람이 리뷰하는 양을 줄이는 방법

## AI에게 코드를 작성하게 하면 발생하는 문제

이번 과제는 AI를 써서 진행했습니다. 그런데 AI가 코드를 많이 뽑아낼수록, 그게 제가 원하는 형태로 나왔는지 사람이 확인해야 할 양도 같이 늘어났습니다.

### 커밋 메시지

과제 하나에도 커밋을 수십 개 쌓게 됩니다. 제가 원하는 포맷은 `feat(reservation): 예약 리스트 페이지 구현` 같은 형태였는데, 이걸 강제하지 않으면 AI가 작성한 커밋 메시지를 매번 열어보고 고쳐야 했습니다.

### 코드 패턴 (정적 분석)

AI가 짠 코드에는 미사용 변수나 위험한 패턴처럼, 리뷰에서 늘 똑같이 지적하게 되는 것들이 섞여 있습니다. 룰로 막아두지 않으면 이 반복 지적을 사람이 매번 손으로 하게 됩니다.

### 포맷 (들여쓰기·따옴표 등)

제가 원하는 포맷이 있는데, AI가 뽑은 코드가 거기 맞는지 하나하나 확인하는 건 여간 번거로운 게 아닙니다. 게다가 포맷 차이가 diff에 섞이면, 진짜 로직 변경이 뭔지가 리뷰에서 흐려집니다.

## 해결 방법

세 가지 모두 공통점이 있습니다. **"AI야 이렇게 해줘"라고 부탁하는 것만으론 강제되지 않고, 결국 사람이 매번 확인하게 된다**는 것입니다.

그래서 문제를 **사람이 리뷰할 양 자체를 줄이자**로 잡았습니다. 무언가를 막는 데엔 비용 순서가 있고, 같은 문제라도 더 저렴한(=더 이른) 계층에서 막을수록 사람 손이 덜 갑니다.

```
사람 리뷰                      (제일 비쌈: 발견 시점이 가장 늦음)
  ↓
AI 검증 (리뷰어 에이전트)
  ↓
Claude hook
  ↓
테스트 코드
  ↓
lint · 정적 분석 · 포맷        (제일 저렴: 커밋 시점에 자동 차단)
```

## 트레이드오프

이런 설정도 공짜가 아닙니다. 버전이 올라가고 룰을 조정하는 등 그 자체로 유지보수 비용이 듭니다.

그럼에도 넣기로 한 근거는 **두 비용이 늘어나는 방식이 다르기 때문**입니다.

- 사람이 매번 확인하는 비용은 **코드 양을 따라 계속 늘어납니다.** 커밋이 100개면 100번 확인해야 합니다.
- 설정을 유지하는 비용은 **코드 양이 늘어도 거의 늘지 않습니다.** 룰 개수나 버전에 붙지, 커밋을 몇 개 짜느냐와는 무관합니다.

그래서 코드가 쌓일수록, 이 설정은 유지비를 치르고도 이득인 선택이 됩니다.
