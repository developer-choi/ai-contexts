# 주석

## 문제

`src/app/api/customers/[id]/route.ts` 같은 라우트 핸들러 위에 이런 주석이 붙어 있습니다.

```ts
/** GET /api/customers/{id} */
export function GET(_request: Request, { params }: { params: { id: string } }) {
```

이 주석이 말하는 정보는 이미 코드에 다 있습니다.

Next.js App Router에선 파일 경로(`app/api/customers/[id]/route.ts`)가 곧 라우트고, export한 함수명(`GET`)이 곧 메서드입니다. 주석은 이 둘을 다시 타이핑한 것뿐입니다.

게다가 라우트가 바뀌면 주석도 같이 고쳐야 하는데, 코드만 고치고 주석은 방치되기 쉽습니다. 그러면 주석이 지금 코드가 아니라 예전 코드를 말하게 되고, 읽는 사람은 둘 중 뭘 믿을지 따져야 합니다.

## 해결

코드에서 이미 파생되는 정보(what/how)를 설명하는 주석은 삭제했습니다.

기준은 "코드에서 파생 가능한가"입니다. 읽으면 바로 아는 내용(what/how)은 코드 자체가 문서 역할을 하게 두고, 코드만 봐선 알 수 없는 의도나 제약(why)이 있을 때만 주석을 남깁니다.

## 트레이드오프

주석을 지우면 여러 `route.ts`를 훑을 때 파일을 열기 전엔 어떤 라우트인지 안 보입니다. 다만 Next.js는 파일 경로가 곧 라우트라 여는 비용이 작습니다.

반면 주석을 유지하면 코드가 바뀔 때마다 같이 고쳐야 하고, 안 고치면 거짓 정보가 남습니다. 파일 한 번 여는 불편보다 코드·주석이 어긋날 위험을 없애는 쪽이 낫다고 판단했습니다.
