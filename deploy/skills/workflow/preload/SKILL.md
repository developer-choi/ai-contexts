# Preload

현재 `workflow`를 읽고, 여러 AI 세션에 미리 읽혀둘 preload 문서를 `outputs/` 아래에 한 번 생성한다.

핵심은 경로 수집이 아니라 구조 복원이다.

- 세션 종류 식별
- 문서에 이미 있는 분류 축 복원
- 세션별 preload 파일 선정
- 파일명 유추

## 원칙

- 기존 축을 먼저 복원한다. 예: 세션 종류, 고정 PR, 마지막 PR, 작업 유형
- 문서에 없는 새 축은 바로 쓰지 말고 사용자에게 제안만 한다
- `bg`, `plan`, `impl`, `write` 같은 라벨을 하드코딩하지 않는다
- 공통 문서만으로 부족할 때만 extras를 만든다
- 기존 `outputs`가 있어도 그대로 복제하지 않는다

## 읽을 범위

최소한 아래를 읽는다.

1. `workflow/SKILL.md`
2. `workflow/steps/*.md`
3. step 문서가 직접 참조하는 스킬/가이드
4. preload 설계에 영향을 주는 관련 맵/가이드

예:
- `requirement-review/recruitment/*`
- `recruitment/SKILL.md`
- `deploy/contexts/coding-standards/map.md`
- MP 같은 외부 패턴 맵

읽을 수 있는 문서와 실제 preload해야 하는 문서를 구분한다.

## 작업 흐름

### 1. 구조 해석

`workflow/SKILL.md`와 `steps/*.md`를 읽고 아래를 정리한다.

- 세션 경계
- step 묶음
- 진입/전환 규칙
- 조건부 경로

### 2. 축 식별

아래 우선순위로 분류 축을 잡는다.

1. 세션 종류
2. 문서에 명시된 고정 작업 단위
3. 마지막 단계 같은 특수 조건
4. 그 외 문서에 드러난 축

새 축이 필요해 보이면 후보로만 두고 사용자에게 묻는다.

### 3. 세션별 preload 후보 추출

각 세션마다 나눈다.

- 반드시 읽어야 하는 공통 파일
- 조건부 파일
- 특정 PR/작업유형 전용 부록 후보

판단 기준:
- 세션 시작 직후 판단 품질에 직접 영향이 있는가
- 세션 안에서 반복 탐색 가능성이 큰가
- 미리 읽어두면 시간/토큰 절감 효과가 큰가

### 4. extras 추출

고정 PR이나 특수 작업 단위가 있으면 별도 extras를 만든다.

extras에는 아래를 적는다.

- 왜 필요한지
- 어떤 세션에서 공통 문서와 함께 읽는지
- 어떤 MP/AC/참조 문서를 추가 preload할지

### 5. 파일명 유도

`outputs/*.md` 파일명은 아래 순서로 유도한다.

1. 작업 컨텍스트 접두사
2. 세션 또는 단위 라벨
3. 필요 시 특수 suffix

파일명은 세션 이름, step 범위, 세션 책임, 특수 조건에서 유도한다.

### 6. 결과 작성

기본적으로 아래 종류를 만든다.

- 전체 운영 가이드 1개
- 세션 종류별 preload 문서
- 필요 시 고정 PR/특수 단위별 extras

문서는 목적, 적용 대상, preload할 파일이 바로 보이게 작성한다.

## 금지

- 기존 `outputs`를 그대로 복제하기
- 새 분류 축을 사용자 확인 없이 추가하기
- 모든 참조 문서를 preload 목록에 넣기
- 생성 후 원본 `workflow`와 자동 동기화가 필요한 것처럼 다루기
