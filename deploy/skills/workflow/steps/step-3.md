## 신규 작업 모드

### 스킬 로드

아래 조건에 해당하는 스킬을 **모두** 로드합니다. 여러 개면 위에서부터 순서대로 하나씩 실행합니다.

| 순서 | 조건 | 스킬 | 실행 | 설명 |
|:---:|------|------|------|------|
| 1 | 버그/성능 문제 포함 | [bug-investigation](../bug-investigation/SKILL.md) | 서브에이전트 | 증상과 Root Cause 분석. 산출물: `/plan/background/bug-analysis.md` |
| 2 | 채용과제 | [prefer-packages](../requirement-review/recruitment/prefer-packages.md) | — | 선호 패키지 참조 |

### 컨벤션 사전 참조

overview.md 작성 전에 아래를 읽고 기술 전략에 반영한다:

- [coding-standards/map.md](../../../contexts/coding-standards/map.md)에서 현재 작업과 관련된 규칙 확인
- 사용자에게 프로젝트별 컨벤션(회사 컨벤션 등)이 있는지 확인하고, 있으면 함께 참조

여기서 파악한 컨벤션은 `overview.md` 하단 「컨벤션·컨텍스트」 섹션에 통합 기술한다 (Step 4의 추가 컨텍스트 수집 결과도 같은 섹션에 누적). Step 4의 "구현 컨텍스트 수집"이 이 섹션을 이어 사용한다.

### overview.md 생성 + 기술 전략 수립

`/plan/pr{N}/` 하위와 `/plan/background/`를 탐색하여 기존 산출물을 읽고, overview.md를 작성한다. 이 단계에서는 읽기만 하며, 원본 산출물을 삭제하지 않는다 (소비는 Step 4에서).

이 단계는 "무엇을 구현할지"를 결정한다. "어떻게 구현할지"는 Step 4에서 다루므로, 여기서는 기술 선택의 근거에만 집중한다.

### 산출물: `/plan/pr{N}/overview.md`

PR별로 아래 항목을 포함한다. **의도 수준만 기술**한다 — 상세 스펙·구체적 기술 키워드(라이브러리명, px값, 토큰명 등)·코드 블록(zod 스키마 본문, JSX, SCSS, 함수 시그니처 외 본문)은 넣지 않는다.

- 참고 자료
- 이 PR의 목표
- 기술 선택과 근거 (선택지 비교 + 결정)
- 범위 요약 (뭘 만드는지의 경계)
- **열려있는 질문** — 본 PR 외부 의존성·다음 PR 이연 항목 (백엔드 합의·디자인 검수·인프라 PR 등). 코드 안 `TODO` 마커가 아닌 본 절에서 영구 관리

참고 자료의 TODO 중, 다른 참고 자료에서 이미 결정된 사항은 해소한다. 결정 근거가 없는 TODO는 미결로 남기고, 판단에 필요한 선택지와 트레이드오프를 기술한다.

**산출물 분담 (overview vs decisions vs markup/logic)**:

| 산출물 | 책임 | 코드 블록 |
|---|---|---|
| `overview.md` | 결정·근거·트레이드오프 (의도 수준) | 짧은 타입 시그니처·인터페이스만. 본문·구현 코드 금지 |
| `decisions.md` | 의사결정 흐름 (사용자 발화 단계 + 거부/채택 사유) | 사용자 발화 인용은 그대로. 코드는 시그니처 수준만 |
| `markup.md` | 마크업 구조·토큰 매핑·디자인 의도 | 짧은 SCSS·JSX 예시 OK |
| `logic.md` | 분기 설계·표시값 매핑·외부 의존성 | 짧은 분기 의사코드 OK |
| `implementation.md` | 구현 순서·커밋 분할·회귀 체크리스트 | 시그니처·함수명 OK |

overview.md가 비대해지면 (예: 40KB 이상) 코드 블록을 markup.md / logic.md / decisions.md로 분배. 결정·근거만 overview.md에 남기고, "이 결정의 사용자 발화 흐름은 decisions.md X 절 참조", "마크업 상세는 markup.md X 절 참조" 형태로 cross-reference만.

「열려있는 질문」 절의 역할:
- 본 PR 안에서 해소 안 되는 외부 의존성 (백엔드 응답 형상 확정, 디자인 검수 라운드 결과, 다음 PR 이연 작업 등)
- step-5에서 코드 안 `TODO` 마커는 본 PR 안에서 모두 해소되어야 하므로, PR 이연 항목은 본 절에서 영구 관리
- plan md 자체가 보존되므로 후속 PR에서 참조 가능

### [CRITICAL] Project 문서 간소화

`/plan/project.md`가 존재하면, 현재 PR의 상세 내용을 삭제하고 제목만 남긴다. TODO는 overview.md에 옮기고, 사용자에게 옮겼다고 안내한다.

### 의사결정 토론

overview.md 작성 후, 토론할 의사결정 항목을 식별하여 사용자에게 안내한다. 자동으로 토론에 진입하지 않으며, 사용자의 명시적 허가가 있을 때에만 진행한다.

- **안내 내용**: overview.md에서 도출한 토론 후보 항목 목록 + 항목별 핵심 쟁점 한 줄 요약. "토론할까요?"만 묻고 끝내지 않는다 — 사용자가 항목을 보고 진행 여부와 범위를 판단할 수 있어야 한다
- **진입 조건**: 사용자가 명시적으로 허가한 경우에만 토론 진입. 사용자가 일부 항목만 선택하면 그 범위로 진행하고, 생략을 원하면 토론 없이 다음 단계로 넘어간다
- **방식 (허가 시)**: 반대 입장 에이전트(opus)를 spawn하여, 메인 에이전트가 overview.md의 기술 선택을 방어한다
- **반대 에이전트 행동 규칙**: [/discussion 원칙](../discussion/SKILL.md) 적용 — 정확성 우선, 모호한 근거 수용 금지
- **종료 조건**: 허가된 항목에 대해 도전이 완료되면 종료

### 산출물: `/plan/pr{N}/decisions.md`

**작성 대상**:

- **토론 결과** (의사결정 토론 진입한 경우): [/discussion의 토론 결과 양식](../discussion/SKILL.md#토론-결과-정리)을 따른다
- **사용자 명시 결정** (토론 미진입이어도 작성): 사용자가 LLM 초안을 거부·수정·확정한 명시 결정이 있으면 decisions.md에 별도 기록

**overview.md와의 분담**:

- **overview.md**: 결정의 본문·근거·컨벤션 인용 (최종 결정만)
- **decisions.md**: 의사결정 흐름 자체 (LLM 초안 → 사용자 발화 → 거부/수정 사유 → 최종 채택안). 사용자 발화 인용은 그대로

**작성 양식 (사용자 명시 결정)**:

```markdown
## 결정 N: {제목}

**LLM 초안**: {초안 내용}

**사용자 발화 (1차)**: "{인용}"

**LLM 보고 (1차)**: {옵션 비교 또는 트레이드오프 제시}

**사용자 발화 (2차)**: "{인용}"

**최종 채택안**: {확정안}

**근거**: {근거}

**연결**: overview.md 결정 X, markup.md "Y 절"
```

decisions.md는 IMPL/리뷰 시점에 "이 결정이 왜 이렇게 됐는지" 추적 자료. 사용자 발화 4단계 이상 이어진 결정은 흐름 보존 가치 있음. 단발 결정은 생략 가능 (overview.md 결정 본문에 합침).

## QA/리뷰 대응 모드

PR을 올린 뒤 리뷰 피드백을 받았을 때 진입한다. [pr-comment-respond](../pr-comment-respond/SKILL.md)의 절차를 따른다.

### 코멘트 수집

`gh` CLI로 PR 코멘트를 자동 수집한다. 사용자가 직접 제공해도 된다.

### 분류 및 대응

각 코멘트를 분류하고 유형별로 대응 전략을 세운다:

- **모르는 것** — 학습 후 기록
- **실수/오류** — 원인 분석 + 재발 방지
- **의견 차이** — 양쪽 근거 정리
- **단순 수정** — 바로 수정

### 지적 축적

한 번 지적받은 것은 두 번 다시 지적받지 않도록 축적한다. 이전 코멘트 이력을 참조하여 같은 패턴의 반복을 방지한다.

### 산출물

수정이 필요한 항목을 정리하여 Step 4~6을 재실행한다.
