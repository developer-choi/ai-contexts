# Step 7: PR 본문 작성

> **이 단계의 목표: PR 본문을 작성한다**

## 준비

`/plan/`에서 이전 PR의 pr-body.md가 있으면 읽고, 섹션 구조·서술 패턴을 맞춘다. step-6 산출물도 참조한다.

## PR 본문 작성 (`/plan/pr{N}/pr-body.md`)

`/plan/pr{N}/` 하위의 모든 산출물과 커밋 로그를 기반으로 PR 본문을 작성한다. 특정 파일명(overview.md 등)을 하드코딩하지 않고, `/plan/pr{N}/`을 탐색하여 존재하는 산출물을 동적으로 참조한다. 의사결정 기록(decisions.md 등)이 있으면 PR 본문에 포함한다.

`/write-doc` 스킬로 톤·구조를 다듬는다.

## 구현 이해도 질문

PR 본문 작성이 끝난 뒤, 작성된 본문과 커밋 diff를 기반으로 사용자에게 질문한다.

- **역할**: 대기업 시니어 개발자의 코드리뷰. 잘못된 점을 짚는 게 아니라, 사용자가 본인의 구현을 정말 이해하고 작성했는지 의심가는 포인트를 질문한다.
- **예시**: "이 부분을 왜 이렇게 구현했는지?", "이 선택의 트레이드오프는?", "다른 방법 대신 이걸 선택한 이유는?"
- **목적**: 사용자가 PR을 올리기 전에 자신의 구현을 한번 더 점검하게 만든다.

## 산출물 정리

PR 본문 작성이 완료되면, `/plan/pr{N}/` 하위의 각 산출물과 PR 본문을 1:1 대조한다.

- 파일명만 보고 판단하지 않는다. 반드시 각 파일의 내용을 Read한 뒤 PR 본문/코드와 항목별로 대조한다.
- 산출물의 모든 내용이 PR 본문 또는 코드에 반영된 경우: 해당 파일 삭제
- 아직 반영되지 않은 판단·결정 내용이 남아 있는 경우: 해당 파일 유지

대조 결과(각 파일의 어떤 내용이 어디에 반영됐는지, 삭제한 파일, 유지한 파일 및 유지 이유)를 사용자에게 보고한 뒤 삭제한다.
