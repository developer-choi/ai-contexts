# Step 7: PR 본문 작성

## 준비

`/plan/`에서 이전 PR의 pr-body.md가 있으면 읽고, 섹션 구조·서술 패턴을 맞춘다.

## PR 본문 작성 (`/plan/pr{N}/consumable/pr-body.md`)

`/write-init pr-body` 스킬로 PR 본문을 작성한다. 작업 컨텍스트로 `/plan/pr{N}/` 하위의 모든 산출물과 커밋 로그를 전달한다. 특정 파일명(overview.md 등)을 하드코딩하지 않고, `/plan/pr{N}/`을 탐색하여 존재하는 산출물을 동적으로 참조한다.

write-init이 패키지 파일을 생성한 후, 사용자가 새 세션에서 `/write-refine <패키지 경로>` 호출해서 톤·구조·분량을 다듬는다.

## 구현 이해도 질문

PR 본문 작성이 끝난 뒤, 작성된 본문과 커밋 diff를 기반으로 사용자에게 질문한다.

- **역할**: 대기업 시니어 개발자의 코드리뷰. 잘못된 점을 짚는 게 아니라, 사용자가 본인의 구현을 정말 이해하고 작성했는지 의심가는 포인트를 질문한다.
- **예시**: "이 부분을 왜 이렇게 구현했는지?", "이 선택의 트레이드오프는?", "다른 방법 대신 이걸 선택한 이유는?"
- **목적**: 사용자가 PR을 올리기 전에 자신의 구현을 한번 더 점검하게 만든다.

## 산출물 정리

PR 본문 작성이 완료되면, `/plan/pr{N}/` 하위의 각 산출물을 PR 본문 및 코드와 대조한다 (대조 절차는 SKILL.md 「자가 검토 필수」 일반 규칙).

- 산출물의 모든 내용이 PR 본문 또는 코드에 반영된 경우: 해당 파일 삭제
- 아직 반영되지 않은 판단·결정 내용이 남아 있는 경우: 해당 파일 유지
- **`/plan/pr{N}/persistent/` 하위는 정리 대상에서 제외** — 영구 보존 자료. 대조도 수행하지 않는다.

대조 자체는 수행하되, 사용자에게는 삭제·유지 파일 목록(유지 시 한 줄 사유)만 보고한 뒤 삭제한다. 내용 매핑을 항목별로 줄줄이 노출하지 않는다.

## 2회차 최종 커밋 정리 (마지막 PR 한정)

본 PR이 **마지막 PR**이면, 머지 직전에 전체 PR 커밋 메시지의 `[PR{N}]` 접두사를 제거하고 메시지를 재작성한다. 마지막 PR 식별은 SKILL.md 「채용과제 마무리」의 식별 방식(`/plan/` 하위 가장 높은 번호의 `pr{N}` 디렉토리가 현재 작업 중인 PR이면 마지막)을 그대로 따른다.

본 PR이 마지막이 아니면 본 절은 건너뛴다.

절차:

- [conventions/commits.md](../conventions/commits.md) 「백업 브랜치」 적용
- 접두사 제거·메시지 재작성 양식은 [conventions/commits.md](../conventions/commits.md) 「`[PR{N}]` 접두사」 참조
- stub drop·슬라이스 분할은 각 PR의 Step 6.5에서 이미 완료되었으므로 본 단계는 메시지 정리에 집중한다
- PR 간 커밋 시간순 섞임 처리는 [conventions/commits.md](../conventions/commits.md) 「PR 간 시간순 섞임 수용」 참조
- 재작성 완료 후 [conventions/commits.md](../conventions/commits.md) 「force-push 요청 보고 양식」 적용
