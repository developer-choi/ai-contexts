# 구현↔리뷰 루프

호출자가 **[구현자 · 재료 · 진실검사 A · 규칙검사 B · 증분 단위]**를 주입하면, 본 스킬은 그 위에서 "구현 → 두 축 검증 → 0건 수렴"을 돌린다. 구현 종류(세션)에 무관하게 골격은 동일하며, 종류 차이는 주입 인자에만 담긴다.

팀 구성은 [team-agent](../../../contexts/team-agent.md), 규칙검사 리뷰 절차는 [code-review](../../code-review/SKILL.md)를 따른다.

## 입력 (호출자가 주입)

| 인자 | 의미 | 비고 |
|---|---|---|
| 구현자 | 증분을 만드는 주체 (+ 모델). 1명 이상 | 종류별로 다름 |
| 재료(context) | 무엇을·어떻게 만들지 — 계획·결정·stub·진실 원천 등 구현 입력 일체 | AI 산출물 포함 |
| **진실검사 A** | (진실 원천 핸들 · 증분 통과기준 · 종료 커버리지 기준 · 검증 메커니즘) | **필수·non-null** |
| 규칙검사 B | 적용할 코딩 스탠다드 묶음 → Reviewer 구성 | 양쪽 공통 골격, 내용물만 주입 |
| 증분 단위 | 한 번에 만들고 검사·커밋하는 변경 묶음 (리뷰 cadence) | 종류별로 다름 |

**재료 ≠ 진실검사 A.** 진실 원천 자체는 재료에도(구현 시 참조) 진실검사 A에도(검사 기준) 들어간다. 그러나 AI가 만든 산출물은 재료에만 들어가고 A의 검사 기준이 될 수 없다 — 검증 대상이 자기 산출물로 자기를 채점하는 자기증명 루프를 차단한다([SKILL.md 「검증 기준 = 진실 원천」](../SKILL.md)).

## 출력

- 워크트리에 **검증 완료된 커밋들** — 모든 증분이 A·B 0건으로 수렴한 상태.
- 호출자에게 **수렴 보고** — 증분별 라운드 결과 + 종료 커버리지 점검 결과 + 수정 사항.

산출물 파일을 만들지 않는다 (리뷰 결과 정리 산출물이 필요하면 호출자의 책임).

## 두 검증 축

### A. 진실원천 충실도 (non-null · 매 증분 게이트)

- 주입된 진실 원천을 **직접 fetch/실행**해 증분과 대조한다. 캐시·AI 산출물을 검사 기준으로 쓰지 않는다.
- 증분이 통과 기준에 미달이면 그 증분은 통과하지 못한다. **어떤 호출도 축 A 없이 종료할 수 없다** — 이 불가침성이 "검증 없이 제출"을 구조로 막는다.
- 검증 메커니즘은 진실 원천의 성질로 주입된다: 실행 오라클이 있으면 그 실행 결과로, 없으면 Reviewer ↔ 구현자 직접 대조 루프로.
- **팀 구성은 A 메커니즘의 함수다.** 오라클형(실행 결과가 곧 게이트)이면 축 A Reviewer를 spawn하지 않는다. 대조형이면 Reviewer ↔ 구현자 직접 루프를 spawn한다. 엔진은 고정 팀을 띄우지 않는다.
- 모든 증분 종료 후 주입된 **종료 커버리지 기준**으로 누락 0건을 1회 점검한다.

### B. 코딩 스탠다드 (공통 루프)

- 주입된 스탠다드로 Reviewer ×N + 자유 리뷰를 0건까지 돌린다. Coding-Standards 지적은 근거 컨벤션 파일 경로를 동반한다. Reviewer 결과는 Lead가 종합한다 — 중복 제거, sonnet 오탐 필터링, 이상 시 사용자 확인, 검증된 이슈만 구현자에게 한번에 전달.
- **축 B는 진실원천 충실도를 판정하지 않는다.** 충실도 의심을 발견하면 스스로 정답 처리하지 말고 **축 A로 넘긴다**. 검증 대상이 검증 기준을 판정하는 자기증명 차단.

## 루프

증분 단위마다:

1. 구현자가 증분을 만든다.
2. 두 축으로 검증한다 — 각 축 **0건까지** (미달 시 구현자 수정 → 재검).
   - 두 축의 순서·병렬은 주입된 **A 메커니즘의 성질**(기계 사실)로 정해진다. A가 **대조형**(실행 오라클 없음)이면 A를 먼저 0건 수렴시킨 뒤 B를 돌린다 — 원본을 오독해 어긋난 증분을 B가 헛리뷰하는 것을 막는다. A가 **오라클형**(실행이 곧 판정)이면 A·B 병렬.
3. 다음 증분으로.

모든 증분 종료 후 **종료 커버리지 점검**(축 A). 누락 발견 시 해당 증분 흐름으로 복귀.

## 우회 (호출자의 사전 점검 — 입력 아님)

다음 두 가지가 **모두 참**일 때만 호출자가 본 엔진을 생략할 수 있다. 둘 다 **기계 판정** — 런타임 의견("간단해 보임" 등) 금지:

1. 구현 전 계획에서 **진실원천 아티팩트가 선언되지 않음** (파일 부재로 검증 가능한 사실)
2. **변경 파일 전부가 프로젝트 자동 검사 도구 스코프 안** (예: lint-staged glob·tsconfig include·포맷 대상 — 스코프 매칭으로 검증 가능한 사실)

하나라도 거짓이면 엔진 강제. 의견 기반 생략은 검증 회피 사고의 직접 원인이므로 차단한다. (도구 셋업 PR처럼 스코프가 아직 일부만 잡힌 경우 ②가 거짓 → 엔진 강제. fails-safe.)
