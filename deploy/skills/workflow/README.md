# Workflow

AI와 함께 효율적으로 코드를 작성하기 위한 단계별 프로세스입니다.

- **사용법**: `/workflow [init | step-N]`

## 무슨 문제를 해결하려고 했나요?

AI와 코드를 작성할 때 "바로 구현"으로 뛰어들면 방향이 흔들리고, PR이 비대해지고, 맥락이 유실됩니다.
사전에 배경 파악 → PR 분할 → 기술 전략 → 구현 → 점검이라는 단계를 밟으면, 각 단계의 산출물이 다음 단계의 입력이 되어 일관된 품질을 유지할 수 있습니다.

## 설계 원리

### 3계층 구조: Workflow → Step → Skill

- **Workflow** — 전체 흐름과 공통 원칙 (승인 대기, 자가 검토 등)
- **Step** — 각 단계의 오케스트레이터. 작업 종류에 따라 어떤 스킬을 쓸지 판단
- **Skill** — 독립 실행 가능한 실제 작업 (code-review, spec-review 등)

### 작업 종류별 오케스트레이션

같은 step이라도 작업 종류(신규 기능, 버그 수정, 채용 과제 등)에 따라 다른 스킬이 호출됩니다. 예: step-1은 신규 기능이면 spec-review, 버그면 bug-investigation을 선택합니다.

### 산출물 체이닝

각 step의 산출물이 다음 step의 입력이 됩니다. step마다 "참고 자료"로 어떤 산출물을 읽는지 명시되어 있습니다.

### 독립 실행

스킬은 워크플로우 없이 단독으로 호출할 수 있습니다. 예: `/code-review`로 남의 코드를 바로 리뷰.

## 진행 흐름

1. **배경 파악 및 문제 정의** — 프로젝트 컨텍스트 수집
2. **PR 분할 전략 수립** — 작업 범위를 PR 단위로 분할
3. **핵심 과제 정의 및 기술 전략 수립** — PR별 목표와 접근 방식 결정
4. **구현 방침 상세화** — 세부 구현 계획
5. **구현** — 코드 작성
6. **최종 점검** — 결과물 검증
7. **PR 본문 작성** — 리뷰어를 위한 PR 설명
8. **마무리** — 워크플로우 회고 + 프로젝트 유형별 제출 마무리

Step 1~2는 초기 1회, Step 3~7은 PR마다 반복, Step 8은 모든 PR 완료 후 1회 수행합니다.

## 폴더 구조

| 경로 | 설명 |
|------|------|

| `template/` | 산출물 양식 (context-setup.md, codebase-audit.md) |
| `TODO.md` | 미완료 개선 사항 |
