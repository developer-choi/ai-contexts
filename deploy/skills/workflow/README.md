# Workflow

제가 일하는 방식을 프롬프트로 옮긴 것입니다. 기획서나 피그마 디자인, 채용 과제 같은 자료에서 출발합니다.

AI가 제 작업 순서를 그대로 밟고, 단계마다 제가 끼어들어 방향을 정하면서 마지막에 PR에 다다릅니다.

## 동기

기획서를 받았다고 바로 코드를 짜면, 배경을 잘못 이해한 채로 작업이 굴러가거나 한 PR이 너무 커져서 리뷰가 어려워집니다. 그래서 저는 먼저 배경을 파악하고, 작업을 PR 단위로 나누고, 구현 방침을 세운 뒤에 코드를 작성합니다. 구현이 끝나면 코드 품질을 점검하고, 리뷰어가 읽기 좋은 PR 본문을 씁니다.

이 순서를 프롬프트로 옮겨서 AI도 같은 방식으로 일하도록 만든 것이 이 워크플로우입니다.

또 하나의 목표는 작업을 가능한 한 병렬로 돌려 기다리는 시간을 줄이는 것입니다. 어떤 PR의 구현과 그 PR에 기대는 PR의 설계를 겹치고, 화면 마크업은 PR 작업과 동시에 진행하고, 리뷰어를 종류별로 나눠 동시에 돌립니다.

## 작업에 맞춘 조절

작업마다 어디까지 파고들지를 조절합니다. 정확성이 더 중요한 작업은 방법을 더 탐색하고 토론과 리뷰를 충분히 거치고, 속도가 더 중요한 작업은 그 과정을 줄여 빠르게 갑니다.

깊이 들어갈수록 결과물의 고점이 높아집니다. 더 나은 방법이 있으면 제가 모르던 것까지 그 자리에서 배워 적용하므로, 처음부터 제가 알던 범위를 넘어선 결과가 나올 수 있습니다.

예를 들어 어떤 문제에 해결 방법 A, B, C가 있고 제가 A와 B만 겪어 봤다고 하겠습니다. 정확성이 중요한 작업은 C까지 배워 셋 중 최선을 고르고, 속도가 중요한 작업은 이미 검증한 B로 바로 갑니다.

어느 쪽이든 선택지는 열려 있고, 그 작업이 무엇을 더 필요로 하느냐에 맞춥니다.

## 작업 종류에 맞는 스킬 호출

각 단계는 작업 종류(채용 과제, 실무, 개인 프로젝트)에 따라 다른 전문 스킬을 불러옵니다. 워크플로우 자체는 "언제 어떤 스킬을 불러올지"를 판단하는 지휘자 역할을 합니다.

작업 종류를 크게 둘로 나눕니다. 채용 과제는 빈 저장소에서 폴더 구조와 코딩 규칙부터 깔아야 하므로 준비 단계를 한 번 더 거치고, 실무는 기존 코드 위에서 시작하므로 배경 파악 직후 바로 마크업과 첫 PR 설계로 갈라집니다.

## 만들고, 검사하고, 0건까지 고치는 엔진

화면 마크업을 만드는 작업과 로직을 구현하는 작업은 겉보기엔 다르지만 골격이 같습니다. "만들고 → 정답과 대조하고 → 틀린 게 0건이 될 때까지 고친다"는 뼈대를 공유합니다. 다른 것은 무엇을 정답으로 보느냐뿐입니다. 마크업의 정답은 피그마 원본이고, 로직의 정답은 실제로 동작하는지(테스트 통과)입니다.

그래서 이 반복 과정을 하나의 엔진으로 빼고, 두 작업이 정답과 검사 기준만 바꿔 끼워 같이 씁니다.

이 엔진은 여러 AI를 한 팀으로 묶어 돌립니다. Claude Code의 [Agent Teams](https://code.claude.com/docs/en/agent-teams) 기능으로, 만드는 역할과 검사하는 역할을 나눠서 만든 사람이 직접 검사하지 않게 합니다.

- **만드는 역할**: 마크업 또는 로직을 작성하고 작업을 저장합니다.
- **정답 대조 역할**: 마크업은 피그마 원본을 직접 열어 대조하고, 로직은 테스트를 실제로 돌려 동작을 확인합니다.
- **코딩 규칙 검사 역할**: 정해 둔 코딩 규칙 문서를 직접 읽고 어긋난 곳을 찾습니다.
- **깊이 있는 품질 검사 역할**: 더 무거운 판단이 필요한 코드 품질을 살핍니다.
- **지휘 역할**: 사용자와 소통하고 검사 결과를 모읍니다.

만드는 역할이 작업을 저장하면 검사 역할이 살펴보고, 어긋난 점이 있으면 만든 역할에게 다시 돌려보냅니다. 이 주고받기를 어긋난 점이 0건이 될 때까지 반복합니다.

## 설계 근거

### 만든 사람과 검사하는 사람을 나눈 이유

코드를 작성한 쪽이 직접 검사하면, 자기가 잡은 방향에 치우쳐서 문제를 놓치기 쉽습니다. 독립된 검사자가 처음 보는 시선으로 살펴야 실제 문제를 잡아냅니다.

### 검사자를 여러 명으로 나눈 이유

먼저 정확도입니다. 지켜야 할 코딩 규칙이 많아서 한 명이 전부 동시에 보면 놓치는 게 생깁니다. 종류별로 나누면 각자 맡은 부분에만 집중할 수 있습니다.

다음은 비용입니다. 검사자를 나눈 김에 일의 성격에 맞게 모델도 다르게 배정했습니다. 기계적으로 대조하는 일은 가벼운 모델에, 깊은 판단이 필요한 검사는 무거운 모델에 맡깁니다.

### 0건까지 반복하는 이유

리뷰에서 찾은 문제를 고친 뒤 다시 확인하면, 여전히 문제가 남아 있는 경우가 있습니다. 한 번 검사로는 충분하지 않기 때문에, 문제가 0건이 나올 때까지 검사와 수정 사이클을 돌립니다.

### 검사 기준은 항상 원본

검사할 때 기준으로 삼는 것은 언제나 원본입니다. 마크업은 피그마 원본 화면, 로직은 실제 테스트 결과, 규칙은 코딩 규칙 문서 그 자체입니다.

AI가 중간에 만들어 둔 정리본을 검사 기준으로 쓰지 않습니다. 그 정리본은 만들 때 빠뜨린 부분이 있을 수 있고, 같은 빠뜨림을 검사자도 똑같이 못 보고 지나치면 문제가 그대로 남기 때문입니다. 정리본은 원본을 빠르게 찾기 위한 메모로만 쓰고, 검사는 항상 원본을 다시 펼쳐 확인합니다.

## 세션 라인업

작업을 여러 세션으로 나눠 각각 별도의 맥락에서 진행합니다. 한 세션이 끝나면 다음 세션을 시작하라는 안내가 나옵니다. 세션 이름은 화면에 그대로 보이는 코드 이름이라 옆에 일상어 설명을 붙입니다.

- **BG** (배경 파악과 PR 확정): 기획서, 피그마, 채용 원본을 읽고 무엇을 왜 해야 하는지 정리합니다. 자료를 다 읽고 한꺼번에 PR을 쪼개는 게 아니라, **읽어나가면서 "이건 더 안 봐도 정할 수 있다" 싶은 PR부터 하나씩 확정**합니다. 확정된 PR은 곧바로 출발할 수 있어서, 확실한 작업이 아직 불확실한 작업을 기다리지 않습니다. 모든 작업의 출발점입니다.
- **FOUNDATION** (채용 전용 기초 다지기): 폴더 구조와 코딩 규칙을 정리합니다. 이 자체가 하나의 PR이라 다른 PR과 똑같이 설계·구현·리뷰를 거칩니다. 빌드·린트 같은 도구 설정은 별개 PR이고, 서로 다른 브랜치라 동시에 출발할 수 있습니다.
- **MARKUP** (화면 마크업): 피그마 시안을 페이지, 구역, 작은 부품 단위로 잘라 마크업 화면을 만들고, 피그마 원본과 0건까지 대조해 검증합니다. 검증을 마친 마크업은 각 PR이 그대로 가져다 씁니다. 시간이 오래 걸리는 작업이라 따로 떼어 일찍 시작합니다.
- **PR_{N}_PLAN** (PR 설계와 가벼운 구현): 그 PR이 무엇을 하는지 정의하고 구현 방침을 정합니다. 가벼운 PR은 이 자리에서 바로 코드를 작성해 저장까지 끝냅니다. 무거워서 한 세션에 다 못 끝내거나 **이 PR에 기대는 다른 PR**이 먼저 그 모양을 알아야 하는 부분만 뼈대만 잡아 두고, 속을 채우는 일은 다음 구현 세션에 넘깁니다. 이렇게 기댈 자리만 미리 잡아 두면, 기대는 쪽 PR의 설계와 이번 PR의 구현이 동시에 굴러갑니다. 기대는 PR이 바로 다음 번호일 필요는 없고, 여럿일 수도 있습니다.
- **PR_{N}_IMPL** (무거운 구현 이어받기): 설계에서 뼈대만 잡아 넘긴 무거운 작업의 속을 채웁니다. 가벼운 PR은 앞 설계 세션에서 이미 끝나므로, 이 세션은 넘어온 무거운 작업만 이어받아 로직에 집중합니다. 화면 마크업은 MARKUP에서 검증한 것을 그대로 가져옵니다.
- **WRITING_IDEATOR / WRITING_REFINER** (PR 본문 작성): 리뷰어가 읽기 좋은 PR 본문을 씁니다. IDEATOR는 설계가 끝나면(각 PR의 step-3 후) 계획만으로 배경·문제·접근을 미리 초안하고, REFINER는 구현이 끝난 뒤(step-6 후) 실제 커밋 목록·변경 요약·테스트 계획을 채워 확정합니다. PR별 세션이 아니라 두 상시 세션이 PR 1~N 본문을 이어서 쓰며, 쓰는 시점은 자유롭습니다.
- **FINALIZE** (마무리 정리): 모든 PR의 구현이 끝난 뒤 마지막에 한 번 돌립니다. 그동안 쌓인 커밋 기록을 정리하고, 각 커밋에 붙였던 PR 번호 표시를 떼고, 엉뚱한 PR에 섞여 들어간 커밋을 제자리로 되돌린 뒤, 합치는 순서를 안내합니다. 앞 PR 위에 얹어 만든 PR은 아래에서 위로 순서대로 합치고, 따로 뻗은 PR은 순서를 지킬 필요가 없습니다.

아래는 자료를 넣어 PR을 합치기 직전까지 가는 큰 흐름입니다. 옆으로 갈라지는 가지는 동시에 진행되거나 시점이 자유로운 작업입니다.

```
BG · 배경 파악 + 확정되는 PR부터 하나씩 내보냄
  │
  ├──→ FOUNDATION · 채용 과제면 폴더·코딩 규칙 정리 (그 자체가 한 PR)
  │
  ├──→ MARKUP · 화면 마크업 (PR 작업과 동시 진행)
  │
  ▼
PR마다  설계·구현  (가벼운 PR은 설계 세션에서 바로 끝내고, 무거우면 구현 세션이 이어받음)
  │
  ├──→ PR 본문 작성
  │
  ▼
모든 PR 구현 완료
  │
  ▼
FINALIZE · 커밋 기록을 정리하고 합치는 순서를 안내
```
