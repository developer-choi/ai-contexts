# Workflow

초고수의 작업 방식을 복제한 단계별 프로세스. AI와 함께 기획서/피그마/버그/채용과제를 PR로 변환한다.

- **사용법**: `/workflow [init | step-N]`

## 무슨 문제를 해결하려고 했나요?

AI와 코드를 작성할 때 "바로 구현"으로 뛰어들면 방향이 흔들리고, PR이 비대해지고, 맥락이 유실됩니다.
사전에 배경 파악 → PR 분할 → 기술 전략 → 구현 → 점검이라는 단계를 밟으면, 각 단계의 산출물이 다음 단계의 입력이 되어 일관된 품질을 유지할 수 있습니다.

## 설계 원리

### 구조: Workflow → Step → Skill

- **Workflow** — 전체 흐름과 공통 원칙 (승인 대기, 자가 검토, 선행조건 자가체크)
- **Step** — 각 단계의 오케스트레이터. 작업 종류에 따라 어떤 스킬을 쓸지 판단
- **Skill** — step 내에서 호출되는 하위 작업 단위

### 작업 종류별 오케스트레이션

같은 step이라도 작업 종류(신규 기능, 버그 수정, 채용 과제 등)에 따라 다른 스킬이 호출됩니다.

### 산출물 체이닝

각 step의 산출물이 다음 step의 입력이 됩니다. step마다 "참고 자료"로 어떤 산출물을 읽는지 명시되어 있습니다.

### 선행조건 자가체크

각 step이 필요한 선행조건을 스스로 확인한다. step 스킵에도 자연스럽게 대응.

## 진행 흐름

1. **배경 파악 및 문제 정의** — 프로젝트 컨텍스트 수집
2. **PR 분할 전략 수립** — 작업 범위를 PR 단위로 분할
3. **과제 정의** — PR별 목표와 접근 방식 결정 (신규 작업 / QA 대응)
4. **구현 방침 상세화** — 세부 구현 계획
5. **구현** — 코드 작성 + 커밋별 컨벤션 리뷰
6. **최종 점검** — 결과물 검증 + 컨벤션별 심층 리뷰
7. **PR 본문 작성** — 리뷰어를 위한 PR 설명
8. **마무리** — 워크플로우 회고 + 프로젝트 유형별 제출 마무리

Step 1~2는 초기 1회, Step 3~7은 PR마다 반복, Step 8은 모든 PR 완료 후 1회. PR 올린 후 리뷰 피드백 받으면 Step 3(QA 모드)~6 재반복.
