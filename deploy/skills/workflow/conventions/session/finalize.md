# FINALIZE (전 PR 최종화)

모든 PR의 IMPL이 끝난 뒤 **1회**, 전 PR 커밋 히스토리를 머지 직전 상태로 정리하는 세션. step 번호 없는 세션(MARKUP과 동렬).

## 전제 — 미머지

메시지 최종화(전역 메시지 재작성)와 오배치 커밋 재배치(replace)가 성립하는 근거는 **아직 머지되지 않았다는 것**이다. 구현 페이즈는 머지 없이 도미노로 진행하므로 이 세션 진입 시점엔 어느 PR도 머지되지 않은 것이 기본이다. 커밋 소속은 접두사가 아니라 **브랜치**(각 PR 브랜치의 base..tip 범위)로 식별한다.

**브랜치 형태는 PR마다 다르다.** 앞 PR에 의존하는 PR은 그 브랜치 위에 얹혀 있고(스택), 독립인 PR은 기본 브랜치에서 각자 뻗어 있다 (base 결정은 [steps/step-4.md](../../steps/step-4.md) 「사전 준비」 — 사용자 확인 사항). 아래 절차는 두 형태를 모두 다룬다. 이미 머지된 PR이 섞여 있으면 「머지된 PR 처리」 참조.

## 진입 조건 (fan-in)

- **모든 PR의 IMPL(step-6) 완료**. IMPL 세션은 step-5(구현)~step-6(최종 점검)이라 step-6까지 끝나야 IMPL 완료다. 특정 PR 하나가 아니라 전 PR이 선행 (fan-in).
- 마지막 IMPL 판정: `/plan/` 하위 가장 높은 번호의 `pr{N}` 디렉토리가 방금 IMPL(step-6)을 끝낸 PR이면 그 IMPL이 마지막 (SKILL.md 「채용과제 마무리」의 project.md 방식과 동일). 마지막 IMPL 세션이 step-6 종료 시 본 세션 진입을 안내한다.
- WRITING(PR 본문)은 per-PR·유연 타이밍이라 본 세션의 선행이 아니다 — IMPL 직후 쓴 PR도, 아직 안 쓴 PR도 섞여 있을 수 있다. 「WRITING·머지 관계」 참조.

## cwd

다중 브랜치를 재작성하므로 단계별로 갈린다.

| 단계 | cwd |
|---|---|
| 백업 브랜치 | 각 PR 워크트리 (또는 메인 repo — 브랜치 전체가 보이는 곳) |
| replace (오배치 재배치) | 재배치 대상 PR 브랜치 워크트리부터 시작. 스택이면 하류 PR 브랜치로 연쇄 |
| 메시지 최종화 | 브랜치별로 각 PR 워크트리. 스택이면 최상단 워크트리에서 전체 히스토리가 한 번에 보인다 |

## 절차

### 백업 브랜치 [CRITICAL]

replace·메시지 최종화 둘 다 history rewriting이므로 시작 전 [commits.md](../commits.md) 「백업 브랜치」를 적용한다. **재작성 대상 브랜치마다** 백업을 뜬다 (단일 브랜치 확장 — 스택이면 오배치 재배치가 대상 PR + 그 하류 PR 브랜치를 모두 건드린다).

### replace — 오배치 커밋을 원래 PR로 재배치

"PR4에 있지만 실은 PR2 작업인 커밋"처럼 **소속이 틀린(오배치)** 커밋을 원래 PR로 되돌린다. 커밋 소속은 **브랜치 기준**(각 PR 브랜치의 base..tip 범위)으로 식별한다 — 어느 브랜치 구간에 있느냐가 소속이고, 오배치 = 그 구간에 있지만 내용상 다른 PR 작업인 커밋. 시간순 섞임(소속은 맞는데 시간만 뒤섞임)과 구분한다 — 후자는 재배치 대상이 아니다 ([commits.md](../commits.md) 「PR 간 시간순 섞임 vs 오배치」).

재배치 방법은 두 브랜치의 관계로 갈린다:

- **스택 관계(대상 PR이 상류)** — 상류에 커밋을 옮기면 그 하류 PR 브랜치들을 **연쇄 rebase**해야 한다 (다중 브랜치 rewrite).
- **독립 브랜치 관계** — 연쇄가 없으므로 rebase로 풀리지 않는다. **대상 브랜치에 cherry-pick → 원본 브랜치에서 그 커밋 drop**(rebase로 제거)의 2단 조작이다. 양쪽 브랜치 모두 재작성되므로 둘 다 백업 대상이고, cherry-pick이 충돌하면 그 커밋이 실은 원본 PR의 맥락에 얽혀 있다는 신호이므로 재배치가 옳은지 사용자와 재확인한다.

전 PR이 다 완성돼야 오배치를 되돌릴 수 있다 — 구현 페이즈 도중엔 대상 PR이 아직 열려 있어 재배치할 곳이 확정되지 않는다. 그래서 종료 페이즈 1회 조작이다.

### 메시지 최종화 — 슬라이스·도메인 재작성 + 톤 정리

전 커밋의 메시지를 슬라이스·도메인 기준으로 재작성하고, **외부 독자(채용담당자·리뷰어·팀원) 친화 톤으로 정리**한다 (작업 중 섞인 캐주얼·은어·비속어를 정중·명확한 기술 문서 톤으로 통일). net-diff 불변(코스메틱). 커밋 메시지 라이프사이클은 [commits.md](../commits.md) 「커밋 메시지 라이프사이클」, 톤 가이드는 [commits.md](../commits.md) 「커밋 메시지 톤 — 외부 독자 친화」 참조.

- 슬라이스 분할·stub drop은 각 PR의 step-6.5(1회차 정리)에서 이미 끝났으므로, 본 세션은 메시지 정리에 집중한다.
- 커밋 메시지와 함께 `project.md` 등 외부 노출 가능 메모도 톤 스캔한다.

### 순서

replace와 메시지 최종화는 동일 커밋을 다시 쓰므로 병렬 불가 — **replace 먼저(소속 교정) → 메시지 최종화(재작성 + 톤)** 순차. replace로 재배치가 끝난 히스토리 위에 메시지 최종화를 얹는다.

### force-push

replace·메시지 최종화 완료 후 [commits.md](../commits.md) 「force-push 요청 보고 양식」을 적용해 사용자에게 요청한다. 재작성된 브랜치가 여럿이면 브랜치별로 백업 이름 + 재정렬 후 커밋 목록을 보고한다.

### consumable 잔존 점검

force-push 요청 후, 전 PR의 `pr{N}/consumable/`을 훑어 잔존 산출물이 있는지 확인한다.

- 잔존 파일이 있으면 경로 목록을 사용자에게 제시하고 **"이 중 교훈으로 남길 게 있는지"** 묻는다.
- 없다고 답하면 그 자리에서 삭제한다. 있다고 답한 파일은 어디로 옮길지(예: `persistent/`로 승격, KA 백로그로 이관) 사용자와 정한다.
- 전수 자동 삭제는 하지 않는다 — 항상 사용자 확인을 거친다.

## WRITING·머지 관계

- **WRITING을 replace 전에 쓴 PR**: replace로 그 PR 커밋 구성이 바뀌면 해당 PR 본문만 사후에 손본다 (하드 게이트 아님 — 사용자 판단. 커밋 링크·수치가 어긋났는지 확인).
- **머지**: 본 세션(replace + 메시지 최종화) 종료 후가 문서 기본 흐름이다. **스택으로 얹힌 PR은 바텀업(아래→위)으로 머지**하고, 독립 브랜치는 순서 제약이 없다. 머지 자체는 사용자가 실행(보호 브랜치 포인터 이동은 AI가 하지 않음).

## 머지된 PR 처리

앞 PR에 독립인 PR은 전체 완성을 기다리지 않고 먼저 머지할 수 있다. 머지된 커밋은 보호 브랜치에 들어가 history rewrite가 불가하므로, **본 세션의 일괄 조작(replace + 메시지 최종화) 대상에서 빠진다.** 즉 2차 정리 "전 커밋 일괄"의 대상은 **미머지분**이다 ([commits.md](../commits.md)에 경계 명시).

- 먼저 머지할 PR은 **머지 직전에 그 PR만 수동으로 메시지 정리**한다 — 본 세션에서 못 고치므로 그때가 마지막 기회다.
- 스택 상류가 섞여 들어가지 않도록 base retarget 또는 상류 선머지로 처리한다.
- 본 세션 진입 시 이미 머지된 PR이 있으면, 그 PR은 건너뛴다는 사실을 사용자에게 보고한다. 조용히 빼지 않는다.

## 후속 안내 (모드 분기)

본 세션 종료 시 SKILL.md 「세션 spawn 안내 메커니즘」으로 후속을 안내한다.

- **채용** — recruitment 마무리 안내([recruitment/SKILL.md](../../recruitment/SKILL.md) 진입: PR 게시·README·메일·채용담당자 리뷰·제출·자산 회수. 세션 자기 회고는 7-3에서 호출). 동일 모드 인자(`채용`) 유지.
- **실무·개인** — 머지 안내(스택은 바텀업, 독립 브랜치는 순서 무관) + 아래 「세션 자기 회고 안내」. 머지는 사용자 실행.

## 세션 자기 회고 안내 (실무·개인)

커밋 최종화(replace + 메시지 최종화 + force-push 요청)가 끝난 사후 시점이므로, 후속 안내에 더해 **각 세션이 자기 자신을 회고**하도록 안내한다 — "workflow를 타고 진행한 각 세션에 다시 들어가 `session-retro`로 자기 회고를 돌리세요(타임라인 뽑기 → 시간소비·위임·병렬 3관점)". 순회 개수는 사용자가 감당 가능한 만큼. 절차는 [session-retro/SKILL.md](../../session-retro/SKILL.md).

안내만 한다 — 본 세션이 회고를 직접 수행하지 않는다(회고 주체는 각 대상 세션 자신).
