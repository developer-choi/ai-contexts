# decisions.md 컨벤션

`/plan/pr{N}/persistent/decisions.md`의 책임·작성 양식·생성 게이트·갱신·정합 점검 단일 출처.

## 책임·위치

본 PR에서 **기록할 가치가 있는** 기술 결정의 근거 + 흐름. 무엇을 기록하고 무엇을 빼는지는 「포함·제외 기준」. 기록 대상이 된 결정은 overview.md에 근거를 적지 않으므로 decisions.md가 유일한 근거 보존처다.

`persistent/` 하위에 둔다. PR·프로젝트 종료 후에도 영구 보존되며, 미래 다른 프로젝트·다른 회사 컨텍스트에서 "이 결정이 왜 이렇게 됐는지" 비교 자료로 쓰인다. consumable 산출물 정리 대상 아님.

## overview.md와의 분담

- **overview.md**: 의도만 — 채택된 결과(기술명·라이브러리·패턴) 나열, 근거 없음
- **decisions.md**: 기록 대상 결정의 근거 + 흐름. 사용자 발화 인용은 그대로

## 포함·제외 기준

**기본값은 "적지 않는다".** 아래 포함 조건에 걸릴 때만 결정으로 남긴다. 옛 "모든 결정을 적는다"는 과수집을 낳았고, MP의 실제 decisions 문서들이 "기각한 대안이 있는 결정"만 기록하는 관행이라 그에 맞춰 뒤집었다.

### 포함 (둘 중 하나라도)

- **사용자 개입 흔적**: LLM 초안을 사용자가 거부·수정했거나 사용자가 직접 선택했다 (대화에 발화가 남는다)
- **기각한 대안을 댈 수 있다**: 둘 이상의 실재 후보를 두고 골랐고 안 고른 쪽도 합리적이었다 — "X 대신 Y, 왜냐면 Z"를 쓸 수 있다

### 제외

- **출처값 그대로 채택**: 1차 소스(공식 문서·MP 베스트프랙티스 등)의 기본값·권장값을 비교 없이 채택 → reference.md 출처 경로로 충분
- **자명한 절차 결과**: 안 할 이유가 없어 결과가 정해진 것 (예: 단일 후보뿐인 라이브러리 채택) → implementation.md 계획에 자연 반영, 별도 기재 불요. 단 **stub 유무는 "자명한 절차 결과"가 아니다** — step-4 §5 판정 소관이고, "외부 시그니처 없음=stub 없음"은 잘못된 디폴트다([stub.md](stub.md) §11: PR이 만들 모든 파일 stub 필수, `*.test.tsx`의 it.todo도 코드 stub). step-3 decisions·계획에서 stub 없음으로 단정하지 않는다.
- **non-issue 확인 (걱정거리 해소)**: "우려했으나 확인해보니 문제 없음"으로 닫힌 항목은 결정이 아니다 (채택한 대안도 기각한 대안도 없음) → 적지 않는다. 필요하면 implementation.md 메모로.
- **PR 분할·범위 변경**: project.md가 PR 분할·이연 인덱스이므로 거기 적는다
- **단순 합의**: 트레이드오프 없이 사용자가 그냥 동의 (선택지가 사실상 하나)

**non-issue 확인 사례** —

**before** (걱정거리 해소를 결정으로 박음):

```markdown
## 결정 N: 오버레이(Dialog·Alert·Confirm) controlled 방식 테스트
채택: open/onClose props 직접 지정 + 콜백 vi.fn() spy. overlay-kit 미사용.
```

→ 사용자가 "뭘 결정했다는 거야"를 반복 질문. 실질은 "overlay-kit 필요할까 걱정했으나 controlled 구조라 불필요"라는 non-issue 확인 — 채택/기각 대안이 없어 결정이 아니다.

**after** (decisions에 안 적고 implementation.md 메모로):

```markdown
(implementation.md) 오버레이는 overlay-kit 불필요 — open prop + 콜백 spy로 충분.
```

### 경계 사례 — AI 단독 판단 금지

포함/제외가 애매하면 AI가 자체 판단으로 박지 않고 "이거 decisions에 적을까요?"로 확인한 뒤 추가한다. 사용자 발화 없이 LLM이 단독 추가하는 결정이 특히 과수집되기 쉽다. 사용자의 즉흥·체념 어조 발화에 대한 확인 절차는 「생성 게이트」 참조.

## 세 갈래 양식

- **사용자 발화 4단계 이상 이어진 결정** (토론 결과 포함): 발화 흐름 인용 + 거부/채택 사유 + 최종 채택안 + 근거. [/discussion 토론 결과 양식](../../../discussion/SKILL.md#토론-결과-정리) 참조
- **사용자 명시 결정** (단발이지만 사용자가 LLM 초안을 거부·수정·확정): 짧은 흐름(LLM 초안 → 사용자 발화 → 채택안) + 근거
- **AI 단독 결정** (사용자 발화 없이 LLM이 컨벤션·관례에 따라 결정): 채택안 + 근거(인용한 컨벤션 라인 등)

## 작성 양식 (사용자 명시 결정)

```markdown
## 결정 N: {제목}

**LLM 초안**: {초안 내용}

**사용자 발화 (1차)**: "{인용}"

**LLM 보고 (1차)**: {옵션 비교 또는 트레이드오프 제시}

**사용자 발화 (2차)**: "{인용}"

**최종 채택안**: {확정안}

**근거**: {근거}

**연결**: overview.md 결정 X, markup.md "Y 절"
```

## 생성 게이트 — 단발 발화 확인

사용자의 단발 발화(특히 "그냥 ~하자", "어쩔수없을거같아", "그렇게라도 하자" 같은 체념·즉흥 결정 어조)로 결정 항목을 추가하기 전에 명시 확인을 받는다.

- 발화 → 곧장 결정 N으로 박지 않는다
- "이 내용을 decisions.md 결정 N으로 박을게요?" 형태로 확정 의사를 묻는다
- 승인을 받은 후에야 항목을 추가한다

사유: persistent 하위라 영구 보존된다. 사용자가 즉흥 어조로 한 발화를 결정으로 굳히면 추후 사용자가 직접 "결정 N은 없애" 같은 삭제 지시를 해야만 회수된다. 대화 흐름의 일시적 합의(메모리상)와 영구 기록의 무게는 다르므로 경계가 필요하다.

**before** — 사용자 "이건 이게 최선이야 어쩔수없을거같아" 한마디로 결정 8을 즉시 작성:

```markdown
## 결정 8: validate API 평생 로딩 케이스 — 별도 대응 없이 현 설계 유지
**사용자 발화**: "이건 이게 최선이야 어쩔수없을거같아"
**최종 채택안**: 타임아웃·재시도·fallback 메커니즘 추가하지 않음
...
```

→ 사용자가 곧이어 "결정 8은 없애"로 삭제 지시. 결정으로 굳힐 의도가 아니었던 발화였음.

**after** — 발화 후 확인:

```
사용자 "이건 이게 최선이야 어쩔수없을거같아"
LLM "decisions.md에 '결정 8: validate 평생 로딩은 별도 대응 없음'으로 박을까요?"
사용자 "ㅇㅋ" → 그제야 추가
```

## 갱신 (구현·리뷰 후)

구현·리뷰 과정에서 새로 발생하거나 초기 작성 시점과 달라진 의사결정을 반영한다.

- 대상: 기술 선택 변경, 트레이드오프 새로 인지한 것, 리뷰 피드백으로 뒤집힌 결정 등
- 단순 코드 변경 이력은 적지 않는다 (git log로 충분). **"왜 그렇게 했는가"만 남긴다.**
- **코드에서 역산 가능한 표면값(커밋 SHA·파일 경로·폴더 구조 등)은 근거·연결 서술에 박지 않는다** (생성·갱신 공통). history rewrite·재설계로 무효화돼 정정 부담만 낳고, 그 값의 진실은 코드·git이 보유한다. "왜"를 설명하려 코드 앵커가 꼭 필요하면 이름 수준(어떤 모듈·어떤 결정)에서 그친다. 이 표면값이 없으면 중간중간 코드가 진화해도 맞출 대상이 없어 갱신은 구현·리뷰 후 한 번으로 수렴한다.

## 정합 점검 게이트 (decisions ↔ 코드)

사용자에게 decisions.md를 읽도록 안내하고 **"결정했던 내용과 지금 코드에 차이가 있는지"** 묻는다.

- 차이가 있고 코드가 맞으면 → decisions.md 갱신
- 차이가 있고 decisions.md가 맞으면 → 코드 수정 흐름
- 일치하면 통과
