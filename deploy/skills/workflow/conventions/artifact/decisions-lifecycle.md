# decisions.md 컨벤션

`/plan/pr{N}/persistent/decisions.md`의 책임·작성 양식·생성 게이트·갱신·정합 점검 단일 출처.

## 책임·위치

본 PR의 **모든 기술 결정**의 근거 + 흐름. 단발 결정도 포함 (overview.md에 근거를 적지 않으므로 decisions.md가 유일한 근거 보존처).

`persistent/` 하위에 둔다. PR·프로젝트 종료 후에도 영구 보존되며, 미래 다른 프로젝트·다른 회사 컨텍스트에서 "이 결정이 왜 이렇게 됐는지" 비교 자료로 쓰인다. consumable 산출물 정리 대상 아님.

## overview.md와의 분담

- **overview.md**: 의도만 — 채택된 결과(기술명·라이브러리·패턴) 나열, 근거 없음
- **decisions.md**: 모든 결정의 근거 + 흐름. 사용자 발화 인용은 그대로

## 세 갈래 양식

- **사용자 발화 4단계 이상 이어진 결정** (토론 결과 포함): 발화 흐름 인용 + 거부/채택 사유 + 최종 채택안 + 근거. [/discussion 토론 결과 양식](../../discussion/SKILL.md#토론-결과-정리) 참조
- **사용자 명시 결정** (단발이지만 사용자가 LLM 초안을 거부·수정·확정): 짧은 흐름(LLM 초안 → 사용자 발화 → 채택안) + 근거
- **AI 단독 결정** (사용자 발화 없이 LLM이 컨벤션·관례에 따라 결정): 채택안 + 근거(인용한 컨벤션 라인 등)

## 작성 양식 (사용자 명시 결정)

```markdown
## 결정 N: {제목}

**LLM 초안**: {초안 내용}

**사용자 발화 (1차)**: "{인용}"

**LLM 보고 (1차)**: {옵션 비교 또는 트레이드오프 제시}

**사용자 발화 (2차)**: "{인용}"

**최종 채택안**: {확정안}

**근거**: {근거}

**연결**: overview.md 결정 X, markup.md "Y 절"
```

## 생성 게이트 — 단발 발화 확인

사용자의 단발 발화(특히 "그냥 ~하자", "어쩔수없을거같아", "그렇게라도 하자" 같은 체념·즉흥 결정 어조)로 결정 항목을 추가하기 전에 명시 확인을 받는다.

- 발화 → 곧장 결정 N으로 박지 않는다
- "이 내용을 decisions.md 결정 N으로 박을게요?" 형태로 확정 의사를 묻는다
- 승인을 받은 후에야 항목을 추가한다

사유: persistent 하위라 영구 보존된다. 사용자가 즉흥 어조로 한 발화를 결정으로 굳히면 추후 사용자가 직접 "결정 N은 없애" 같은 삭제 지시를 해야만 회수된다. 대화 흐름의 일시적 합의(메모리상)와 영구 기록의 무게는 다르므로 경계가 필요하다.

**before** — 사용자 "이건 이게 최선이야 어쩔수없을거같아" 한마디로 결정 8을 즉시 작성:

```markdown
## 결정 8: validate API 평생 로딩 케이스 — 별도 대응 없이 현 설계 유지
**사용자 발화**: "이건 이게 최선이야 어쩔수없을거같아"
**최종 채택안**: 타임아웃·재시도·fallback 메커니즘 추가하지 않음
...
```

→ 사용자가 곧이어 "결정 8은 없애"로 삭제 지시. 결정으로 굳힐 의도가 아니었던 발화였음.

**after** — 발화 후 확인:

```
사용자 "이건 이게 최선이야 어쩔수없을거같아"
LLM "decisions.md에 '결정 8: validate 평생 로딩은 별도 대응 없음'으로 박을까요?"
사용자 "ㅇㅋ" → 그제야 추가
```

## 갱신 (구현·리뷰 후)

구현·리뷰 과정에서 새로 발생하거나 초기 작성 시점과 달라진 의사결정을 반영한다.

- 대상: 기술 선택 변경, 트레이드오프 새로 인지한 것, 리뷰 피드백으로 뒤집힌 결정 등
- 단순 코드 변경 이력은 적지 않는다 (git log로 충분). **"왜 그렇게 했는가"만 남긴다.**

## 정합 점검 게이트 (decisions ↔ 코드)

사용자에게 decisions.md를 읽도록 안내하고 **"결정했던 내용과 지금 코드에 차이가 있는지"** 묻는다.

- 차이가 있고 코드가 맞으면 → decisions.md 갱신
- 차이가 있고 decisions.md가 맞으면 → 코드 수정 흐름
- 일치하면 통과
