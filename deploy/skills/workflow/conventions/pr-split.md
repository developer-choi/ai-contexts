# PR 분할 — 확정 기준 + 분할 원칙 + 의존 태깅

큰 작업을 여러 PR로 가르는 판단의 단일 출처. **일괄 분할 시점은 없다** — step-1에서 자료를 읽어나가며 확정 가능한 PR부터 하나씩 확정한다.

## 작업 크기 판단

- **작은 작업** (커밋 3개 이하): 분할하지 않는다
- **큰 작업** (여러 도메인 포함): 아래 기준으로 분할

작업이 작다고 판단되면 사용자에게 분할 생략을 제안한다.

---

## PR 확정 기준 — "다른 문서를 더 안 봐도 확정 가능한가"

step-1 진행 중 자료를 읽어나가며 `background/consumable/project.md`에 PR을 **하나씩 확정해 append**한다. 모든 PR이 정해질 때까지 기다리지 않는다.

- **아직 안 읽은 자료가 이 PR의 경계를 바꿀 수 있으면** → 확정하지 않는다
- **바꿀 수 없으면** → 확정하고 append. 그 PR은 곧바로 출발 후보가 된다

기준이 자기 자신을 규제하므로 별도 게이트가 필요 없다. 예시로 이렇게 갈린다:

- 툴링 PR — 안 읽은 기획서·시안이 린트 룰·포매터 설정을 바꿀 일이 없다 → 확정 가능
- 도메인 페이지 PR — 안 읽은 기획서가 "목록과 상세가 컴포넌트를 공유한다"고 말할 수 있다 → 아직 아님

**왜 일괄 분할을 안 하나**: PR 순서를 정하는 시점엔 각 PR 속을 아직 안 짜서 공용물(훅·유틸·API·타입)의 뼈대/실물을 정밀히 알 수 없다. 모아서 확정해도 정보가 더 정확해지지 않고, 확실한 PR(툴링)이 불확실한 PR(도메인 분석)을 기다리는 대기만 생긴다.

### 확정 ≠ 출발 가능

둘은 별개 게이트다.

| 게이트 | 질문 | 기준 |
|---|---|---|
| 경계 확정 | 이 PR이 무엇인지 정해졌나 | 다른 자료를 더 안 봐도 되나 (위) |
| 출발 가능 | 지금 시작할 수 있나 | 앞 PR의 껍데기만 필요한가, 실물이 필요한가 (아래 「의존 태깅」) |

확정은 append-only로 쌓이고, 출발은 각 PR이 자기 선행이 풀릴 때 개별 발동한다.

## 채용과제: 기본 PR 구조 (가설)

채용과제는 아래를 **기본 가설**로 잡는다 — 정적으로 못박힌 순서가 아니라 사용자 확정으로 override되는 출발점이다. 구체적 항목은 각 PR의 step-3에서 정의한다.

| 프리셋 | 경계 |
|----|------|
| PRESET_FOUNDATION (프로젝트 베이스) | 폴더 구조 마이그레이션 + 코딩 스탠다드 마이그레이션. FOUNDATION 세션이 자기 브랜치에서 완결한다 ([session/foundation.md](session/foundation.md)) |
| PRESET_SETUP (프로젝트 세팅) | 빌드·린트·포맷 등 static checking 도구와 설정. 런타임 코드 없음 |
| PRESET_INFRA (공통 인프라) | 런타임 의존성 설치 + Provider·API 클라이언트 등 앱 인프라 구성 |
| PRESET_COMPONENTS (공통 컴포넌트) | MARKUP이 「공통 컴포넌트 확정」으로 산출한 범용(도메인 안 묻은) 재사용 UI 컴포넌트. 공통성 판정(2군데 이상)은 MARKUP이 하고 여기서는 그 산출을 소비한다(단방향) — 공통성을 재판정하지 않는다 |

순서·경계가 실제 의존과 맞지 않으면 사용자가 override한다 (아래 「판단거리 노출」).

## PR 분할 원칙

- **한 번에 하나의 PR, 하나의 구체적인 기능**만 다룬다.
- **기존 자산 변경 ≠ 신규 기능**: 한 PR이 기존 자산의 개명·이동·구조 변경(마이그레이션)과 새 기능 추가·확장을 동시에 하려 하면 두 PR로 가른다 — ① 마이그레이션 PR(동작 변화 없음, 개명·파일 이동·참조 갱신만), ② 기능 PR(그 위에 새 동작). 마이그레이션을 먼저.
- **의존성 순서**: 공통 모듈 → 개별 페이지
- **기술적 과제는 각 PR의 step-3에서** — 분할 판단은 경계에만 집중
- **stub 시점부터 병렬 (껍데기 의존)**: `PR_{N}_PLAN.step-4` 종료 시점에 stub 커밋이 외부 시그니처를 확정하면, 다운스트림 PR이 껍데기(시그니처)에만 기대는 한 PR_{N+1}_PLAN과 PR_{N}_IMPL을 동시 spawn 가능
- **숨은 실물 의존이면 직렬**: 다운스트림 PR이 앞 PR의 **실물 산출물**(추출·리팩토링·이동 결과)이 있어야 착수 가능하면 앞 IMPL 완료를 선행으로 직렬화한다. 껍데기(시그니처)만 필요한지 실물이 필요한지가 병렬/직렬을 가른다 (앞 PR이 미머지인 경우 머지는 종료 페이즈 FINALIZE 몫이라 선행이 아니다)

## 의존 태깅 — 판단거리 노출

PR 순서를 정적으로 못박지 않는다. "PRESET_SETUP·PRESET_INFRA는 인프라성이라 순차" 같은 고정 엣지도 실제론 유동적이다. 확정 시점엔 각 PR 속을 아직 안 짜서 공용물의 뼈대/실물을 정밀히 알 수 없으므로, AI가 무거운 의존 분석을 스스로 하면 과설계다.

AI는 "판단"이 아니라 "**판단거리 노출**"만 한다: 각 PR이 앞 PR에서 당겨 쓰는 것을 **[껍데기 의존 / 실물 의존]**으로 러프 태깅해(값싼 신호만) 사용자가 순차/병렬을 1초에 확정하게 한다. 판단은 사용자 몫.

- **값싼 신호**: 새로 만드는 것 = 뼈대(껍데기)면 병렬 후보 / 기존 걸 뽑아내는(공통화·리팩토링) = 실물 필요라 직렬 후보. 주제 단위로 끊으며 "대략 예상되는 만큼만" 러프 예측 → 사용자가 도메인 지식으로 교정.
- **금지 (함정)**: [껍데기/실물] 태깅을 생략하고 "선행 → 후행 순서 리스트"로 미리 구우면 의존의 **정체**(껍데기냐 실물이냐)가 숨어 실패가 재발한다. 순서로 미리 굽지 않는다.

브랜치 base 결정도 이 태깅을 판단거리로 쓴다 — 결정 자체는 사용자 확인 사항이다 ([steps/step-4.md](../steps/step-4.md) 「사전 준비」).

## 기획 점검 (TODO 식별)

PR을 확정할 때마다 `project.md`의 미분류 절(PR 경계 확정 전에 쌓인 TODO) 중 그 PR에 속하는 항목을 해당 PR 절로 재배치한다. 그 외 각 PR별로 사용자에게 사전 확인이 필요한 새 TODO가 있는지 추가로 확인한다.

미분류 절은 유지한다 — 아직 확정 안 된 PR의 TODO가 계속 쌓이는 보관소다.

## 산출물: `/plan/background/consumable/project.md`

확정된 PR별로 아래 항목을 포함한다:
- PR 제목
- 범위
- 참고 background 파일
- 기획 확인 필요 사항 (TODO)
- 앞 PR 의존 태깅 [껍데기 / 실물]

PR을 확정할 때마다 사용자에게 보고한다:
- 확정한 PR의 범위 한 줄 요약
- 앞 PR과의 의존 관계 (껍데기/실물)
- 사용자 판단이 필요한 열린 질문 (있는 경우)
