# 커밋 컨벤션

워크플로우 단계별 커밋 메시지 양식 + history rewriting 안전 절차 단일 출처.

프로젝트의 commitlint(`type(scope): subject`, scope-enum, subject 한글 필수 등) 위에 얹는 워크플로우 추가 룰만 본 컨벤션이 정의한다.

**commitlint 도입 책임**: 채용은 PRESET_SETUP 정식 구축 단계에서 도입한다 (foundation.md 「PRESET_SETUP 정식 구축 항목」 참조). 실무·개인은 프로젝트 셋업 초기에 이미 깔려있는 것이 전제 — 신규 프로젝트면 첫 PR에 포함한다.

## 커밋 메시지 라이프사이클

| 시점 | 양식 |
|---|---|
| stub 커밋 (PR 골조 생성) | `chore(<scope>): 초기 골조 stub` |
| 잔존 md 커밋 (`/plan/pr{N}/` 산출물) | `docs(<scope>): <산출물 요약>` |
| IMPL·리뷰 수정 커밋 | `feat(<scope>): <슬라이스 메시지>` |
| 1차 정리 (각 PR step-6.5, 슬라이스별 재정렬) | 슬라이스 단위로 묶음 |
| 2차 정리 (FINALIZE, 전 PR 일괄 1회) | 슬라이스·도메인 기준 메시지 재작성 + 외부 독자 톤 최종화 |

- `<scope>`는 프로젝트 commitlint scope-enum에 따라 결정
- 커밋 소속(어느 PR인지)은 접두사가 아니라 **브랜치**(각 PR 브랜치의 base..tip 범위)로 식별한다 — PR=브랜치

## stub 커밋

- PR 골조 모든 stub 파일을 하나의 커밋으로 묶음
- 양식: `chore(<scope>): 초기 골조 stub`
- stub 파일만 담는다. md 산출물(`/plan/pr{N}/` 하위)은 아래 「잔존 md 커밋」 별도 분리

## 잔존 md 커밋

- `/plan/pr{N}/` 하위 산출물(decisions.md·reference.md·markup.md·implementation.md·overview.md 등) 작성·갱신을 묶는 커밋
- 양식: `docs(<scope>): <산출물 요약>` (예: `docs(plan): decisions·reference·implementation 초기 작성`)
- stub 파일과 같은 커밋에 섞지 않는다 (글로벌 규칙 「커밋 단위」). 한 step에서 여러 md를 함께 만들었으면 한 docs 커밋으로 묶어도 OK

## 슬라이스 분할 정리 예시

```
git commit -m "feat(<scope>): <슬라이스 1 메시지>"
git commit -m "feat(<scope>): <슬라이스 2 메시지>"
```

## 머지 후 수정은 원 커밋에 폴드

머지(또는 2차 정리) 후 리뷰·수정이 발생해 히스토리를 다시 쌓는 경우, 수정분을 후속 커밋으로 남기지 않고 **해당 원 커밋에 폴드**한다. 기준은 "처음부터 그렇게 작성한 히스토리" — 사용자가 재스택을 지시하면 폴더 구조·코드 내용·카피까지 전부 이 기준으로 재구성한다.

**before** — 구조만 고치고 리뷰 수정을 후속 커밋으로 잔존:

```
feat(examples): 제출 생명주기 데모 + 도착 페이지
...
refactor(examples): [리뷰 반영] 데모 네이밍·카피·주석 정리
fix(examples): 머무름 데모 reset 버그 수정
```

**after** — 리뷰 수정분을 원 커밋에 폴드, 후속 커밋 없음:

```
feat(examples): 제출 생명주기 데모 + 도착 페이지   ← 네이밍·버그 수정 내장
```

## 커밋 메시지 톤 — 외부 독자 친화

커밋은 외부 독자(채용담당자·리뷰어·팀원)가 읽는다. 작업 중 섞인 캐주얼·은어·비속어를 정중·명확한 기술 문서 톤으로 통일한다.

- 은어·감탄사·비속어("존나", "촤르륵", "죽음", "그짓" 등) → 중립 기술 표현으로 치환하거나 삭제
- 무엇을·왜 바꿨는지가 드러나는 명확한 서술로. 감정·구어체는 걷어낸다
- **모드 공통** — 실무·개인도 리뷰어·팀원이 커밋을 본다. 채용은 강도만 더한다(은어 0건 목표)
- 최종 스캔 시점: FINALIZE 2차 정리(replace + 메시지 재작성과 함께 수행). 커밋 메시지뿐 아니라 `project.md` 등 외부 노출 가능 메모도 함께 스캔한다

## 전제 — 미머지

아래 메시지 최종화(전역 메시지 재작성)·replace(오배치 재배치)가 성립하는 근거는 **아직 머지되지 않았다는 것**이다. 구현 페이즈는 머지 없이 도미노로 진행하므로 종료 페이즈(FINALIZE) 진입 시점엔 어느 PR도 머지되지 않은 것이 기본이다. 커밋 소속은 브랜치(base..tip)로 식별한다.

**브랜치 형태는 PR마다 다르다** — 앞 PR에 의존하면 그 브랜치 위에 얹히고(스택), 독립이면 기본 브랜치에서 각자 뻗는다. base 결정은 사용자 확인 사항이다 ([../steps/step-4.md](../steps/step-4.md) 「사전 준비」). 형태에 따라 replace 방법이 갈린다(아래).

## PR 간 시간순 섞임 vs 오배치

두 현상을 구분한다 — 처리가 정반대다.

- **시간순 섞임 (소속은 맞음)**: 커밋이 원래 자기 PR에 있는데 시간 순서만 다른 PR 커밋과 뒤섞임. **재배열하지 않는다** — PR 소속은 브랜치(base..tip)로 식별되고 커밋 시간순은 식별 가치가 없으므로, 시간순 재배열은 순이익 없는 추가 history rewrite 부담이다.
- **오배치 (소속이 틀림)**: "PR4 브랜치에 있지만 실은 PR2 작업인 커밋"처럼 커밋이 잘못된 PR에 얹혀 있음. 원래 PR로 **재배치(replace)** 한다 — 이건 소속을 바로잡는 조작이라 위 "재배열 금지"의 예외가 아니라 별개 사안이다.

### replace 절차 (오배치 재배치)

FINALIZE에서 1회 수행한다 ([conventions/session/finalize.md](session/finalize.md) 「replace」). 두 브랜치의 관계로 방법이 갈린다:

- **스택 관계** — 대상 PR(상류)에 커밋을 옮기면 하류 PR 브랜치들을 연쇄 rebase해야 한다(다중 브랜치 rewrite).
- **독립 브랜치 관계** — 연쇄가 없어 rebase로 안 풀린다. 대상 브랜치에 **cherry-pick → 원본 브랜치에서 그 커밋 drop**의 2단 조작이다.

전 PR이 완성돼야 대상 PR이 확정되므로 종료 페이즈 1회 조작이다. 「history rewriting 안전 절차」의 백업 브랜치를 **재작성되는 모든 브랜치**에 적용한다 (스택이면 대상·하류, 독립이면 대상·원본).

### 2차 정리 대상 경계 — 머지된 PR 제외

FINALIZE 2차 정리(replace + 메시지 최종화) "전 커밋 일괄" 대상은 **미머지분**이다. 머지된 커밋은 보호 브랜치에 들어가 history rewrite가 불가하므로 일괄 조작에서 빠진다. 앞 PR에 독립인 PR은 전체 완성 전에 먼저 머지할 수 있으므로 이 경우가 실제로 생긴다.

먼저 머지할 PR은 머지 직전 **그 PR만 수동으로 메시지 정리**한다(FINALIZE에서 못 고치므로 그때가 마지막 기회) + 스택 상류가 섞이지 않도록 base retarget 또는 상류 선머지로 처리한다.

## history rewriting 안전 절차

다음 작업 직전에 적용한다:

- stub 커밋 drop + 슬라이스별 재정렬
- 커밋 메시지 재작성 (FINALIZE 2차 정리 등)
- rebase·soft reset으로 커밋 history를 다시 쓰는 모든 작업

### 백업 브랜치 [CRITICAL]

history rewriting 시작 전 반드시 백업 브랜치를 뜬다.

```
git branch backup/<현재브랜치>-<YYYYMMDD-HHmm>
```

백업 없이 history rewriting을 시작하지 않는다. 사고 시 다음 명령으로 즉시 복구:

```
git reset --hard backup/<현재브랜치>-<YYYYMMDD-HHmm>
```

**다중 브랜치(FINALIZE) 확장**: replace는 여러 브랜치를 재작성하므로(스택이면 대상 PR + 하류, 독립이면 대상 + 원본), **재작성 대상 브랜치마다** 백업을 뜬다. 한 브랜치만 백업하고 연쇄 rebase나 cherry-pick+drop에 들어가지 않는다.

### force-push 요청 보고 양식

재작성 완료 후 사용자에게 force-push 요청. 다음을 보고에 포함:

- 백업 브랜치 이름
- 재정렬 후 커밋 목록 (`git log` 결과)
