# 채용과제 리뷰 가이드

채용 과제의 요구사항을 분석하여, 이후 PR 분할과 구현의 기반이 되는 분석 문서를 작성합니다.

## 슬롯

### 준비 방식

AI가 사용자에게 공고, 메일, 과제요구사항 자료를 요청하고, `/plan/interview-prep.md`를 생성한다(준비 단계의 중간 산출물 — 최종 산출물이 아니라 분석 과정에서 참조용으로 사용). 원본 사이트 URL이 있으면 같이 요청한다.

### 입력 자료

사용자가 `/plan/background/`에 작성한 원본 자료:

- `공고.md` — 채용공고 원본 (공고가 닫히면 접근 불가하므로 보관 필수)
- `메일.md` — 채용 담당자가 보낸 메일 본문
- `과제요구사항.md` — 과제 요구사항 문서

이후 step(과제 정의, 리뷰 등)에서 원본을 직접 참조하여 누락/개선점을 확인한다.

### 체크리스트

없음 — 채용과제는 체크리스트 대신 분석 과정의 전용 절차를 따릅니다.

### 산출물

- `/plan/background/cross-analysis.md` — 분석 결과
- `/plan/background/service-analysis.md` — 대상 서비스 분석 ([가이드](service-analysis.md))

### 분석 과정

#### 패키지 설치 PR 가이드

채용과제에서 패키지 설치 PR을 논의할 때, [install-packages.md](install-packages.md)를 **반드시** 참조한다.
이 가이드의 순서와 필수 항목을 기준으로 설치 범위를 사용자와 논의한다.

#### 핵심 원칙: 원본과의 중복 금지

`cross-analysis.md`는 원본 파일(`공고.md`, `메일.md`, `과제요구사항.md`)에 **이미 적혀 있는 내용을 재진술하지 않는다.**
원본은 이후 step에서 직접 참조되므로, cross-analysis.md에는 원본을 **교차 분석하여 도출한 인사이트만** 담는다.

- **적어야 하는 것**: 놓치기 쉬운 세부 스펙, 원본 간 교차 분석, 추론한 평가 기준, 기술 결정, 모호한 부분
- **적지 않는 것**: 요구사항 요약, 제출 기한/방식, 기술 스택 제약, 평가 우대사항 목록 등 원본에 그대로 있는 정보

#### 1. 공동 탐색 + 서비스 분석

사용자가 자료를 제출하면, **공고 → 메일 → 과제요구사항 → 사이트(있는 경우)** 순서로 같이 정독하며 논의한다.

정독과 병렬로 AI는 분석을 수행한다:
- 원본 3개 교차 분석 (놓치기 쉬운 세부 스펙, 원본 간 교차 분석)
- [service-analysis.md](service-analysis.md) 가이드에 따른 서비스 분석
- API/도메인 분석
- **테스트 코드 분석** — 프로젝트에 테스트 설정(Playwright, Jest 등)이 있으면 테스트 내용을 읽고 채점 기준을 역추론한다
- **프로젝트 파일 분석** — 제공된 프로젝트의 Dockerfile, packageManager, 빌드 설정, .gitignore, 기존 디렉토리 구조 등을 확인한다. 빌드/실행 환경 제약을 조기에 파악하기 위함
- **기존 소스 코드 탐색** — "퍼블리싱 완료" 등 기존 코드가 있다고 명시된 부분은 실제 src를 확인하여 작업 범위를 파악한다

정독과 병렬 분석이 모두 끝나면 발견 사항을 공유한다.

정독 중 공고·과제요구사항이 중시하는 역량을 파악하고, 그에 맞는 디테일 TODO를 서로 제안한다. `/plan/project.md`가 이미 존재하면 거기에 쌓고, 없으면 cross-analysis.md의 TODO 섹션에 임시로 기록한다 (step-2에서 project.md 생성 시 이관). 최소한 아래 관점은 점검한다:
- 완성도 (404 페이지, 파비콘, page.title, OG 태그, 실제 이미지·텍스트, 로딩/에러 상태)
- UI/UX (뒤로가기, 취소 버튼, 푸터 링크, 빈 상태, 호버/포커스, 토스트 피드백)
- 공고에서 강조하는 역량에 연결되는 어필 포인트
- 페이지별 기술적·비기술적 강점 어필 포인트

공고가 중시하는 역량에 매칭되는 기존 베스트 프랙티스를 MP/DC에서 찾아 `/plan/background/`에 참조 링크를 걸어둔다. MP/DC에 접근할 수 없으면 사용자에게 알리고 넘어간다.

#### 2. 평가 기준 추론

원본에 명시된 평가 기준을 재진술하지 않고, 과제 성격과 채용공고에서 **암묵적 평가 기준**을 추론합니다:

- 공고의 자격요건/우대사항과 과제 요구사항을 대조하여 숨겨진 기대치 도출
- 과제 구조(테스트 자동화, 디자인 제공 범위 등)에서 역추론한 평가 포인트

#### 3. 프로젝트 결정 사항

사용자와 논의하여 확정한다:
- 기술 스택 (프레임워크, 주요 라이브러리)
- 디자인 시스템
- 폴더 구조
- 공통 패키지 ([install-packages.md](install-packages.md) 참조)
- PR 1 구성 (린트, tsconfig, husky 등 초기 셋업 범위)
- 마크업 우선 여부

#### 4. 모호한 부분 및 질문 목록

과제 설명에서 불명확한 부분을 짚고, 채용 담당자에게 질문할지 / 본인 판단으로 결정할지 사용자와 논의합니다.
