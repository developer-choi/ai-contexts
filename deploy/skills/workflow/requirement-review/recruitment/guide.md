# 채용과제 리뷰 가이드

채용 과제의 요구사항을 분석하여, 이후 PR 분할과 구현의 기반이 되는 분석 문서를 작성합니다.

## 슬롯

### 준비 방식

AI가 사용자에게 공고, 메일, 과제요구사항 자료를 요청한다. 원본 사이트 URL이 있으면 같이 요청한다.

### 입력 자료

사용자가 `/plan/background/persistent/`에 작성한 원본 자료:

- `공고.md` — 채용공고 원본 (공고가 닫히면 접근 불가하므로 보관 필수)
- `메일.md` — 채용 담당자가 보낸 메일 본문
- `과제요구사항.md` — 과제 요구사항 문서

이후 stage(과제 정의, 리뷰 등)에서 원본을 직접 참조하여 누락/개선점을 확인한다.

### 입력 자료 저장 시 제외 항목

원본 자료를 `/plan/background/persistent/`에 저장할 때, 아래 정보는 모두 제외한다:

- 제출 기한, 마감일, 잔여 시간
- "X일까지", "X시간 내", "X일 이내 제출" 등 시간 압박 표현
- 메일 수신 일자/타임스탬프
- 공고 마감일·응시 기한

이유: 시간 압박 정보는 AI가 작업 범위·완성도를 자의적으로 축소하는 근거가 되므로 차단한다. 시간 배분은 사용자가 판단한다.

### 체크리스트

없음 — 채용과제는 체크리스트 대신 분석 과정의 전용 절차를 따릅니다.

### 산출물

- `/plan/background/consumable/cross-analysis.md` — 분석 결과
- `/plan/background/consumable/service-analysis.md` — 대상 서비스 분석 ([가이드](service-analysis.md))

### 분석 과정

본 「분석 과정」 5단 절차는 워크플로우 step-1.md의 **Step 1.2 requirement-review 본체** 영역이다. step-1.1(자료 받기)이 종료되고 「Step 1.1 종료 — 분기점」 절차(SKILL.md 「세션 spawn 안내 메커니즘」 발동)가 끝난 뒤 진입한다. 자료 저장 직후 곧장 본 절차에 진입하지 않는다 — 분기점 spawn 안내가 누락된다.

#### 핵심 원칙: 원본과의 중복 금지

`cross-analysis.md`는 원본 파일(`공고.md`, `메일.md`, `과제요구사항.md`)에 **이미 적혀 있는 내용을 재진술하지 않는다.**
원본은 이후 stage에서 직접 참조되므로, cross-analysis.md에는 원본을 **교차 분석하여 도출한 인사이트만** 담는다.

- **적어야 하는 것**: 놓치기 쉬운 세부 스펙, 원본 간 교차 분석, 추론한 평가 기준, 기술 결정, 모호한 부분
- **적지 않는 것**: 요구사항 요약, 제출 기한/방식, 기술 스택 제약, 평가 우대사항 목록 등 원본에 그대로 있는 정보

#### 1. 공동 탐색 + 서비스 분석

사용자가 자료를 제출하면, **공고 → 메일 → 과제요구사항 → 사이트(있는 경우)** 순서로, 각 자료의 핵심 내용을 사용자에게 설명하며 같이 정독한다. 각 자료마다 사용자의 반응을 확인한 뒤 다음으로 넘어간다. 자료를 혼자 분석한 뒤 결과만 보고하지 않는다 — 자료별로 멈추고 사용자와 논의한다.

정독과 병행하여 AI는 아래 분석을 수행하되, 분석 결과는 모든 자료의 정독이 끝난 뒤에 한꺼번에 공유한다 (정독 중에는 자료 이해와 사용자 논의에 집중):
- 원본 3개 교차 분석 (놓치기 쉬운 세부 스펙, 원본 간 교차 분석)
- [service-analysis.md](service-analysis.md) 가이드에 따른 서비스 분석
- API/도메인 분석
- **테스트 코드 분석** — 프로젝트에 테스트 설정(Playwright, Jest 등)이 있으면 테스트 내용을 읽고 채점 기준을 역추론한다
- **프로젝트 파일 분석** — 제공된 프로젝트의 Dockerfile, packageManager, 빌드 설정, .gitignore, 기존 디렉토리 구조 등을 확인한다. 빌드/실행 환경 제약을 조기에 파악하기 위함
- **기존 소스 코드 탐색** — "퍼블리싱 완료" 등 기존 코드가 있다고 명시된 부분은 실제 src를 확인하여 작업 범위를 파악한다

과제요구사항에 UI 목업/시안이 포함되어 있으면, 페이지(화면)별로 분석한다:
- 각 화면의 구성 요소와 기능 범위를 식별
- 목업에서 명시된 것과 추론이 필요한 것을 구분
- 화면 간 네비게이션 흐름을 파악

UI 시안이 포함되어 있는데 design 플래그가 켜져 있지 않으면, 사용자에게 design 플래그를 켤지 확인한다.

정독과 병렬 분석이 모두 끝나면 발견 사항을 공유한다.

정독 중 공고·과제요구사항이 중시하는 역량을 파악하고, 그에 맞는 디테일 TODO를 서로 제안한다. `/plan/background/consumable/project.md`가 이미 존재하면 거기에 쌓고, 없으면 cross-analysis.md의 TODO 섹션에 임시로 기록한다 (step-2에서 project.md 생성 시 이관). 최소한 아래 관점은 점검한다:
- 완성도 (404 페이지, 파비콘, page.title, OG 태그, 실제 이미지·텍스트, 로딩/에러 상태)
- UI/UX (뒤로가기, 취소 버튼, 푸터 링크, 빈 상태, 호버/포커스, 토스트 피드백)
- 공고에서 강조하는 역량에 연결되는 어필 포인트
- 페이지별 기술적·비기술적 강점 어필 포인트

공고가 중시하는 역량에 매칭되는 기존 베스트 프랙티스를 MP/DC에서 찾아 `/plan/background/consumable/`에 참조 링크를 걸어둔다. MP/DC에 접근할 수 없으면 사용자에게 알리고 넘어간다.

#### 2. 평가 기준 추론

원본에 명시된 평가 기준을 재진술하지 않고, 과제 성격과 채용공고에서 **암묵적 평가 기준**을 추론합니다:

- 공고의 자격요건/우대사항과 과제 요구사항을 대조하여 숨겨진 기대치 도출
- 과제 구조(테스트 자동화, 디자인 제공 범위 등)에서 역추론한 평가 포인트

#### 3. 프로젝트 결정 사항

사용자와 논의하여 확정한다:
- 기술 스택 (프레임워크, 주요 라이브러리)
- 디자인 시스템 라이브러리
- 폴더 구조
- 사용자 직접 진행 항목 (프로젝트 초기화, 에디터 설정 등 AI가 대행할 수 없는 것)

기술 스택·디자인 시스템 라이브러리는 [prefer-packages.md](prefer-packages.md)의 사용자 선호를 **기본 선택지로 안건화하여 확정**한다. 스타일링·데이터 페칭·상태관리 같은 기초 스택은 MARKUP·이후 PR이 모두 의존하므로 BG에서 확정한다 (step-3은 PR별 패키지 참조, BG는 기초 스택 확정 — 같은 파일을 두 시점에서 사용).

- 기존 코드베이스에 설치·사용 중인 스택을 "결정된 사실"로 흡수하지 않는다. "기존이 X를 쓴다" ≠ "X로 결정됐다" (create-next-app 디폴트·이전 작업자 선택일 수 있음). prefer-packages.md 선호와 다르면 안건으로 올려 확정한다.
- **공고 우대 스택은 결정 근거가 아니다.** 결정 근거 = ① 과제요구사항 강제 명시(오버라이드) → ② prefer-packages.md 사용자 선호. 공고 우대사항은 평가 디테일 추론(#2)에만 쓰고 스택 결정 근거로 leak시키지 않는다.

#### 4. 페이지 자료 수집 + 디자인 시스템 설계

섹션 3에서 디자인 시스템 라이브러리가 결정된 후 진행한다.

**페이지별 자료 요청**: 사용자에게 페이지별 자료를 요청한다. 형식은 자유 — 페이지 캡처 이미지 묶음, 파일별 링크, 피그마 링크 등. 사용자가 주는 대로 받는다.

**자료 정리**: 받은 자료를 [conventions/plan-folder.md](../../conventions/plan-folder.md) 「피그마 URL·캡처 캐싱」 정책에 따라 정리한다 — URL은 `background/retained/figma-url.md`에 누적, 캡처 이미지는 `background/retained/figma/[meaningful-name].[이미지확장자]`에 저장.

**디자인 시스템 설계**: 전체 자료를 훑어 필요한 컴포넌트 종류와 각 컴포넌트의 props 요구사항을 설계하여 `/plan/background/consumable/design-system.md`에 저장한다. step-3 PR3 정의와 step-4 PR3 구현 방침의 입력 재료로 쓰이고, 사용 후 삭제되는 중간 산출물이다. PR4+ 구현 시에는 페이지 이미지 파일을 `markup.md` 참조 자료로 사용한다.
