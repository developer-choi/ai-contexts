# URL / 라우팅 / 렌더링 점검 가이드

페이지 단위 기획에서 URL 구조와 렌더링 전략이 정의되어 있는지 점검한다.

---

## 1. URL 설계

- 각 페이지의 URL 경로가 정의되어 있는가?
- URL만 보고 어떤 페이지인지 유추할 수 있는가?
- URL 계층 구조가 페이지 간 관계를 반영하는가?
- 동적 파라미터(`:id` 등)가 필요한 페이지는 식별되어 있는가?
- 동적 파라미터에 올 수 있는 값이 명세되어 있는가? (예: `[language]`에 `en`이 오는지 `english`가 오는지 — AI가 값을 추측하면 잘못된 구현으로 이어진다)
- 동적 값이 pathname에 있어야 하는지, 쿼리스트링으로 분리해야 하는지 근거와 함께 검토했는가? (참고: SSG가 필요한 값은 반드시 pathname이어야 한다 — 쿼리스트링은 정적 생성을 지원하지 않는다)

## 2. 페이지 렌더링 전략

- 서버 캐싱 전략(static / revalidate / dynamic)의 선택 근거가 타당한가?
- 컴포넌트 유형(Server / Client)의 선택 근거가 타당한가?
- 페이지 내에서 Server/Client가 혼합되는 경우, 경계가 식별되어 있는가?

## 3. 페이지(라우트) 분리 판단

화면이 별도 라우트로 존재해야 하는지 다음 3가지를 점검한다. 모두 "아니오"면 별도 라우트가 아니라 컴포넌트 상태로 처리한다.

- 뒤로가기 했을 때 이 화면이 나와야 하는가?
- URL을 직접 입력해 진입했을 때 이 화면이 나와야 하는가?
- 새로고침 했을 때 이 화면이 유지되어야 하는가?

## 4. 중첩 라우트 / 레이아웃

- 여러 페이지가 공유하는 레이아웃이 있는가?
- 중첩 라우트가 필요한 구간이 있는가?
- 레이아웃 내 공통 UI 요소(네비게이션, 브레드크럼 등)가 식별되어 있는가?
