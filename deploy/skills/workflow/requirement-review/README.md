# Requirement Review

기획서, 디자인 시안, 채용 과제 요구사항을 구현 전에 개발자 관점에서 살펴보고, 모호한 부분과 예외 상황을 미리 찾아내는 스킬입니다. 작업 워크플로우의 과제 정의 단계에서 부릅니다.

## 동기

요구사항을 받자마자 코드를 짜면, 빈틈은 구현 도중에야 드러났습니다. "이 상태에서 뒤로 가면?" 같은 질문이 개발 중에 떠오르고, 그때마다 기획자에게 확인을 요청하면 작업이 멈추곤 했습니다.

그 빈틈을 코드를 짜기 전에 질문으로 먼저 끄집어내 두자는 것이 이 스킬입니다.

## 살펴보는 자료

세 종류의 자료를 다루며, 필요한 것만 골라 켤 수 있습니다.

- **기획서**: 페이지 단위로 화면의 목적, 데이터 흐름, 예외 상황을 정리합니다.
- **디자인 시안**: 반응형, 상태별 화면, 인터랙션을 검토합니다.
- **채용 과제**: 요구사항에서 숨은 평가 기준을 추론하고 구현 방향을 잡습니다.

기획서와 디자인 시안은 한 페이지를 받아 같이 분석하고, 논의하고, 확정한 뒤 다음 페이지로 넘어가는 식으로 한 페이지씩 진행합니다. 둘 다 켜져 있으면 같은 페이지에서 기획서를 먼저 확정한 뒤 디자인을 살펴, 같은 페이지를 두 번 돌지 않습니다.

채용 과제는 페이지 개념이 없어, 자료를 함께 살피고 평가 기준을 추론한 뒤 구현 방향을 정하는 자체 순서를 한 번에 따릅니다.

## 산출물

예를 들어 좌석 예약 기획서를 살펴보면, 기획서가 다루지 않은 예외 상황을 질문으로 뽑아냅니다.

```markdown
## 기획자/디자이너 질문 목록

- 좌석을 선택한 뒤 결제하기 전에 다른 사용자가 같은 좌석을 예약하면 어떤 화면을 보여주나요?
- 3단계 플로우(좌석 선택 → 정보 입력 → 결제) 도중 브라우저를 닫았다가 다시 열면
  1단계부터 다시 시작하나요, 이어서 진행하나요?
- 뒤로 가기로 이전 단계로 돌아갔을 때 선택했던 좌석이 그대로 유지되나요?
- 공연명이 매우 길 때(예: 40자 이상) 목록에서 말줄임 처리하나요, 줄바꿈하나요?
```

이렇게 정리된 결과는 이후 PR을 어떻게 나눌지, 어디서부터 구현할지 판단하는 데 씁니다.
