# Requirement Review

기획서, 디자인 시안, 채용과제 요구사항을 구현 전에 개발자 관점에서 리뷰하여, 모호한 부분과 예외 상황을 선제적으로 찾아내는 스킬입니다. workflow의 과제 정의 단계에서 호출됩니다.

## 왜 만들었나요?

요구사항을 받자마자 코드를 짜면, 빈틈은 구현 도중에야 발견되었습니다. "이 상태에서 뒤로 가면?" 같은 질문이 개발 중에 떠오르고, 그때마다 기획자에게 확인을 요청하면 작업이 멈추는 경험을 했습니다.

세 가지 입력 타입을 지원하며, 자유 조합으로 켤 수 있습니다.

- **planning** — 기획서를 페이지 단위로 분석하여 흐름, 데이터, 예외 상황을 정리합니다
- **design** — 디자인 시안을 분석하여 반응형, 상태별 UI, 인터랙션을 검토합니다
- **recruitment** — 채용과제 요구사항을 분석하여 평가 기준을 추론하고 구현 방향을 잡습니다

planning과 design은 **페이지 단위 루프**로 진행합니다. 한 페이지를 받아 같이 분석 → 논의 → 확정한 뒤 다음 페이지로 넘어가는 사이클을 반복합니다. 두 플래그가 같이 켜져 있으면 동일 페이지에서 planning을 먼저 확정한 뒤 design을 분석하므로, 루프를 두 번 돌지 않습니다. recruitment는 페이지 개념이 없어 자체 절차(공동 탐색 → 평가 기준 추론 → 프로젝트 결정 → 모호한 부분)를 한 번에 따릅니다.

각 입력 타입의 가이드는 5개 슬롯(준비 방식 / 입력 자료 / 체크리스트 / 산출물 / 분석 과정)을 채우는 형태로 작성되어 있어, 새 입력 타입을 추가할 때도 같은 슬롯 양식을 따르면 페이지 단위 루프에 그대로 끼워집니다.

예를 들어 좌석 예약 기획서를 리뷰하면, 기획서가 다루지 않은 예외 상황을 찾아냅니다:

```markdown
## 기획자/디자이너 질문 목록

- 좌석을 선택한 뒤 결제하기 전에 다른 사용자가 같은 좌석을 예약하면 어떤 화면을 보여주나요?
- 3단계 플로우(좌석 선택 → 정보 입력 → 결제) 도중 브라우저를 닫았다가 다시 열면
  1단계부터 다시 시작하나요, 이어서 진행하나요?
- 뒤로 가기로 이전 단계로 돌아갔을 때 선택했던 좌석이 그대로 유지되나요?
- 공연명이 매우 길 때(예: 40자 이상) 목록에서 말줄임 처리하나요, 줄바꿈하나요?
```

분석 결과는 구조화된 산출물로 정리되므로, 이후 PR을 어떻게 나눌지, 어디서부터 구현할지 판단하는 데 활용할 수 있습니다.

> 자세한 내용은 [SKILL.md](SKILL.md)를 참고하세요.
