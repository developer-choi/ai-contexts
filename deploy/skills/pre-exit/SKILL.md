# 세션 마무리

핵심 원칙: 사용자가 지적한 것은 다음 세션에서 반복되지 않아야 한다.

## Step 0. 팀 에이전트 회고 수집

활성 팀 에이전트가 있으면, 각 에이전트에게 세션 회고를 요청하고 결과를 수집한다. 에이전트가 발견한 개선 사항도 Step 1에서 함께 사용자에게 제시한다. 수집 완료 후 에이전트를 종료한다.

## Step 1. 회고

이번 세션을 되돌아본다.

- **반복된 실수 패턴**: 같은 유형의 실수가 2회 이상 발생했는가? 발생했다면, 재발 방지를 위해 어디에 규칙화할지 제안한다 (AC 스킬? 규칙? 메모리?).
- **프롬프트 개선 기회**: 세션 중 발견했지만 아직 반영하지 않은 개선 사항이 있는가?
- **사용자가 반려하거나 수정한 것**: 왜 반려/수정됐는지 되짚고, 같은 실수를 반복하지 않도록 피드백 메모리에 저장할 필요가 있는가?
- **사용자 지적 규칙화**: 세션 중 사용자가 AI의 행동을 지적한 모든 건을 모은다. 각 지적에 대해 규칙화 위치(스킬 수정, 규칙 추가, 메모리 저장)와 구체적 규칙 문안 초안을 제시한다.
- **혼낸 원인 분석**: 사용자가 불만을 표한 경우 원인을 분류한다 — (1) 기술스택 이해 부족 (2) 프로젝트 코드구조 이해 부족 (3) 한번에 너무 많은 것을 시킴 (4) 설명이 덜 명확. 원인에 따라 대응이 다르다 (1,2는 AI 학습 필요, 3,4는 사용자 피드백).
- **방향 수정 이력 검증**: 사용자가 방향을 여러 번 수정한 경우, 최초 지시와 최종 결과의 일관성을 검증한다. 불일치가 있으면 사용자에게 알린다.

발견한 개선 사항을 사용자에게 제시하고, 승인받은 것만 반영한다.

## Step 2. 선호 프로파일링

세션 중 드러난 사용자의 선호를 `deploy/contexts/user-preferences.md`에 축적한다.

- 학습 스타일 (예: "비유로 설명해야 이해 잘 함", "면접 질문 형태 선호")
- 작업 스타일 (예: "추상적 질문 싫어함", "코드보다 방향성 먼저")
- 이미 기록된 선호와 중복되면 추가하지 않는다
- 새로운 선호를 발견하면 사용자에게 확인받고 추가한다

## Step 3. 메모리 업데이트

메모리에 저장하거나 업데이트할 것이 있는지 확인한다.

- 새로 알게 된 사용자 정보, 프로젝트 상태, 피드백
- 기존 메모리 중 업데이트가 필요한 것
- 더 이상 유효하지 않아 삭제해야 할 것

## Step 4. 미커밋 변경사항 확인

`git status`로 커밋되지 않은 변경사항이 있는지 확인한다. 있으면 사용자에게 알린다.
