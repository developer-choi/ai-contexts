# 세션 마무리

핵심 원칙: 사용자가 지적한 것은 다음 세션에서 반복되지 않아야 한다.

## Step 0. 팀 에이전트 회고 수집

활성 팀 에이전트가 있으면, 각 에이전트에게 세션 회고를 요청하고 결과를 수집한다.
에이전트가 발견한 문제도 Step 1에서 함께 포함한다. 수집 완료 후 에이전트를 종료한다.

특히 **editor 팀 에이전트**가 활성 상태이면, 세션 중 사용자가 교정한 말투·톤·표현 사례를
빠짐없이 반환하도록 요청한다. 이 사례는 Step 2에서 writing-guide에 규칙+before/after로 심는다.

## Step 1. 문제 리스트업

세션에서 **문제만** 추출한다. 잘한 것은 다루지 않는다.

- 사용자가 지적하거나 반려한 것
- 사용자가 방향을 수정한 것
- 같은 유형의 실수가 2회 이상 발생한 것

각 문제를 한 줄로 요약하여 사용자에게 제시한다.

## Step 2. 규칙화 + 사례 축적

Step 1의 각 문제에 대해:

1. **분류** — 문제의 성격에 맞는 맥락을 특정한다:
   - 프롬프트 작성 문제 → `deploy/contexts/writing-guide/`
   - 코딩 컨벤션 문제 → `deploy/contexts/coding-standards/` 또는 MP `docs/best-practices-map.md`
   - 스킬 동작 문제 → 해당 스킬의 SKILL.md 또는 specialized 체크리스트
   - 워크플로우 문제 → 해당 워크플로우 스킬
   - 일반 AI 동작 교정 → `deploy/rules/global.md`

2. **규칙 문안** — "이런 경우 이렇게 하지 마세요" 형태의 규칙 초안을 작성한다.

3. **실패 사례** — 규칙에 대응하는 before/after 사례를 작성한다.
   사례는 규칙과 같은 맥락에 둔다 (예: writing-guide 규칙이면 writing-guide에 사례도 함께).

사용자에게 위치·규칙·사례를 제시하고, 승인받은 것만 반영한다.

## Step 3. 미커밋 변경사항 확인

`git status`로 커밋되지 않은 변경사항이 있는지 확인한다. 있으면 사용자에게 알린다.

## Step 4. 워크트리 정리

`git worktree list`로 이 세션에서 사용한 워크트리가 있는지 확인한다.
더 이상 필요 없는 워크트리가 있으면 사용자에게 정리할지 확인한다.
