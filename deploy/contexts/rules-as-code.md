# 코드로 표현 가능한 규칙은 코드로

## 핵심

규칙을 "지키도록 부탁"하는 방식(프롬프트·메모리)은 매 턴 LLM의 준수에 의존해 확률적이고, 어겨도 아무도 안 잡는다. 어기는 게 구조적으로 불가능하거나 번거롭게 만드는 쪽으로, 가능한 만큼 항상 왼쪽으로 민다. 규율(discipline)에서 설계(design)로.

## 강제 사다리

```
없앤다(codegen·SSOT)  >  불가능하게(타입·제약)  >  강제(정적분석·hook)  >  부탁(프롬프트)
```

밑에 깔린 축은 **보장 강도**다. 왼쪽일수록 강하다.

- **없앤다** — 한쪽을 다른 쪽에서 생성하면 어긋날 짝이 사라진다 (codegen·단일 출처).
- **불가능하게** — 타입·권한·제약으로 위반 자체가 표현 불가능해진다.
- **강제** — 정적분석·hook·lint이 위반을 결정론적으로 잡는다. 같은 강제 칸 안에서도 **발동 범위**로 우선순위가 갈린다: 정적분석·lint(eslint·tsconfig·stylelint)은 사람·CI·타 도구 어디서나 걸리고, hook은 에이전트 런타임 안에서만 발동한다 → 코드 규칙은 정적분석이 먼저, hook은 그다음. (에이전트 행동·채팅 인터셉트는 정적분석이 적용 불가하므로 hook이 그 도메인에서 가장 강한 수단.)
- **부탁** — 프롬프트로 적어두고 매 턴 준수를 바란다. 가장 약하다.

칸 이름(정적분석·hook…)은 수단의 예시일 뿐이다. validate 린터처럼 "이름엔 없지만 더 강한 보장을 주는" 수단은 강도 축으로 판정해 제 칸에 둔다.

## 내릴 수 있는가 — 단일 게이트

규칙을 한 칸 왼쪽으로 내릴 수 있는 기준은 하나다.

> **LLM 판단이 1도 안 남게 완전히 코드로 표현되는가.**

판단이 조금이라도 남으면 못 내린다. 0이면 내린다. 비용(작성·유지·오탐)은 별도 기준이 아니다 — 코드로 내리면 어차피 더 안전하고, 유지비는 프롬프트로 둬도 든다. "지금 짤 시간이 있나"는 일반 작업 우선순위지 이 게이트가 아니다.

### "코드로 표현 가능"은 거의 항상 과장된다

규칙의 기계적인 부분만 보고 판단이 남는 부분을 빼먹는 함정이 흔하다. 판단이 일부라도 남으면 게이트 통과 실패다.

### hook은 발동만 보장한다

판단 기반 행동(맞춤법·표현 교정 등)은 기계 판정이 불가능하다. 이런 규칙에 hook을 걸어도 hook은 *발동*(그 순간 LLM을 깨우는 것)만 보장하고, 실제 동작은 여전히 LLM이 판단해 수행한다. 강제 칸처럼 보이지만 보장 강도는 부탁에 가깝다 — 이 한계를 알고 선택한다.

## 없앤다 칸의 경계 — 파생 가능성

`없앤다` 칸은 **B가 A에서 파생 가능할 때만** 열린다.

- what/how 주석 = 코드에서 파생 가능 → 없앤다 ✅
- why 주석(의도·근거·외부 제약·판단) = 파생 불가능 → 못 없앤다. 한 칸 내려와 강제·부탁 ❌
- 짝꿍 동기화: "어떤 파일이 짝인가(사실)"는 데이터로 내리고, "제대로 고쳤나(판단)"는 사람 몫.

## 수단 매핑 — 무엇을 무엇으로

게이트를 통과한 규칙을 어느 도구로 내릴지:

- TypeScript 타입·컴파일 옵션 → `tsconfig.json` (strict, noUncheckedIndexedAccess, exactOptionalPropertyTypes 등)
- JS/TS 코드 패턴 → ESLint 룰 (no-restricted-syntax, no-restricted-imports, 커스텀 룰 등)
- CSS/SCSS 패턴 → Stylelint 룰
- 커밋 메시지 형식·범위 → commitlint 룰
- 파일 구조·금지 경로 → Husky hook·정책 hook·CI lint
- 마크다운·문서 형식 → markdownlint, prettier, remark 룰

도구로 잡히는 것은 도구로, 의미·맥락 판단이 필요한 것만 프롬프트 규칙으로 남긴다. 양쪽 다 필요한 경우(도구가 부분만 잡는 경우) 보완 관계를 명시한다.
