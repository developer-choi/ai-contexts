# 문서 작성 톤

PR body, 안내문 등 **외부에 공개되는 문서**에 적용합니다.

---

## 1. 기본 톤

- **습니다체** 사용 (합니다, 됩니다, 있습니다)
- **반말 금지** — 한다, 이다, 그렇다 등 사용하지 않음
- **하세요** 금지 — 명령형 존댓말은 사용하지 않음. 단, **~해 보세요**(권유형)는 허용

## 2. 모든 문장은 독자에게 역할이 명확해야 함

평서문을 나열하지 않습니다. 독자가 "이걸 왜 나한테 말하는 거지?"라고 느끼면 실패입니다.

모든 문장은 다음과 같은 역할을 가져야 합니다:
- **안내**: 독자에게 해보라고 유도 ("클릭해 보세요", "확인할 수 있습니다")
- **조건/상황 설명**: 전제를 깔아줌 ("진입하신 다음,", "진입한 경우,")
- **근거/이유**: 왜 이런지 설명 ("~하기 때문입니다", "~한 결과입니다")

**❌ Bad**

```
목록에서 스크롤을 내립니다.
이 페이지로 진입합니다.
Network 탭을 엽니다.
링크를 클릭합니다.
```

→ 보고서처럼 읽힘. 독자와 연결이 없음.

**✅ Good**

```
5페이지 이상 데이터를 쌓고, Network 탭을 연 뒤 링크를 클릭해 보세요.
리페칭 없이 1페이지부터 새로 로드됩니다.
```

→ 독자에게 해보라고 유도하고, 결과를 알려줌. 문장마다 역할이 명확함.

## 3. 기술 용어와 사실 관계는 정확하게 사용

동작의 주체나 역할을 추측하여 서술하지 않습니다. 확인된 사실만 적고, 모르는 부분은 주체를 특정하지 않는 표현을 사용합니다.

"~이 필요하다", "~해야 한다", "~를 써야 한다" 같은 단정도 사실 여부를 확인합니다. 구현에 따라 달라질 수 있는 것을 하나의 방식으로 단정하면 오류입니다.

**Bad**

```
CDN이 `?s={size}&f=webp` 쿼리스트링으로 리사이징을 지원하므로
```

→ CDN은 캐싱/배포 역할이고, 리사이징은 뒤의 이미지 처리 서비스가 담당할 수 있습니다. 프론트엔드 코드만으로는 리사이징 주체를 알 수 없습니다.

**Good**

```
CDN URL에 `?s={size}&f=webp` 쿼리스트링을 붙이면 리사이즈된 WebP 이미지를 받을 수 있으므로
```

→ 주체를 특정하지 않고, 프론트엔드에서 확인할 수 있는 사실(URL에 쿼리를 붙이면 리사이즈된 이미지가 온다)만 서술합니다.

## 4. 문장 반려 시 여러 시안 제시

사용자가 문장을 반려하면, 단일 수정안이 아니라 여러 시안을 가져와서 선택하게 합니다.

## 5. 링크 참조 시 내용 요약 먼저

링크를 걸기 전에 맥락이나 내용을 먼저 요약합니다. 독자가 링크를 클릭하지 않아도 핵심을 파악할 수 있어야 합니다.

**❌ Bad**

```
[에러 처리는 왜 필요한가?](링크)에서 정리했듯이, 예방이 더 중요합니다.
```

→ 링크를 읽어야만 맥락을 알 수 있음. 불친절함.

**✅ Good**

```
에러 처리는 과거에 고민을 마친 상태이니 확인해주시면 감사하겠습니다 — [에러 처리는 왜 필요한가?](링크)
```

→ 내용이 먼저 전달되고, 링크는 근거 자료로 뒤에 위치함.

## 6. 외부 문서에 내부 맥락 금지

외부에 공개되는 문서에 내부 프로젝트 약어나 내부 작업 이력을 넣지 않는다.

## 7. 리뷰 시 모든 문장의 이해도를 점검

문장을 읽었을 때 한 번에 이해가 안 되면 이슈로 올립니다. 톤 규칙의 형식적 항목(습니다체, 하세요 금지 등)만 통과하면 OK가 아닙니다. "이 문장이 독자에게 한 번에 전달되는가?"가 최종 기준입니다.

## 8. 문서 작업 시 양식을 먼저 확정

내용을 채우기 전에 문서의 구조(섹션 순서, 제목 체계, 래핑 구조)를 먼저 확정합니다. 양식이 확정되지 않은 상태에서 내용을 리뷰하면 구조 변경 시 작업이 낭비됩니다.

## 9. 톤 리뷰 시 원본을 직접 확인

원본이나 레퍼런스 문서가 존재하면 반드시 직접 읽고 톤을 대조합니다. 다른 에이전트가 제공한 맥락 요약("내부 문서이므로 한다체 허용" 등)만으로 톤을 판단하지 않습니다.
