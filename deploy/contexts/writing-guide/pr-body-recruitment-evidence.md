# PR 본문 작성 구조 — 증거형

성능 개선, 렌더링 최적화 등 **측정 가능한 결과를 증거로 보여주는 PR**에 적용합니다.

> 예시: [렌더링 전략 최종 선택: Streaming](https://github.com/developer-choi/monorepo-playground/pull/6)

---

## 1. 요약은 결론만

요약에는 최종 기술 선택과 핵심 구현만 나열합니다. 배경 설명 없이 "이 PR이 뭘 했는지"만 적습니다.

---

## 2. 개선 섹션 반복: 증거 → 설명 → 의사결정

각 개선 포인트마다 아래 구조를 반복합니다:

1. **Before/After 비교표** — 스크린샷, Network Waterfall, Timing 등 시각 자료를 테이블로 배치
2. **커밋 링크 + 개선 내역** — 무엇을 바꿨고, 구체적으로 어떤 효과가 있는지
3. **의사결정 과정** — 왜 이 방식을 선택했는지, 트레이드오프는 무엇인지

독자가 비교표만 봐도 "개선됐구나"를 체감하고, 텍스트에서 "왜 이렇게 했는지"를 이해할 수 있어야 합니다.

---

## 3. 시각 자료는 대비를 극대화

**Bad**

> 스크린샷 하나만 첨부하고 "개선되었습니다"

**Good**

> | SSR Blocking | Streaming |
> |---|---|
> | (스크린샷) | (스크린샷) |
>
> 수치 테이블:
> | | Before | After |
> |---|---|---|
> | 서버 응답 대기 | **1.04s** | **10.87ms** |

→ 동일 조건에서의 Before/After를 나란히 배치하여 차이를 즉시 인지할 수 있게 합니다.

---

## 4. 수치에는 측정 조건을 명시

"1초 개선"만 적으면 신뢰할 수 없습니다. 어떤 조건에서 측정했는지를 함께 적습니다.

**Bad**

```
Streaming 적용 후 응답이 빨라졌습니다.
```

**Good**

```
API에 1초 딜레이를 추가한 상태에서 측정했습니다.
Blocking은 서버 응답 대기 1.04s, Streaming은 10.87ms.
```
