# 구조 교정 사례

AI가 작성한 문장과 사용자가 확정한 문장을 기록합니다.
같은 실수를 반복하지 않기 위한 참고 자료입니다.

---

## 목적 먼저, 구현 나중에

**Bad (AI)**

```
구조: mode 옵션 설명 → 코드 → 데모 → "왜 onSubmit인가"
소개: "react-hook-form의 useForm()은 mode 옵션으로 에러 노출 시점을 결정합니다."
라벨: onChange, onBlur, onSubmit
```

- 라이브러리 기능 설명부터 시작 → "뭘 왜 하는 건지" 파악 전에 "어떻게"가 먼저 나옴
- 기술 용어(onChange 등)로 라벨링 → 목적이 아닌 구현 관점

**Result (사용자 확정)**

```
구조: UX 관점(에러 시점 3가지) → 라이브러리 해결법 → 데모 → 코드 → 한계
소개: "에러를 보여주는 시점은 크게 세 가지입니다."
라벨: 입력 시점, 포커스가 빠진 시점, 제출 시점
```

- 목적(언제 에러를 보여줄 것인가)을 먼저 설명하고, 구현(mode 옵션)은 다음에
- UX 관점의 라벨 → 기술 용어 없이도 의미 전달

## 독자가 추론할 수 있는 결론 삭제

> 관련: tone.md "독자가 이미 알거나 추론할 수 있는 내용은 쓰지 않는다"

**Bad (AI)**

```
제출 버튼을 누르는 건 사용자 스스로 "다 채웠다"고 판단한 시점입니다.
이때 에러를 보여주는 게 가장 자연스럽습니다.
```

```
정답은 없지만, 라이브러리가 기본값을 onSubmit으로 설정한 데는 동의합니다.
대부분의 폼에서 가장 자연스러운 선택입니다.
```

- "다 채웠다고 판단한 시점"이라고 설명했으면 "자연스럽다"는 결론은 독자가 스스로 내릴 수 있음
- AI가 결론까지 달아주면 설교조가 됨
- "대부분의 폼에서 가장 자연스러운 선택입니다" — 직전에 "정답은 없지만"이라고 했으면서 "가장"이라고 단정하면 모순

**Result (사용자 확정)**

각각 해당 문장 삭제.

- 설명이 충분하면 결론은 독자의 몫
- "가장" 같은 단정은 "정답은 없지만"과 모순

## 불필요한 디테일/요소 삭제

> 관련: tone.md "독자가 이미 알거나 추론할 수 있는 내용은 쓰지 않는다"

**Bad (AI)**

```
react-hook-form은 제출 시 첫 번째 에러 필드로 자동 포커스하고
```

```
추천 조합: mode: 'onSubmit' + reValidateMode: 'onChange' (둘 다 기본값).
제출 전까지 에러를 보여주지 않아 편안한 입력 경험을 제공하되,
에러 발생 후에는 수정할 때마다 즉시 피드백합니다.
```

```
// MODE_CODE 10줄+: mode, reValidateMode, 옵션별 동작 주석 전부 나열
```

**Result (사용자 확정)**

```
react-hook-form은 제출 시 에러 필드로 자동 포커스하고
```

```
(추천 조합 Callout 삭제)
```

```
// MODE_CODE 3줄
const { register, handleSubmit } = useForm({
  mode: 'onSubmit', // 'onSubmit' | 'onBlur' | 'onChange'
});
```

- "첫 번째" — 불필요한 디테일. 핵심은 "에러 필드로 포커스"
- 추천 조합 Callout — 각 섹션에서 이미 설명한 내용의 반복
- 코드 스니펫 — 독자가 알아야 할 핵심만 남기고 나머지 삭제

## 독자가 이해할 수 없는 내부 구조 서술

> 관련: tone.md "독자가 이미 알거나 추론할 수 있는 내용은 쓰지 않는다"

**Bad (AI)**

```
개별 props를 명시적으로 내부 요소에 전달 + `{...inputProps}` bag으로 추가 props를 후순위 spread
```

```
`Select.js` — `React.cloneElement`로 `OutlinedInput`에 `inputComponent={SelectInput}` 주입
```

- 특정 라이브러리(MUI)의 내부 구조를 알아야만 이해할 수 있는 문장
- 독자가 MUI 소스를 읽어본 적이 없으면 의미 전달 불가

**Result (사용자 확정)**

해당 섹션 전체 삭제.

- 선택지 A/B의 코드 예시와 장단점만으로 결정 근거가 충분했음
- 이미 설득된 논점에 외부 라이브러리 권위를 빌리는 것은 불필요한 디테일

## 목적 먼저, 수단 나중 (섹션 순서)

**Bad (AI)**

```
섹션 순서: 에러 클래스(수단) → 에러 문구(UX) → 에러 잡는 구조(목적)
```

**Result (사용자 확정)**

```
섹션 순서: 에러 잡는 구조(목적) → 에러 문구(UX) → 에러 클래스(수단)
```

- 독자는 "왜"를 먼저 알아야 "어떻게"에 납득합니다.
- 수단·구현 세부사항이 먼저 나오면 목적을 파악하기 전에 피로해집니다.
- 섹션 내 문장 순서뿐 아니라 섹션 배치 순서도 이 원칙을 따릅니다.

## 괄호 내 용어 나열로 주절이 묻힘

> 관련: tone.md "괄호 안에 긴 용어 나열 금지"

**Bad (AI)**

```
`eslint src/` 전체 실행은 type-aware 규칙(`no-unnecessary-condition`, `restrict-template-expressions` 등)이 크로스파일 영향을 받을 수 있기 때문입니다.
```

- "type-aware 규칙이 크로스파일 영향을 받는다"가 한 번에 이해되지 않음
- 괄호 안에 긴 규칙명을 나열하면서 주절의 의미가 묻힘

**Result (사용자 확정)**

```
`eslint src/` 전체 실행은 type-aware 규칙 때문입니다. `no-unnecessary-condition`, `restrict-template-expressions` 등은 타입 정보를 기반으로 판단하므로, 다른 파일의 타입이 바뀌면 staged 파일만으로는 정확하게 검사할 수 없습니다.
```

- 주절에서 핵심 의미("type-aware 규칙 때문")를 먼저 전달
- 구체적 규칙명과 인과관계를 별도 문장으로 분리

## 한 문단에 논점 3개 — 문단 분리 + 접속사

> 관련: tone.md "한 문단 = 한 논점"

**배경**: README 기술적 의사결정 섹션. 서버 프리페치 설명.

**Bad (AI)**

```
클라이언트에서 API를 호출하면 HTML → JS → API 순서로 대기가 발생합니다. 서버에서 첫 페이지 데이터를 미리 가져와 이 대기를 제거했습니다. API가 느려도 스켈레톤을 즉시 전송하고 데이터가 준비되면 점진적으로 채우는 Streaming 방식을 채택했습니다.
```

- 문제 → 해결 → 추가 전략이 한 문단에 뭉쳐 있어 한 번에 읽히지 않음
- 문장 간 관계(인과, 추가)가 암묵적

**Result (사용자 확정)**

```
클라이언트에서 API를 호출하면 HTML → JS → API 순서로 대기가 발생합니다. 따라서, 서버에서 첫 페이지 데이터를 미리 가져와 이 대기를 제거했습니다.

추가로, API가 느려도 스켈레톤을 즉시 전송하고 데이터가 준비되면 점진적으로 채우는 Streaming 방식을 채택했습니다.
```

- 논점 전환 지점에서 문단 분리
- "따라서,", "추가로," 접속사로 문장 간 관계 명시

## 비교 나열을 쉼표로 연결 — 리스트로 분리

> 관련: tone.md "비교/나열은 리스트로"

**배경**: README. API 1초 고정 제약에 대한 로딩 전략 선택지 비교.

**Bad (AI)**

```
실시간 검색 미리보기는 타이핑할 때마다 1초를 기다려야 하므로 입력이 반영된 건지 알 수 없고, 이전 데이터를 유지하는 방식은 1초간 화면이 전혀 변하지 않아 검색이 실행된 건지 피드백이 없습니다. 검색 실행 즉시 스켈레톤을 보여주어 "로딩 중"이라는 피드백을 바로 제공하는 방식을 선택했습니다.
```

- 선택지 2개 + 결론이 한 문장에 쉼표로 연결
- 어디서 끊어 읽어야 하는지 불분명

**Result (사용자 확정)**

```
1. 실시간 검색 미리보기는 타이핑할 때마다 1초를 기다려야 하므로 입력이 반영된 건지 알 수 없고,
2. 이전 데이터를 유지하는 방식은 1초간 화면이 전혀 변하지 않아 검색이 실행된 건지 피드백이 없습니다.
3. 따라서, 검색 실행 즉시 스켈레톤을 보여주어 "로딩 중"이라는 피드백을 바로 제공하는 방식을 선택했습니다.
```

- 각 선택지를 리스트로 분리하여 개별 비교 가능
- 마지막 항목에 "따라서,"로 결론 연결

## 전제+목적+비교가 한 문장에 — 볼드 분리 + 문단 나누기

> 관련: tone.md "한 문단 = 한 논점"

**배경**: PR 본문. DDD 폴더 구조 선택 이유.

**Bad (AI)**

```
라포랩스 담당자님이 위에서부터 폴더를 탐색할 때, "이 로직은 어떤 도메인인가?" 한 가지 질문만으로 파일을 찾을 수 있게 하려고 했습니다. FSD 같은 구조는 Layer/Slice/Segment 규칙을 먼저 알아야 탐색이 가능한데, 이 규모에서는 러닝커브 대비 이점이 적다고 판단했습니다.

**Result (사용자 확정)**

```
**라포랩스 담당자님이 위에서부터 폴더를 탐색할 때**, 러닝커브 없이 파일을 찾을 수 있게 하려고 했습니다.

```

- 핵심 전제를 볼드로 강조하여 독자의 시선을 잡음
- 대안 비교는 별도 문단으로 분리
- 불필요한 중간 설명 삭제, 직접적 표현으로 압축

## 서술 사례와 코드 예제의 주체 불일치

> 관련: tone.md "리뷰 시 모든 문장의 이해도를 점검"

**배경**: 기술 필기. 일반 주장을 펼치는 서술 뒤에 코드 예제가 따라오는 흐름.

**Bad (AI)**

```
초기 배치를 잘못 하면 나중에 해체 비용이 큽니다. 실제로 langdy-admin의 outro 타입이 그랬습니다. 메인 챕터 파일에 타입이 정의되어 있고 문법·발음 도메인이 그걸 직접 import하는 구조였습니다.

```ts
// article-types.ts
export interface ArticleContent { ... }

// notice-logic.ts (남의 도메인)
import type { ArticleContent } from '../article/article-types';
```
```

- 서술은 langdy-admin outro 사례인데 뒤따르는 코드는 Article·Notice·FAQ — 주체가 다름
- 독자가 "사례 얘기 따로, 예제 얘기 따로"로 맥락을 잃음

**Result (사용자 확정)**

```
초기 배치를 잘못 하면 나중에 해체 비용이 큽니다. 예를 들어 형제 도메인 중 하나에 타입이 정의되어 있고 나머지가 그걸 직접 import하는 구조를 생각해 봅니다. Article·Notice·FAQ 세 도메인으로 단순화하면 다음과 같습니다.

```ts
// article-types.ts
export interface ArticleContent { ... }

// notice-logic.ts (남의 도메인)
import type { ArticleContent } from '../article/article-types';
```
```

- 서술을 일반화하여 코드 예제와 주체 일치
- "세 도메인으로 단순화하면 다음과 같습니다" 연결 문구로 서술→예제 흐름 명시

## 결론 → 디테일 → 자랑 순서 위반

> 관련: tone.md "읽는 순서 = 이해 순서"

**배경**: 이력서 한줄소개 2번 항목. 첫 bullet에 "어떻게 구현했고 코드는 n줄"이라는 자랑부터 등장. 독자는 "뭘 한 건지(Sentry 연동)" 결론을 끝에서야 받음.

**Bad (AI)**

```
2. 에러의 체계적 관리
    - beforeSend() 훅 하나와 커스텀 에러 클래스 필드만 추가하는 n줄 규모의 최소 구조로 Sentry를 연동했습니다. 동료는 Sentry 스펙을 몰라도 기존 throw 흐름만 유지하면 모든 에러의 단서가 자동 수집됩니다.
    - 원인 기반 커스텀 에러 클래스 n개와 4단계 우선순위 정책을 도입하여, Sentry 이슈 98%를 해결했습니다.
    - 페이지 n개·버튼 n개에서 발생하는 에러를 n줄의 공통 로직으로 처리했습니다.
```

- 첫 bullet이 "n줄 규모의 최소 구조" 자랑으로 시작 — 독자는 "뭘 한 건지(Sentry 연동)"를 끝에서야 만남
- 첫 bullet 한 문장에 "어떻게(beforeSend 추가) + 규모(n줄) + 무엇(Sentry 연동) + 효과(자동 수집)"가 한꺼번에 들어가 무거움

**Result (사용자 확정)**

```
2. 에러의 체계적 관리
    - Sentry를 도입해, 동료가 Sentry 스펙을 몰라도 기존 throw 흐름만 유지하면 모든 에러의 단서가 자동 수집되도록 했습니다.
    - 원인 기반 커스텀 에러 클래스 n개와 4단계 우선순위 정책을 도입하여, Sentry 이슈 98%를 해결했습니다.
    - 페이지 n개·버튼 n개에서 발생하는 에러를 n줄의 공통 로직으로 처리했습니다.
    - beforeSend() 훅 하나와 커스텀 에러 클래스 필드, 총 n줄만으로 위 모든 효과가 동작했습니다.
```

- 첫 bullet = "무엇을 했는가/어떤 효과인가"(Sentry 도입 + 자동 수집)
- 마지막 bullet = "구현 규모/자랑"(n줄)
- 위에서 아래로 읽으면 결론 → 디테일 → 자랑 순서가 자연스러움

## 문서 첫머리 메타 독자 안내 섹션

> 관련: tone.md "메타 독자 안내 섹션 금지"

**배경**: 히스토리 문서 H1 바로 아래에 `## 이 문서를 읽는 독자에게` 메타 헤더를 달고 "~을 정리한다" 류 메타 종결 문장으로 문단을 마감.

**Bad (AI)**

```

## 상위 구조와 중복되는 시간 디테일

> 관련: tone.md "독자가 이미 알거나 추론할 수 있는 내용은 쓰지 않는다"

**배경**: 히스토리 문서에서 섹션 헤더에 이미 `(2025-10-29)` 날짜가 있는데 본문에서 다시 "같은 날 오후"로 시각까지 세분화.

**Bad (AI)**

```
### 1단계 — 신규 생성 차단 (2025-10-29)

(생략)

**(b) "+따라읽기" 버튼의 생성 대상 교체**

같은 날 오후, `+따라읽기` 버튼 자체는 남겨두되 클릭 시 만들어지는 객체의 타입을 영&따로 교체했습니다.
```

- 섹션 헤더가 이미 날짜를 담고 있음 — 본문의 "같은 날 오후"는 상위 구조와 중복
- 히스토리 문서에서 "그날 오전·오후"까지의 시간 세분화는 독자에게 무의미한 디테일

**Result (사용자 확정)**

```

(생략)

**(b) "+따라읽기" 버튼의 생성 대상 교체**
