# 프로세스 교정 사례

AI가 작성한 문장과 사용자가 확정한 문장을 기록합니다.
같은 실수를 반복하지 않기 위한 참고 자료입니다.

---

## 문서 성격에 맞는 제목

**배경**: 폼 에러 피드백 예제 페이지. 기술 레퍼런스가 아니라 UX 관점에서 "왜 이렇게 해야 하는가"를 설명하는 에세이 형태의 문서.

**Bad (AI)**

```
1. Validation Mode
```

- 기술 레퍼런스/API 문서라면 기능명 제목이 맞음
- 하지만 이 문서는 독자에게 UX 판단 근거를 설명하는 에세이 — 기능명을 나열하면 매뉴얼이 됨

**Result (사용자 확정)**

```
1. 유효성검증은 언제 해야 하는가?
```

- 독자의 질문을 제목으로 → 읽는 목적과 일치
- 문서 성격(에세이/설명문)에 맞는 질문형 제목

---

## 내용보다 양식이 먼저

> 관련: tone.md "문서 작업 시 핵심 메시지 → 양식 → 내용 순서"

**배경**: AI가 문서 내용 리뷰를 먼저 진행하고, 여러 라운드의 수정을 거친 뒤 사용자가 양식(구조/제목 체계)을 변경하여 작업이 낭비됨.

**Bad (AI)**

```
1. 내용 리뷰 (톤, 전방 참조, 라인번호 등 여러 라운드)
2. 양식 논의 (사용자가 "당연히 처음부터지"라고 지적)
3. 양식 확정 후 재작성
```

- 양식이 바뀌면 내용 배치도 바뀌므로, 양식 확정 전 내용 리뷰는 낭비
- 사용자가 지적하기 전까지 순서 문제를 인식하지 못함

**Result (사용자 확정)**

```
1. 양식 확정 (섹션 순서, 제목 체계, 래핑 구조)
2. 내용 작성/리뷰
```

- 양식을 먼저 확정하고 내용을 채움
- 확정한 양식을 전달할 때 제목 체계, 래핑 구조 등 형식 요소도 명시

---

## 핵심 메시지 파악 없이 형식 리뷰에 진입

> 관련: tone.md "문서 작업 시 핵심 메시지 → 양식 → 내용 순서"

**배경**: Turbopack/Webpack 환경 차이 대응 문서. 사용자의 핵심 메시지는 "silent failure가 근본 문제"였으나, AI는 이를 파악하지 못하고 톤·중복·부록 분리 같은 형식 리뷰부터 진행. 사용자가 직접 양식을 재배치한 뒤에야 따라감.

**Bad (AI)**

```
1. 톤 리뷰 (습니다체, 평서문 나열 등 형식 이슈 5건 제출)
2. 구조 제안 ("알려진 차이"를 부록으로 분리)
3. 사용자가 "내가 집중하고 싶은 건 silent failure 그 자체"라고 지적
4. 사용자가 직접 양식 재배치 (기존 배경 → 최초 발견 → 근본 문제 → 집중할 문제)
5. AI가 뒤따라 말투만 정리
```

- 문서가 독자에게 전달하려는 핵심 메시지를 파악하지 않고 형식부터 리뷰
- "내용보다 양식이 먼저"의 상위 패턴: 양식조차도 핵심 메시지를 먼저 파악해야 제안할 수 있음
- 결과적으로 AI의 구조 제안(부록 분리 등)이 사용자의 최종 양식과 달라 작업이 낭비됨

**Result (사용자 확정)**

```
1. 핵심 메시지 파악 ("이 문서가 독자에게 뭘 전달하려는가")
2. 핵심 메시지에 맞는 양식 제안 (섹션 순서, 제목 체계)
3. 양식 확정 후 톤·분량 리뷰
```

- 핵심 메시지를 모르면 양식도 제안할 수 없음
- "양식 먼저"를 한 단계 더 확장: 양식 전에 핵심 메시지가 먼저

## 삭제 섹션 부활

> 관련: templates/pr-body/recruitment-narrative.md

**상황**: 사용자가 초안에서 "배경", "AppProvider", "sass+디자인토큰" 섹션을 삭제한 뒤 피드백을 남김. 메인이 재작성하면서 삭제된 섹션을 다시 추가함.

**Bad (AI)**

```
재작성 시 사용자가 삭제한 섹션("배경", "AppProvider", "sass+디자인토큰")을 다시 포함
```

**Result (사용자 확정)**

```
삭제한 섹션은 재작성에서도 제외
```

- 사용자가 삭제한 섹션은 "이 내용이 불필요하다"는 명시적 의사입니다.
- 재작성 시 삭제된 섹션을 부활시키지 않습니다.
