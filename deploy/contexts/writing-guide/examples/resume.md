# 이력서 경력기술서 교정 사례

AI가 작성한 문장과 사용자가 확정한 문장을 기록합니다.
같은 실수를 반복하지 않기 위한 참고 자료입니다.

---

## 방법(how) vs 성과(outcome) 구분

> 관련: resume-guide.md "방법 vs 성과 구분"

**배경**: "AI 하네스 엔지니어링" 키워드가 필수로 들어가야 한다는 피드백을 받고, 성과 섹션에 키워드를 넣으려다 사용자 교정 받음. 하네스 엔지니어링은 **방법**이지 성과가 아님.

**Bad (AI)**

```
성과: AI 하네스 엔지니어링 구조 확립. QA당 7건이던 오류가 0건으로 감소
```

- 방법 서술("하네스 엔지니어링 구조 확립")이 성과 섹션에 섞임
- 성과의 역할은 수치·효과 중심인데 방법 설명이 앞을 차지

**Result (사용자 확정)**

```
해결: 작업과 리뷰를 서로 다른 에이전트에 맡겨, 이슈 0건이 될 때까지 AI끼리 검증·재작업이 오가는 AI 하네스 엔지니어링 구조 도입
성과: QA당 평균 7건이던 오류가 0건으로 감소
```

- "AI 하네스 엔지니어링"을 해결에 배치 (방법)
- 성과는 수치만 (결과)

## 시제 혼재 조건화

> 관련: tone.md "시제는 문서 유형에 일관되게 적용"

**배경**: 이력서 한줄소개 bullet에서 시제가 섞임(`담아뒀습니다(과거) / 구조입니다(현재) / 해결했습니다(과거)`). "bullet 내 시제 통일"을 단순 규칙으로 적용하려다, bullet을 합쳐 "정의 → 사용" 흐름으로 묶을 때는 시제 변화가 자연스럽다는 맥락을 놓침.

**Bad (AI, 1차 판단 — 과잉 일반화)**

세 bullet이 시제 섞여 있으니 모두 과거형으로 통일해야 한다고 강하게 주장.

**Result (사용자 확정)**

앞의 두 bullet을 **한 bullet 두 문장**으로 묶어 "정의 → 사용" 흐름으로 전환. 이때는 시제 변화가 한 호흡 안에서 자연스럽게 읽힘.

```
- 커스텀 에러 클래스에 request·response 같은 디버깅 단서를 필드로 담아뒀습니다. 이 객체가 throw되면 beforeSend() 훅이 필드를 꺼내 Sentry 이벤트에 자동으로 붙이는 구조입니다.
```

**규칙 조건화**

- bullet이 **독립 단위**면 시제 통일 적용 (과거형 일관)
- bullet을 합쳐 **한 호흡의 정의 → 사용 흐름**으로 묶으면 시제 변화 허용 (과거 완료 → 현재 상태 서술)

## 디테일은 해결에서 성과로 옮기기

> 관련: resume-guide.md "수치·디테일은 성과에 모으기"

**배경**: 해결 섹션에 `62개 페이지·54개 버튼·266줄 공통 로직` 같은 구현 디테일을 박아놓으니 해결이 빽빽해지고 성과는 허전해짐. 사용자가 "62·54·266을 성과에 보내고, 해결은 담백하게"로 교정.

**Bad (AI)**

```
해결: 원인·해결 방법·탈출 수단·에러 코드 4가지 UX 정의 + 페이지 62개·버튼 54개에서 발생하는 모든 에러를 266줄의 공통 로직으로 처리
성과: 운영팀이 개발팀 호출 없이 원인 파악 가능
```

- 해결에 수치 3종(62, 54, 266)이 몰려 빽빽
- 성과가 정성적 서술만 있어 어필이 약함

**Result (사용자 확정)**

```
해결: 에러 화면에 원인·해결 방법·탈출 수단·에러 코드 4가지가 모두 드러나도록 UX를 먼저 정의하고, 에러가 발생할 수 있는 모든 경로를 266줄의 공통 로직으로 처리하는 7단계 전략으로 체계화
성과: 페이지 62개·버튼 54개에 걸친 모든 에러를 일괄 대응, 운영팀이 개발팀 호출 없이 화면만으로 원인 파악 가능
```

- 페이지·버튼 수치를 성과로 옮김 → 성과에 구체 규모감
- 해결은 방법 중심으로 담백

## 평범한 단어 회피 — "구축" → "도입"

> 관련: resume-guide.md "방법 vs 성과 구분" (방법 단어 선택)

**배경**: 제목에 "구축"을 쓰면 "시스템을 만들었다"는 평범한 신호만 전달됨. 사용자가 "구축은 어필하고 싶지 않아"라고 교정 → 구체적 동작·효과를 담는 단어로 교체.

**Bad (AI)**

```
## Sentry 구축 및 이슈 98% 해결
```

- `구축`은 "시스템 설치"의 평범한 서술
- 이미 있던 Sentry를 쓸만하게 만든 것이므로 `구축`보다 `도입`이 정확

**Result (사용자 확정)**

```
## Sentry 도입 및 이슈 98% 해결
```

- `도입`으로 "가져와 자리 잡게 했다"는 행위가 명확
- 이력서 제목은 평범한 단어를 피하고 구체 행위·효과를 드러냄

## 주제 간 재료 중복 방지

> 관련: resume-guide.md "시안 간 재료 중복 방지"

**배경**: 여러 주제를 시리즈로 다듬을 때, 한 주제에서 쓴 수치·키워드를 다른 주제에도 또 써버리면 어필이 희석됩니다. 주제별 관점을 나눠 재료를 배분합니다.

**Bad (AI)**

한 세션에서 02 `Sentry 도입` 성과에도 `페이지 62개·버튼 54개·266줄 공통 로직`을 쓰고, 03 `에러 처리 전략` 성과에도 같은 수치를 사용.

- 같은 수치가 두 주제에 반복되어 "한 항목이 다른 항목을 베낀 것 같은" 인상
- 각 주제의 차별화된 어필 포인트를 잃음

**Result (사용자 확정)**

주제별 관점을 분리:

- 02 Sentry = **개발팀 관점 트래킹 효율** (이슈 98% 감소, throw 자동 수집)
- 03 에러 처리 전략 = **사용자·운영팀 관점 에러 피드백** (62·54·266줄 일괄 대응)

- 수치 재료(62·54·266)는 03에만 배치
- Sentry 관련 재료(98%, beforeSend, 에러 클래스 수)는 02에만 배치
- 주제 관점을 먼저 나누면 재료도 자연스럽게 나뉨
