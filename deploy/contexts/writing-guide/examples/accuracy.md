# 내용 정확성 교정 사례

AI가 작성한 문장과 사용자가 확정한 문장을 기록합니다.
같은 실수를 반복하지 않기 위한 참고 자료입니다.

---

## 구현에 따라 달라지는 것을 단정

> 관련: tone.md "기술 용어와 사실 관계는 정확하게 사용"

**Bad (AI)**

```
하지만 TextAreaField는 `<textarea>`, SelectField는 `<button>`이 필요해서
```

- SelectField의 내부 요소가 `<button>`이어야 한다고 단정
- 실제로는 라이브러리에 따라 `<button>`일 수도 `<input>`일 수도 있음
- 같은 문서의 다른 곳에서 "라이브러리마다 다름"이라고 적어놨으므로 모순

**Result (사용자 확정)**

```
하지만 TextAreaField는 `<textarea>`가 필요하고, SelectField도 구현에 따라 `<button>` 등 `<input>`이 아닌 요소가 들어갈 수 있어서
```

- "필요하다"를 "들어갈 수 있다"로 변경 — 단정이 아닌 가능성
- 문서 내 다른 서술과 모순 없음

## 외부 라이브러리 근거 인용 시 실제 구조 확인

> 관련: tone.md "기술 용어와 사실 관계는 정확하게 사용"

**배경**: InputBase의 children 주입 방식(방법 2)의 근거로 Radix SelectTrigger 코드를 인용. 하지만 Radix Select는 베이스 컴포넌트를 공유하는 구조가 아니라 아예 독립 구현이므로, "children 주입"의 근거가 아님.

**Bad (AI)**

```
Radix도 Select 계열에서는 이 방향을 택하고 있습니다. SelectTrigger는 TextField와 공유하는 컴포넌트가 하나도 없고, 내부 폼 요소로 `<button>`을 사용합니다:
(SelectTrigger 코드)
```

- Radix Select는 TextField와 베이스를 공유하지 않으므로, "이 방향을 택했다"는 근거가 성립하지 않음
- 구조 자체가 다른 코드를 같은 선택지의 근거로 인용하면 오해를 줌

**Result (사용자 확정)**

해당 문단 삭제. 대신 방법 4(번외)로 Radix의 독립 구현 방식을 별도 섹션에 소개.

- 외부 라이브러리를 근거로 인용할 때, 그 라이브러리가 실제로 해당 방법을 쓰는지 확인
- 구조가 다른 경우 근거가 아니라 별도 방식으로 분류

## PR body 요약에서 내용을 부정확하게 축약

> 관련: tone.md "기술 용어와 사실 관계는 정확하게 사용"

**Bad (AI)**

```
처음에는 화면마다 필터 조합이 다를 것을 대비해 팩토리 훅 방식으로 설계했습니다.
```

- "필터 조합이 다를 것"은 원문의 의미를 축소 — 실제로는 디자인, 종류, 로직 전반이 달라질 것을 대비한 것

**Result (사용자 확정)**

```
처음에는 화면마다 필터 디자인과 종류, 로직이 달라질 것을 대비해 팩토리 훅 방식으로 설계했습니다.
```

- 원문의 범위를 정확하게 반영
- 요약하더라도 의미 범위를 축소하지 않는다

## AI 맥락에서 "학습" 대신 "분석"

> 관련: tone.md "기술 용어와 사실 관계는 정확하게 사용"

**배경**: 이력서·기술 문서에서 AI에게 코드를 읽히는 작업을 설명할 때, `학습`이라고 적으면 기계학습(모델 훈련)을 연상시켜 실제 작업 내용을 왜곡합니다.

**Bad (AI)**

```
AI로 라이브러리 원본 코드를 학습해 공통 동작을 도출
```

- `학습`은 모델을 추가로 훈련시킨다는 뉘앙스를 포함
- 실제로는 AI가 기존 지식과 프롬프트 맥락으로 코드를 읽고 분석한 것이지, 모델 가중치를 갱신한 것이 아님

**Result (사용자 확정)**

```
AI로 라이브러리 원본 코드를 분석해 공통 동작을 도출
```

- `분석`으로 실제 작업(읽고 해석함)을 정확히 서술
- 모델 훈련과의 혼동 방지

## 원본 수치의 실제 의미를 근거 문서로 검증

> 관련: tone.md "기술 용어와 사실 관계는 정확하게 사용", resume-guide.md "수치 검증"

**배경**: 이력서 경력기술서 `성과`에 원본 수치를 그대로 가져올 때, 원본 표현이 실제 의미를 정확히 반영하지 않을 수 있습니다. 근거 자료(enrich 문서·PR·커밋)와 대조가 필요합니다.

**Bad (AI)**

```
성과: 1건당 평균 7회 발생하던 수정 지시가 0회로 감소
```

- 원본 이력서 한 줄을 그대로 수용
- 실제 `7`은 "수정 지시 횟수"가 아니라 "QA에서 발견된 오류 건수"였음 (근거 문서 확인 결과)
- 면접에서 "수정 지시가 왜 7회씩?" 같은 질문을 받으면 실제 의미와 어긋남

**Result (사용자 확정)**

```
성과: QA당 평균 7건이던 오류가 0회로 감소
```

- 근거 문서와 대조해 "QA 오류 건수"로 의미를 정확히 표현
- 수치 뒤의 사실을 한 번 더 검증하는 루프는 이력서 작업에서 특히 중요 (resume-guide.md "수치 검증" 참조)

## 비교 서술의 차별점이 실제로는 양쪽에 모두 성립

> 관련: tone.md "기술 용어와 사실 관계는 정확하게 사용"

**배경**: 두 방식(`<label>` + `htmlFor` vs `<div>` + `handleClick`)의 장단점을 비교하는 ADR. 방법 2의 "장점"으로 `DOM 순서 무관`을 꼽았으나, 방법 1도 `htmlFor` + `id`로 DOM 순서와 무관하게 타겟 지정이 가능하므로 차별점이 아님.

**Bad (AI)**

```
- **포커스 타겟을 소비자가 자유롭게 지정 가능 — DOM 순서 무관.** `inputRef`가 가리키는 대상이 DOM의 어느 위치에 있든 포커스가 그쪽으로 갑니다. leading에 select가 먼저 와도 `inputRef`가 메인 input을 가리키면 그쪽으로 위임됩니다.
```

- "DOM 순서 무관"이 방법 2의 고유 이점인 것처럼 서술
- 실제로는 방법 1도 `htmlFor` + `id`로 동등하게 DOM 순서 무관. "장점"으로 비교에 올리면 독자에게 잘못된 차별점을 심음

**Result (사용자 확정)**

```
- **포커스 타겟을 소비자가 자유롭게 지정 가능.** `inputRef`가 가리키는 대상이 DOM의 어느 위치에 있든 포커스가 그쪽으로 갑니다. 다만 이 유연성은 방법 1도 `htmlFor` + `id`로 동등하게 얻을 수 있고, 배선이 더 간단한 쪽은 label입니다.
```

- 유연성 자체는 사실이므로 유지하되, 양쪽에 동등하게 성립하는 점을 솔직하게 명시
- 단정조 "고유 장점 없음" 대신 "동등 + 배선은 label이 간단" 수준으로 톤 완화
- 비교 서술에서 "A만의 장점"을 올릴 때는 실제로 B에서는 안 되는지 직접 검증해야 함
