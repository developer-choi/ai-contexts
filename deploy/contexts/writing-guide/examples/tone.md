# 톤 교정 사례

AI가 작성한 문장과 사용자가 확정한 문장을 기록합니다.
같은 실수를 반복하지 않기 위한 참고 자료입니다.

---

## "덜어주는 UX"

> 관련: tone.md "리뷰 시 모든 문장의 이해도를 점검"

**Bad (AI)**

```
마우스 클릭이나 모바일 터치 한 번을 줄여 사용자의 불편함을 덜어주는 UX입니다.
```

```
페이지에 진입하자마자 바로 타이핑할 수 있는 것이 좋은 UX라고 생각합니다.
```

- "불편함을 덜어주는 UX" — 의미가 모호하고 주어-목적어 관계가 어색함
- "좋은 UX라고 생각합니다" — 뭘 말하려는 건지 전달이 안 됨

**Result (사용자 확정)**

```
이런 작은 불편함을 줄여주는 것이 좋은 UX 중 하나라고 생각합니다.
```

- "불편함을 줄여주는 것" — 동작이 명확
- "좋은 UX 중 하나" — 단정이 아닌 의견으로 톤 조절

## 한계 분석 후 개인 의견 추가

**Bad (AI)**

```
onBlur/onChange는 필드를 하나하나 채워 넘어가므로 이 문제가 원천적으로 없습니다.
(끝 — 개인 의견 없이 분석만으로 마무리)
```

- 한계를 지적만 하고 끝나면 "그래서 어쩌라고?"가 됨
- 필자의 입장이 빠져 있음

**Result (사용자 확정)**

```
onBlur/onChange는 필드를 하나하나 채워 넘어가므로 이 문제가 원천적으로 없습니다.
정답은 없지만, 라이브러리가 기본값을 onSubmit으로 설정한 데는 동의합니다.
```

- 분석 → 개인 의견으로 마무리
- "동의합니다" — 단정이 아닌 개인 입장

## 명령조 → 개인 의견

> 관련: tone.md "기본 톤"

**Bad (AI)**

```
isValid로 버튼을 비활성화하면 사용자가 뭘 고쳐야 하는지 알 수 없습니다.
항상 활성화하고, 제출 시 에러 피드백으로 안내하세요.
```

- "안내하세요" — 명령조. 필자가 독자에게 지시하는 톤
- "isValid로" — UX 설명인데 기술 용어가 먼저

**Result (사용자 확정)**

```
제출 버튼을 조건이 안 맞다고 비활성화시켜 버리면 사용자는 뭘 고쳐야 하는지 알 수 없습니다.
실제로 부모님 세대에서 "제출 버튼이 왜 안 눌리냐"고 문의를 주시는 경우가 있습니다.
버튼은 항상 활성화하고, 제출 시 에러 피드백으로 안내하는 게 좋다고 생각합니다.
```

- "비활성화시켜 버리면" — 사용자 관점의 표현
- "좋다고 생각합니다" — 명령이 아닌 개인 의견
- 실제 경험 근거 추가

---

## 비개발자 독자 용어 순화

> 관련: tone.md "~화 계열 한자어 회피"

**Bad (AI)**

```
Coding-Standards Reviewer만 N명이 병렬로 리뷰하고, 나머지 단계 간은 순차입니다.
```

**Good (사용자 확정)**

```
컨벤션 검증만 여러 리뷰어가 동시에 수행하고, 나머지 단계는 순서대로 진행됩니다.
```

- "N명", "병렬", "순차"는 개발자에게는 명확하지만 비개발자에게는 의미 불분명
- 같은 뜻을 일상 용어로 바꾸면 정보 손실 없이 접근성이 올라감

## 과장된 비유 표현

> 관련: tone.md "자기 산출물 과장 금지"

**Bad (AI)**

```
props 폭발 — `placeholder`, `autoComplete`, `type` 등 `<input>` 속성을 InputBase가 중계해야 함
```

- "폭발"은 과장된 비유로, 구체적으로 무엇을 뜻하는지 독자마다 다르게 해석
- 문서 전체가 차분한 한다체인데 이 부분만 과장 표현

**Result (사용자 확정)**

```
props가 많아짐 — `placeholder`, `autoComplete`, `type` 등 `<input>` 속성을 InputBase가 중계해야 함
```

- 과장 없이 사실 그대로 서술
- 문서 톤과 일관

---

## 결론부 톤 불일치 — 단답 결정 vs 분석 톤

> 관련: tone.md "결론/결정 문장도 본문 톤과 일치"

**배경**: ADR 문서. 본문에서 장단점을 분석하는 톤으로 진행했는데, 결론만 일방적 결정 톤으로 마무리.

**Bad (AI)**

```
**방법 2(children)로 진행합니다.**
```

- 본문 전체는 "~할 수 있습니다", "~가 필요해집니다" 같은 분석 톤
- 결론만 "진행합니다"로 단답 결정 — 어감이 끊김
- 필자가 일방적으로 결정을 내려버리는 톤

**Result (사용자 확정)**

```
확장성과 API 단순함을 고려하면 **방법 2(children)가 적합하다고 판단했습니다.**
```

- 판단 근거를 한 줄 깔고 결정
- "판단했습니다" — 단정이 아닌 개인 의견 톤
- 본문의 분석 톤과 일관

## ~화 용어 순화

**Bad (AI)**

```
에러 처리 이원화 설계: Error 클래스(개발자/Sentry용) + getErrorInfo(사용자 노출용) 분리
```

**Result (사용자 확정)**

```
커스텀 에러 클래스 도입으로 에러 발생 시 개발자와 사용자 각각에게 필요한 정보를 전달
```

- "이원화"는 독자에게 생소한 한자어. 구체적으로 풀어씁니다.
- ~화 계열 한자어는 쉬운 동사·명사로 대체합니다. (tone.md "~화 계열 한자어 회피" 참고)

## README에 코드 레벨 용어 사용

> 관련: readme-guide.md

**배경**: 채용과제 README의 "기술적 의사결정" 섹션. 필터 재사용성 구조를 설명하는 문맥.

**Bad (AI)**

```
과잉 추상화로 판단하여, `enableFilters`로 필터를 ON/OFF하는 방식으로 단순화했습니다.
```

- `enableFilters`는 코드 변수명 — README 독자(채용담당자)가 이해할 수 없음
- README는 비개발자도 읽는 문서 (readme-guide.md 참고)

**Result (사용자 확정)**

```
과잉 추상화로 판단하여, 일부 필터를 통째로 활성화/비활성화할 수 있는 옵션을 만드는 방식으로 단순화했습니다.
```

- 코드 용어 없이 동작을 일상 용어로 설명
- 정보 손실 없이 접근성 향상

## 예제 프로젝트에서 "유효함을 확인"

> 관련: tone.md "자기 산출물 과장 금지", "독자가 이미 알거나 추론할 수 있는 내용은 쓰지 않는다"

**Bad (AI)**

```
이후 브랜드 화면에서 설정만 바꿔 재사용하며 이 구조가 유효함을 확인했습니다.
```

- 채용과제 수준의 예제 페이지에서 "구조가 유효함을 확인"은 과장
- 독자가 추론할 수 있는 결론을 AI가 달아주면 설교조

**Result (사용자 확정)**

해당 문장 삭제.

## 전문 용어를 동작으로 풀어쓰기

> 관련: tone.md "전문 용어보다 동작 설명"

**배경**: README. 필터 재사용 구조 설명.

**Bad (AI)**

```
처음에는 화면마다 필터 디자인과 종류, 로직이 달라질 것을 대비해 팩토리 훅 방식으로 설계했습니다.
```

- "팩토리 훅 방식"을 모르면 뭘 했는지 알 수 없음

**Result (사용자 확정)**

```
처음에는 화면마다 필터 디자인과 종류, 로직이 달라질 것을 대비해 중간에서 로직을 한번 더 조립하는 방식으로 설계했습니다.
```

- 패턴 이름 대신 실제 동작을 서술
- 전문 용어를 몰라도 이해 가능

## "두었으니" — 시혜적 어감

> 관련: tone.md "기본 톤"

**배경**: PR 본문에서 외부 링크를 안내하는 문장. 지원자가 채용담당자에게 보내는 문서.

**Bad (AI)**

```
각 전략의 비교는 별도로 정리해 두었으니 참고해주시면 감사하겠습니다 — [링크]
```

- "정리해 두었으니" — "준비해 두었으니 보세요"와 같은 시혜적 어감
- 지원자가 채용담당자에게 쓸 톤이 아님

**Result (사용자 확정)**

```
각 전략의 비교는 별도로 정리했었는데 참고해주시면 감사하겠습니다 — [링크]
```

- "정리했었는데" — 시혜적 뉘앙스 없이 겸양 톤
- "했으니"는 지시적 어감이 남을 수 있음

## 열거 속 반말체 잔존

> 관련: tone.md "기본 톤"

**배경**: 본문 전반을 습니다체로 전환하는 작업. 종결어만 스캔하고 열거·쉼표로 이어진 절의 서술어는 놓침.

**Bad (AI, 1차 수정 결과)**

```
이름이 같다, 스키마가 같다, 지금은 똑같다 — 공유의 근거가 아닙니다.
```

- 문장 끝 "아닙니다"는 습니다체로 바꿨지만, 앞쪽 열거 "~다"는 반말체 그대로
- 종결어만 보고 검수하면 열거 속 반말을 지나침

**Result (사용자 확정)**

```
이름이 같음, 스키마가 같음, 지금은 똑같음 — 공유의 근거가 아닙니다.
```

- 열거 절을 명사형으로 전환하여 반말체 제거
- 원문의 리듬은 유지

## 헤딩-본문 시제 어긋남

> 관련: tone.md "시제는 문서 유형에 일관되게 적용"

**배경**: 이력서 한줄소개. 본문은 과거 성과를 `~했습니다`로 서술. 헤딩만 `~합니다` 현재형이라 시제가 어긋남.

**Bad (AI)**

```
1. 성능을 최적화합니다.
    - Streaming SSR로 전환하여 TTFB를 39% 단축했습니다.

2. 에러를 체계적으로 관리합니다.
    - 원인 기반 커스텀 에러 클래스를 도입하여 Sentry 이슈 98%를 해결했습니다.
```

- 헤딩은 현재형(`~합니다`), bullet 본문은 과거형(`~했습니다`) — 한 문서에서 시제 충돌
- 이력서는 과거 성과를 서술하는 문서이므로 본문은 과거형이 자연스러움

**Result (사용자 확정)**

```
1. 성능 최적화
    - Streaming SSR로 전환하여 TTFB를 39% 단축했습니다.

2. 에러의 체계적 관리
    - 원인 기반 커스텀 에러 클래스를 도입하여 Sentry 이슈 98%를 해결했습니다.
```

- 헤딩을 명사형으로 통일 — 본문 과거형과 시제 충돌 없음
- 항목 라벨 역할에 명사형이 더 자연스러움

## 영어 기술 용어를 음차로 옮김

> 관련: tone.md "외래어 음차 회피"

**배경**: 히스토리 문서 섹션 제목에서 "Soft Deprecated" 개념을 한글 음차로 표기.

**Bad (AI)**

```
### 1단계 — 소프트 디프리케이트 (2025-10-29)
```

- "소프트 디프리케이트"는 영어 발음을 한글로 옮긴 음차 — 의미가 직관적으로 전달되지 않음
- 기술 용어를 음차로 옮기면 독자가 원문을 역추적해야 의미를 파악 가능

**Result (사용자 확정)**

```
### 1단계 — 신규 생성 차단 (2025-10-29)
```

- **원문 영어** ("Soft Deprecated") 또는 **의미 번역** ("신규 생성 차단") 중 택일
- 이 사례는 뒤이어 "생성 대상 교체", "삭제 정책 완화" 같은 우리말 조치가 이어지므로 의미 번역이 일관성 측면에서 자연스러움

## 한자어 뭉침 — "유사·파편화·명세 정의"

> 관련: tone.md "딱딱한 한자어 회피"

**배경**: 이력서 경력기술서의 `문제` 서술. 한 문장에 한자어가 여러 개 몰리면 "AI가 쓴 것처럼" 딱딱하게 읽히고, 일상어로 풀 수 있는 지점을 놓침.

**Bad (AI)**

```
페이지마다 유사 컴포넌트가 파편화되고, 명세 정의에도 시간 소모
```

- `유사`·`파편화`·`정의`가 한 문장에 몰려 독자가 숨 돌릴 틈이 없음
- 외부 공개 문서(이력서·README)에서 한자어 밀도가 높으면 거리감이 생김

**Result (사용자 확정)**

```
페이지마다 비슷한 컴포넌트가 여기저기 중복되고, 디자이너와 정책을 바닥부터 세우느라 시간이 오래 걸림
```

- `유사` → `비슷한`, `파편화` → `여기저기 중복`, `명세 정의` → `정책을 바닥부터 세우느라`
- 일상어로 풀어도 의미 손실 없음
