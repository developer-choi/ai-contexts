# README 작성 가이드

README는 채용담당자(비개발자일 수 있음)가 읽는 문서입니다. GitHub 프로필에서 핀된 레포를 클릭해서 들어오는 사람이 대상입니다.

톤은 tone.md를 따르되, 1인칭 서술("제가 일하는 방식을", "제가 실무에서")을 사용합니다.

## 성격과 1차 소스

README는 그 프로젝트나 스킬의 SKILL.md(또는 그에 준하는 핵심 문서)가 정의한 내용을, 비개발자가 알아볼 수 있게 번역한 대표 창구입니다.

- 1차 소스는 SKILL.md입니다. 최근 커밋에서 무엇이 바뀌었는지(diff)가 아니라, SKILL.md가 정의한 현재 모습을 기준으로 씁니다. README는 변경 이력을 모아두는 곳이 아닙니다.
- SKILL.md에 이미 있는 내용은 사용자에게 되묻지 않고 번역합니다. 사용자 확인이 필요한 빈칸을 최소화합니다.
- 자랑하거나 파는 문서가 아닙니다. 무엇을 만들었고 무엇을 하는지 담백하게 소개합니다. 어필 포인트를 앞세우거나 과장하지 않습니다. (담백하게 쓰는 구체 기준은 tone.md를 따릅니다.)

## 전체 흐름 개요

여러 단계나 구성요소로 이뤄진 프로젝트·스킬의 README에는 전체 그림을 한눈에 보여주는 개요를 둡니다. 비개발자가 이 한 토막만 읽어도 "무엇을 넣으면 무엇이 나오는지" 감이 와야 합니다.

- 입력에서 결과까지를 흐름으로 보여줍니다. 단순하면 화살표 한 줄로(예: 기획서 → 배경 파악 → 설계 → 구현 → PR), 분기·병렬이 있으면 위에서 아래로 흐르는 세로 그림으로 갈라지는 구조를 드러냅니다. 자동 생성 막대 차트(간트 등)는 진행 시간만 보일 뿐 이 구조가 드러나지 않으므로 쓰지 않습니다.
- 가지 기호는 한글 텍스트 뒤가 아니라 공백 들여쓰기 뒤에만 오게 합니다. 단계 이름은 각자 제 줄에 두고, 세로줄(`│`)·갈림(`├──→` `└──→`)·아래 화살표(`▼`)는 줄 앞 같은 칸(예: 공백 2칸)에 맞춥니다. 한글 뒤에 가지를 붙이는 가로 계단형은 한글 너비(2칸)를 세어 맞춰야 해서 어긋나기 쉬우므로 쓰지 않습니다.
- 한 그림에 모든 단계·분기·반복을 욱여넣지 않습니다. 핵심 병렬 구조만 남기고 나머지는 본문으로 풉니다. 예시:

```
배경 파악 · PR 분할
  │
  ├──→ 화면 마크업  (PR 작업과 동시에 진행)
  │
  ▼
PR마다  설계 → 구현·리뷰 → 본문 작성
  │
  ▼
완성된 PR
```
- SKILL.md의 의존성 그래프를 내부 표기(step-1.1, stub 등)째로 베끼지 않습니다. 노드에 시스템이 정의한 세션·단계 이름을 쓸 때는 옆에 일상어 설명을 붙이고(예: `BG · 배경 파악과 PR 분할`), 개발자 전용 기호는 풉니다.
- 그래프는 본문(세션 목록 등) 뒤, 문서 아래쪽에 둡니다. 세션 코드 같은 이름이 그것을 설명하는 절보다 먼저 나오면 비개발자가 막히기 때문입니다.

## 금지

- 트리 형식 다이어그램 (폴더 구조, 조직도 등). 단, 입력에서 결과까지의 흐름 개요(분기·병렬 포함)는 「전체 흐름 개요」대로 허용합니다.
- 설치 방법 (npm run, git clone 등)
- 호출·사용법 문법 (명령어 인자, 슬래시 커맨드 등). 비개발자는 실행할 일이 없습니다.
- AI 지시문 톤 ("로드 시점", "frontmatter 태그" 등)
- 파일 경로 테이블
- 정의 없이 던지는 기술 약어 (KA, AC 등). 풀어서 쓰거나 링크합니다. 단, 시스템이 정의한 세션·단계·역할 이름은 바로 옆에 일상어 설명을 붙이면 그대로 둘 수 있습니다 (예: `BG — 배경 파악과 PR 분할`).
- 코드·개발 용어 (변수명·함수명·설정값, stub·spawn 같은 작업 용어). 일상 용어로 풀어씁니다.
- 커밋 해시, git 명령어 출력, git log 스니펫. 비개발자는 의미를 알 수 없습니다.
- 개발자용 내부 문서 링크 (코드, 기술 노트 등). 비개발자 독자가 클릭해도 이해할 수 없습니다.
- 예시나 인용 블록 안의 기술 확장 괄호 부연. 핵심 흐름만 남기고 삭제합니다.
- 외국어 인용 원문 (영어 등). 비개발자가 못 읽으니 한국어로 풀거나 출처 링크만 남깁니다.

## 스킬 README인 경우

- 만든 동기를 문제와 해결 구조로 담습니다. SKILL.md에 목표·동기가 적혀 있으면 그것을 도입부 중심에 두고 「어떤 문제가 있어서 → 이렇게 푼다」 순으로 풉니다. 목표를 여러 기능 중 하나처럼 뒤로 흩뜨리지 않습니다. 독자가 '왜 이게 필요한가'를 먼저 이해해야 나머지가 읽힙니다.
- SKILL.md 링크를 하단에 둡니다: `> 자세한 내용은 [SKILL.md](SKILL.md)를 참고하세요.`
