# PR 본문 작성 구조 — 서사형

사고 과정과 서사로 설득하는 PR에 적용합니다.

> 예시: [Feature static checking](https://github.com/developer-choi/frontend-test-v2-for-tester/pull/2)

---

## 1. 서사 구조: 목표 → 로드맵 → 연결

배경 섹션은 아래 순서를 따릅니다:

1. **최종 목표** 선언 — 무엇을 달성하려는지
2. **로드맵** 제시 — 목표를 위한 단계들
3. **이미 완료된 것** 언급 — 과거 고민/작업 (링크)
4. **이번 PR에서 한 것** — 로드맵 중 어디를 실행했는지
5. **다른 목표와의 연결** — 이 작업이 다른 가치에도 기여하는 점

독자가 "이 PR이 왜 존재하는지"를 배경만 읽고 파악할 수 있어야 합니다.

---

## 2. 설정 디테일은 불필요

읽는 사람은 `parserOptions.project` 연결이나 프리셋 이름 같은 설정 방법에 관심 없습니다. **왜 이런 구조를 선택했는지**, **어떤 문제를 해결하는지**를 서술합니다.

**❌ Bad**

```markdown
### parserOptions.project 연결

ESLint는 기본적으로 AST만 보고 타입 정보를 모릅니다.
parserOptions.project로 tsconfig를 연결하면...
```

**✅ Good**

```markdown
### 왜 2단계인가?

커밋할 때마다 전체 파일을 대상으로 돌리면 작업 속도가 느려집니다.
따라서 커밋 시점에서는 변경된 파일 위주로 빠르게 검사하고...
```

---

## 3. 예시로 임팩트를 보여주기

추상적 설명보다 코드 예시로 "이걸 막았다"를 보여줍니다.

**❌ Bad**

```markdown
타입 기반 정적 분석으로 코드 품질이 향상됩니다.
```

**✅ Good**

```markdown
예를 들어, 아래와 같은 코드는 이제 커밋 자체가 불가능합니다:

// ❌ await 없이 Promise 호출 → 에러가 조용히 삼켜짐
fetchData();
```

→ 독자가 "아, 이런 걸 막아주는구나"를 코드로 바로 체감할 수 있습니다.

---

## 4. 목적은 기술적 이유가 아니라 도입 동기

각 도구·구조 섹션에서 "왜 도입했는가"를 설명할 때, 기술적 동작 방식이 아니라 도입 동기(팀에 어떤 가치를 주는가)를 서술합니다. 기술 선택이라면 대안과 비교하여 왜 이것을 골랐는지까지 포함합니다.

**❌ Bad**

```markdown
lint-staged에서 Prettier를 먼저 실행하고 ESLint --fix를 이어서 실행합니다.
이 순서로 포맷팅과 린트를 동시에 처리합니다.
```

**✅ Good**

```markdown
AI가 작성한 코드의 포매팅이 항상 통일되도록 커밋 시점에 강제합니다.
```

→ "어떻게 동작하는가"가 아니라 "팀에 어떤 문제를 해결하는가"를 씁니다.

**❌ Bad — 선택 근거 없이 결과만 기재**

```
DDD 기반 폴더 구조를 도입했습니다.
```

**✅ Good — 도입 동기 + 대안 비교**

```
담당자님이 위에서부터 폴더를 탐색할 때, "이 로직은 어떤 도메인인가?" 한 가지 질문만으로 파일을 찾을 수 있게 하려고 했습니다. FSD 같은 구조는 Layer/Slice/Segment 규칙을 먼저 알아야 탐색이 가능한데, 이 규모에서는 러닝커브 대비 이점이 적다고 판단했습니다.
```

---

## 5. 목적/의도 먼저, 수단/방법은 나중에

각 섹션 내에서도, 섹션 순서 자체도 이 원칙을 따릅니다.

- 독자는 "왜"를 먼저 알아야 "어떻게"에 납득합니다.
- 에러 처리 예시: "에러 잡는 구조(목적) → 에러 문구(UX) → 에러 클래스(수단)" 순으로 배치합니다.
- 수단·구현 세부사항이 먼저 나오면 독자가 목적을 찾기 전에 피로해집니다.

---

## 6. 사용자가 삭제한 섹션은 부활시키지 않음

사용자가 초안에서 특정 섹션을 삭제했으면, 그것은 "이 내용이 불필요하다"는 명시적 의사입니다. 재작성 시 해당 섹션을 다시 추가하지 않습니다.

---

## 7. 결론은 강조

섹션의 핵심 메시지는 bold로 강조합니다. 독자가 스캔하듯 읽어도 결론을 놓치지 않게 합니다.

```markdown
AI가 이런 코드를 작성하더라도 커밋 시점에 자동으로 차단되므로,
리뷰어가 이런 항목까지 눈으로 확인할 필요가 없습니다.
**코드 작성 속도와 리뷰 속도가 모두 올라갑니다.**
```
