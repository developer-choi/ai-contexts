# decision-guide

기술적 의사결정 문서(`type: decision`) 전용 작성·검토 규칙입니다. `write-init`에서 `type: decision`으로 입력받아 `write-refine`이 다듬을 때 함께 로드됩니다.

여러 구현 방법을 비교하고 선택 근거를 남기는 문서입니다. 결론만 단독으로 읽어도 "왜 다른 방법이 탈락했는지" 파악되어야 합니다.

## 비교 구조 강제 규칙

뼈대 단계와 다듬기 단계 모두에서 다음 3가지를 검증합니다. 위반 시 보강을 요청합니다.

- **방법 2개 이상.** `### 방법 1.`, `### 방법 2.` 헤딩이 최소 2개. 1개면 "비교 대상이 하나면 의사결정 문서로 성립하지 않습니다. 방법 2 이상 추가해주세요"로 보강 요청. 주력 2개 + `### 방법 N. (번외) [이름]` 형태의 보조 비교는 허용하며, (번외) 라벨도 방법 개수에 포함합니다.
- **각 방법에 장점·단점 둘 다 채움.** `**장점:**` `**단점:**` 블록이 누락·공란인 방법이 있으면 보강 요청.
- **결론에 선택 이유와 비선택 이유가 모두 드러남.** 형식은 자유(단락·서브섹션·bullet). 결론 섹션만 단독으로 읽어도 "왜 다른 방법이 탈락했는지" 파악되어야 합니다. "방법 N 선택" 명시만 있고 탈락 사유가 본문·서브섹션·bullet 어디에도 없으면 보강 요청.
- **사용자가 직접 채움.** AI는 장단점·탈락 사유 블록을 임의로 채우지 않습니다. 비어 있으면 사용자에게 항목별로 질문해서 받아 적습니다. 사용자가 명시적으로 "AI가 초안 채워라"고 요청한 경우에만 AI가 채우되, 사용자 검토·승인을 받아야 확정합니다.

## 서술 규칙

### 감수한 단점은 정당화하지 않는다

채택안의 단점을 감수하기로 했으면, 그 단점이 "작다 / 문제없다"고 변명하지 않습니다. 감수 사실과 근거(이번 요구 범위 밖)만 적습니다. 단점을 작아 보이게 만드는 서술은 자기합리화이며, 다음 사람이 "정말 작은가"를 재검증하게 만듭니다.

**before** — 감수한 단점을 손해가 작다고 변명:
> `<label>` 중복이라는 시맨틱 비용은 감수합니다. 시각적·기능적 문제를 일으키지 않고, 두 `<label>`이 같은 input을 가리키므로 a11y 손해는 크지 않다고 봅니다.

**after** — 감수 사실 + 요구 범위 밖이라는 근거만:
> `<label>` 중복이라는 시맨틱 비용은 감수합니다. 시맨틱 정합성은 이번 요구사항이 아니기 때문입니다.

### 비선택 방법의 비용은 기능으로 적고 구현을 단정하지 않는다

탈락한 방법의 비용을 설명할 때 특정 구현을 유일 정답처럼 "~해야 한다"로 박지 않습니다. 필요한 기능만 서술하고, 구체적 구현은 예시(예: …)로 낮춥니다. 다른 구현 여지가 있는데 하나로 단정하면 비교의 공정성이 깨집니다.

**before** — 특정 가드 구현을 필수로 단정:
> `e.target === e.currentTarget` 가드까지 포함해 InputBase에 얹혀야 padding 클릭과 children 내부 클릭이 구분됩니다.

**after** — 기능만 서술, 구현은 예시로:
> padding 클릭과 children 내부 클릭을 구분하는 처리도 함께 들어갑니다(예: 클릭 대상이 컨테이너 자신인지 확인).
