# 문장 교정 사례

AI가 작성한 문장과 사용자가 확정한 문장을 기록합니다.
같은 실수를 반복하지 않기 위한 참고 자료입니다.

---

## 사례 1: "덜어주는 UX"

**Bad (AI)**

```
마우스 클릭이나 모바일 터치 한 번을 줄여 사용자의 불편함을 덜어주는 UX입니다.
```

```
페이지에 진입하자마자 바로 타이핑할 수 있는 것이 좋은 UX라고 생각합니다.
```

- "불편함을 덜어주는 UX" — 의미가 모호하고 주어-목적어 관계가 어색함
- "좋은 UX라고 생각합니다" — 뭘 말하려는 건지 전달이 안 됨

**Result (사용자 확정)**

```
이런 작은 불편함을 줄여주는 것이 좋은 UX 중 하나라고 생각합니다.
```

- "불편함을 줄여주는 것" — 동작이 명확
- "좋은 UX 중 하나" — 단정이 아닌 의견으로 톤 조절

## 사례 2: 문서 성격에 맞는 제목

**배경**: 폼 에러 피드백 예제 페이지. 기술 레퍼런스가 아니라 UX 관점에서 "왜 이렇게 해야 하는가"를 설명하는 에세이 형태의 문서.

**Bad (AI)**

```
1. Validation Mode
```

- 기술 레퍼런스/API 문서라면 기능명 제목이 맞음
- 하지만 이 문서는 독자에게 UX 판단 근거를 설명하는 에세이 — 기능명을 나열하면 매뉴얼이 됨

**Result (사용자 확정)**

```
1. 유효성검증은 언제 해야 하는가?
```

- 독자의 질문을 제목으로 → 읽는 목적과 일치
- 문서 성격(에세이/설명문)에 맞는 질문형 제목

## 사례 3: 목적 먼저, 구현 나중에

**Bad (AI)**

```
구조: mode 옵션 설명 → 코드 → 데모 → "왜 onSubmit인가"
소개: "react-hook-form의 useForm()은 mode 옵션으로 에러 노출 시점을 결정합니다."
라벨: onChange, onBlur, onSubmit
```

- 라이브러리 기능 설명부터 시작 → "뭘 왜 하는 건지" 파악 전에 "어떻게"가 먼저 나옴
- 기술 용어(onChange 등)로 라벨링 → 목적이 아닌 구현 관점

**Result (사용자 확정)**

```
구조: UX 관점(에러 시점 3가지) → 라이브러리 해결법 → 데모 → 코드 → 한계
소개: "에러를 보여주는 시점은 크게 세 가지입니다."
라벨: 입력 시점, 포커스가 빠진 시점, 제출 시점
```

- 목적(언제 에러를 보여줄 것인가)을 먼저 설명하고, 구현(mode 옵션)은 다음에
- UX 관점의 라벨 → 기술 용어 없이도 의미 전달

## 사례 4: 독자가 추론할 수 있는 결론 삭제

**Bad (AI)**

```
제출 버튼을 누르는 건 사용자 스스로 "다 채웠다"고 판단한 시점입니다.
이때 에러를 보여주는 게 가장 자연스럽습니다.
```

```
정답은 없지만, 라이브러리가 기본값을 onSubmit으로 설정한 데는 동의합니다.
대부분의 폼에서 가장 자연스러운 선택입니다.
```

- "다 채웠다고 판단한 시점"이라고 설명했으면 "자연스럽다"는 결론은 독자가 스스로 내릴 수 있음
- AI가 결론까지 달아주면 설교조가 됨
- "대부분의 폼에서 가장 자연스러운 선택입니다" — 직전에 "정답은 없지만"이라고 했으면서 "가장"이라고 단정하면 모순

**Result (사용자 확정)**

각각 해당 문장 삭제.

- 설명이 충분하면 결론은 독자의 몫
- "가장" 같은 단정은 "정답은 없지만"과 모순

## 사례 5: 한계 분석 후 개인 의견 추가

**Bad (AI)**

```
onBlur/onChange는 필드를 하나하나 채워 넘어가므로 이 문제가 원천적으로 없습니다.
(끝 — 개인 의견 없이 분석만으로 마무리)
```

- 한계를 지적만 하고 끝나면 "그래서 어쩌라고?"가 됨
- 필자의 입장이 빠져 있음

**Result (사용자 확정)**

```
onBlur/onChange는 필드를 하나하나 채워 넘어가므로 이 문제가 원천적으로 없습니다.
정답은 없지만, 라이브러리가 기본값을 onSubmit으로 설정한 데는 동의합니다.
```

- 분석 → 개인 의견으로 마무리
- "동의합니다" — 단정이 아닌 개인 입장

## 사례 6: 명령조 → 개인 의견

**Bad (AI)**

```
isValid로 버튼을 비활성화하면 사용자가 뭘 고쳐야 하는지 알 수 없습니다.
항상 활성화하고, 제출 시 에러 피드백으로 안내하세요.
```

- "안내하세요" — 명령조. 필자가 독자에게 지시하는 톤
- "isValid로" — UX 설명인데 기술 용어가 먼저

**Result (사용자 확정)**

```
제출 버튼을 조건이 안 맞다고 비활성화시켜 버리면 사용자는 뭘 고쳐야 하는지 알 수 없습니다.
실제로 부모님 세대에서 "제출 버튼이 왜 안 눌리냐"고 문의를 주시는 경우가 있습니다.
버튼은 항상 활성화하고, 제출 시 에러 피드백으로 안내하는 게 좋다고 생각합니다.
```

- "비활성화시켜 버리면" — 사용자 관점의 표현
- "좋다고 생각합니다" — 명령이 아닌 개인 의견
- 실제 경험 근거 추가

## 사례 7: 불필요한 디테일/요소 삭제

**Bad (AI)**

```
react-hook-form은 제출 시 첫 번째 에러 필드로 자동 포커스하고
```

```
추천 조합: mode: 'onSubmit' + reValidateMode: 'onChange' (둘 다 기본값).
제출 전까지 에러를 보여주지 않아 편안한 입력 경험을 제공하되,
에러 발생 후에는 수정할 때마다 즉시 피드백합니다.
```

```
// MODE_CODE 10줄+: mode, reValidateMode, 옵션별 동작 주석 전부 나열
```

**Result (사용자 확정)**

```
react-hook-form은 제출 시 에러 필드로 자동 포커스하고
```

```
(추천 조합 Callout 삭제)
```

```
// MODE_CODE 3줄
const { register, handleSubmit } = useForm({
  mode: 'onSubmit', // 'onSubmit' | 'onBlur' | 'onChange'
});
```

- "첫 번째" — 불필요한 디테일. 핵심은 "에러 필드로 포커스"
- 추천 조합 Callout — 각 섹션에서 이미 설명한 내용의 반복
- 코드 스니펫 — 독자가 알아야 할 핵심만 남기고 나머지 삭제

## 사례 8: 비개발자 독자 용어 순화

**Bad (AI)**

```
Convention Reviewer만 N명이 병렬로 리뷰하고, 나머지 단계 간은 순차입니다.
```

**Good (사용자 확정)**

```
컨벤션 검증만 여러 리뷰어가 동시에 수행하고, 나머지 단계는 순서대로 진행됩니다.
```

- "N명", "병렬", "순차"는 개발자에게는 명확하지만 비개발자에게는 의미 불분명
- 같은 뜻을 일상 용어로 바꾸면 정보 손실 없이 접근성이 올라감
