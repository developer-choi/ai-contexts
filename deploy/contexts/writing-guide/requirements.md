# 요구사항 문서 작성 가이드

요구사항 문서의 목적은 **무엇이 되어야 하는가**를 정의하는 것이다. 테스트 케이스와 목적이 같다 — 읽는 사람이 구현 없이도 "통과/실패"를 판단할 수 있어야 한다.

## 금지: 구현 방법 서술

요구사항에 구현 방법(how)을 적지 않는다. 어떻게 만들지는 구현자가 결정할 일이다.

### Bad

> 드래그 시 `useMotionValue`로 progress를 계산하고, `useTransform`으로 scale과 borderRadius에 연결한다. pan progress에 비례하여 `dimOpacity.set()`으로 즉시 반영한다.

### Result

> 아래로 드래그할수록 오버레이가 축소되고 모서리가 둥글어진다. 백드롭은 드래그 진행도에 비례하여 밝아진다.

---

### Bad

> `clip-path: inset(top right bottom left round 16px)` → `inset(0px 0px 0px 0px round 0px)`로 확장하며, `DOMRect`를 `triggerRect` prop으로 받아 `toInset()` 헬퍼로 변환한다. opacity 0→1 동시 진행.

### Result

> 열릴 때 트리거 버튼의 위치와 크기에서 시작하여 전체화면으로 확장된다. 닫힐 때 원래 버튼 위치로 오므라든다.

---

## 실제 문서 예시

FullScreenOverlay 애니메이션 요구사항 중 일부.

```markdown
## A1. 열기

- 오버레이는 사용자가 탭한 버튼의 위치와 크기에서 시작하여 전체화면으로 확장됩니다.
- 확장되면서 동시에 점점 불투명해집니다.

## A3. 드래그 피드백

- 아래로 드래그할수록 오버레이가 축소되고 모서리가 둥글어집니다.

## A6. 백드롭

- 열기: 배경이 서서히 어두워집니다.
- 드래그 중: 드래그 진행도에 비례하여 배경이 밝아집니다.
- 드래그 복귀: 배경이 열린 상태의 밝기로 서서히 되돌아옵니다.
```

## 올바른 요구사항의 조건

1. **관찰 가능한 동작**만 서술한다 — 유저나 QA가 눈으로 확인할 수 있는 것
2. **통과/실패를 판단**할 수 있어야 한다 — "부드럽게"처럼 모호한 표현은 피한다
3. **왜(why)는 가능하면 포함**한다 — 엣지케이스 판단 근거가 됨
4. 구현 기술(라이브러리, 함수명, 변수명)을 언급하지 않는다
