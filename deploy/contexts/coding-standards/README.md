# 코딩 스탠다드

개발 과정에서 반복적으로 마주치는 판단들을 문서화한 기준 모음입니다.

## 어떤 기준을 다루고 있나요

다음과 같은 주제에 대한 기준을 정리하고 있습니다.

- **React 컴포넌트 패턴** — 컴포넌트 설계, props 구조, 상태 관리 방식
- **TypeScript 타입 설계** — 타입 안전성과 유지보수성을 고려한 패턴
- **코드 품질 원칙** — 가독성, 유지보수성, 테스트 용이성에 대한 기준
- **테스트 작성 규칙** — 테스트 범위, 작성 방식, 네이밍 컨벤션
- **Git 워크플로우** — 커밋 메시지, 브랜치 전략, PR 작성 기준
- **HTML/CSS 마크업** — 시맨틱 마크업, 스타일링 패턴

## 상황에 맞는 기준 적용

모든 기준을 모든 프로젝트에 일률적으로 적용하지는 않습니다.

회사 프로젝트에서는 팀원 누구나 동의할 수 있는 범용적인 기준만 적용하고, 개인 프로젝트에서는 네이밍, 폴더 구조, 테스트 커버리지 등 더 세밀한 개인 기준까지 적용합니다. 컨텍스트에 따라 적절한 수준의 기준을 선택하는 것도 엔지니어링 판단의 일부라고 생각합니다.

## 왜 코딩 스탠다드를 문서화하나요

코딩 스탠다드는 단순한 규칙 목록이 아니라, 반복된 경험에서 얻은 엔지니어링 판단을 체계화한 것입니다. "이 상황에서는 이렇게 하는 게 낫다"는 결론에 도달하기까지의 맥락과 근거를 함께 기록합니다.

이렇게 문서화하면 새로운 상황에서도 일관된 판단을 내릴 수 있고, 팀에 합류하는 사람에게 "왜 이렇게 하는지"를 설명하기도 수월해집니다.
