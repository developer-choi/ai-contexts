# TypeScript Advanced Patterns

## 안전한 타입 단언 (`satisfies` vs `as`)

### 원칙: `satisfies` 우선 사용
타입을 확정하면서도 원래 값의 구체적인 정보(Literal Type 등)를 유지하고 싶을 때, `as` 대신 `satisfies`를 우선적으로 검토해야 합니다.

**❌ Bad (Type Assertion)**
- `as`를 쓰면 컴파일러가 타입을 맹신하게 되어, 잘못된 속성이나 오타를 잡아내지 못할 수 있음.

**✅ Good (satisfies)**
- 타입 호환성은 체크하되, 실제 값의 타입 정보를 유지함.

### `as` 사용 시 검토 프로세스
코드에서 `as`를 사용하거나 발견했을 때, 다음 단계를 반드시 따르세요.

1.  **대체 가능성 확인**: `satisfies`로 변경했을 때 동일한 목적을 달성할 수 있는지 확인합니다.
    - 가능하다면 → 사용자에게 `satisfies`로 변경을 제안합니다.
2.  **의도 확인 (Double Check)**: `satisfies`로 대체가 불가능하여 `as`가 꼭 필요한 상황(예: 외부 라이브러리 타입 오버라이딩 등)으로 보인다면,
    - 사용자에게 **"여기에 `as`를 사용했는데, 정말 의도하신 것이 맞나요?"** 라고 1회 되묻습니다.

## `as unknown as` 절대 금지

`as unknown as T` 이중 단언 패턴은 **어떠한 경우에도** 사용하지 않습니다. 타입 시스템이 두 타입의 호환을 거부한다는 신호를 강제로 무력화하는 행위이며, 런타임 오류의 직접 원인이 됩니다.

코드를 작성하다 `as unknown as`가 필요하다고 느낀다면, 그것은 작성 중인 코드가 잘못되었다는 신호로 간주합니다. 다음 대안을 **반드시** 찾아 적용합니다.

- 타입 정의 자체를 수정 (서버 응답 스키마, 라이브러리 타입 보강, 제네릭 추가)
- 런타임 검증으로 타입 좁히기 (zod parse, 사용자 정의 타입 가드)
- 데이터 변환 함수를 명시적으로 작성 (값을 새로 빚어서 목표 타입을 만족)
- 설계 변경 (호환되지 않는 두 타입을 억지로 잇지 않도록 경계 재설정)

대안 탐색 없이 `as unknown as`를 그대로 두지 않습니다. 외부 라이브러리 한계 등으로 어떤 대안도 불가능하다고 판단되면, 코드에 추가하기 전에 사용자에게 상황과 시도한 대안을 보고하고 명시적 승인을 받습니다.
