# Workflow

workflow 스킬의 각 Step에서 발견된 개선점들을 모아둔 백로그.


## [ideation] PLAN 중간산출물(step-4 stub) 꼭 필요한가

### 동기

PR_{N}_PLAN 등에서 설계 stub·중간산출물을 만드는데, 이 산출물 단계가 정말 필수인지 재검토하고 싶다 — 불필요하면 워크플로 단순화 여지.

### 핵심 생각

작성 시점 잠정 결론: 이 중간산출물이 있어야 구현 세션(PR_{N}_IMPL)이 그걸 인수인계받아 구현한다 → 세션 간 핸드오프 매개로는 필요해 보인다. 다만 stub 없이 PLAN→IMPL을 직접 잇는 경로도 의존성 그래프에 이미 있어(`PR_{N}_PLAN.step-4 stub 안 만든 경우 ──→ PR_{N}_IMPL`), "stub이 필수인 경우 vs 생략 가능한 경우"의 경계를 명확히 가를 수 있는지 검토 대상.

관련 잔여(2026-07-01): "코드표현 가능하나 stub 불가한 편집 산문전사 금지" 룰은 벤치 검증 후 반영됨(step-4 §5 조건3 — commit fd3e403). 그 룰의 **강한 버전**("trivial rename/이동이면 워크트리+PLAN/IMPL 세션 분리 자체를 생략, 그 자리 실행")은 이번에 미반영 — 조건3는 문서 산문 억제 + "그 자리 실행" 문구까지만이고, **세션·워크트리 collapse 구조 변경**은 본 항목의 "stub 생략 가능 경계"와 같은 뿌리라 여기서 함께 다룬다.


## [ideation] 커밋(PR) 순서 강제 — PR 도미노 굳이 필요한가

### 동기

workflow가 의존성 그래프·PR 도미노(`PR_{N}_PLAN → PR_{N}_IMPL → PR_{N}_WRITING → PR_{N} 머지 → PR_{N+1}_PLAN`)로 PR 처리 순서를 직렬로 강제하는데, 이 순서 강제가 모든 작업에서 굳이 필요한지 의문.

### 핵심 생각

PR 간 코드 의존이 실재할 때는 순서가 정당하지만, 독립적인 PR끼리도 도미노에 묶여 직렬화되면 병렬화 손해. "순서 강제가 의존성에서 비롯된 경우 vs 관성으로 직렬화된 경우"를 가려, 후자는 풀 수 있는지 검토.


## [draft] step-1.1 컨벤션 소스 수집 게이트 — 재현 불가로 보류

### 동기

BG step-1.1의 컨벤션 소스 수집 절차가 누락되면, 레포의 핵심 컨벤션 1차 소스를 초반에 인덱싱하지 못하고 작업 후반(또는 사용자 지적 후)에야 발견해 설계가 어긋난다. 2026-06-13 `/workflow BG 개인`("design-system 컴포넌트 전부 유닛테스트 추가") 세션에서 실제 발생: 메인 LLM이 `steps/step-1.md`를 Read했음에도 컨벤션 수집 3단계를 건너뛰고 곧장 PR1 구현 설계로 진입, 테스트 컨벤션 1차 소스 3건(`docs/patterns/setup/TestSetup.md`, `docs/patterns/testing/WhatToTest.md`, `docs/patterns/testing/TestWriting.md`)을 못 챙겨 사용자 지적 후에야 뒤늦게 수집함.

### 시도한 것과 결과 (2026-06-13)

`steps/step-1.md` 「Step 1.1 종료 — 분기점」에 `[CRITICAL] 컨벤션 수집 게이트` 문안을 추가해 (a)이름 스캔 선제안·채택 (b)"추가 자료?" 질문 (c)`conventions-index.md` 작성 3단계를 강제하려 했다. 효과를 sonnet 서브에이전트로 2회 벤치(게이트 없음 vs 있음)했으나 **변별이 안 됐다 — 게이트 없어도 안 건너뛰었고(베이스라인 통과), 게이트 넣어도 똑같이 통과.** 둘 다 정상이라 약 효과를 볼 수가 없었다.

**재현 실패 원인**: 원본 위반은 실제 메인 에이전트가 task 모멘텀에 밀려 구현으로 점프한 것인데, 벤치를 서브에이전트로 돌리니 "워크플로 따르기"로 정렬돼 애초에 그 실패를 안 일으킨다. 즉 **고칠 병(건너뛰기)이 재현 안 되니 약 효과도 측정 불가.** 게이트 작업물(`workflow-conv-gate` 브랜치, 커밋 068a7c8)은 변별 못 해 master에 안 부었고 폐기했다(브랜치 소멸).

### 보류 사유 — 버그 재현 대기

이 게이트가 막으려는 "모멘텀에 밀려 절차 건너뛰기"가 **어떤 구체적 조건에서 발생하는지** 안정적으로 재현할 하네스가 없다. 같은 문제가 실제로 다시 관측되면, 그때의 상황 조건(작업 지시·모드·대화 흐름)을 잘 기록·재현해서 게이트 효과를 다시 측정한다.

### 종료 조건

작성 시점 **2026-06-16**. 이후 한 달(≈2026-07-16) 안에 동일 "절차 건너뛰기"가 재발하지 않으면 재현 가능성이 낮다고 보고 이 백로그를 그냥 삭제한다. 재발하면 그 세션 조건으로 재현 하네스를 짜서 게이트 측정으로 진행.


## [ready] step-4·SKILL.md — PLAN을 "문서·핸드오프"로 프레이밍해 실행을 IMPL로 미루게 함

scope: global (deploy/skills/workflow/steps/step-4.md, SKILL.md, steps/step-3.md 분담표)

### 동기

PLAN 세션이 코드로 표현 가능한(=실행하면 끝나는) 작업을 "이건 IMPL 몫 아닐까"로 이연하는 근본 원인이 문서 프레임에 있다. "걱정마"류 금지를 추가하는 대신 그 프레임 자체를 걷어내야 한다.

### 기대상황 (필수)

PLAN 세션이 실행 가능한 작업을 만났을 때 망설임 없이 그 자리에서 실행한다. 이는 "IMPL 침범 걱정마"류 문구 추가가 아니라, PLAN을 "방침·문서 생성 후 다음 세션 핸드오프"로 규정한 프레임을 걷어내서 달성한다. **PLAN/IMPL 세션 분리는 무거운 작업을 위한 선택적 분해로 재정의**되고(기본 핸드오프 아님), step-4는 구현 수행을 기본으로 하되 무거우면 IMPL로 분리한다.

### 현재상태 (필수)

발생: **2026-07-01~02** workflow PR_1_PLAN(MP `apps/examples`) 세션. 순수 rename PR에서 실행 가능한 작업을 IMPL 세션으로 이연(워크트리 생성·IMPL spawn 안내)함. 이후 사용자가 "코드표현 가능한 걸 왜 문서화·핸드오프했나" 지적. **AI의 첫 수정 제안은 step-4에 "코드표현 가능한 건 지금 하라 / IMPL 침범 걱정마 / 충돌 시 PLAN 우선" 섹션을 추가**하는 것이었으나 — 사용자: "'하지마' 전에 애초에 안 하게 만들어라. 그렇게 행동하게 된 배경을 찾아 바꿔라."

근본 프레임 (문서가 PLAN을 핸드오프로 규정 — 인용 임베드):

- step-4 제목 "구현 **방침** 상세화"; intro "파생 산출물(stub 코드 + 잔존 md)을 **생성**한다" → 산출은 문서·stub
- step-4 §2 "**다음 세션에서** Lead가 팀에게 컨텍스트를 주입 … 파생 산출물을 작성" → 구현은 명시적으로 다음 세션 일
- step-4 사전 준비 "사용자에게 워크트리 디렉토리에서 **새 세션을 띄워 이어가도록 안내**"
- step-4 §5 stub 커밋 "**다음 단계의 작업 기반**이 되며"
- (뿌리) SKILL.md 세션 표: PR_{N}_PLAN 출력=decisions/overview/implementation(전부 문서), PR_{N}_IMPL="**코드 변경 + 커밋**" → 코드+커밋이 IMPL 칸에 통째 배정 → PLAN에서 실행가능 작업 만나면 "IMPL 칸이잖아"가 당연한 추론

즉 실행 이연은 성격 문제가 아니라 문서의 노동분담 프레임에서 나온 정상 추론. 관련: 본 파일 "[ideation] PLAN 중간산출물" 항목(collapse 구조 미반영 언급)과 같은 뿌리이며, 이미 반영된 §5 조건3(산문전사 금지, commit fd3e403)은 같은 뿌리의 *문서-내용* 증상만 덮음.

채점 기준 (프레임 교체가 됐다면):

- step-4가 "구현 수행, 무거우면 IMPL 분리"로 읽히고, "다음 세션이 구현/새 세션 안내/다음 단계 기반" 서술이 무거운-PR 분기로 내려감
- "IMPL 몫이라 미루지 말라"는 **금지 문구가 없어도** 프레임만으로 PLAN이 실행을 택함
- SKILL.md 세션 표가 "PLAN=문서/IMPL=코드+커밋" 경직 분담이 아니라 "IMPL은 PLAN이 못 끝낸 무거운 구현을 이어받음"으로 읽힘
