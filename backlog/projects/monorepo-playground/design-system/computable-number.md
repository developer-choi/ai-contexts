# ComputableNumber

연산 가능한 숫자 인풋 위젯. 금액·수량처럼 실제 계산에 쓰이는 값을 입력받는 케이스.

## Features

### 음수 / 양수 선택기능

양수만 가능하게할지, 음수도 가능하게할지 옵션으로 제공하되 기본값을 양수로 해야함.

- 마이너스 입력받는 곳을 본적이 한번도없음.

### 소수 입력 활성화 여부 선택기능

1. 자연수만 입력받아야하는 케이스는 소숫점 입력안되게 막아야하고, (기본)
2. 소수의 경우에는 소숫점을 딱 한번만 입력이 가능해야함. (type text로 직접 구현하는 경우)

### 옵션별 입력 가능한 문자 요약

항상 입력이 불가능한 문자 ⇒ 플러스, e, E (큰숫자표기법), 콤마  
음수 옵션이 활성화 된 경우에만 ⇒ 마이너스 입력가능  
소수 옵션이 활성화 된 경우에만 ⇒ 소수점 입력가능

### 키보드 포커스 위치

입력한 값에 키보드 포커스를 중간이나 맨 좌측으로 옮겨서 입력하더라도 키보드 포커스 위치가 유지되야함.  
1234**|**56789 << 저 자리에 대고 0을 입력을 했을 때  
12340**|**56789 << 키보드 포커스가 저기로 유지되야함.

- 내 솔루션은 저 상황에서 맨 우측으로 키보드 포커스가 이동됨.

### 최대, 최소

최대값은 기본값으로 Number.MAX_SAFE_INTEGER까지.  
최소값은

1. 음수 허용 아니면 0 (기본값)
2. 음수 허용이면 MIN_SAFE_INTEGER (기본값)

연산할 수 없는 숫자인풋은, 자리수 제한 안해도 되지만,  
연산할 수 있는 숫자는 대부분 9000조 넘을 일이 없을거라 판단. (정말 큰 단위를 쓰는 암호화폐 거래소, 은행이라 하더라도)

- [코인원](https://coinone.co.kr/exchange/trade/btc/krw)도 그렇게함.

- ㄴㄴ 이거 별로임. 자연스럽지못함. 차라리 콤마 포함해서 복사시켜주고 붙여넣어줄 때 그 인풋에서 숫자만 보이는게 맞음.

"123asd456asd789" 붙여넣기 했을 때 숫자만 반영해주기  
1.2.3.4 이런식으로 소수점 2개이상 들어간 경우도 붙여넣기 했을 때 기능이 망가지지않도록만 하기.

모바일 키패드는 숫자로 올라와야함.  
기왕이면 숫자만 있고 숫자관련 기호 다 빠져있는걸로.

- type tel 하면 inputMode tel과 같은 키패드가 올라오고,
- 소수 활성화 = inputMode decimal 쓰면되고
- 소수 비활성화 = inputMode numeric 쓰면될듯.

방향키 위아래 눌러서 +1 -1되는거 비활성화해야.

### [선택] 정상적인 값이 될 수 있는 가능성이 존재하는 경우 인터렉션을 방해하지않음.

어쨌든 결과적으로, 폼 제출했을 때

1. "001" 입력해서 기능이 망가지거나,
2. "." 입력해서 기능이 망가지면 안됨.

내 생각에는,

1. dot 하나만 입력해도 인터렉션 방해하지않는게 좋아보임. 뒤에 숫자, 앞에 숫자입력하면 1.2 이런식으로 되니까.
2. 000만 입력해도 인터렉션 방해하지않는게 좋아보임. 앞에 1 붙이면 1000 되니까.

하지만 어쨌든 무슨 라이브러리, 무슨 구현을 하더라도 결과적으로 저 케이스에서 버그만 안생기면 됨.

## [Auto Numeric 라이브러리](https://docs.autonumeric.org/)

[옵션 목록](https://docs.autonumeric.org/Documentation/configuration%20options/#options)에 의하면,  
decimalPlaces 2 하면 123.00 이런식으로 뒤에 노출되는 소수점 갯수가 조절됨.  
그래서 0으로 하면 자연수만 입력가능한 상태가 됨.  
dot 입력하면 같이 막혀버림.

minimumValue 0 하면, 음수 입력 뿐만 아니라 마이너스도 입력이 불가능한 상태가 됨.

"123asd456asd789"를 복사해서 붙여넣으면 별다른 옵션 안써도 "123,456,789"가 들어감.  
1.2.3.4 이런 소수점 여러개 들어간 값 붙여넣으면 1.2 까지만 들어감.

키보드 포커스 위치 너무 잘잡혀있고!  
이상한 의도하지않은 숫자관련 문자도 너무 잘 막혀있음.

최대제한보다 크거나, 최소제한보다 작은 값을 입력하려고 하면 아무런 반응이 없는걸로 예외처리가 됨.

modifyValueOnUpDownArrow 이거는 false로 끄면 위아래방향키로 1씩 증감되는거 막을 수 있음.

0 입력하고나서 0 또 입력하려고 하면 아무런 반응이없음. 00 입력안됨.  
dot 하나만 입력하면 0. 으로 자동으로 붙어버림. 인터렉션을 좀 조작하는게 있긴함.  
그치만 어쨌든 제출할 때는 문제 안생기게 처리됨.

- "0." 상태에서 제출하려고 Submit 버튼을 누르면 0.0으로 조정됨. (인풋에서 포커스 간 상태에서 0.을 만들고 서브밋 버튼을 누르는 순간 onBlur로 감지하는것으로 추정중)
- 하지만 "0." 상태에서 input에 포커스 유지한 상태로 단축키로 엔터 누르면 "0." 그대로 제출이 되긴함.
- 하지만 이 2가지 케이스에서 어쨌든 Number()로 변환하면 문제는 안생기기때문에 ㄱㅊ
