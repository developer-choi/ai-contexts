# 채용과제 단일레포 셋업 — 잔여

단일레포용 static-checking 설정 번들을 MP `templates/recruitment/`에 두고(eslint base+next+vite, prettier, editorconfig, gitattributes, commitlint, husky, check-file), MP 동기화 규율(CLAUDE.md 「설정 파일 변경 시 문서 동기화」·static-checking.md·.prettierignore·formatter.md)에 편입 완료. ProjectSetup.md는 이 번들을 가리키는 포인터로 교체. (기존 ProjectSetup 인라인-보완 항목 흡수 — 문서 인라인 대신 실파일 번들로 결론)

## 잔여

### /workflow PR1이 템플릿을 자동 소비하도록 배선

PR1이 `monorepo-playground/templates/recruitment/`를 복사해 셋업하도록 workflow에 명시한다.

- **비블로커**: 템플릿이 이미 존재해 수동 복사로 즉시 가용. 자동 배선은 편의 개선이지 전제 아님. (검증: PR_IMPL을 흉내 낸 에이전트가 `eslint.config.next.mjs`의 import를 보고 `base.mjs`까지 빠짐없이 골라옴 — 구조상 누락 시 `Cannot find module`로 즉시 깨져 fail-loud)
- **착수 절차**: `step-2.md`는 PR 경계 선언만 하므로 소비 지점이 아님. PR1 도미노 가이드(FOUNDATION / PR_1_PLAN / IMPL)를 읽어 "린트 셋업 = `templates/recruitment/` 복사 후 과제 범위 밖만 제거" 지시를 넣을 정확한 위치를 먼저 특정.
- **주의**: 사용자가 workflow를 병행 재설계 중일 수 있음 — 그 작업과 조율해 진행(충돌 회피).

### ts/tsx file-level disable 가드

MP `scripts/check-file-level-disable.sh`는 현재 scss/css `stylelint-disable`만 검사. ts/tsx의 `@ts-nocheck`·`eslint-disable` file-level 가드는 MP에 없어 템플릿에도 미포함(템플릿은 MP 실제값만 담는 원칙).

- **출신**: 삭제된 ProjectSetup 인라인-보완 항목의 Action 7. 동기 = shadcn·starter 데모 파일에 `@ts-nocheck`·`eslint-disable`이 깔린 채 남아도 작업 시 정리를 강제할 수단이 없던 문제.
- **착수**: MP `check-file-level-disable.sh`에 ts/tsx 분기 추가(`@ts-nocheck`·`eslint-disable` 검사, `ESLINT_INIT=1` 최초 도입 스킵). 추가 후 템플릿의 check-file 스크립트도 같이 반영.
