# 리스트(목록) 페이지 요구사항 총괄

## 요구사항

- 목록 갯수의 경우의수를 모두 반영
  - 없을 수 있는가? ⇒ 가능성을 배제할 것인가?
  - 무한개가 가능한가? ⇒ 가능성을 배제할 것인가?
  - 최대갯수가 제한되어 있는가?
- Pagination 기반의 목록페이지에서 추가 / 삭제 / 수정 / 기타설정(ex: 차단) 했을 때 피드백
- Infinite scroll 기반의 목록페이지에서 추가 / 삭제 / 수정했을 때 피드백

## 목록 갯수의 경우의수를 모두 반영

### 없을 수 있는가?

- 목록이 없습니다 (안내)
- 또는 목록 자체를 숨김
  - 배너리스트, 이벤트리스트는 이런 UI도 상당히 괜찮은 듯.

### 무한개가 가능한가?

- 일부 목록만 보여주는 기술 도입
  - [Pagination with filter & sort & search](https://docs.google.com/document/d/18uixuUU9rpzDBMUEgxfB2TBx9Tr9SApG18c1rFoaOJs/edit)
  - [Get more data](https://docs.google.com/document/d/1Ayskq-VCn6TbXAruk82xYPnI423PQ4LTkmafpCxlIhQ/edit)
  - [Infinite Scroll](https://docs.google.com/document/d/1IeMIvPc-18TKEscvuRYmktziMxieeKW_wJGB779nOXg/edit)

### 최대갯수가 제한되어 있는가?

- 최대갯수가 제한되어 있으면 그냥 목록을 모두 노출한다.
  - 관리자에서 일부만 지정해서 목록으로 노출되는 성격인 경우
  - 애초에 시스템상 상품후기를 작성할 때 첨부파일을 5개까지만 올릴 수 있다거나

### 무한개의 가능성을 배제할 것인가?

- 진행중인 이벤트 목록 페이지는, 이론상 무한개가 가능하지만 가능성을 배제하고 모든 목록을 노출해도 지장이 없다.
  - 사용자가 아닌 관리자가 등록하기 때문.

### 없을 수 있는 가능성을 배제할 것인가?

- 만약 목록을 모달의 형태로 보여주는 경우가 있다면
  - 해당 경우의 수에서 모달 자체가 안 뜨게 구현하되
  - 모달에서는 목록이 없음에 대한 경우의 수를 고려하지 않는 방식으로 구현할 수도 있다.

## 목록을 필터링 하기

목록 수가 많은 경우 라디오버튼, 셀렉트박스 같은 필터폼이나 검색폼이 있는데, 액션 후 1페이지로 보내줘야 함.

추가로, 제일 위 항목으로 스크롤 보내야 하는 경우가 있음.
