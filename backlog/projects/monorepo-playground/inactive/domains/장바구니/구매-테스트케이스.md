# 장바구니 / 구매하기 테스트케이스

## 구매하기 / 장바구니 버튼을 클릭했을 때

- 상단의 [애초에 구매 못하는지 체크 로직] 실행 (주문하기 버튼, 장바구니 버튼 모두 동일하게 실행됨)
- 나머지 유효성 검증
  - 선택된 상품이 없는 경우
    - PC: "장바구니에 담을 상품이 없습니다." 얼럿
    - Mobile: 옵션 고를 수 있는 모달 뜸
- API 호출
  - 실패 (주문하기 버튼, 장바구니 버튼 모두 동일하게 실행됨)
    - API에서 응답하는 에러 메시지 그대로 노출 (재고 초과 등)

### 성공 > 주문하기인 경우

- (사은품 재고 **있**는 경우) "사은품이 제공되는 상품입니다. 사은품은 주문서에서 선택하실 수 있습니다." 얼럿
- (사은품 재고 **없**는 경우) "사은품이 제공되는 상품이나, 현재 사은품의 재고가 0개 입니다. 주문을 계속하시겠습니까?" 얼럿
  - 여기서 사용자가 취소하면 이후 주문 / 장바구니 로직은 실행되지 않는다.
- 비회원인 경우 로그인 페이지로 이동
- 회원인 경우 주문서 페이지로 이동

### 성공 > 장바구니인 경우

- API 응답에서 동일한 상품에 대한 컨펌을 보여달라는 응답이 온 경우
  - "장바구니에 동일한 상품이 이미 존재합니다. 추가하시겠습니까?" 컨펌창 뜸
- 장바구니 이동 컨펌 물어봐야 하는 경우 "상품이 장바구니에 담겼습니다. 지금 확인하시겠습니까?" 얼럿
  - 여기서 확인하기 누르면 장바구니 페이지로 이동한다.

## 상품 데이터에 따른 구매 관련 UI 변화

- 판매상태: 품절 (최근 본 상품 / 위시리스트는 이 경우 모달 자체가 안 떠서 영향 없음)
  - 최대 할인가 0으로 표기
  - 옵션목록 삭제
- 판매상태: 판매중지 (최근 본 상품 / 위시리스트의 장바구니 담기 모달도 영향받음)
  - 옵션목록 삭제
  - 판매중지 버튼 보이고 구매/장바구니 버튼 숨김
- 옵션이 없는 경우 선택된 상품목록에 단일 목록이 나오고 수량만 조절할 수 있음
- 선택한 옵션과 수량에 따라 총 예상 결제 금액이 계산되어야 함

## 옵션목록 UI

- 이미 선택한 목록이 있는 경우 "이미 추가된 상품입니다. 주문 수량을 조정해주세요" 얼럿
- 그 외에는 하단에 선택한 옵션으로 [선택된 상품목록]에 떠야 함
