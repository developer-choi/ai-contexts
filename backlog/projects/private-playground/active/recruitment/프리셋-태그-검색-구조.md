# 프리셋 discovery를 축(파일) 기준에서 태그(개념) 기준으로

> PP = `~/WebstormProjects/main/private-playground/`. 경로는 PP 레포 상대.

## 동기

역량 축으로 프리셋을 찾을 때 파일 하나(competency.md)만 보고 "없음"으로 단정해, 두 축에 걸친 개념(UX)의 블록을 놓쳤다. 프리셋을 개념(태그) 기준으로 찾게 하면 축 분리로 놓치는 일이 사라진다.

## [ideation] 태그 기반 프리셋 검색 구조 도입 검토

핵심 아이디어: 프리셋의 검색 키를 **축(공감/역량 = 파일)**이 아니라 **개념(태그)**으로 바꾼다. 스킬 전반(recruitment 외 다른 컨텍스트 자산)에도 이 방식을 도입할지 함께 검토.

### 기대상황

- 개념(태그, 예: `ux`) 기준으로 프리셋을 찾으면 그 개념이 공감 축이든 역량 축이든 **한 번에 도달**한다. "역량 파일에 없으니 프리셋 없음" 같은 축-분리 오판이 구조적으로 불가능해진다.

### 현재상태 (발생 문제 — 재현 가능)

- **구조**: `presets/`가 축으로 갈려 있다 — `empathy.md`(공감 문단1·2, **도메인 태그**: edutech·ecommerce·ux 등), `competency.md`(역량 문단3·4, **역량 태그**: ai·error-handling·performance·search-ux 등), `closing.md`. 태그 어휘는 `presets/README.md` 레지스트리.
- **문제**: `ux`는 성격이 이중이다 — 도메인 공감이 없을 때 여는 '가치관 공감'(empathy 축)이자, 공고 요구 '역량'(competency 축). 그런데 프리셋은 `empathy.md`의 `## ux` 한 곳에만 있다. 역량으로 UX를 찾으면 `competency.md`만 뒤지다 놓친다.
- **발생 세션**: 2026-07-02 앤톡(antock) 프론트엔드 서류지원. 단락2·3 역량을 UX로 정한 뒤, AI가 "competency.md에 ux 태그 없음"만 보고 "ux 프리셋 없음"으로 단정 → 프리셋(empathy.md `## ux`, fitt·krm 출처: 터치 줄이기 UX관 + 숫자키패드·커서 예시 + 꼼꼼함 문단2)을 안 쓰고 손으로 압축 → 리뷰어 3라운드 낭비. 사용자 지적: "이상하네 진짜 프리셋이 없었어?"
- **근본**: 검색 키가 '축'이라 축으로 파일이 갈리면 교차 개념이 한 파일에만 산다.

### 현재 생각중인 방법 (사용자 제안 — 검토 대상)

전제(제약): **마크다운 frontmatter는 파일 단위**라 블록(`## ux`)마다 frontmatter 태그를 못 단다. "frontmatter tag로 검색"을 성립시키려면 태그를 파일 단위로 내려야 한다. 그래서 두 갈래:

- **A. 태그 줄 + grep (현 2파일 유지, 가벼움)**: `## ux` 헤딩 밑에 검색용 줄(`axes: 공감, 역량` / `tags: ux, ...`) 추가. 규칙을 "역량 프리셋 찾을 때 `presets/` 전체를 태그로 grep"으로. — 장점: 구조 안 건드림, 실패 지점 직접 차단. 단점: 진짜 frontmatter 아님(grep 컨벤션), 태그 줄 수기 관리.
- **B. 태그(개념)당 파일 1개로 재편 (frontmatter tag 성립, 무거움)**: `presets/ux.md`·`ai.md`·`error-handling.md` … 각 파일이 그 개념의 공감(문단1·2)·역량(문단3·4)을 안에 다 담고 frontmatter에 `tags`·`axes`. 파일명=검색키. — 장점: 한 개념=한 파일, 축 혼동 원천 소멸. 단점: `empathy.md`·`competency.md` 전량 해체·재작성(큰 마이그레이션), 문단 역할별로 보던 편의는 파일 내 섹션으로 대체.
- **작성 세션 의견**: A로 즉시 막고 B는 별건으로 — 지원 흐름 도중 벌일 규모가 B는 아니다. 단, 사용자는 이 태그 기반 방식을 **recruitment 프리셋을 넘어 스킬 전체 컨텍스트 자산에 도입할지** 검토하고 싶어 함 → 그 판단이 이 ideation의 핵심.

### 검토 포인트 (이 ideation이 답할 것)

- A vs B 중 recruitment 프리셋에 무엇을 적용할지 (마이그레이션 비용 대비 재발 방지 효과).
- 태그 기반 discovery를 다른 스킬 컨텍스트 자산(예: KA·qna 등 태그로 찾는 자산)에도 **일반 패턴**으로 도입할 가치가 있는지.

## 첫 행동

`presets/README.md` 태그 레지스트리와 `empathy.md`·`competency.md`의 현재 태그 수를 세어 B(파일 재편) 마이그레이션 규모를 가늠하고, A/B 트레이드오프를 사용자와 확정한다.

## 종료 조건

프리셋 discovery가 개념(태그) 기준으로 바뀌어 축-분리 오판이 사라지고, 이 방식의 스킬 전체 도입 여부가 결정되면 이 항목 삭제.
