# recruitment-application 스킬 동작 검증 — 다음 실제 지원 때 계측 관찰

> PP = `~/WebstormProjects/main/private-playground/`.

## 왜 자동 벤치가 아니라 실제 지원 관찰인가 (2026-07-02 결론)

이 스킬은 자동 서브에이전트 벤치로 검증이 **안 된다**:

- **오케스트레이터 구조**: `motivation-criteria.md`가 "시안 생성은 Writer 서브에이전트(opus)에 위임, 메인(Reviewer)이 검증"을 규정. 서브에이전트는 다시 서브에이전트를 못 스폰하므로(`scw/eval-delegation.md §1`) 단일 서브에이전트로 통짜 벤치 불가.
- **인터랙티브 의존**: `[ROOT_FIRST]` 뿌리 1:1 문답, 시안 다중 제시→사용자 선택, 표현 반려→리뷰어 게이트 — 전부 실사용자가 있어야 발동. 가짜 사용자 = 가짜 동기 = 산출물 품질 판단 불가.
- **주관적 출력**: 지원동기 진정성·오글 판정의 최종 오라클은 사용자 감각. skill-creator 가이드도 "주관적 출력 스킬은 정량 테스트가 안 맞는다".

1인용·주관적 출력 스킬에 풀 벤치 하네스를 짓는 건 ROI가 나쁨 → 다음 실제 지원 1회를 아래 체크리스트로 관찰하는 것으로 갈음.

## 관찰 체크리스트 (다음 실제 지원동기 작성 시)

- [ ] **선제 매칭**: 시안 만들기 전에 공고 키워드→태그→`presets/` 재사용 블록을 먼저 꺼내 제시하는가 (사용자가 "ux로 써둔 거 없냐"고 짚어주게 만들지 않는가)
- [ ] **축 라우팅**: 도메인 공감이 있을 때 `ux` 같은 겸용 태그가 역량 축으로 밀리는가, 신설한 `design-system` 태그가 실제로 활용되는가
- [ ] **단락순서 게이트**: 5단락을 한 번에 쏟지 않고 그룹씩(뿌리 확정→공감 1·2→경험 3→입사후 4→목표 5) 시안 제시·선택을 거치는가
- [ ] **Writer+Reviewer 2-agent**: 시안 생성을 Writer 서브에이전트(opus)에 위임하고 메인이 평가기준으로 검증 후 제출하는가 (미달 시안을 그대로 넘기지 않는가)
- [ ] **cringe 리뷰어 게이트**: 표현을 1회 반려하면, 그 회사 지원동기가 끝날 때까지 모든 시안이 리뷰어 clean 통과 후에만 노출되는가
