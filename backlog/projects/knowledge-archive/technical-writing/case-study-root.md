# Sentry 도입기 — 86종 이슈를 4개 목표로 정리한 기록

## 1. 문제 발견

<!-- 사례 전체 Before + 근본 원인 + 비즈니스 임팩트 -->

고객이 오류를 제보해도 개발 환경에서 재현되지 않아 캡처와 추가 정보를 반복 요청해야 했음. 구매 페이지가 깨진 사례에서는 약 2,000만 원 규모의 손실이 발생 — 개발팀의 인지가 늦은 게 원인.

**근본 원인**: 프론트엔드 에러를 자동 수집·중앙화하는 모니터링 시스템 부재. 사용자 제보가 유일한 인지 경로.

## 3. 해결 결과

### G1. 누락 없이 수집

- 선택: 글로벌 핸들러(`window.onerror`, `unhandledrejection`) + React `ErrorBoundary`
- 검토했던 후보:
  - 각 핸들러마다 `try-catch` + `captureException` 수동 호출 — 호출 누락 위험·보일러플레이트
  - Interceptor 패턴(요청·응답 가로채기) — 비-API 에러(런타임·렌더) 못 잡음
- 구현 핵심: 글로벌 이벤트 리스너 등록 + 트리 최상위 ErrorBoundary 한 곳에 일괄
- [→ 1편의 #해결방법-리스트업](./case-study-sub.md#3-해결방법-리스트업)

### G2. 경중 구분 + 긴급 알림

- 선택: 에러 등급 정책(Fatal/Error/Warning/Info) + Slack 차등 알림
- 검토했던 후보:
  - 전부 같은 등급 + 메일 일괄 발송 — 노이즈에 묻혀 긴급도 식별 불가
  - 키워드 매칭 자동 등급 — 새 에러 케이스 누락·오분류
- 구현 핵심: 에러 클래스마다 등급 필드 → Sentry alert rule이 Fatal만 Slack 채널 즉시 발송
- [→ 2편의 #해결방법-리스트업](./case-study-sub-g2.md#3-해결방법-리스트업)

### G3. 풍부한 단서 자동 첨부

- 선택: 커스텀 에러 클래스 + `beforeSend` 훅에서 단서 매핑
- 검토했던 후보:
  - 각 호출 지점에서 `withScope.setTag` 수동 호출 — 호출 누락 시 단서까지 소실
  - 콜스택만 의존 — 비즈니스 컨텍스트(요청·응답·입력) 부재
- 구현 핵심: 에러 클래스가 단서 필드 보유 → `beforeSend`가 `event.extra`로 자동 추출
- [→ 3편의 #해결방법-리스트업](./case-study-sub-g3.md#3-해결방법-리스트업)

### G4. Sentry 모르는 동료도 유지보수

- 선택: 에러 클래스 추상화 (3편 패턴 재사용)
- 검토했던 후보:
  - 사내 위키에 SDK 사용법 문서화 — 학습 비용·코드 리뷰 부담 지속
- 구현 핵심: 동료는 `throw new CustomError(...)`만 알면 됨. Sentry API는 노출 X
- [→ 4편의 #해결방법-리스트업](./case-study-sub-g4.md#3-해결방법-리스트업)
