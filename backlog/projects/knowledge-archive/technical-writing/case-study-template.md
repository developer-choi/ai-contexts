# Sentry 도입기 1편 — 모든 에러를 누락 없이 잡기

## 1. 문제 발견

<!-- 증상(무엇이 일어났나) / 발견 경로(어떻게 알았나) / 비즈니스 임팩트(왜 해결해야 했나) -->

캐치되지 않은 에러로 사용자가 직접 제보할 때까지 개발팀이 인지하지 못함.
실제로 구매 페이지가 깨진 사례에서 약 2,000만 원 규모의 손실 발생.
근본 원인: 수십 페이지·수백 개 버튼·핸들러에 Sentry 호출이 일관되게 들어가있지 않음.

## 2. 목표 정의

<!--

에러가 발생할 수 있는 모든 지점에서 누락 없이 Sentry로 전송된다.

## 3. 해결방법 리스트업

<!--
이 목표를 달성할 수 있는 후보 N개 + 각각의 trade-off.
trade-off 분류: 성능 / 복잡도 / 유지보수 비용 / 학습 비용 / 운영 부담 / 확장성 등.
-->

### A. 각 핸들러마다 try-catch + captureException 수동 호출

- 장점: 발생 지점 컨텍스트가 명확
- Trade-off:
  - 보일러플레이트 (복잡도·유지보수 비용)
  - 호출 누락 위험 (목표의 정반대)
  - 동료가 Sentry SDK 학습 필요 (학습 비용)

### B. Interceptor 패턴 (요청·응답 가로채기)

- 장점: 한 곳에서 관리
- Trade-off:
  - API 외 에러는 못 잡음 (확장성)
  - UI 렌더 에러·비동기 콜백 에러 누락

### C. 글로벌 핸들러 + ErrorBoundary

`window.onerror`, `unhandledrejection`로 캐치 안 된 에러 일괄 수집. React 렌더 에러는 `ErrorBoundary`.

- 장점: 모든 캐치 안 된 에러 자동 포착
- Trade-off:
  - 발생 지점의 의미적 컨텍스트가 부족 → 단서 보강이 별도로 필요

## 4. 선택 및 근거

<!--

C 선택. 누락 0이 이 목표의 절대 조건이라 A의 누락 위험, B의 커버리지 부족은 즉시 탈락.
C의 단서 부족 문제는 [3편] 패턴으로 보완.

## 5. 구현 과정

<!-- 어떻게 코드로 옮겼나. 핵심 메커니즘 + 엣지 케이스. -->

- `window.addEventListener('error', ...)` + `window.addEventListener('unhandledrejection', ...)`
- React 트리 최상위 `ErrorBoundary` 컴포넌트
- 의도적 throw 지점은 그대로 두고 글로벌 핸들러가 받음

## 6. 성과

<!-- 2번 목표 달성 여부 + 정량/정성 결과 + 남은 한계. -->

- 누락 0: 어디서 에러가 나든 Sentry로 들어옴
- 신규 페이지·버튼 추가 시 Sentry 전용 코드 불필요
- 남은 한계: 발생 지점의 의미적 컨텍스트는 부족 → [3편] 패턴으로 보완
