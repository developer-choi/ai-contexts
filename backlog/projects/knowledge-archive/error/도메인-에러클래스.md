# 도메인 에러클래스는 언제·어떻게 만들어야 하는가

dada 회고 중 발견. 도메인 에러클래스를 만들다가 포기한 사례에서 출발한 사고 흔적.

## 출발

"도메인 에러클래스를 만들어볼까?" 시도했다가 포기. 어디서·왜 만드는 게 적절한지 명확하지 않음.

## 사고 흐름

### 1차 결론 — 에러 메시지 변환 시점이 적절해 보임

API 응답에서 에러를 꺼내서 사용자 노출용 메시지로 변환하는 지점에서 도메인 에러클래스를 쓰는 게 자연스럽다. 그 외 시점에는 굳이 만들 동기가 약함.

### 1차 결론에 대한 반론

- 일반 에러클래스였다면 "사용처가 늘어나는데 클래스가 수정돼야 한다? 별로다"라고 평가했을 것
- 그런데 도메인 에러클래스는 API가 20개로 늘면 에러 메시지 변환도 20개로 늘어남
- 같은 "사용처 증가에 따라 코드가 늘어난다"는 문제를 일반 에러클래스로 보면 안티패턴이라고 봐놓고, 도메인 에러클래스만 다르게 평가할 근거가 약함

### 결정적 반론 — 유지보수 동선이 안 맞음

유지보수 상황을 생각해보면:

- 에러 메시지 하나를 바꾸고 싶을 때, "이 도메인 에러클래스를 바꿔야 한다"라고 떠올리지 못함
- 보통은 **에러 메시지가 노출되는 화면**을 기점으로 찾아 들어감
- 즉 클래스 추상화가 실제 수정 동선과 맞지 않음

## 잠정 결론

- 도메인 에러클래스를 만드는 비용 대비 유지보수 효용이 약하다
- 에러 메시지 변환 지점을 기점으로 코드를 찾는 동선이 자연스럽다면, 그 자리에 직접 변환 로직을 두는 게 합리적
