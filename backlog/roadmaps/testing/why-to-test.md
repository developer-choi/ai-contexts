## 핵심 명제 — 테스트의 목적은 확신(confidence)

"사용자가 우리 서비스를 쓸 때 문제 없을 것"이라는 확신을 얻기 위해.

> "We write tests to be confident that our application will work when the user uses them." (`testing-library-philosophy.md`)

- 이 확신에 기여하지 않는 테스트는 쓰지 않는다 → 이하 전개.

## 그 확신은 자동화 테스트라야 채워진다 — 사람 손으로는 못 한다

확신을 얻으려면 매번·전부 다시 검증해야 하는데, 그 반복을 사람은 신뢰성 있게 못 한다.

### 영향범위(blast radius) — 내가 안 본 B가 깨진다

- 내가 *방금 고친* 코드의 재확인은 수동으로도 된다 (어디 고쳤는지 아니까).
- 테스트만의 대체불가 가치는 여기: **A 모듈을 고쳤는데, 정작 깨지는 건 열어보지도 않은 B 모듈**이다. 사람은 자기 변경의 파급을 완전히 열거하지 못하고, 열거해도 매번 빠뜨린다. 단, 자동화가 파급을 *발견*해주는 건 아니다 — 테스트로 한번 표현해둔 기대를 **빠짐없이·매번** 재실행해줄 뿐이다 (테스트 없는 B는 자동화도 못 잡는다 → 그래서 What "무엇을 테스트하나"로 이어진다).

### QA가 있는데도 왜? — 회귀(자동화) vs 탐색(QA)

- **자동화 = 알려진 정답을 싸게 반복 검증** (회귀 방어). 사람 QA는 매 릴리스마다 전체를 손으로 못 돌아 "고친 근처만" 본다 → 멀리 떨어진 영향범위 회귀를 놓친다. 자동화는 수초에 전부 매번 돌린다.
- **QA = 정답이 뭔지 모르는 영역을 탐색**(exploratory). 레이아웃 깨짐·어색한 UX·명세 자체의 모순 등, 코드는 "정상 동작"하지만 제품이 틀린 경우 — 사람만 잡는다.
- 대체재가 아니라 상호보완. 한눈 비교:

| 항목 | QA | 테스트 코드 |
|---|---|---|
| 주체 | 사람(QA 엔지니어) | 자동화된 코드 |
| 시점 | 기능 완성 후 | 코드 작성과 동시에 / CI |
| 검증 대상 | 사용자 시나리오, UX, 시각적 요소 | 함수·컴포넌트 단위 동작, 회귀 |
| 반복 비용 | 매번 사람이 다시 수행 | 한 번 작성하면 자동 반복 |
| 잡는 결함 | 사용성·UX·예상 못한 흐름 | 로직 오류·회귀 |

## 그렇다고 아무 테스트나 확신을 주진 않는다

### 확신은 "실제 사용을 닮은 정도"에 비례한다

테스트가 실제 사용자의 사용 방식을 그대로 재현할수록, 통과가 곧 "사용자에게도 동작한다"를 뜻해 확신이 커진다. E2E > Integration > Unit 순으로 확신이 큰 건 등급이 높아서가 아니라 그 순서로 실제 사용에 가까워지기 때문이다.

- 예: 같은 "로그인"을 ① 토큰 발급 함수 유닛 ② API+DB 통합 ③ 브라우저에서 직접 클릭(E2E)으로 각각 테스트하면, ③이 사용자 행동을 가장 그대로 재현하므로 "로그인이 실제로 된다"는 확신이 가장 크다.
- ⚠ 단, 위로 갈수록 비용·속도·취약성(flakiness)도 같이 커진다 → 무조건 E2E가 아니라 균형. 어디에 얼마나 둘지는 What.

### 확신을 못 주는 테스트는 안 쓴다 — 구현상세

앞의 명제(확신 = 실제 사용을 닮은 정도)를 뒤집으면 나온다: 구현상세(내부 state·메서드 등)는 사용자가 보지도 쓰지도 않는다 → 검증해봐야 사용자 동작과 연결이 없어 확신에 기여하지 않는다. 게다가 동작은 그대로인데 구조만 바꾼 리팩터링에서 깨져 거짓 경보를 내고, 정작 동작이 깨져도 통과할 수 있다.

> "Writing tests that include implementation details is all downside and no upside." — https://kentcdodds.com/blog/avoid-the-test-user
