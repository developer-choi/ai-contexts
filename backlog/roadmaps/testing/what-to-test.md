## 무엇을, 어디까지? — Why에서 넘어오는 다리

Why의 결론("확신 = 실제 사용을 닮은 정도, 구현상세는 안 쓴다")을 한 문장으로 받고, 그 원칙이 *측정 기준*이지 *대상 목록*은 아니라고 선언. 이 글은 그 기준을 실제 대상 선정에 적용한다. 이번 글이 다루는 건 **컴포넌트 테스트 + 함수 유닛테스트** 둘(E2E·통합·모킹은 범위 밖, E2E는 개념만 한 줄 각주).

- **예시 코드 메모**: 코드블록 없음(브릿지 문단).

## 기준은 유스케이스다 — 코드 라인이 아니라

무엇을 테스트할지의 잣대는 if/else·loop 같은 코드 구조가 아니라 "이 코드가 떠받치는 **유스케이스**"다. 같은 함수라도 사용자 시나리오가 곧 테스트 목록이 된다.

### "이 라인은 어떤 use case를 받치나" — 던질 질문

테스트 없는 코드 앞에서 "이 분기를 어떻게 통과시키지"가 아니라 "이 코드는 **어떤 사용자 상황**에서 실행되나, 그 상황을 검증하는 테스트가 있나"를 묻는다. (trophy.md:344-358 — 자동화된 Use Case Coverage 리포트는 없고 우리가 직접 만들어야 한다)

- **예시 코드 메모**: 본인 함수 하나를 놓고 before(라인 기준: "이 if를 타게 입력 맞추기") / after(유스케이스 기준: "할인 코드가 만료됐을 때"라는 상황 → 그 상황을 검증하는 `it('만료된 쿠폰은 거부한다')`) 대비.

## 어느 높이에서 묶나 — 트리 레벨과 tradeoff

컴포넌트는 어느 레벨에서 테스트하느냐가 곧 "무엇을 테스트하나"를 가른다. 너무 낮으면 사용자 행동과 멀고, 너무 높으면 실패 추적이 어렵다.

### 단독 vs 묶어서 — high enough up the tree

실제 사용자 상호작용을 흉내 내려면 "충분히 높은 레벨"에서 테스트하는 게 보통 낫고, 더 높이/낮게 추가로 테스트할지는 tradeoff 판단이다. (rtl-overview.md:124-132) 자식 컴포넌트를 격리해 "props로 잘 넘긴다"만 검증하면, 그 자식이 실제로 깨지는 경우를 놓친다.

- **예시 코드 메모**: 본인 폼 컴포넌트를 ① 입력 필드 자식만 격리 렌더 vs ② 폼 전체를 렌더해 `userEvent`로 채우고 제출 결과 검증 — 후자가 "사용자가 실제로 쓰는 방식"에 가깝다는 대비.

### 타입체커로 갈 것 vs 유닛으로 갈 것

모든 방어를 테스트로 하지 않는다. "문자열을 넣으면?" 같은 타입 위반은 유닛테스트가 아니라 **타입체커(TS)**가 가장 싸게 막는다. (trophy.md:248 — `add(string)`은 테스트가 아니라 타입의 일) 잡을 수 있는 가장 낮은 레벨을 고른다.

- **예시 코드 메모**: 같은 함수에 대해 ① 잘못된 타입 인자를 막는 건 시그니처(TS)로, ② 값의 경계·계산 오류는 유닛테스트로 — 두 방어가 각각 어디 사는지 코드로. (E2E·통합 비교는 deferred, 한 줄 언급만.)

## 함수는 어디까지 — 경우의 수와 경계

한 함수를 테스트할 땐 "경우의 수가 빠짐없나"를 세 축으로 본다.

### 한 함수 안에서: 로직·경계·에러 세 갈래

(1) 모든 분기 경로(logic), (2) 일반·경계·범위 밖 입력(boundary), (3) 잘못된 입력의 처리(error handling). (unit-test-strategies.md:16-37)

- **예시 코드 메모**: 0~10 허용 함수에 5(일반)/0·10(경계)/-1·11(범위 밖)을 `it.each`로 나열한 본인 테스트. (unit-test-strategies.md User Annotation의 경계 예시를 코드화)

## 검증된 기반은 다시 테스트하지 않는다 — 델타만

이미 공통(기반)에서 검증된 동작은, 확장·래핑한 쪽에서 **다시 테스트하지 않는다.** 내가 *추가한 부분(델타)*만 테스트한다.

### 함수: 코어 → 상세 전파

상세 함수는 코어가 이미 테스트한 케이스를 중복하지 않고, ① 코어로 잘 전파되는지만 짧게 + ② 상세 고유 로직만 테스트한다. (재료: [부모-자식-테스트-분배.md](../../projects/knowledge-archive/testing/부모-자식-테스트-분배.md))

- **예시 코드 메모**: `validateIncludeString`이 내부에서 `validateString` 호출 — 코어 케이스 중복 없이 "전파 확인 + 포함검사 고유 로직"만 테스트하는 본인 커밋(585ed1f). `calculateTotal`/`calculateNumberTotal`(54fddbf)도.

### 컴포넌트: 래퍼는 델타만

radix UI `Select`를 우리 스타일로 감싼 래퍼. radix가 이미 검증한 동작(키보드 네비·옵션 선택·포커스)은 다시 테스트하지 않고, 우리가 추가한 스타일·prop 전달만 테스트한다.

- **예시 코드 메모**: radix 래퍼 테스트 — 키보드 네비 검증은 없고(= radix 책임), `className` 적용·커스텀 prop 전달만 검증하는 코드.

## 보조 렌즈로서의 커버리지 — 도구이지 기준이 아니다

커버리지는 "무엇을 테스트하나"의 **기준이 아니라**, 세운 유스케이스를 빠뜨렸는지 점검하는 보조 렌즈다. (trophy.md — 커버리지는 useful tool이자 not a perfect metric)

### 빈 라인 앞에서 던지는 질문

미커버 라인을 라인 그 자체로 보지 말고 "이 라인이 받치는 유스케이스가 무엇인지" 되묻는 환원 절차. 위 "기준은 유스케이스다"를 커버리지 도구에 적용한 모습.

- **예시 코드 메모**: 본인 커버리지 리포트의 빨간 라인 + "이건 어떤 사용자 상황?" 주석 → 그 상황을 검증하는 테스트 추가. (라인을 억지로 덮는 게 아니라 유스케이스로 환원)

### 100%가 맞는 2조건 — 높은 파급력 + 낮은 달성비용

커버리지 100% 의무화는 보통 의미 없는 테스트를 양산한다. 단 (1) 깨지면 다수 소비처에 파급되고(높은 파급력) (2) 작아서 100%가 싼(낮은 달성비용) 재사용 라이브러리·공통 함수에선 합리적이다. (trophy.md:329-338 — "공통 함수 파급력"과 직결)

- **예시 코드 메모**: 라인만 거치고 assertion 없는 "가짜 100%" vs 본인의 작은 공통 유틸에 100%를 적용한 사례 — 두 코드 대비.

## 안 짜도 되는 곳 — 유닛이 덜 유익한 3상황

모든 코드에 유닛테스트를 강요하지 않는다. Why의 "확신에 기여 안 하면 안 쓴다"가 함수 레벨에 착지하는 지점. (unit-test-strategies.md:64-85)

- (1) 시간 제약 하의 단순 작업, (2) look&feel 위주(로직 없는 UI·애니메이션), (3) 요구사항이 자주 뒤집히는 영역.
- **예시 코드 메모**: 로직 분기가 풍부해 테스트 가치 큰 함수 vs 위 3상황에 해당해 의식적으로 테스트를 생략한 본인 판단 — 코드 + 한 줄 근거.
