# Estimation

## [not-ready] 일정 산정

- 기획/디자인 리뷰 산출물을 기반으로 개발 일정을 산정
- 작업을 분해하고 공수를 산출
- 피그마 스펙 조정/문의/리뷰에 드는 시간도 꽤 됨
- 이걸 근거로 제시하려면 실제 쟁점이 될 만한 것 하나 준비해가는 것도 좋음
- 메인 업무만 하는 게 아니므로 보정해서 잡아야 함

### 입력

- spec-review (구 plan-review) 산출물: 사용자 플로우, 상태 전이, 엣지케이스, 필요 API, 일정 영향도 분류
- design-review 산출물: 레이아웃 구조, 컴포넌트 복잡도

### 작업 분해

산출물에서 구현해야 할 작업을 최소 단위로 분해:

- 신규 컴포넌트
- 기존 컴포넌트 수정
- API 연동
- 상태 관리
- 폼 유효성 검증
- 엣지케이스 처리
- 테스트

### 가중치

- 불확실성: 기획 미확정 항목이 많을수록 버퍼 추가
- 의존성: 백엔드 API 대기, 디자인 확정 대기 등 외부 의존 별도 표기
- 복잡도: 처음 다루는 기술, 까다로운 엣지케이스 등

### 산출물 템플릿

- "바로 착수 가능" / "기획 확인 후 착수 가능" 분리
- 가중치 표 (요소, 가중치, 근거)
- 합계: 단순 합산, 가중치 적용 후, 외부 의존 대기

### 디자이너 커뮤니케이션 공수

- 디자이너랑 왔다갔다 많이 해야 하는 항목이 있는지 식별
- 엣지케이스, 애니메이션 수치 조정 등은 디자이너랑 손잡고 해야 함
- 애니메이션 수치 조정(얼마나 내렸을 때 닫히게 할 것인가 등)은 예상보다 시간 오래 걸림

### 라이브러리 선정

- 일정 산정 전에 라이브러리 선정을 미리 해야 함
- 베이스가 뛰어난 라이브러리가 아닌 경우 다양한 기기 테스트 구간에서 망할 수 있음
- 라이브러리 품질이 일정에 직접 영향
