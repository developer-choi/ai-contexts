# 일일 작업 회고 피드백 가이드

## 개요

일일 작업을 회고하여 개선 포인트를 발견하고 성장하기 위한 가이드입니다.

**사용 상황**:
- 하루 작업을 마치고 커밋 히스토리를 분석하고 싶을 때
- 작업 흐름과 커밋 품질을 개선하고 싶을 때

---

## AI 행동 가이드

### 배경 지식

**스캔할 프로젝트 경로**:
- ~/workspace/langdy/* (langdy-admin, langdy-student, langdy-teacher, design-system, langdy-utils)
- ~/WebstormProjects/* (dsa-playground, react-playground, test-playground, forworkchoe)

**분석 기준**:

1. **작업 흐름 & 효율성**
   - 커밋 순서와 작업 진행 방향이 효율적이었는지
   - 막힌 부분이 있었다면 다른 접근법은 없었을까
   - 시행착오를 줄일 수 있는 방법은
   - 작업 우선순위 설정이 적절했는지

   예시 - Legacy 코드 처리 미흡으로 인한 QA 범위 확대:
   - **문제 상황**: 기획/디자인 변경으로 기존 모듈을 직접 수정하여 QA 범위가 지나치게 커짐
   - **개선책**: Deprecated 모듈과 새 모듈을 분리. 기존 모듈 격리를 먼저 수행
   - **효과**: Git Conflict 차단, 코드 구분, QA 리소스 절감

2. **커밋 품질**
   - 커밋 메시지의 명확성
   - 커밋 단위가 적절했는지 (너무 크거나 작지 않은지)
   - 기능별로 잘 분리되어 있는지
   - 커밋 히스토리가 리뷰어가 이해하기 쉬운 스토리를 만드는지

### 진행 방식

1. **커밋 수집**: 위 프로젝트 경로에서 본인 커밋만 스캔
2. **분석**: 위 분석 기준에 따라 작업 흐름과 커밋 품질 분석
3. **리포트 작성**: 출력 형식에 맞춰 회고 리포트 생성
4. **저장**: `~/Downloads/daily-review-YYYY-MM-DD.md`에 자동 저장

### [CRITICAL] 지켜야 할 원칙

- **모든 브랜치 스캔**: `git log --branches --since="날짜"` 사용
- **Author 필터링 필수**: `forworkchoe@gmail.com` 또는 `yujin.choe@langdy.net` 커밋만
- **reflog 제외**: `--all` 대신 `--branches` 사용
