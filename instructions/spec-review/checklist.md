# 기획서 리뷰 체크리스트

개발자가 피그마 기획서를 받았을 때, AI가 순서대로 점검하는 항목.

## 역할 분담

- **AI**: 기획서를 분해하고, 모호한 부분을 즉시 짚어주며, UX 개선 제안도 함께 한다.
- **개발자**: AI의 질문에 답하고 추가 캡처/설명을 제공하며, 기획자에게 돌아갈 질문을 확정한다.

## 진행 방식

1. **리뷰 시작 시 분석 문서를 먼저 생성한다.** 개발자에게 파일명을 안내하고, 이후 모든 내용을 이 문서에 실시간으로 누적한다.
2. 개발자가 피그마 캡처를 하나씩 제공한다.
3. AI는 캡처를 받을 때마다 **즉시 모호한 부분을 짚고, 분석 문서에 기록한다.** 전체 분석을 한꺼번에 쏟아내지 않는다.
4. 개발자가 답변/설명을 주면 AI는 분석 문서를 업데이트하고, 다음 캡처를 요청한다.
5. 모든 캡처를 받은 후, 누적된 분석 문서를 종합 정리한다.

## AI의 세 가지 역할

### 1. 모호한 부분 짚기 (필수)

기획서에서 정의되지 않았거나 불명확한 부분을 찾아 질문한다.

### 2. 엣지케이스 선제시 (필수)

캡처를 받을 때마다 해당 화면에서 발생 가능한 엣지케이스를 먼저 제시한다. 개발자가 떠올리기 전에 AI가 먼저 짚어준다. [edge-cases.md](./edge-cases.md)의 카테고리를 기반으로 점검한다.

### 3. UX 개선 제안 (선택)

기획 의도는 이해하되, 더 나은 대안이 있을 때 **선택지를 제시하는 수준**으로 제안한다. 단정하지 않고 장단점을 함께 설명한다.

> 예: "언어 선택 후 '계속하기'를 누르는 구조인데, 선택지가 3개뿐이라 탭하면 바로 이동하는 것도 고려할 수 있습니다. 다만 현재 방식은 실수 방지와 단계 일관성 측면에서 장점이 있습니다."

## 점검 항목

### 1. 목적/배경 파악

기획서에서 이 기능의 목적과 배경을 추출한다.

- 왜 이 기능이 필요한지 명시되어 있는가?
- 이 기능이 해결하려는 사용자 문제가 무엇인가?
- 성공 기준(KPI, 목표 지표)이 정의되어 있는가?

> 목적이 불명확하면 구현 방향 자체가 흔들린다. 가장 먼저 확인한다.

### 2. 사용자 플로우 검증

상세 점검은 [flow.md](./flow.md) 참조.

- 진입점(entry point)이 명확한가?
- 시작부터 완료까지의 전체 경로가 그려져 있는가?
- 분기 조건(if/else)이 모두 명시되어 있는가?
- "뒤로가기"와 "취소" 시 동작이 정의되어 있는가?

### 3. 상태 정의 확인

상세 점검은 [flow.md](./flow.md) 참조.

- 화면/데이터가 가질 수 있는 상태가 모두 나열되어 있는가?
- 상태 간 전이 조건이 명확한가?
- 각 상태에서 사용자가 할 수 있는 액션이 정의되어 있는가?

### 4. 엣지케이스 확인

상세 점검은 [edge-cases.md](./edge-cases.md) 참조.

- 빈 상태(empty state) UI가 정의되어 있는가?
- 에러 상태별 UI와 복구 방법이 정의되어 있는가?
- 권한별 화면 차이가 정의되어 있는가?
- 데이터가 극단적일 때(0건, 1000건, 매우 긴 텍스트) 처리가 정의되어 있는가?

### 5. 데이터/API 확인

상세 점검은 [data.md](./data.md) 참조.

- 화면에 필요한 데이터가 무엇인지 파악 가능한가?
- 해당 데이터를 제공하는 API가 존재하는가, 새로 만들어야 하는가?
- 사용자 입력값의 유효성 검증 규칙이 정의되어 있는가?

### 6. 비기능 요구사항

- 성능 요구사항이 있는가? (로딩 시간, 대량 데이터 처리 등)
- 보안 관련 고려사항이 있는가? (인증, 인가, 민감 데이터 등)
- 접근성 요구사항이 있는가?
- 다국어/로컬라이제이션 대응이 필요한가?

### 7. 일정 영향도 판단

위 1~6의 결과를 종합하여:

- 명확하여 바로 구현 가능한 부분
- 기획자 확인이 필요한 부분 (질문 목록)
- 불명확하여 일정 리스크가 되는 부분

을 분류한다.
