# 상태/플로우 점검 가이드

기획서에서 사용자 플로우와 상태 전이가 충분히 정의되어 있는지 점검한다.

---

## 사용자 플로우 점검

### 1. 진입 경로

- 사용자가 이 화면에 어떻게 도달하는가?
- 진입 경로가 여러 개인 경우, 각 경로별 초기 상태가 다른가?
- 딥링크/알림/외부 링크로 직접 진입 가능한가?

### 2. 핵심 플로우 (Happy Path)

- 시작 → 완료까지의 정상 경로가 빠짐없이 그려져 있는가?
- 각 단계에서 사용자의 액션과 시스템의 반응이 1:1로 대응되는가?
- 단계 간 데이터 전달이 명확한가? (이전 단계에서 입력한 값이 다음 단계에서 어떻게 쓰이는지)

### 3. 분기와 조건

- 조건 분기가 있는 곳에서 모든 경우(if/else)가 정의되어 있는가?
- 조건의 판단 기준이 명확한가? (구체적인 값, 상태, 권한 등)
- 분기 후 합류(merge) 지점이 있는 경우, 어디서 합류하는지 명확한가?

### 4. 이탈/복귀

- "뒤로가기" 시 이전 입력값이 유지되는가, 초기화되는가?
- "취소" 시 확인 다이얼로그가 필요한가?
- 중간에 이탈했다가 다시 돌아왔을 때 이어서 진행 가능한가?
- 브라우저 새로고침 시 상태가 어떻게 되는가?

---

## 상태 전이 점검

### 1. 상태 나열

기획서에서 파악 가능한 모든 상태를 나열한다. 일반적으로 확인해야 할 상태:

- 초기(initial)
- 로딩(loading)
- 성공(success) / 데이터 있음
- 빈 상태(empty)
- 에러(error)
- 부분 로딩(partial / skeleton)
- 비활성(disabled)
- 권한 없음(unauthorized)

### 2. 전이 조건

각 상태 간 전이가 언제 발생하는지 확인한다:

- 어떤 이벤트로 상태가 바뀌는가? (사용자 액션 / 서버 응답 / 시간 경과)
- 되돌릴 수 있는(reversible) 전이인가, 단방향(one-way) 전이인가?
- 동시에 여러 상태가 될 수 있는가? (예: 로딩 중이면서 부분 데이터 표시)

### 3. 동시성

- 같은 데이터를 여러 사용자가 동시에 수정할 수 있는가?
- 여러 요청이 동시에 발생할 때 처리 방식이 정의되어 있는가?
- 낙관적 업데이트(optimistic update) vs 비관적 업데이트(pessimistic update) 중 어떤 방식인가?
