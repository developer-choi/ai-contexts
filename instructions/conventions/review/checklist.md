# 코드 리뷰 체크리스트

## 목적 및 사용 시점
이 문서는 **작업 도중 작성한 코드 블록이나 단일 파일**을 빠르게 검증하기 위한 체크리스트입니다.

*   **사용 대상**: 함수 하나, 컴포넌트 하나, 또는 단일 커밋.
*   **PR/통합 리뷰**: 전체적인 문제 해결 논리나 PR 단위의 리뷰가 필요하다면 **[`pull-request.md`](./pull-request.md)**를 참고하세요.

---

## 1. 공통 점검 사항
- **[필수]** `../coding` 폴더 하위의 모든 규칙(Naming, TypeScript, Component 등)을 준수했는지 확인합니다.
- **[필수]** `any`, `as` 타입 단언이 불필요하게 사용되지 않았는지 확인합니다.
- **[필수]** `ts-ignore`, `eslint-disable` 등 경고 무시 코드가 없는지 확인합니다.

## 2. CSS / 스타일 점검
- 불필요한 스타일이 없는지 확인합니다.
    - `width: 100%` (Block 요소에 불필요)
    - `display: flex` (자식 요소 배치나 정렬 목적 없이 컨테이너에만 적용된 경우)
- SCSS 모듈 시스템(`@use`)을 올바르게 사용했는지 확인합니다.

## 3. 테스트 코드 리뷰 가이드
**테스트 파일 검증**
- 테스트 파일이 대상 파일과 같은 폴더(또는 직관적인 위치)에 있는지 확인합니다.
- 파일 상단의 `// yarn test [경로]` 주석이 **실제 파일 경로**와 정확히 일치하는지 검증합니다. (오타, 경로 불일치 엄격 체크)

**코드 내용 검증**
- 함수명, 변수명이 직관적이고 문법적으로 올바른 영어인지 확인합니다. (어색한 표현 수정 제안)
- 누락된 테스트 케이스(Edge case, Boundary case)가 있는지 확인합니다.
    - 누락 시: `it('한글 설명')` 형태로 구체적인 케이스 추가를 제안합니다.
- 시간/공간 복잡도 주석이 정확한지 계산하여 검증합니다.
- 알고리즘 구현체의 경우, 함수 상단에 시간 복잡도 주석이 포함되었는지 확인합니다.

## 4. 로직 및 구조 점검
- 함수가 너무 길거나 복잡하지 않은지(단일 책임 원칙) 확인합니다.
- 중복된 로직이 유틸리티로 분리 가능한지 확인합니다.
- React 컴포넌트의 경우, `useEffect` 남용이나 불필요한 리렌더링 요소가 없는지 체크합니다.
