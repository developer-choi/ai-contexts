# 목표

프롬프트를 잘 작성합니다.

## 방법

1. 프롬프트 엔지니어링, Skill에 대해 공식문서를 읽고 전문성을 갖춥니다.
2. 프롬프트를 잘 유지보수합니다. AI가 이를 돕습니다.
   - 프롬프트는 계속 내용이 많아지기 때문에 관리가 필요합니다.
   - AI 모델별로 성능이 다르고 앞으로 계속 발전하므로, 프롬프트 양도 모델 성능을 고려하며 유지보수해야 합니다.

### 현재 방법론

- **검증을 사람 대신 AI로**: 프롬프트 품질 규칙을 체크리스트로 만들고(`README.md`), AI가 체크리스트를 돌려 먼저 검증합니다. 사람은 AI의 검증 결과를 리뷰만 합니다.
- **AI 자가검진**: 체크리스트 순회 후 자유 검토 및 프로세스 회고(`retrospect.md`)를 실행합니다. AI가 체크리스트가 놓친 문제를 찾고, 체크리스트 자체에 항목 추가를 제안합니다. 이를 통해 검증 규칙이 스스로 진화하는 구조입니다.

### 예상 문제: 규칙 증가에 따른 약한 모델의 노이즈

규칙이 늘어날수록 약한 모델은 규칙을 제대로 실행하지 못해 노이즈를 만듭니다.

**검토한 방향:**

- 반려 데이터를 메모리 파일에 축적하고, 반려가 많은 규칙을 별도 파일로 분리합니다 (예: `consistency.md` → `consistency-full.md`)
- 검증 실행 시 옵션(`--full` 등)으로 강한 모델은 전체 규칙, 약한 모델은 기본 규칙만 로드합니다
- 같은 파일 내 별도 섹션은 토큰을 소비하므로 별도 파일이 낫습니다

**미해결 과제:**

- 반려 횟수만으로는 부족하고 반려율이 필요하지만, 실행 횟수 추적이 복잡해집니다. 임계값을 보수적으로 잡으면 횟수만으로도 가능할 수 있습니다
- 체크리스트 항목별 유일 식별자(ID)가 필요하지만, ID 체계 관리 비용이 큽니다
- 규칙이 충분히 쌓이고 반려 패턴이 실제로 체감될 때 구체화합니다
