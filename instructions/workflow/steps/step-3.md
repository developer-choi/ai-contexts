# Step 3: 핵심 과제 정의 및 기술 전략 수립

이 단계는 **하나의 PR**에 대한 핵심 기술적 과제를 정의하고, 최적의 도구(라이브러리)를 선정하는 단계입니다.

> 대기업 기술면접관에게 기술 선택 근거를 설명할 수 있는 수준으로 작성하세요.
> [문제 해결 로드맵](../../self-help/roadmap.md) 참고.

## 작성 지침
- **간결함**: 화려한 수식어나 마케팅적 표현은 배제하고 담백하게 작성하세요.
- **개요 중심**: 기획서의 내용을 /plan/overview.md에 모두 옮겨 단권화하지 마세요.

---

## 작성 양식 (`/plan/overview.md`)

### 0. 참고 자료
- `/plan/background.md`
- `/plan/codebase-audit.md` (있는 경우만)

### 1. 이 PR의 목표
- 이번 PR에서 구현할 기능만 나열

### 2. 디자인 및 레이아웃 컨셉 - [선택적]
> **피그마가 있는 경우**: 피그마 링크만 참고 자료에 명시하고 생략
>
> **피그마가 없는 경우**: 전반적인 레이아웃 컨셉만 **간략히** 작성

#### 예시
- **레이아웃 구조**:
  - **Sidebar (좌측 고정)**: 상단 로고, 서비스 메뉴(Dashboard, Posts), 하단 사용자 프로필 요약
  - **Header (상단 고정)**: 브레드크럼, 알림 아이콘, 테마 토글, 로그아웃 버튼
  - **Main Content (본문)**: 페이지 타이틀, 액션 버튼(우측 상단), 데이터 테이블 또는 차트 영역
- **인터랙션**: 모달 중심

### 3. 참고 컨벤션

`instructions/conventions/coding/` 하위 파일 **제목**을 훑고, 이번 PR 작업 내용과 관련된 파일을 리스트업합니다. 파일 내용은 읽지 않고 제목만 보고 판단합니다.

```markdown
- `react/basics.md`
- `markup/style.md`
- `naming.md`
```

> 구현 단계(Step 5)에서 이 목록의 파일을 읽고 코드를 작성합니다.

### 4. 핵심 기술적 과제 및 기술 전략

#### 작성 방법
[문제 해결 로드맵](../../self-help/roadmap.md)의 2~3단계(선택지 탐색, 결정의 근거) 참고.

---

**⚠️ 핵심 구현 방법은 작성하지 않습니다.** 어떤 도구를 왜 쓰는지, 무엇과 비교했는지, 프로젝트 맥락에서 왜 최선인지만 명시하세요.

## [CRITICAL] Project 문서 간소화

`/plan/project.md`가 존재하면, 현재 PR의 상세 내용을 삭제하고 제목만 남기세요.

```markdown
### PR #1: 네이티브 브릿지 API
```
