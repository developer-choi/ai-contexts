# Step 3: 핵심 과제 정의 및 기술 전략 수립

이 단계는 **하나의 PR**에 대한 핵심 기술적 과제를 정의하고, 최적의 도구(라이브러리)를 선정하는 단계입니다.

> **Plan mode 필수**. 코드베이스를 탐색하여 근거 기반으로 핵심 과제를 도출하세요. AI가 제시한 과제는 사용자 승인을 거쳐야 하며, **승인된 과제만 Step 4에서 구현 방침을 작성**합니다.

> 대기업 기술면접관에게 기술 선택 근거를 설명할 수 있는 수준으로 작성하세요.
> [문제 해결 로드맵](../../self-help/roadmap.md) 참고.

## 작성 지침
- **간결함**: 화려한 수식어나 마케팅적 표현은 배제하고 담백하게 작성하세요.
- **개요 중심**: 기획서의 내용을 overview.md에 모두 옮겨 단권화하지 마세요.

---

## 작성 양식 (`/plan/pr{N}/overview.md`)

### 0. 참고 자료
- `/plan/background.md`
- `/plan/codebase-audit.md` (있는 경우만)

### 1. 이 PR의 목표
- 이번 PR에서 구현할 기능만 나열

### 2. 디자인 및 레이아웃 컨셉 - [선택적]
> **피그마가 있는 경우**: 피그마 링크만 참고 자료에 명시하고 생략
>
> **피그마가 없는 경우**: 전반적인 레이아웃 컨셉만 **간략히** 작성

#### 예시
- **레이아웃 구조**:
  - **Sidebar (좌측 고정)**: 상단 로고, 서비스 메뉴(Dashboard, Posts), 하단 사용자 프로필 요약
  - **Header (상단 고정)**: 브레드크럼, 알림 아이콘, 테마 토글, 로그아웃 버튼
  - **Main Content (본문)**: 페이지 타이틀, 액션 버튼(우측 상단), 데이터 테이블 또는 차트 영역
- **인터랙션**: 모달 중심

### 3. 핵심 기술적 과제 및 기술 전략

#### 작성 방법
[문제 해결 로드맵](../../self-help/roadmap.md)의 2~3단계(선택지 탐색, 결정의 근거) 참고.

---

**⚠️ 핵심 구현 방법은 작성하지 않습니다.** 어떤 도구를 왜 쓰는지, 무엇과 비교했는지, 프로젝트 맥락에서 왜 최선인지만 명시하세요.

---

## [CRITICAL] Project 문서 간소화

`/plan/project.md`가 존재하면, 현재 PR의 상세 내용을 삭제하고 제목만 남기세요.

```markdown
### PR #1: 네이티브 브릿지 API
```

단, 사용자가 남긴 TODO가 있으면 `overview.md`에 원본 그대로 옮긴 뒤, 사용자와 논의하여 구체화하세요.

> "project.md에 남기신 TODO를 overview.md로 옮겼습니다.
>
> - [ ] iOS/Android 권한 거부 시 에러 코드 확인
>
> 이 중 현재 PR에서 해결해야 할 항목이 있나요? 구체적인 방향을 정해주시면 overview.md에 반영하겠습니다."

---

## [CRITICAL] 사용자 확인 필수

overview.md 작성 후 사용자에게 확인 요청:

> "overview.md 작성 완료.
>
> 다음 문서들을 모두 이해하셨나요?
> - `/plan/background.md`
> - `/plan/codebase-audit.md` (있는 경우)
> - `/plan/pr{N}/overview.md`
>
> ⚠️ **모르는 부분이 있으면 다음 단계부터 문제가 발생합니다.**
>
> 이해하셨으면 승인해주세요."
