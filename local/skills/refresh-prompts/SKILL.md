# Refresh Prompts

## 목적

모델이 똑똑해지면 how(어떻게)를 덜 적어도 알아서 하므로, 레포 프롬프트 전체에서 what·why만 남기고 군더더기 how를 걷어낸다. "how 제거 = 개선"은 가설이며 벤치로 증명된 것만 채택한다.

발동은 둘이다 — 모델 업그레이드(낡은 모델로 정비한 분야 따라잡기), 또는 한동안 안 본 분야의 주기 정비. 어느 쪽이든 전량을 다시 돌리지 않는다 — 안 건드린 파일엔 새 how가 안 쌓여 재정비가 낭비고, 사용자 리뷰 비용도 크다. 마지막 정비 이후 **많이 바뀐 곳**을 찾아 범위를 선제안하고(고변경 = 고ROI), 분야별 상태를 state.json에 기록해 세션을 걸쳐 나눠 정비한다.

## 기준 (위임)

잘 쓴 프롬프트 판정은 `/scw` 「how 최소화 원칙」 + skill-creator를 따른다. 본 스킬은 **프롬프트 정비 절차**만 담고 방법론(변형 비교 벤치 · 트리거 정확도 측정 · 격리 · 0건 될 때까지 반복)은 scw에 위임한다.

## state.json

분야별 정비 상태는 `backlog/refresh-prompts/state.json`에 기록한다. 프롬프트 **분야**(rules·skills·contexts 등 최상위 영역)를 키로 둔다.

형식:

```json
{
  "<분야>": { "model": "<정비한 모델 ID>", "sweptAt": "<ISO 8601 날짜>" }
}
```

- `sweptAt`은 마지막 정비 시점. **이 시점부터 HEAD까지 커밋을 훑어 변경량을 집계하는 기준점**이다(step 1). 정비 후 그 분야의 값을 현재로 전진시킨다.
- `model`은 어느 모델로 정비했나. 기록 모델이 현재보다 낮으면 모델 업그레이드 발동 — 안 바뀐 옛 파일까지 광범위 정비를 2순위로 제안할지 판단하는 신호다(step 1).
- 분야는 독립 전진한다 — 한 분야만 정비해 그 엔트리만 갱신해도 되므로, 세션을 걸쳐 분야 단위로 나눠 정비할 수 있다.

커밋: `backlog/` 하위이므로 AC **백로그 브랜치**에만 커밋한다 (master 금지).

## 절차

### 1. 목표 선정 (커밋 활동 → 선제안)

매번 전량을 정비하지 않는다 — 안 건드린 파일엔 새 how가 안 쌓이므로 지난 정비분을 또 도는 건 낭비고 리뷰 비용만 든다. 마지막 정비 이후 바뀐 곳을 찾아 범위를 선제안한다:

1. state.json에서 각 분야의 마지막 정비 시점(`sweptAt`)을 읽는다.
2. `sweptAt`부터 HEAD까지 커밋을 훑어 프롬프트 파일(규칙·스킬·contexts)별 변경량(커밋 수·변경 라인)을 집계한다. 새로 생긴 파일은 변경량 큰 쪽으로 본다(한 번도 정비 안 됨).
3. 변경량 큰 순으로 랭킹해 **"이번엔 이 범위가 많이 바뀌었는데 여기부터 정비할까요?"** 형태로 사용자에게 제안한다. 안 바뀐 분야는 제안에서 뺀다(이미 깨끗).
4. 모델 업그레이드 발동(기록 `model` < 현재)일 때만: 안 바뀐 옛 파일도 더 똑똑한 모델 기준으로 뺄 how가 남을 수 있으니, 고변경 정비 뒤 **2순위 옵션**으로 "옛 파일 광범위 정비"를 덧붙여 제안한다. 자동으로 돌지 않는다.

정비 이득이 낮은 분야는 제안에서 제외한다 — 현재 제외: `coding-standards/` (후보 대부분이 코드 예시로 자명한 무해 부연이라 ROI가 낮고 사용자가 대부분 되돌렸다).

사용자가 범위를 확정·조정하면, 그 범위만 읽기전용 서브에이전트 여럿에 나눠 맡겨 how 후보를 뽑는다. 후보별 1차 분류: 군더더기 / 벤치의심 / 필수유지.

### 2. 3갈래 분류

- 군더더기 → 즉석리뷰 (사용자가 노이즈인지 즉석 판정, 벤치 없음)
- 벤치의심 → 벤치
- 필수유지 → 유지

AI가 노이즈를 단독 판정하지 않는다(scw). 사용자가 확신 못 하는 후보는 벤치로 올린다. 전체를 벤치에 넣지 않는 게 핵심 — 대부분은 즉석리뷰·유지로 빠진다.

### 3. 직접삭제 먼저 (즉석리뷰 갈래)

군더더기 갈래를 벤치보다 먼저 비운다. 후보를 일괄로 떼어낸 모습을 사용자에게 보여주고 즉석 판정을 받아 노이즈 확정분을 삭제한다. AI가 노이즈 여부를 단독 판정하지 않는다(scw 「how 최소화 원칙」). 먼저 빼는 이유: 노이즈를 걷어내면 벤치가 다룰 후보가 줄고 의심 후보가 또렷해진다.

### 4. 벤치 (의심 후보)

남은 벤치의심 후보를 **같은 가설을 공유하는 묶음** 단위로 돈다(후보 1개 단위 아님). 묶음 안에서 후보마다 시나리오 1개를 붙여 어느 후보 때문에 차이가 났는지 구분한다. 격리·순서는 **대상이 어떻게 로드되는지**로 갈린다 (`/scw` `specialized/bench-operations.md` 「대상 로드 방식별 격리」):

- 호출형 스킬 → 일반 세션에서 깨끗이 격리됨, 병렬 가능 → 먼저
- 자동로드 규칙(배포된 `rules`·`CLAUDE.md`) → 워커에 그대로 상속돼 "규칙을 뺀 변형"이 무효가 됨 → API/외부 주입 → 나중

묶음마다 **붙여넣기용 프롬프트 md 1개**를 만든다. 각 프롬프트에 담을 것: 대상·후보 출처, 고유 측정 워크트리 이름(병렬 충돌 방지), 변형 설계, 평가 모델(sonnet), 회수 양식(판정·제안 전부 결과 md에 적고 대화 회신엔 안 씀 — 메인이 묶음별 result md를 모아 종합하므로, scw 「측정 vs 권고 분리」), 결과 md 영속 경로(`backlog/this/<campaign>-bench/<묶음id>-result.md` 등 teardown 안 닿는 곳 — 메인 종합 전까지 살림, scw 「벤치 산출물 정리」), 후속 처리(적용/추가벤치/접기 — scw 「벤치 후 처리」 위임), 인벤토리 비수정(판정은 result·커밋에만 기록, 인벤토리 판정칸은 손대지 않음), 완료 후 프롬프트 자가삭제(결과 md는 남김). 사용자가 묶음 수만큼 세션을 열어 실행하고 결과를 회수한다. 호출형 스킬 묶음은 동시 실행 가능.

### 5. 적용·정합

직접삭제분과 벤치 확정분을 작업 워크트리에 영역별로 커밋한다. 바뀐 이름·경로를 참조하는 다른 곳도 맞는지 점검한다. 묶음 세션이 회수한 판정은 메인이 인벤토리 판정칸에 일괄 반영한다 — 인벤토리는 단일 작성자(메인)만 쓰므로 세션마다 갱신 여부를 되묻거나 병렬 충돌이 생기지 않는다. 메인은 묶음별 result md의 `## 권고`·`## 추정`을 모아 종합해 사용자에게 제시하고(승인 게이트 대상), 종합을 마친 뒤 result md를 삭제한다(scw 「벤치 산출물 정리」 — 영속 결과 md는 메인이 정리). 한 분야의 적용이 끝나면 그 분야의 state.json 엔트리를 현재 모델·날짜로 갱신한다(분야별 독립 전진).

### 6. 배포 (마지막)

모든 직접삭제·벤치·적용이 끝난 뒤 sync 배포를 한 번에 한다(사용자 실행). 중간 배포 금지: 배포된 자동로드 규칙이 벤치 워커에 상속돼 "규칙을 뺀 변형"을 무효로 만들기 때문이다(scw bench-operations 「대상 로드 방식별 격리」). 벤치 전에 배포하면 남은 묶음의 격리가 깨진다.

## 산출물

`backlog/this/`에 로드맵·인벤토리·묶음 계획·묶음별 프롬프트를 둔다. 형식은 자유, 필수 항목: 후보별 판정 칸, 묶음별 대상·후보·고유 워크트리 이름.
