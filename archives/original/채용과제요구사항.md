# 프론트엔드 채용 과제 — 수업 통화 모니터링 어드민

## 과제 배경

랭디는 학생과 선생님을 화상 통화로 연결해 주는 1:1 수업 플랫폼입니다. 수업 중에는 마이크·스피커·네트워크 등 여러 요인으로 통화 품질 이슈가 발생할 수 있고, 운영팀은 이슈가 보고된 수업을 빠르게 찾아 원인을 진단해야 합니다.

이 과제에서는 운영팀이 **특정 수업의 통화 상태를 진단**할 때 사용하는 내부 어드민의 핵심 화면을 구현해 주시면 됩니다. 디자인 시안은 별도 Figma 링크로 공유드리며, 데이터는 모두 mock으로 처리합니다.

---

## 구현 페이지

크게 두 페이지를 구현합니다.

1. 수업 검색 페이지 (어드민 진입점)
2. 수업 통화 상세 페이지 (검색 결과 카드 클릭 시 진입)

---

## 1. 수업 검색 페이지

### 역할

운영팀이 **수업 ID로 특정 수업을 찾아** 통화 진단 정보로 진입하는 입구입니다.

### 화면 구성

- 페이지 타이틀: **수업**
- 검색 영역
  - 라벨: `수업 ID`
  - 단일 입력 필드 (placeholder: `수업 ID를 입력해 주세요.`)
  - **조회하기** 버튼 (좌측 돋보기 아이콘)
- 결과 영역 (아래 3가지 상태 중 하나)

### 결과 영역의 상태

#### A. 초기 진입 상태

- 화면 중앙에 안내 문구 노출
  - 타이틀: `데이터를 조회해 주세요`
  - 설명: `필터와 검색으로 원하는 데이터의 범위를 설정하고 조회할 수 있어요.`

#### B. 조회 결과 없음

- 입력한 수업 ID로 조회했으나 일치하는 수업이 없을 때
- 화면 중앙에 메시지 노출
  - 헤더: `앗!`
  - 설명: `해당하는 결과가 없어요. 필터를 다시 설정하거나 검색어를 변경해 주세요`
  - **필터 초기화** 버튼 (주황색 강조)
- 버튼 동작: 입력값과 결과를 모두 비우고 A. 초기 진입 상태로 복귀

#### C. 조회 결과 있음

- 해당 수업의 모니터링 카드 그리드 노출 (자세한 내용은 "모니터링 카드" 섹션 참고)

### 비즈니스 규칙

- 수업 ID는 숫자 문자열 (예: `134585`)
- 입력값이 비어 있는 상태에서 **조회하기** 버튼은 비활성화 처리
- 동일한 수업 ID로 재조회 허용 (refetch)
- 조회한 수업 ID는 URL에 반영해 새로고침·링크 공유 시에도 동일 결과가 복원되어야 함

## 2. 수업 통화 상세 페이지

### 역할

검색한 수업의 **통화 품질을 항목별로 진단**합니다. 6개 카테고리의 카드를 통해 각 항목이 정상인지, 어디서 문제가 발생했는지 한눈에 파악합니다.

### URL

수업 ID로 직접 접근 가능한 별도 URL을 가짐 (구체 경로는 구현자가 정함)

### 화면 구성

- 좌측 상단 뒤로가기 링크 `< 수업` — 검색 페이지로 복귀
- 페이지 헤더: `{수업 ID} 수업 상세 정보`
- 메타 정보 영역 (한 줄)
  - 수업 ID (예: `134585`)
  - 수업 시간 (예: `2026. 04. 22 09:30`)
  - 선생님 ID / 선생님 이름 (예: `464622` / `Bob`)
  - 학생 ID / 학생 이름 (예: `161611` / `Karina`)
- 모니터링 카드 그리드 (가로 4열 기준, 카드 수에 따라 자동 줄바꿈)

### 비즈니스 규칙

- 존재하지 않는 수업 ID로 직접 URL 접근 시 404 또는 검색 페이지로 리다이렉트
- 모든 카드 데이터는 **수업 단위**로 함께 조회 (개별 API 호출 가정 X)

## 모니터링 카드

### 공통 구조

모든 카드는 동일한 골격을 따릅니다.

- 좌상단: **카드 타이틀**
- 카드 타이틀 아래: **설명 문구**
- 우상단: **상태 아이콘** (4종 중 하나)
- 본문: 항목 리스트 (각 항목은 `라벨` + `값` 쌍)

### 상태 아이콘 4종

| 상태 | 아이콘 | 의미 |
|---|---|---|
| 정상 | 초록 체크 | 진단 항목 전부 이상 없음 |
| 오류 | 빨간 ! | 진단 항목 중 하나라도 문제가 발생함 |
| 경고 | 노란 ⚠️ | 규정 범위를 벗어났거나 주의가 필요함 |
| 데이터 없음 | 회색 점선 원 | 데이터 만료·미수집으로 진단할 수 없음 |

### 카드 종류 (6개)

#### 1. 학습자 기기

학생이 사용한 통화 기기 정보를 표시합니다.

| 항목 | 설명 |
|---|---|
| 현재 통화 기기 | 학생이 현재 통화 시점에 사용하는 기기 모델 |
| 당시 통화 기기 | 학생이 이전 통화 때 사용한 기기 모델 |

- 기기 모델명은 모델 정보 외부 링크로 연결 (예: `iPhone 17,6` → 모델 페이지)
- 두 값이 모두 존재하면 **정상**
- 둘 중 하나라도 비어 있으면 **데이터 없음**

#### 2. 마이크 및 알림 권한

학생 단말이 마이크와 알림 권한을 허용했는지 진단합니다.

| 항목 | 가능한 값 |
|---|---|
| 당시 통화 기기 마이크 권한 | `허용` / `거부` |
| 현재 통화 기기 마이크 권한 | `허용` / `거부` |
| 현재 통화 기기 알림 권한 | `허용` / `거부` |

- 모든 항목이 `허용`이면 **정상**
- 하나라도 `거부`면 **오류**

#### 3. 마이크 및 스피커 활성화 상태

통화 중 학생 단말의 마이크·스피커가 정상 활성화됐는지 진단합니다.

| 항목 | 가능한 값 |
|---|---|
| 마이크 에러 | `발생함` / `발생하지 않음` |
| 스피커 에러 | `발생함` / `발생하지 않음` |

- 모든 항목이 `발생하지 않음`이면 **정상**
- 하나라도 `발생함`이면 **오류**

#### 4. 전화 재시도 횟수

통화 중 학생이 전화를 재시도한 횟수를 표시합니다.

- 표시값: 재시도 횟수 (정수)
- 부가 라벨
  - 재시도 횟수가 **3회 미만**: `규정 범위 이내` + **정상**
  - 재시도 횟수가 **3회 이상**: `규정 범위 초과` + **경고**

> 규정 범위를 초과한 경우 전화가 정상적으로 연결되지 않을 수 있다는 운영팀 내부 기준에 따른 규칙입니다.

#### 5. 학습자 네트워크 품질

학생 단말의 통신 품질을 표시합니다.

| 항목 | 가능한 값 |
|---|---|
| 네트워크 상태 | `좋음` / `보통` / `나쁨` |

- `좋음` → **정상**
- `보통` / `나쁨` → **오류**
- **수업 종료 후 2주가 경과한 데이터는 조회할 수 없음** → **데이터 없음**

#### 6. 선생님 네트워크 품질 및 기기

선생님 단말의 통신 품질과 운영체제·브라우저 정보를 표시합니다.

| 항목 | 가능한 값 |
|---|---|
| 네트워크 상태 | `좋음` / `보통` / `나쁨` |
| 운영체제 및 브라우저 | 예: `Windows / Chrome 147.0.0` |

- 상태 판정 기준은 학습자 네트워크 품질과 동일
- **수업 종료 후 2주가 경과한 데이터는 조회할 수 없음** → **데이터 없음**

### 공통 비즈니스 규칙

- 카드 내부 항목 값이 비어 있을 때는 자리에 `정보 없음` 표시
- **데이터 없음** 상태일 때는 본문을 회색 처리하고 우상단 아이콘을 회색 점선 원으로 표시
- 카드 전체를 클릭해도 동작 없음 (정보 표시 전용). 단, 항목 내 링크(예: 기기 모델명)는 새 탭으로 열림

## 데이터 / API

- 실제 백엔드 API는 제공되지 않습니다.
- Next.js **Route Handler**를 직접 구현해 mock 응답을 내려주고, 클라이언트에서는 일반 API 호출과 동일하게 fetch로 소비해 주세요.
- 클라이언트 코드 안에 데이터를 정적으로 박아 두지 말고, 실제 서비스와 동일하게 **네트워크 경계를 두고** 동작하도록 구성해 주세요.
- mock 응답 데이터는 코드 내 정적 정의, JSON 파일 분리 등 자유롭게 구성하면 됩니다.
