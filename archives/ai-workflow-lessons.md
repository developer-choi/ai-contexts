# AI 활용 작업 시 시행착오 최소화 원칙

## 목표

잘 모르는 코드베이스에서 AI를 활용해 작업할 때, 시행착오로 인한 커밋 누적과 정리 비용을 최소화한다.

## 예시

### 실무 (langdy-admin order 리팩토링)

- 사내 자체 컨벤션(SafeAsync, UseCase, api/types vs model/types 등)에 따라 기존 코드를 FSD 구조로 이동하는 작업
- 컨벤션 문서는 있었지만 팀원마다 적용 방식이 달랐고, 구두로만 공유된 내용도 있었음
- AI에게 "이 파일 어디에 둬야 해?" 식으로 물어보며 진행 → 파일을 A→B→C로 반복 이동
- 커밋 30개 쌓인 후 rebase/squash 정리에 시간 폭발

### 개인 과제

- 채용 과제 등 처음 보는 요구사항을 AI에게 시키는 경우
- 내 코드베이스, 익숙한 스타일이라 컨벤션 문제는 적음 (시행착오 10%)
- 하지만 요구사항 이해가 부족한 상태에서 커밋을 쌓으면 동일한 패턴 발생
- 커밋 10개 쌓고 순서 조정 + 리뷰하다 시간 낭비

## 발생한 문제

1. **이해도 70~80% 상태에서 작업 시작이 불가피했다**
2. **AI에게 "이거 어디에 둬야 해?" 식으로 답을 구했고, AI는 확신 없이도 답을 줬다**
3. **AI가 가져온 답을 출처 파일까지 확인했지만, 그 판단 자체가 틀렸는지 검증할 수 없었다**
4. **틀린 판단 기반으로 커밋이 쌓이고, 나중에 리뷰 단계에서야 발견됨**
5. **소급 수정 + 커밋 정리에 시간이 폭발적으로 소모됨**

## 핵심 교훈

AI에게 **답을 구하지 말고, 비교를 시켜라.**

- "이 파일 어디에 둬야 해?" (X) → AI가 확신 없이도 답을 줌 → 틀려도 그럴듯해 보임
- "컨벤션 문서 + 참고 폴더 기준으로, 내가 이렇게 하려는데 차이점이나 문제 찾아줘" (O) → AI가 사실을 나열 → 판단은 내가 함

## 왜 이게 중요한가

- AI에게 "답을 내려라"와 "비교해서 차이를 찾아라"는 완전히 다른 작업
- 전자는 틀려도 자신있게 답함
- 후자는 기계적으로 차이를 찾아내므로 신뢰도가 높음
- 비교 결과를 보고 **판단은 내가 하는 구조**가 되어야 함

## 근본 원인

> **확신 없는 상태에서 대량으로 작업을 진행했다.**

이 패턴은 반복적으로 실패했다:

- 사내 order 리팩토링: 확신 없이 커밋 30개 → 정리 폭발
- 개인 과제: 확신 없이 커밋 10개 → 순서 조정 + 리뷰에 시간 낭비
- AI 워크플로우 설계: AI에게 PR 하나 단위로 커밋 열몇개 한방에 쌓게 시킴 → 한번에 리뷰 → 왔다갔다 → 개고생

**"한방에 많이 쌓고 나중에 정리"는 매번 실패한다.**

## 실천 원칙

1. **AI에게 답을 구하지 말고 비교를 시킨다**
2. **비교 결과를 보고 판단은 내가 한다**
3. **새로운 패턴이 등장할 때마다 리뷰를 돌린다** (같은 패턴의 반복 적용은 리뷰 불필요)
4. **확신을 먼저 만들고, 확신이 생긴 범위만 작업하고, 확신이 맞는지 자주 검증한다**
5. **첫 패턴 적용이 확정된 후에 나머지 반복 작업을 시작한다** (확정 전에 대량으로 쌓지 않는다)
