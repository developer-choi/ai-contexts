# Step 2: PR 분할 전략 수립

이 단계는 **여러 PR로 나뉠 수 있는 큰 프로젝트**의 PR 분할 전략을 수립하는 단계입니다.

## 언제 Step 2를 진행하나?

[Step 1](./step-1.md)에서 background.md 작성 후, 다음 중 하나라도 해당되면 Step 2를 진행하세요:

1. **여러 페이지/컴포넌트에 걸친 작업**: 2개 이상의 페이지나 컴포넌트가 영향을 받음
2. **공통 모듈 개발 필요**: 여러 곳에서 재사용될 컴포넌트, 훅, 유틸 등이 필요
3. **외부 협업 필요**: 앱팀, 백엔드팀 등 다른 팀의 작업이 선행되어야 함
4. **단계적 배포 필요**: 리스크 분산을 위해 기능을 나눠서 배포해야 함
5. **복잡한 기술적 의존성**: 특정 기능이 완성되어야 다른 기능을 시작할 수 있음

### 예시

**❌ Step 2 불필요 (바로 Step 3으로)**
- "이벤트 페이지에 필터 추가"
- "프로필 수정 폼 검증 로직 개선"

**✅ Step 2 필요**
- "마케팅 수신 동의 독려 기능" (이벤트, 공지사항, 알림설정 3개 페이지 + 공통 모달 + 네이티브 브릿지)
- "게시판 CRUD 전체 개발" (리스트, 상세, 작성, 수정, 삭제 → 여러 PR로 분할)

## Step 2에서 할 일

### PR 분할 전략 수립
- 몇 개의 PR로 나눌 것인가?
- 각 PR의 범위는 무엇인가?
- PR 간 의존성은 어떻게 되는가?
- 병렬로 진행 가능한 PR은 무엇인가?

#### PR 분할 원칙
1. **독립성**: 각 PR은 독립적으로 배포 가능해야 함 (가능하면)
2. **작은 단위**: 리뷰와 테스트가 용이하도록 작게 나눔
3. **의존성 순서**: 공통 모듈 → 개별 페이지 순으로 진행
4. **리스크 분산**: 복잡한 기능은 단계적으로 배포

**기술적 과제는 Step 3에서 다룹니다.** Step 2는 PR 분할에만 집중하세요.

## 작성 양식 (`/plan/project.md`)

### PR 분할 전략

#### 예시
```markdown
## PR 분할 전략

### PR #1: 네이티브 브릿지 API (앱팀 협업)
**범위:**
- 알림 권한 상태 조회 API
- 알림 권한 요청 트리거 API

**담당:** 앱팀
**의존성:** 없음
**예상 기간:** 1주

### PR #2: 공통 모달 컴포넌트
**범위:**
- 독려 모달 (가운데 모달)
- 알림 OFF 안내 모달 (바텀시트)
- 2주 재노출 로직

**담당:** 웹팀
**의존성:** PR #1 (mock으로 병렬 진행 가능)
**예상 기간:** 1주

### PR #3: 이벤트/공지사항 페이지 통합
**범위:**
- 이벤트 페이지에 모달 통합
- 공지사항 페이지에 모달 통합

**담당:** 웹팀
**의존성:** PR #1, #2 완료 필요
**예상 기간:** 3일
```

## Step 2 완료 후

`/plan/project.md` 작성이 완료되면, **첫 번째 PR부터 [Step 3](./step-3.md)**으로 진행합니다.

**예시:**
```
Step 2 완료 → project.md 생성
  ↓
PR #1 시작 → Step 3 진행 → overview.md 생성 (PR #1 상세 계획)
  ↓
PR #1 완료 → overview.md 삭제
  ↓
PR #2 시작 → Step 3 진행 → overview.md 생성 (PR #2 상세 계획)
  ↓
...
```

**핵심:**
- `project.md`는 전체 프로젝트 동안 유지
- `overview.md`는 각 PR마다 생성/삭제 반복
