# Step 6: 최종 점검 및 PR 준비

이 단계는 모든 기능 구현(상세 명세 작성 완료) 및 커밋이 끝난 후, **PR(Pull Request) 생성 또는 QA 요청 직전**에 수행하는 최종 리허설 단계입니다.

---

## 1. Gap Analysis 및 추가 커밋 검증
상세 명세서(Task Detail)보다 실제로는 더 많은 커밋이 발생했을 가능성이 높습니다. (버그 수정, 엣지 케이스 발견 등)
이 단계에서는 **계획되지 않았던 추가 커밋들**을 식별하고 집중적으로 검증합니다.

- **커밋 로그 분석**: `git log`를 통해 원래 계획된 커밋 외에 추가된 커밋들을 식별합니다.
- **추가 검증 수행**: 계획에 없던 코드가 추가되었다면, 해당 로직을 커버하는 테스트 케이스도 함께 추가되었는지 확인합니다.

## 2. 최종 검증 보고서 작성 (Final Verification Report)
PR을 생성하거나 QA를 요청하기 전, **개발자가 스스로 수행한 최종 리허설 결과**를 기록합니다.
이 문서는 "내 코드는 검증되었습니다"라고 증명하는 **품질 보증서** 역할을 합니다.

- **작업 파일**: `/plan/review.md`
- **분석 대상**: 이번 브랜치(`feature/...`)에 추가된 **모든 커밋**
- **작성 원칙**:
  - 단순한 '할 일 목록'이 아닌, **실제 검증 수행 결과(Pass/Fail)**를 기록해야 합니다.
  - 성공(Happy Path)뿐만 아니라 실패/예외(Edge Case) 시나리오 검증 결과를 반드시 포함하세요.

**작성 양식 예시 (`/plan/review.md`)**:
```markdown
# 최종 검증 보고서 - [기능명]

## 1. 개요
- **검증 환경**: (예: Chrome / iOS Simulator / Localhost)
- **검증 날짜**: YYYY.MM.DD

## 2. 시나리오별 검증 결과
### 2.1. 기본 기능 (Happy Path)
- [x] **[시나리오]** "다음에" 버튼 클릭 시 동작
    - **절차**: 모달 오픈 -> "다음에" 클릭
    - **기대 결과**: 모달 닫힘, 로컬 스토리지 저장됨
    - **실제 결과**: Pass ✅

### 2.2. 예외 케이스 (Edge Cases)
- [x] **[시나리오]** 권한 거부 상태에서 "수신 동의" 클릭
    - **기대 결과**: 기기 설정 유도 모달 오픈
    - **실제 결과**: Pass ✅
```

## 3. 통합 무결성 및 코드 품질 검증
개별 Task가 아닌 프로젝트 전체 관점에서 무결성을 확인하고, 스스로 '엄격한 리뷰어'가 되어 코드를 점검합니다.

> **상세 기준 및 체크리스트는 아래 문서를 따릅니다:**
> - **[코드 품질 및 리뷰 기준](../for-all-projects/README.md)**: `coding`, `testing`, `review` 폴더 내의 규칙 준수 여부 확인.
> - **[작업 회고 및 셀프 리뷰](../self-help/review-work.md)**: 작업 효율성, 커밋 품질, 리팩토링 포인트 점검.

위 문서들을 참조하여 다음 항목들을 수행하세요:
- **전체 테스트 및 빌드 수행**: 사이드 이펙트 확인.
- **잔여물 제거**: `console.log`, 주석 처리된 코드, 임시 파일 삭제.
- **컨벤션 준수 확인**: 네이밍, 구조, 스타일 가이드 위반 사항 수정.

## 4. 워크플로우 회고 및 개선
작업을 마무리하며, **'일하는 방식'** 자체를 점검하고 업그레이드합니다.

- **프로세스 점검**: Step 1~6를 진행하면서 불필요한 절차, 모호한 지시, 혹은 AI가 헤맸던 부분이 있었는지 회고합니다.
- **가이드 업데이트**: 개선이 필요한 부분이 있다면 `docs/workflow/*.md` 파일을 어떻게 수정하면 좋을지 사용자에게 제안합니다.
