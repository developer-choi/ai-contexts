# 요청 내용
사용자가 추가로 제공하는 아래 정보를 바탕으로, `/plan/tasks/` 폴더 내의 해당 Task Detail 문서를 업데이트해주세요.

## [CRITICAL] 필수 정보 누락 시 절대 진행 불가
**아래 정보가 누락된 경우, 작업을 즉시 중단하십시오.** 추측으로 코드를 작성하는 것은 엄격히 금지됩니다.
반드시 사용자에게 해당 정보를 요구하고, 정보를 받은 후에만 Task Detail 문서를 작성하십시오.

### 1. 컴포넌트 개발인 경우 (필수)
- **Props Interface (타입 정의)**: 어떤 데이터를 받는지 명확해야 합니다.
- **피그마 링크 (또는 디자인 스펙)**: 시각적 구현 정보가 있어야 합니다.

> **⛔ 경고:** Props 타입이 정의되지 않으면 개발을 시작할 수 없습니다. "알아서 해주세요"는 허용되지 않습니다.

### 2. 함수 / 훅 / 클래스 개발인 경우 (필수)
- **Input / Output 타입 정의 (시그니처)**: 함수의 인자와 반환값이 무엇인지 정확해야 합니다.

> **⛔ 경고:** 입출력 타입이 없으면 로직을 짤 수 없습니다. 사용자에게 타입을 정의해달라고 요청하세요.

## 사용자가 제공해야 할 Context (작업 유형별)
사용자는 작업의 종류에 따라 위 필수 정보를 포함하여 아래 내용을 AI에게 제공해야 합니다.
AI는 정보 수신 후 Task Detail 문서를 작성하세요.

### 1. 컴포넌트 개발인 경우
- **참고 템플릿**: `./template/component.md`
- **필수 포함 항목**:
  - Props Interface
  - 피그마 링크

### 2. 함수 / 훅 / 클래스 개발인 경우
- **참고 템플릿**: `./template/function-hooks-class.md`
- **필수 포함 항목**:
  - Input / Output 타입 정의

### 3. 공통 사항 (기존 리소스 활용)
- 새로 만들지 않고 재사용할 수 있는 **공통 모듈**이나 **유틸리티**가 있다면 명시
- 비슷한 작업을 했던 커밋이 있다면 해시 전달

## 작성 시 주의사항
- 사용자가 준 정보를 그대로 복사하기보다, **개발자의 관점에서** 누락된 엣지 케이스나 예외 상황을 찾아서 보완해주세요.
- 최종 완성된 문서는 개발자가 코드를 작성할 때 **추가 질문 없이 보고 바로 짤 수 있을 정도**로 구체적이어야 합니다.

# step 4를 통해 완성된 Task Detail 문서 양식 (결과물 예시)
AI는 최종적으로 아래와 같은 구조로 문서를 완성해야 합니다.

### 개요
- **작업 파일**: `src/components/.../FileName.tsx`
- **목적**: 이 작업이 왜 필요한지 세 줄 미만 요약.

### 참고자료 (STEP 3, 4에서 사용자가 제공한 정보)
- 기획, 피그마, 회의록
- 참고할 파일, 커밋 등

### 예상 영향 범위
- **수정되는 파일**: 기존 코드를 수정해야 한다면 해당 파일 목록.
- **테스트 포인트**: 이 작업으로 인해 꼭 확인해야 할 기능.

### 스펙 (Specification)
아래 내용을 작성합니다.
- 성공/실패/엣지 케이스 시나리오
- 테스트 케이스 (Checklist)
- 핵심 코드 스니펫 (Interface, 시그니처 등)

유형은 2가지가 있습니다.
- [Component](./template/component.md)
- [Function, Hooks, Class](./template/function-hooks-class.md)
