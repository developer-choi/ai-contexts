## 1. 작업 흐름 & 효율성

### 분석 포인트
- 커밋 순서와 작업 진행 방향이 효율적이었는지
- 막힌 부분이 있었다면 다른 접근법은 없었을까
- 시행착오를 줄일 수 있는 방법은
- 작업 우선순위 설정이 적절했는지

### 예시
- **Legacy 코드 처리 미흡으로 인한 QA 범위 확대 및 충돌**
  - **문제 상황**: 기획/디자인 변경으로 기존 모듈을 수정해야 했는데, 원본 파일을 직접 수정하다 보니 해당 모듈을 사용하는 모든 곳이 영향을 받아 QA 범위가 지나치게 커짐.
  - **개선책**: 기존 코드 수정 대신, Deprecated 모듈과 새 모듈을 나눠야 했음. + 작업을 시작하기 전에 **기존 모듈 격리**를 먼저 수행하면 커밋도 아주 깔끔해짐. 
    1. 기존 컴포넌트/함수에 `@deprecated` 어노테이션 추가
    2. 파일명과 심볼명에 `Deprecated` 접두사 일괄 적용
    3. 별도 `deprecated` 폴더로 이동하여 기존 코드 분리
    4. 이후 새로운 요구사항에 맞춘 새 모듈을 생성하여, 변경 요청이 있는 페이지에서만 부분적으로 교체
  - **효과**:
    1. 기존 파일과의 Git Conflict을 원천 차단하고
    2. 구형 코드와 신규 코드를 명확히 구분 가능
    3. 영향 범위를 최소화하여 QA 리소스를 획기적으로 절감

## 2. 커밋 품질

### 분석 포인트
- 커밋 메시지의 명확성
- 커밋 단위가 적절했는지 (너무 크거나 작지 않은지)
- 기능별로 잘 분리되어 있는지
- 커밋 히스토리가 리뷰어가 이해하기 쉬운 스토리를 만드는지

### 예시 질문
- "하나의 큰 커밋보다 여러 개의 작은 커밋으로 나눴으면 좋았을까?"
- "커밋 메시지가 변경 의도를 명확히 전달하는가?"

## 피드백 형식

피드백은 다음 형식으로 제공됩니다:

### 잘한 점
- 구체적으로 어떤 부분이 좋았는지
- 이유와 근거

### 개선 제안
- 어떤 부분을 개선하면 좋을지
- 구체적인 대안 제시

### 학습 포인트
- 오늘 작업에서 배울 수 있는 점
- 다음 작업에 적용할 인사이트

### 액션 아이템
- 당장 또는 나중에 처리할 구체적인 작업
- 우선순위 표시
