## 1. 타인 PR 리뷰하기

**목적:** 남의 코드를 빠르게 이해하고 의미 있는 리뷰 제공

### 5단계 적용 방법

**1단계: 문제의 발견**
- PR Description에서 해결하는 문제 파악
- 연결된 이슈 번호 확인
- "왜 이 코드가 필요한가?" 먼저 이해

**2단계: 선택지 탐색**
- "나라면 어떻게 구현했을까?" 먼저 생각
- 다른 구현 방법이 있는지 고민
- 더 간단한 방법은 없었을까?

**3단계: 결정의 근거**
- 저자가 선택한 방법이 합리적인가?
- 트레이드오프가 명확한가?
- 프로젝트 상황에 맞는 선택인가?

**4단계: 구현의 여정**
- 코드를 라인별로 읽기
- 엣지 케이스 처리 확인
- 예상치 못한 버그는 없는가?

**5단계: 결과와 학습**
- 테스트 코드가 충분한가?
- 문서화가 되어 있는가?
- 성능 영향은 없는가?

### 리뷰 코멘트 작성 팁

```markdown
## 이해한 내용
이 PR은 [1단계: 문제]를 해결하기 위해 [3단계: 선택한 방법]을 사용했습니다.

## 질문
- [2단계] 왜 [대안A] 대신 [현재 방법]을 선택하셨나요?
- [4단계] line 42에서 [엣지 케이스]는 어떻게 처리되나요?

## 제안 (선택)
[3단계] ~한 트레이드오프를 고려하면 ~방법도 고려해볼 수 있을 것 같습니다.
```

- [overlay-kit 설계 분석](../../../langdy/langdy-utils/overlay-kit-design-decision.md) - 코드 리뷰 형식의 분석 예시

## 2. 내 PR 작성 전 검증하기

**목적:** 리뷰어가 궁금해할 질문에 미리 답변 준비

### 자가 검증 체크리스트

- [ ] **1단계:** "왜 이 작업이 필요한지" PR Description에 명확히 작성했는가?
- [ ] **2단계:** 다른 구현 방법도 고려했음을 코멘트로 남겼는가?
- [ ] **3단계:** 왜 이 방법을 선택했는지 3줄로 설명 가능한가?
- [ ] **4단계:** 엣지 케이스를 처리했으며, 테스트 코드를 작성했는가?
- [ ] **5단계:** 성능/메모리 영향을 측정했는가?

### PR Description 구조 예시

```markdown
## 문제 (1단계)
[이슈 #123] 사용자가 결제 실패 시 중복 주문이 생성되는 문제

## 해결 방법 (2-3단계)
다음 방법들을 고려:
- ❌ 프론트엔드에서 중복 클릭 방지: UX 저하
- ❌ DB unique constraint: 기존 데이터 영향
- ✅ **Idempotency Key 사용**: API 레벨에서 안전하게 처리

트레이드오프:
- 장점: 네트워크 재시도 상황에서도 안전
- 단점: 클라이언트가 UUID 생성 필요

## 주요 변경 (4단계)
- `OrderAPI.ts:45`: Idempotency-Key 헤더 추가
- `server/order.ts:78`: 중복 요청 감지 로직

## 테스트 (5단계)
- [x] 단위 테스트 추가 (5개)
- [x] 재현 시나리오 100회 테스트 → 중복 0건
- [x] 성능 영향 없음 (응답시간 +5ms)
```

## 3. 새로운 기술 학습하기

**목적:** 단순 튜토리얼을 넘어 깊이 있는 이해

### 학습 노트 작성법

**1단계: 문제의 발견**
- 이 기술이 왜 등장했는가?
- 어떤 문제를 해결하는가?
- 역사적 배경은?

**2단계: 선택지 탐색**
- 기존에는 어떻게 해결했는가?
- 경쟁 기술은 무엇인가?
- 각각의 차이는?

**3단계: 결정의 근거**
- 이 기술의 장점은?
- 단점 및 제약사항은?
- 언제 써야 하고, 언제 쓰지 말아야 하는가?

**4단계: 구현의 여정**
- 직접 토이 프로젝트 만들어보기
- 내부 동작 원리 파악
- 어려운 부분은?

**5단계: 결과와 학습**
- 핵심 개념 3줄 정리
- 실무에 적용 가능한가?
- 다음에 무엇을 학습해야 하는가?

### 실제 예시

```markdown
# React Query 학습 노트

## 1단계: 문제의 발견
- 서버 상태를 클라이언트에서 관리하는 게 복잡함
- useEffect + useState 조합의 보일러플레이트
- 캐싱, 동기화, 리페칭을 직접 구현해야 함

## 2단계: 선택지 탐색
- Redux + Redux Thunk/Saga: 클라이언트 상태 관리 중심
- SWR (Vercel): 경량, 단순
- Apollo Client: GraphQL 전용
- React Query: REST API 지원, 풍부한 기능

## 3단계: 결정의 근거
React Query 학습 이유:
- ✅ 자동 캐싱 및 백그라운드 리페칭
- ✅ Devtools로 디버깅 쉬움
- ✅ REST API 지원 (우리 프로젝트와 맞음)
- ❌ 추가 의존성 (번들 사이즈 +40KB)
- ❌ 학습 곡선 있음

## 4단계: 구현의 여정
[토이 프로젝트 코드]

내부 원리 파악:
- Query Key로 캐시 식별
- staleTime vs cacheTime 차이
- Observer 패턴으로 컴포넌트 자동 리렌더링

어려웠던 점:
- Optimistic Update 개념
- Query Invalidation 타이밍

## 5단계: 결과와 학습
핵심 정리:
1. 서버 상태 ≠ 클라이언트 상태 (분리 필요)
2. 선언적 데이터 페칭으로 코드 간결
3. 낙관적 업데이트로 UX 향상

실무 적용: ✅ 다음 프로젝트에 적용 예정
다음 학습: React Query의 Query Invalidation 심화
```

## 4. 기존 코드 분석하기

**목적:** 레거시 코드나 오픈소스 라이브러리 이해

### 분석 절차

**1단계: 문제의 발견**
- README, 문서에서 목적 파악
- 이 코드가 해결하는 문제는?
- 왜 만들어졌는가?

**2단계: 선택지 탐색**
- 커밋 히스토리에서 과거 시도 확인
- 이슈, PR에서 논의된 대안 찾기
- 다른 구현 방법은 없었을까?

**3단계: 결정의 근거**
- 왜 이 패턴을 사용했는가?
- 당시 제약사항은?
- 트레이드오프는?

**4단계: 구현의 여정**
- 코드 라인별 분석
- 테스트 코드 확인
- 주석에서 힌트 찾기

**5단계: 결과와 학습**
- 이 코드에서 배울 점은?
- 개선 가능한 부분은?
- 우리 프로젝트에 적용할 만한 패턴은?

### 실제 예시

**Material-UI Modal vs overlay-kit 비교 분석:**

```markdown
# overlay-kit과 Material-UI의 설계 차이 분석

## 1단계: 문제의 발견
overlay-kit의 useOverlay는 closeOnBackdropClick을 옵션으로 받음.
반면 Material-UI Modal은 onClose prop만 받고 옵션이 없음.
왜 두 라이브러리의 설계가 다른가?

## 2단계: 선택지 탐색
- 방식 1 (overlay-kit): 훅에서 옵션으로 UI 동작 제어
- 방식 2 (Material-UI): 컴포넌트에서 onClose 호출 시점 결정

## 3단계: 결정의 근거
Material-UI가 컴포넌트 레벨을 선택한 이유:
- 컴포넌트별 독립적 동작 (Modal, Dialog, Drawer 각각 다름)
- 옵션 폭발 방지 (새 컴포넌트마다 옵션 추가되는 문제)
- 복잡한 로직 가능 (boolean 외에 조건부 로직)

## 4단계: 구현의 여정
[실제 코드 분석 - useOverlay.tsx:78-80, Modal-simplified.md 등]

발견한 문제:
- 타입 불일치 (isOpen 미반환)
- 메모리 누수 (history 미제거)

## 5단계: 결과와 학습
학습한 설계 원칙:
"훅은 언제 열고 닫을지만 결정한다. 어떻게 닫을지는 컴포넌트가 결정한다."

적용:
- 우리 프로젝트의 모달도 컴포넌트 레벨로 리팩토링 고려
```

## 5. 설계 의사결정 문서화 (ADR)

**목적:** 아키텍처 결정 이유를 팀과 공유

### ADR (Architecture Decision Record) 형식

- **1단계 → Context**: 왜 이 결정이 필요한가?
- **2단계 → Considered Options**: 고려한 옵션들
- **3단계 → Decision**: 최종 결정 및 이유
- **4단계 → Consequences**: 이 결정의 영향
- **5단계 → Validation**: 검증 방법 및 회고 일정

### 실제 예시

```markdown
# ADR-001: API 상태 관리 라이브러리 선택

Date: 2025-01-10
Status: Accepted

## Context (1단계)
프로젝트에 20개 이상의 API 엔드포인트가 있으며, 각각 캐싱/로딩/에러 상태를 관리해야 함.
현재 useEffect + useState로 처리 중이며, 코드 중복이 심함 (각 컴포넌트마다 50줄 이상).

## Considered Options (2단계)
1. **Redux Toolkit + RTK Query**
   - 장점: Redux 생태계 통합
   - 단점: 우리는 Redux 사용 안 함, 학습 비용 큼

2. **SWR**
   - 장점: 경량 (11KB), 간단한 API
   - 단점: Mutation 지원 약함, devtools 없음

3. **React Query**
   - 장점: 풍부한 기능, 좋은 devtools, 큰 커뮤니티
   - 단점: 번들 사이즈 큼 (40KB), 학습 곡선

## Decision (3단계)
**React Query를 선택**

이유:
- 우리 프로젝트는 CMS로 Mutation이 많음 → SWR 부적합
- Redux 사용 안 함 → RTK Query 부적합
- Devtools로 캐시 상태 확인 필요 (디버깅 중요)
- 팀원 중 2명이 경험 있음 (학습 비용 완화)

Trade-offs:
- ✅ 개발 속도 50% 향상 예상 (보일러플레이트 감소)
- ✅ 자동 캐싱으로 불필요한 API 호출 감소
- ❌ 번들 사이즈 +40KB (현재 500KB → 540KB, 8% 증가)
- ❌ 팀 교육 2주 필요

## Consequences (4단계)
**긍정적 영향:**
- 기존 API 호출 코드 50% 감소 예상
- 캐싱으로 서버 부하 20% 감소 기대

**부정적 영향:**
- 초기 마이그레이션 2주 소요
- 기존 코드와 혼재되는 과도기 발생

**위험:**
- 팀원들이 Query Invalidation 타이밍을 잘못 이해하면 stale data 이슈

**완화 방안:**
- 내부 세미나 1회 진행
- Best Practice 문서 작성
- Code Review에서 중점 확인

## Validation (5단계)
**성공 지표:**
- [ ] API 관련 코드 라인 수 50% 감소
- [ ] 평균 API 호출 횟수 30% 감소 (DevTools 측정)
- [ ] 팀원 만족도 설문 4.0/5.0 이상

**검증 방법:**
- 1개월 후: 중간 점검 (마이그레이션 50% 완료)
- 3개월 후: 최종 회고 및 지표 측정
- 문제 발생 시: 대안 재검토

**회고 일정:** 2025-04-10
```

## 6. 버그 분석 보고서 작성하기

**목적:** 버그 원인과 해결 과정을 체계적으로 문서화

### 보고서 구조

**1단계: 문제의 발견**
- 버그 증상 (스크린샷, 에러 메시지)
- 재현 방법
- 영향 범위 (사용자 수, 빈도)

**2단계: 선택지 탐색**
- 가능한 원인들 (가설 나열)
- 각 가설의 근거

**3단계: 결정의 근거**
- 실제 원인 특정
- 근거 (로그, 디버깅 결과)

**4단계: 구현의 여정**
- 수정 방법
- 재발 방지 조치
- 코드 diff

**5단계: 결과와 학습**
- 검증 결과 (재현 테스트)
- 모니터링 결과
- 비슷한 버그 예방법

### 실제 예시

```markdown
# 버그 리포트: 결제 실패 후 중복 주문 생성

## 1단계: 문제 발견
**증상:**
- 결제 API 타임아웃 발생 시 주문이 2개 생성됨
- 사용자에게 "주문 실패" 메시지 표시되지만 실제로는 주문 완료

**재현 방법:**
1. 상품을 장바구니에 담기
2. 결제 진행
3. 네트워크를 3초 지연시킴 (Chrome DevTools)
4. 결과: 주문 2건 생성 (100% 재현)

**영향 범위:**
- 일평균 10건 발생
- CS 비용 증가 (환불 처리)
- 사용자 불만 (중복 결제)

## 2단계: 원인 가설
1. **가설 1: 프론트엔드 중복 클릭**
   - 근거: 사용자가 로딩 중에 버튼 재클릭?
   - 검증: 로그 확인 → 클릭은 1회만 발생

2. **가설 2: API 재시도 로직 문제**
   - 근거: axios retry 설정 의심
   - 검증: 로그 분석 → 동일 요청 2번 전송 확인 ✅

3. **가설 3: DB 트랜잭션 미적용**
   - 근거: race condition?
   - 검증: 서버 로그 → 2개의 다른 요청 확인

## 3단계: 실제 원인
**가설 2번이 원인**

**근거:**
- API 서버 로그: 동일 사용자의 주문 요청 2번 수신
- 요청 타임스탬프: 3초 간격 (타임아웃 설정과 일치)
- axios 설정 확인: retry 로직에 idempotency key 누락

**코드 문제점:**
```typescript
// OrderAPI.ts:42 (문제 코드)
axios.post('/api/order', orderData, {
  timeout: 3000,
  // idempotency key 없음!
});
```

타임아웃 발생 시 axios-retry가 자동으로 재시도하는데,
서버는 이것을 새로운 요청으로 인식하여 중복 주문 생성.

## 4단계: 수정 내용
**해결 방법:**
```typescript
// OrderAPI.ts:42 (수정)
import { v4 as uuidv4 } from 'uuid';

axios.post('/api/order', orderData, {
  timeout: 3000,
  headers: {
    'Idempotency-Key': uuidv4(), // 추가
  },
  'axios-retry': {
    retries: 3,
    retryCondition: (error) => {
      // 5xx 에러만 재시도 (타임아웃은 재시도 안 함)
      return error.response?.status >= 500;
    },
  },
});
```

```typescript
// server/order.ts:78 (서버 측 추가)
const idempotencyKey = req.headers['idempotency-key'];
const existingOrder = await Order.findOne({ idempotencyKey });

if (existingOrder) {
  return res.json(existingOrder); // 기존 주문 반환
}

// 새 주문 생성...
```

**재발 방지:**
- [ ] 모든 POST/PUT/DELETE API에 idempotency key 필수화
- [ ] ESLint 규칙 추가 (idempotency-key-required)
- [ ] API 가이드 문서 업데이트

## 5단계: 검증 및 학습
**검증 결과:**
- [x] 재현 시나리오 100회 테스트 → 중복 0건
- [x] 1주일 모니터링 → 중복 주문 0건
- [x] 다른 API 5개도 동일 패치 완료

**성능 영향:**
- 응답 시간: 변화 없음
- DB 조회 1회 추가: +5ms (무시 가능)

**학습 내용:**
1. **외부 API는 멱등성 보장 필수**
   - 네트워크는 언제든 실패 가능
   - 재시도는 불가피함

2. **타임아웃 ≠ 실패**
   - 타임아웃은 "응답 못 받음"일 뿐
   - 서버에서는 정상 처리되었을 수 있음

3. **Idempotency Key 패턴**
   - UUID로 요청 식별
   - 서버에서 중복 체크
   - 멱등성 보장

**적용:**
- 다른 프로젝트의 결제/주문 API도 동일 패턴 적용 예정
```

## 7. 면접 답변 준비하기

### 5단계 답변 구조 (STAR 기법 변형)

**1단계 (Situation) - 15초**
- 문제 상황 설명
- 비즈니스 임팩트 강조

**2단계 (Task) - 20초**
- 고려한 해결 방법들
- 각 방법의 간략한 설명

**3단계 (Action) - 30초** ⭐ 가장 중요!
- 최종 선택 및 이유
- 트레이드오프 설명
- 왜 그 방법이 최선이었는지

**4단계 (Action 상세) - 25초**
- 구현 과정에서의 난관
- 어떻게 해결했는지

**5단계 (Result) - 10초**
- 수치로 증명 가능한 결과
- 배운 점

**총 시간: 100초 (1분 40초)** - 면접 답변 적정 길이

### 실제 답변 예시

```
[1단계 - 15초]
저희 서비스는 실시간 채팅 기능이 있는데요,
동시 접속자 1만 명을 넘어가면서 메시지 지연이 3초까지 발생했습니다.
사용자 이탈률이 30% 증가하는 심각한 문제였습니다.

[2단계 - 20초]
해결 방법으로 크게 세 가지를 고민했습니다.
첫째, DB 쿼리 최적화로 부하를 줄이는 방법
둘째, Redis Pub/Sub을 도입해서 메시지 브로드캐스팅 효율화
셋째, WebSocket 서버를 스케일아웃하는 방법이었습니다.

[3단계 - 30초]
최종적으로 Redis Pub/Sub을 선택했습니다.
쿼리 최적화만으로는 근본적 해결이 어렵다고 판단했고,
서버 증설은 비용이 3배 이상 들어갔습니다.
Redis는 인메모리 데이터베이스라 속도가 빠르고,
Pub/Sub 패턴으로 여러 서버 간 메시지 동기화가 효율적이었습니다.
물론 Redis 장애 시 전체 채팅이 중단되는 SPOF 위험이 있었지만,
Redis Sentinel로 고가용성을 구성해서 위험을 완화했습니다.

[4단계 - 25초]
구현 과정에서 메시지 순서 보장이 어려웠습니다.
여러 서버에서 Redis로 메시지를 보내다 보니 순서가 뒤섞였거든요.
처음에는 Redis Streams를 고려했지만 복잡도가 높아서,
각 메시지에 timestamp를 넣고 클라이언트에서 정렬하는 방식으로 해결했습니다.

[5단계 - 10초]
결과적으로 평균 지연시간이 3초에서 0.3초로 90% 개선되었고,
동시 접속 3만 명까지 안정적으로 처리하게 되었습니다.
이 경험을 통해 병목 지점 분석과 트레이드오프 판단의 중요성을 배웠습니다.
```

## 기존 방식의 문제점 (1단계 상세)
[Before 코드 예시]

문제:
- 보일러플레이트 많음
- 캐싱 직접 구현 필요
- 리페칭 로직 복잡

## 해결 방법 비교 (2단계)
여러 라이브러리를 검토했습니다:

| 라이브러리 | 장점 | 단점 |
|-----------|------|------|
| Redux Toolkit | 상태 관리 통합 | Redux 의존 |
| SWR | 경량 | Mutation 약함 |
| React Query | 풍부한 기능 | 번들 사이즈 |

## 왜 React Query인가? (3단계)
최종적으로 React Query를 선택했습니다.

**선택 이유:**
1. Mutation 지원이 강력함 (우리는 CMS 프로젝트)
2. Devtools로 디버깅 편함
3. 커뮤니티가 크고 문서가 좋음

**Trade-offs:**
- ✅ 개발 속도 50% 향상
- ❌ 번들 사이즈 +40KB (허용 가능)

## 실제 적용 방법 (4단계)
### 1. 설치
```bash
npm install @tanstack/react-query
```

### 2. 기본 설정
```tsx
// App.tsx
import { QueryClient, QueryClientProvider } from '@tanstack/react-query';

const queryClient = new QueryClient();

function App() {
  return (
    <QueryClientProvider client={queryClient}>
      <YourApp />
    </QueryClientProvider>
  );
}
```

### 3. 사용 예시
```tsx
function UserProfile({ userId }) {
  const { data, isLoading, error } = useQuery({
    queryKey: ['user', userId],
    queryFn: () => fetchUser(userId),
  });

  if (isLoading) return <Spinner />;
  if (error) return <Error message={error.message} />;

  return <Profile data={data} />;
}
```

**주의할 점:**
- Query Key를 일관되게 관리하세요
- staleTime vs cacheTime 이해 필수
